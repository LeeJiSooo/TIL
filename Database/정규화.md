## 데이터베이스 정규화

#### 가져가야 할 핵심

1. 삽입, 갱신, 삭제 이상 현상은 데이터의 중복과 잘못된 종속 관계에서 비롯된다.
2. 1NF, 2NF, 3NF가 각각 어떤 중복과 종속성 문제를 제거하는 단계인지 명확히 구분할 수 있어야 한다.
3. 정규화와 성능은 어느 정도 트레이드 오프 관계를 가진다. 따라서 성능 향상을 위해 의도적으로 반정규화를 선택하기도 한다.
4. 정규화 규칙은 좋은 원칙이지만 절대적으로 따라야 하는 것이 아니며, 비즈니스 상황에 따라 타협해 나가는 과정이 필요하다.

정규화는 관계형 데이터베이스 설계에서 중복을 최소화하도록 데이터를 구조화하는 프로세스이다. 

데이터의 무결성을 유지하고 저장 용량을 줄이며, 궁극적으로 데이터 이상 현상을 방지하기 위해 사용한다.

- 중복의 최소화 : 데이터가 중복되면 저장 공간 낭비 뿐만 아니라 데이터 불일치를 유발한다.(ex. 고객의 주소가 여러 곳에 중복 저장되어 있을 때, 이사로 인해 주소를 변경하는 과정에서 한 곳이라도 누락하면 과거 주소와 새 주소가 공존하는 불일치가 발생한다. 정규화는 데이터가 단 한 곳에만 저장되도록 개선하여 이 문제를 해결한다.)
- 데이터 구조화 : 정규화는 무작정 테이블을 쪼개는 것이 아니다. ‘함수적 종속성’이라는 논리적 근거에 따라 데이터 간의 관계를 명확히 파악하고 테이블을 재구성한다. 이를 통해 데이터의 의미가 명확해지고, 구조를 이해하기 쉬워지며, 유지보수성과 확작성이 향상된다.

## 데이터 이상 현상(Anomaly)

데이터베이스 설계가 잘못되었을 때(정규화가 제대로 일어나지 않았을 때), 데이터를 삽입/수정/삭제하는 과정에서 발생하는 부작용을 의미한다.

<img width="1458" height="436" alt="image" src="https://github.com/user-attachments/assets/092338e5-78ce-4568-95c5-956f70d4266d" />


#### 삽입 이상

- 데이터를 삽입하려고할 때, 원하지 않는 불필요한 정보까지 억지로 입력해야 하거나 입력이 불가능한 현상이다.
- 새로운 프로젝트(P04, 빅데이터 플랫폼)가 개설되었으나 아직 배정된 사원이 없는 경우, 사원 정보 칸을 NULL이나 임의의 가짜 데이터로 채워야만 등록이 가능하거나 아예 등록 자체를 못하는 상황이 발생한다.

#### 갱신 이상

- 데이터 중복으로 인해 데이터를 수정할때 일관성이 깨지고 정보가 불일치해지는 현상이다.
- ‘개발1팀’의 부서명이 ‘AI개발팀’으로 변경될 경우, 테이블 내 모든 ‘개발1팀’행을 찾아서 수정해야한다. 데이터가 수천, 수만 건이라면 매우 비효율적이며, 일부 행을 누락하면 부서명이 서로 달라지는 문제가 발생한다.

#### 삭제 이상

- 특정 데이터를 삭제할 때, 의도하지 않는 다른 유용한 정보까지 함께 삭제되어 데이터 손실이 발생하는 현상이다.
- ‘기획팀’의 102번 박영희 사원이 퇴사하여 해당 행을 삭제하면, 박영희 정보 뿐만 아니라 ‘기획팀이 P01 차세대 시스템 프로젝트에 참여하고 있었다’는 프로젝트 정보까지 함께 사라져 버린다.

→ 이러한 3가지 이상 현상은 데이터 중복과 잘못된 종속 관계 때문에 발생하며, 정규화는 테이블을 논리적으로 분리해 각 데이터의 올바른 위치를 찾아줌으로써 이를 해결한다.

## 단계별 정규화 과정(1NF → 2NF → 3NF)

#### 제 1정규형(1NF)

테이블의 모든 컬럼은 반드시 원자값(더 이상 쪼갤 수 없는 값)을 가져야 한다. 즉, 하나의 칸에는 딱 하나의 데이터만 들어가야 한다.

**[제1정규화 전] users table**

<img width="1374" height="204" alt="image" src="https://github.com/user-attachments/assets/382495f9-3ed5-4c16-a6f8-ef70677065ee" />


- 검색의 어려움 : 특정 번호로 사용자를 찾으려면 `WHERE phone_number = '010-1234-1234'` 같은 단순 쿼리를 쓰지 못하고 `LIKE` 연산 등을 써야 하므로 인덱스를 타지 못해 비효율적이다.
- 관리의 복잡성 : 번호 하나만 수정/삭제하려 해도 전체 문자열을 가져와 분리하고 다시 합치는 복잡한 과정이 필요하다.
- 의미의 모호성 : 두 번호 중 어떤 것이 주 연락처인지 비상 연락처인지 구분이 불가능하다.

#### 제 2정규형(2NF)

1정규형을 만족하고, 기본키가 아닌 모든 일반컬럼들이 기본키 전체에 완전 함수 종속되어야 한다. 부분 함수 종속을 제거하는 단계이다.

→ 이 문제는 주로 기본키가 2개 이상의 컬럼으로 구성된 복합키일 때 발생한다. 기본키가 단일 컬럼이면 1NF만족 시 자동으로 2NF도 만족한다.

- 함수 종속(A→B) : A값을 알면 B값이 유일하게 결정되는 관계(ex. 과목ID가 CS101이면 과목명은 ‘컴퓨터 개론’으로 고정됨)
- 부분 함수 종속 : 복합키를 구성하는 여러 컬럼 중, 전체가 아닌 ‘일부 컬럼’에만 종속되는 일반 컬럼이 존재하는 경우

**[수강 테이블 (2NF 위반)]**

<img width="1224" height="374" alt="image" src="https://github.com/user-attachments/assets/0a04a28f-350f-4b24-bcab-68f6d9a99020" />


- 성적은 학생과 과목을 모두 알아야 하므로 복합키 전체에 종속된다.(완전 함수 종속)
- 과목명과 담당교수는 학생ID와 상관 없이 오직 과목ID에 의해서만 결정된다.(부분 함수 종속)

#### 제 3정규형(3NF)

제 2 정규형을 만족하고, 기본키가 아닌 모든 일반 컬럼들이 기본키에 이행적으로 종속되지 않아야 한다.

→ 일반 컬럼끼리 종속 관걔가 발생하는 것을 막는 것이다. 중간에 다른 일반 컬럼을 걸쳐서 알게되는 정보라면 별도의 테이블로 떼어내야 한다.

이행적 종속 : A→B이고 B→C일때, 결과적으로 A→C가 성립하는 관계

- 사원번호를 알면 부서번호를 알 수 있고, 부서번호를 알면 부서명과 부서위치를 알 수 있다. 즉, 부서명과 부서위치는 기본키인 사원번호에 직접 종속되는 것이 아니라 일반 컬럼인 부서번호를 거쳐 간접적으로 종속되어 있다.(이행적 종속)

- 삭제 이상 : 특정 부서의 유일한 팀을 삭제하면 부서번호, 부서명, 위치 정보까지 함께 삭제된다.

## 반정규화

정규화 원칙을 의도적으로 위배하여 조회 성능을 높이기 위해 데이터 중복을 허용하거나 테이블을 합치는 작업이다.

#### 반정규화가 필요한 이유

- 정규화를 거쳐 테이블을 잘게 쪼개다 보면, 원하는 화면에 데이터를 보여주기 위해 여러 테이블을 엮는 JOIN연산을 많이 사용하게 된다.
- JOIN연산은 데이터 양이 많아질수록 시스템에 부하를 주며, 이로 인해 대량의 읽기작업이 빈번한 비즈니스 환경(ex. 쇼핑몰 상품 목록 조회 시 상품, 판매자, 카테고리 테이블을 매번 JOIN해야 하는 상황 등)에서는 성능이 저하될 수 있다.

#### 반정규화 예시와 주의점

**정규화 구조:** `게시글(post_id, title, content, user_id)` / `사용자(user_id, nickname, email)`

- 게시글을 조회할 때마다 사용자 테이블과 JOIN 하여 닉네임을 가져와야 하므로 부하가 걸릴 수 있다.

**반정규화 구조:** `게시글(post_id, title, content, user_id, nickname)` / `사용자(user_id, nickname, email)`

- 게시글 테이블에 `nickname`을 중복 저장하여 JOIN 없이 빠르게 조회한다.
- **치명적인 단점:** 사용자가 닉네임을 변경하면 과거에 작성한 모든 게시글의 `nickname` 컬럼도 일일이 수정해주어야 하는 **갱신 이상**이 발생한다.

개발자의 판단 기준 : 반정규화는 데이터 갱신 빈도가 매우 낮고, 조회 성능이 극도로 중요할 때 신중하게 고려하는 선택지이다. 기획자나 비즈니스 상황을 파악하여 “이 값이 얼마나 자주 수정되는가”, “나중에 수정될 여지가 있는가?”를 명확히 판단하고 도입해야 한다.

## 정리

- 정규화는 중복을 없애고 종속 관계를 바로잡아 이상 현상을 방지하고 데이터 무결성을 확보하는 최적의 설계 과정이다.
- 1NF는 모든 컬럼의 원자값 확보, 2NF는 복합키에서의 부분 함수 종속 제거, 3NF는 일반 컬럼 간의 이행적 종속 제거를 목표로 한다.
- 이론적으로 완벽한 정규화가 이상적이지만, 실제 서비스 성능(조회 속도)향상을 위해 비즈니스 상황에 맞춰 의도적으로 중복을 허용하는 반정규화를 전략적으로 타협하여 사용한다.
