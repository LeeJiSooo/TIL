## 가져가야 할 핵심

- 데이터 일관성과 무결성을 지키기 위한 동시성 제어의 필요성
- 비관적 락과 낙관적 락에 대한 이해
- 데드락의 발생 원인과 이해

## 동시성 제어와 필요성

### 동시성 제어

여러 사용자가 동시에 데이터베이스에 접근할 때 발생할 수 있는 데이터 충돌을 방지하고, 데이터의 일관성과 무결성을 유지하기 위한 기술이다.

- 일관성 : 트랜잭션 전후에 데이터베이스가 비즈니스 규칙과 스키마 제약 조건을 항상 만족하는 상태이다. (예 : “어떤 사용자도 잔액이 0원 미만이 될 수 없다”는 규칙은 트랜잭션이 아무리 몰려도 깨져서는 안 된다.)
- 무결성 : 데이터가 어떠한 상황에서도 정확하고 신뢰할 수 있으며 왜곡되지 않은 상태를 유지하는 성질이다.

### 레이스 컨디션과 갱신 손실(Lost Update)

로컬 환경에서 혼자 테스트할 때는 동시성 문제가 터지지 않는다. 하지만 실제 서비스 배포 후, 남은 쿠폰이 딱 1장인 상황에서 10명의 사용자가 0.0001초 사이에 쿠폰 발급 API를 동시에 호출하는 상황이 발생하면 데이터베이스에 병렬 트랜잭션이 쏟아진다.

### 성능(Throughput)과 정합성의 트레이드 오프

가장 무식하고 안전한 방법은 모든 트랜잭션을 일렬로 줄 세우는 최고 격리 수준인 `Serializable`을 거는 것이다. 안정성은 100%보장되지만, 동시 처리량(성능)이 안 좋아지며 유저 응답 속도가 느려진다. 

반대로 제어 장치를 완전히 풀면 속도는 빨라지지만 데이터가 전부 파괴된다. 백엔드 개발자는 비즈니스 요구사항에 맞춰 최적의 동시성 제어 기법을 선택하고 조율해야 한다.

## 비관적 락(Pessimistic Lock) : 물리적 자물쇠 채우기

상황을 극도로 비관적으로 바라보는 방식이다. “내가 이 데이터를 읽고 고치는 동안, 분명히 다른 사람도 들어와서 데이터를 고칠 것이다. 그러니 처음부터 원본 데이터에 자물쇠를 채우고 시작하자”는 방어적 접근이다.

- 화장실 열쇠 비유 : 술집에서 화장실에 갈 때 카운터에서 하나뿐인 물리적인 화장실 열쇠를 아예 손에 쥐고 들어가는 것과 같다. 내가 볼일을 다 보고 열쇠를 반납할 때까지 다른 사람들은 화장실 문 앞에서 아무것도 못 하고 대기해야한다.
- DB 내부 동작 : 한 트랜잭션이 특정 데이터 행에 비관적 락을 거는 순간, 동일한 행을 수정하려는 다른 트랜잭션들은 앞의 트랜잭션이 커밋이나 롤백으로 락을 풀 때까지 대기상태(Blocking)에 빠진다. → 선착순 쿠폰 발급처럼 충돌이 빈번하게 일어나는 핵심 로직에 반드시 필요하다.

### 락의 종류

#### 공유 락(Shared Lock, S-Lock)

- 여러 트랜잭션이 동시에 데이터를 읽을 수 있지만, 쓰기는 철저히 차단하는 락이다.
- “나 이거 읽는 중이니까 아무도 수정하지 마. 대신 다른 애들이 같이 읽는 건 허용해 줄게.”
- S-Lock끼리는 서로 완벽히 호환된다. 여러 트랜잭션이 하나의 데이터 행에 동시에 S-Lock을 걸고 병렬 조회를 할 수 있다.

#### 독점 락(Exclusive Lock, X-Lock)

- 읽기와 쓰기를 모두 차단하여 완벽한 단독 접근을 보장하는 강력한 락이다.
- “나 이거 수정할 거니까 내가 다 끝낼 때까지 읽지도 말고 쓰지도 마.”
- 그 어떤 락과도 호환되지 않는다. X-Lock이 걸린 데이터를 다른 S-Lock이나 X-Lock을 추가로 걸 수 없다.
- (MySQL InnoDB의 특성) : X-Lock이 걸려있어도 MySQL같은 MVCC 기반 DB의 일반적인 무조건 조회(SELECT)는 막히지 않는다. 왜냐하면 원본 데이터를 직접 쳐다보는 것이 아니라, 이전에 커밋 완료된 과거의 스냅샷(Undo로그)을 읽어오기 때문이다.

#### 의도 락(Intent Lock)

- 하위 자원(Row 레벨)에 락을 걸 계획이 있음을 상위 레벨(Table 레벨)에 미리 알려두는 일종의 표식이다.
- “지금 이 테이블 안의 어떤 특정 행들에 누군가 락을 걸고 작업 중이다.”
- DBA가 테이블 구조를 바꾸기 위해 테이블 전체를 잠그려고 할 때, 테이블 안의 수백만 건의 행을 하나씩 찾아가며 락이 걸려있는지 확인하는 것은 불가능하다. 이때 테이블 레벨에 있는 인텐트 락의 유무만 확인하면 효율적으로 락 충동을 판단할 수 있다.

#### 레코드락 & 갭 락

- 레코드 락 : MySQL InnoDB에서 행 자체를 잠그는 게 아니라, 데이터가 물리적으로 위치한 인덱스 레코드에 자물쇠를 채우는 방식이다.
- 갭 락 : 조건 조회 시 데이터와 데이터 사이에 존재하는 빈 공간까지 통째로 잠가버려서 새로운 데이터가 끼어들지(INSERT) 못하게 막는 락이다.
- 넥스트 키 락 : 레코드 락과 갭 락이 결합된 형태이며, MySQL Repeatable Read 격리 수준에서 Phantom Read를 원천 차단한다.

### 일반 SELECT vs 비관적 락

일반 SELECT는 락 규칙을 무시하고 스냅샷을 읽는다.

트랜잭션 A가 특정 데이터에 X-Lock을 걸고 수정 중일 때, 뒤이어 온 트랜잭션 B의 행동에 따라 결과가 달라진다.

#### 일반 SELECT를 날린 경우(Non-locking Read)

SELECT * FROM item WHERE id = 1;

→ 대기하지 않고 즉시 읽는다. MVCC기술 덕분에 A가 수정하기 전의 과거 스냅샷 데이터를 그냥 읽어와 버린다.

→ A가 락을 풀고 커밋한 후, B가 아까 읽은 옛날 스냅샷 기준으로 수정을 시도하면 데이터가 꼬이는 Lost Update가 발생한다.

#### 비관적 락을 날린 경우(Locking Read)

`SELECT * FROM item WHERE id = 1 FOR UPDATE;` (X-Lock 요청)

→ 스냅샷 안 읽고 대기. A가 락을 쥐고 있으면 B는 쿼리 실행 자체가 멈추고 대기실(Queue)에서 줄을 서서 기다린다.

→ A가 커밋해서 락을 놓으면, 그제서야 B가 들어가서 A가 고쳐놓은 최신 데이터를 안전하게 읽는다.

## 비관적 락의 부작용 : 데드락(Deadlock)

비관적 락을 남발하면 두 개 이상의 트랜잭션이 서로가 쥔 자물쇠가 풀리기만을 무한정 기다리며 데드락 현상이 발생한다.

**시나리오 1**

트랜잭션 A, B가 동시에 (S-Lock)으로 재고를 조회한다.

S-Lock끼리는 호환되니까 둘 다 동시에 S-Lock을 획득한다.

트랜잭션 A가 “나 이제 수정할게. X-Lock주라” → DB : “B가 S-Lock 쥐고 있어서 안된다. 기다려라” (A 대기)

트랜잭션 B가 “나도 수정할래. X-Lock주라” → DB : “A가 S-Lock 쥐고 있어서 안된다. 기다려라” (B 대기)

결과 : 서로 락을 풀어주기만 기다리며 무한 대기 상태에 빠짐

**시나리오 2**

트랜잭션 A : [사용자 테이블 1번 행]에 X-Lock을 성공적으로 획득했다. 그다음 [주문 테이블 1번 행]에 락을 걸려고 시도한다.

트랜잭션 B : 진짜 찰나의 동시 순간에 [주문 테이블 1번 행]에 X-Lock을 먼저 획득했다. 그다음 [사용자 테이블 1번 행]에 락을 걸려고 시도한다.

→ 결과적으로 트랜잭션 A는 B가 주문 락을 풀어주길 기다리고, 트랜잭션 B는 A가 사용자 락을 풀어주길 기다린다. 서로가 서로의 자원을 요구하며 영원히 끝나지 않는 대기 상태에 돌입한다.

- 해결 및 대처 : 데이터베이스 엔진이 ‘순환 대기 감지 알고리즘’을 통해 데드락을 모니터링한다. 교착 상태가 감지되면 DB는 둘 중 손해가 적은 한쪽 트랜잭션을 강제로 희생(취소/롤백)시킨다.
- 방어 컨디션 : 실무에서는 데드락 빈도를 줄이기 위해 개발자들끼리 “테이블에 접근할 때는 [사용자]테이블 먼저 조회한 뒤 [주문]테이블로 가야한다”와 같이 순서를 고정하는 개발 컨벤션을 약속하여 정렬하기도 한다.

## 동시성 표준과 비관적 락 예시

### 2PL(2-Phase Locking)프로토콜

트랜잭션이 락을 획득하고 해제하는 시점을 통제하여데이터 직렬성을 보장하는 표준 학술 이론이다. 두 단계로 명확히 분리하여 락을 다룬다.

- 확장 단계 : 트랜잭션이 필요한 락들을 계속해서 얻기만 하는 단계이다. 이 단계에서는 들고 있는 락을 절대 풀 수 없다.
- 수축 단계 : 트랜잭션이 들고 있던 락을 단 하나라도 해제(반납)하기 시작하는 단계이다. 수축 단계에 접어들면 그 즉시 새로운 락을 절대로 추가 획득할 수 없다.
- Strict 2PL : 실무 DB엔진이 채택하는 더 엄격한 규칙이다. 수축 단계를 트랜잭션이 완벽히 종료(`COMMIT` 또는 `ROLLBACK`)되는 시점까지 미뤄두었다가, 마지막 종료 순간에 획득했던 모든 락을 한 번에 다 풀어버리는 방식이다.

### 비관적 락 실제 SQL 구현 예시(재고 차감)

```sql
START TRANSACTION;

-- 1. 재고 조회와 동시에 해당 행(id=1)에 강력한 X-Lock을 걸어버린다.
-- 다른 트랜잭션이 이 상품을 UPDATE하거나 똑같이 FOR UPDATE 하려고 하면 대기(Blocking)를 태운다.
SELECT stock_quantity 
FROM product 
WHERE id = 1 
FOR UPDATE;

-- 2. 애플리케이션 로직에서 재고 검증 후 차감 연산 진행
UPDATE product 
SET stock_quantity = stock_quantity - 1 
WHERE id = 1;

COMMIT; -- 이 순간 락이 해제된다 (Strict 2PL)
```

`FOR UPDATE` 구문을 호출하여 X-Lock을 쥐고 있는 시간이 길어지거나 대기하는 트랜잭션이 많아지면, 순식간에 **DB 커넥션 풀이 고갈**되어 대형 장애로 번진다. 따라서 비관적 락은 아주 좁은 범위의 데이터에만, 짧은 트랜잭션 시간 동안만 타이트하게 유지해야 한다.

## 낙관적 락(Optimistic Lock) : 애플리케이션 레벨의 가벼운 통제

비관 락의 커넥션 고갈 위험과 데드락 리스크를 극복하기 위해 고안된 대안이다. 상황을 아주 긍정적으로 바라본다.

“내가 데이터를 고치는 동안 설마 다른 사람이 동시에 건드리겠어? 일단 DB 물리 락 걸지 말고 가볍게 가자. 대신 마지막에 수정 완료(UPDATE)할 때, 내가 처음 봤던 데이터 상태가 맞는지 검사만 해보자”는 방식이다.

→ DB의 `FOR UPDATE` 같은 물리적인 락 메커니즘을 쓰지 않는다. 대신 테이블에 `version`이라는 컬럼을 하나 두고, 애플리케이션 레벨(코드 레벨)에서 소프트웨어적으로 정합성을 체크한다. 

→ 데이터 충돌 빈도가 낮고, 읽기 작업이 압도적으로 많을 때 성능 효율을 낸다.

### 낙관적 락 작동 프로세스

#### 1단계 : 첫 데이터 조회(버전 체크)

백엔드 서버가 게시글 데이터를 조회한다. 이때 버전 번호도 함께 읽어온다.

```sql
SELECT id, title, content, version FROM article WHERE id = 100;
-- 결과: [id=100, version=1] 로 데이터를 깔끔히 읽어옴
```

#### 2단계 : 유저가 본문 수정 후 저장 API 호출(업데이트 검증)

개발자는 업데이트 쿼리를 날릴 때 `WHERE` 조건절에 내가 처음 읽었던 버전 번호(version=1)를 명시하고, 수정이 성공하면 버전 번호를 1증가시키도록 쿼리를 설계한다.

```sql
UPDATE article 
SET content = '정한 내용', version = version + 1 
WHERE id = 100 AND version = 1; -- ★ 핵심 조건문
```

#### 3단계 : 충돌 발생 시 처리 메커니즘

- 만약 1번 유저가 글을 고치고 저장 버튼을 누리기 딱 0.0001초 전에, 2번 유저가 먼저 수정을 완료해서 커밋을 했다면 테이블의 진짜 `version` 은 이미 2로 올라가 있는 상태다.
- 1번의 `UPDATE` 쿼리는 `WHERE version = 1` 조건을 만족하지 못하므로, 수정된 행의 개수가 0개(Affected Rows = 0)가 된다.
- 이를 감지한 백엔드 프레임워크(Spring, JPA)는 데이터 충돌을 인지하고 즉시 **`OptimisticLockException`** 예외를 던진다.

→ 낙관적 락은 충돌을 원천 차단하는 게 아니라 감지하는 기술이므로, 예외가 터지면 백엔드 개발자가 잡아내서 데이터를 DB에서 새로 다시 읽어와 재시도 조치(`@Retryable`)를 취하거나, 유저에게 "다른 사용자가 먼저 수정했습니다"라고 안내창을 띄워야 한다. 충돌이 잦은 환경에 낙관적 락을 쓰면 무한 재시도 연산 때문에 서버 CPU가 터지므로 주의해야 한다.

## 데이터베이스 조회 최적화 : Buffer Pool

### 사라진 쿼리 캐시(Query Cache)

과거 MySQL은 유저가 날린 SQL 쿼리 문장 문자열 자체를 Key로, 결과 데이터 통째를 Value로 메모리에 저장해 두는 '쿼리 캐시' 기능이 있었다. 똑같은 쿼리가 오면 디스크를 안 거쳐서 매우 빨랐다.

- **삭제된 이유:** 만약 캐시해 둔 테이블에 새로운 데이터가 단 1건이라도 `INSERT`되거나 `UPDATE`되면, 데이터 정합성을 맞추기 위해 **해시 맵에 저장된 해당 테이블 관련 쿼리 캐시 수만 건을 통째로 파기해야 하는 메모리 오버헤드가 발생**했다. 실무 서비스는 쓰기 연산이 계속 일어나므로 캐시 관리 비용이 성능을 오히려 더 갉아먹어 **MySQL 8.0부터 완전히 폐기**되었다.

### 버퍼 풀(Buffer Pool)

쿼리 문자열을 통해 캐싱하지 않고, 디스크에 저장된 실제 데이터 블록 페이지와 인덱스 페이지 자체를 메모리에 캐싱해 두는 메모리 영역이 버퍼풀이다. 데이터베이스 성능 튜닝의 핵심으 느린 ‘디스크I/O’를 줄이고 빠른 ‘메모리 접근’으로 전환하는 것이다.

- **읽기 최적화:** 쿼리가 들어오면 InnoDB 엔진은 디스크로 가지 않고 **버퍼 풀 메모리를 먼저 찾는다.** 데이터 페이지가 메모리에 있으면 즉시 반환하고, 없을 때만 디스크에서 페이지를 올려 버퍼 풀에 안착시킨 뒤 반환한다.
- **쓰기 최적화:** 데이터를 수정할 때마다 매번 디스크의 물리적인 파일에 기록하면 엄청나게 느려진다. 그래서 변경 사항이 생기면 **버퍼 풀 메모리에 올라와 있는 페이지의 데이터만 먼저 수정**한다. 디스크 원본과 메모리 값이 달라진 이 시점의 메모리 페이지를 더티 페이지라고 부른다. 이후 백그라운드 쓰레드가 주기를 돌며 이 더티 페이지들을 모아 디스크에 한꺼번에 일직선으로 밀어내어 동기화하는 **플러시(Flush)** 연산을 수행해 성능을 극대화한다.

## 비관적 락과 낙관적 락의 차이점과 각각 언제 사용해야 하냐?

- 차이점 - 비관적 락은 데이터 읽는 시점부터 걸고 낙관적 락은 업데이트 시도하는 시점에서 충돌 검사를 한다.
- 비관적 락은 x lock 같은 물리적인 락을 쓰는데 낙관적 락은 컬럼을 이용하여 어플리케이션 레벨에서 처리한다.
- 충돌이 자주 일어나고 정합성이 중요하면 비관적 락을 사용하고 충돌이 적고 읽기 성능이 중요하면 낙관적 락을 사용한다.

## 정리

- 동시성 제어는 다중 트랜잭션 환경에서 데이터의 일관성과 무결성을 지키기 위해 필수적이며, 성능과 정합성간의 확실한 트레이드 오프를 조율하는 기술이다.
- 비관적 락은 `FOR UPDATE` 구문을 써서 원본 데이터에 물리적인 X-Lock을 채워 충돌을 원천 차단한다. 충돌이 잦은 선착순 로직에 필수적이나 데드락 리스크와 커넥션 풀 고갈 위험이 있다.
- 낙관적 락은 물리 락 없이 `version` 컬럼을 활용해 애플리케이션 레벨에서 충돌을 감지한다. 데드락 위험이 없고 가볍지만, 충돌 발생 시 개발자가 재시도(@Retryable) 로직을 직접 구현해야 하며 충돌이 잦으면 성능이 급락한다.
- 과거의 쿼리 캐시는 무용지물로 삭제되었으며, 현재는 디스크 페이지 자체를 메모리에 올려두고 대량의 읽기/쓰기 성능을 해결해주는 버퍼풀아키텍처가 데이터베이스 최적화의 핵심으로 작동한다.
