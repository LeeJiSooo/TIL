## 이진 탐색 트리(Binary Search Tree)

<img width="1168" height="940" alt="image" src="https://github.com/user-attachments/assets/91efb42f-d358-4bd5-9209-d7812761b66c" />


루트 노드를 기준으로 왼쪽 서브트리에는 나보다 작은값, 오른쪽 서브트리에는 나보다 큰 값을 배치한다.

- 장점 : 찾고자 하는 값이 현재 노드보다 작으면 왼쪽, 크면 오른쪽으로 이동하면 되기 때문에 탐색할 때마다 검색 대상이 절반으로 줄어든다. O(long n)
- 단점(편향 트리 문제) : 데이터가 정렬된 순서로 들어오면 트리가 한쪽으로만 길게 뻗는다. 이렇게 되면 값을 찾기 위해 모든 노드를 다 거쳐야 하므로 일반 배열을 풀스캔하는 것과 다를 바가 없다.

## B-Tree : 디스크 기반 DB에 최적화된 균형 트리

<img width="1218" height="388" alt="image" src="https://github.com/user-attachments/assets/36ef91d1-77ed-42d8-b88e-5cba1f6bdbbe" />


- **디스크 I/O의 장벽** : RAM접근 속도에 비해 디스크 접근(디스크I/O)속도는 수만 배 이상 느리다. 데이터베이스 성능 튜닝의 핵심은 이 “**디스크 I/O 횟수를 얼마나 줄이느냐**”이다. 이진 탐색 트리는 노드를 하나씩 타고 내려갈 때마다 디스크 I/O가 매번 발생할 수 있어서, 데이터가 수백만건이면 깊이만큼 수십 번의 디스크I/O가 생겨서 성능이 저하된다.
- B-Tree의 해결책 : 트리의 높이를 낮춘다.
    - 하나의 노드에 데이터를 단 하나만 담는 게 아니라 여러개를 저장한다.
    - 자식을 최대 2개만 갖는 이진 트리와 달리, B-Tree는 하나의 노드가 수백, 수천 개의 자식을 가질 수 있다.

→ 노드 하나가 DB의 기본 읽기/쓰기 단위인 페이지 또는 블록 크기에 맞춰 설계된다. 덕분에 디스크 I/O한 번에 수백 개의 키 정보를 한번에 메모리로 가져올 수 있다. 1000만개의 데이터가 있어도 트리의 높이가 3~4단계로 압축되므로, 단 3~4번의 디스크 읽기만으로 원하는 데이터를 도달할 수 있게 된다.

## B+Tree : 관계형 데이터베이스(RDB)의 표준 자료구조

MySQL의 InnoDB등 대부분의 RDB는 B-Tree를 더 발전시킨 B+Tree를 인덱스 기본 구조로 채택하고 있다.

### B-Tree와의 차이점

1. **모든 실제 데이터 주소는 최하단 ‘리프노드’에만 저장**
- B-Tree는 루트나 중간 노드에도 실제 데이터의 물리적 주소가 섞여 있어서 공간을 많이 차지한다.
- B+Tree는 중간 노드에 오직 자식을 찾아가기 위한 길잡이 역할(Key값)만 넣는다. 공간이 널널해지기 때문에 중간 노드 하나에 훨씬 더 많은 키를 넣을 수 있고, 트리의 높이는 더욱 낮아져 검색 효율이 극대화된다.

1. **리프 노드끼리 인접 포인터로 연결된 링크드 리스트 구조**
- 범위검색에 뛰어나다 : 예를들어 `HERE age BETWEEN 10 AND 20` 또는 `WHERE age > 30` 같은 범위 쿼리를 날렸다.
- B-Tree였다면 10 찾고 다시 루트로 올라가서 11찾고..이 것을 반복해야 해서 비효율적이다.
- 반면 B+Tree는 루트에서 시작해 딱 10이 있는 리프 노드 위치를 한 번 찾으면, 그 다음부터는 루트로 안 올라가고 리프 노드끼리 연결된 포인터를 타고 20이 나올 때까지 쭉 옆으로 밀면서 읽으면 된다.

→ 기한 검색, 가격 범위 검색에서 유리

## 비클러스터형 인덱스

기본키에 자동으로 붙는 클러스터형과 달리 개발자가 필요에 의해 일반 컬럼에 직접 추가한다.

- 실제 데이터는 테이블 공간에 순서대로 그대로 두고, 별도의 물리적 공간에 인덱스 페이지를 새로 생성하는 방식이다.
- 인덱스 페이지에는 정렬된 `인덱스 키`값과 함께, 그 데이터가 실제 위치한 곳을 가리키는 `물리적인 포인터(페이지 번호/위치)`가 짝을 이뤄 저장된다.
- 테이블 당 하나만 가질 수 있는 클러스터형(PK)과 달리 비클러스터형 인덱스는 개발자가 조회 성능 향상을 위해 **한 테이블에 여러 개를 생성**할 수 있다.

동작 과정

1. 먼저 별도로 생성된 인덱스 페이지에서 내가 찾고자 하는 키 값을 탐색한다.
2. 매핑된 물리 포인터를 확인한 뒤, 그 주소를 따라 실제 데이터가 저장된 데이터 페이지로 이동해서 최종 데이터를 가져온다.

- 장점 : 데이터가 삽입/삭제될 때 실제 데이터 자체를 물리적으로 매번 재정렬할 필요 없이, 별도의 인덱스 페이지만 갱신해주면 되기 때문에 상대적으로 CUD리스크가 분산된다.
- 단점 : 인덱스 탐색 한 번, 데이터 페이지 탐색 한 번, 총 2단계를 거쳐야 하므로 바로 PK를 조회하는 것 보다는 미세하게 느리다. 또한 인덱스를 만들 때마다 별도의 디스크 공간이 추가로 소모된다.

## 인덱스의 단점(트레이드 오프)

인덱스가 무조건 좋은 것은 아니다.

- 저장 공간의 낭비 : 테이블 외에 별도 저장 공간을 차지하므로, 원본 데이터가 1GB면 인덱스 용량은 수백 MB이상 늘어날 수 있다. 인덱스가 많아질수록 서버 용량 부담될 수 있다.
- CUD(쓰기)성능 저하 : 데이터를 추가, 수정, 삭제할 때마다 테이블 데이터만 바꾸는 게 아니라, 이 테이블에 걸려있는 인덱스 구조도 정렬을 유지하기 위해 매번 같이 갱신해야된다.
    - 한 테이블에 인덱스가 5개 걸려있다면, 데이터 딱 한 줄 추가할 때 내부적으로는 총 6번의 쓰기 작업(테이블 1번 + 인덱스 페이지들 5번)실행된다.
    - 따라서 조회가 거의 없고 로그성 데이터처럼 쓰기 작업만 일어나는 테이블이라면 인덱스를 안 거는 것이 효율적이다.

## EXPLAIN으로 데이터베이스 실행 계획 분석하기

```jsx
EXPLAIN SELECT * FROM users WHERE name = 'User_99996';
```

우리가 작성한 쿼리를 DB옵티마이저가 어떻게 할지 미리 보여주는 도구이다. 

<img width="1332" height="188" alt="image" src="https://github.com/user-attachments/assets/12467f34-3260-4c70-b702-c332c8bd9e3c" />


1. `type`
- `const / system` : PK나 Unique Key로 조회하여 딱 1건의 데이터만 정확히 짚어냈을 때 나온다.(최상)
- `ref` : 고유하지 않은 일반 인덱스로 동등비교했을 때 나타난다.(매우 좋음)
- `range` : 인덱스를 사용해 특정 범위(BETWEEN, >, <)를 타서 탐색한다.(좋음)
- `index` : 풀 인덱스 스캔. 인덱스 테이블을 타긴 했는데, 인덱스 페이지의 처음부터 끝까지 다 훓는다. 풀스캔인 ALL보다는 미세하게 낫지만 비효율적이다.(나쁨)
- `ALL` : 인덱스를 전혀 타지 못하고 데이터 테이블 전체를 한 줄씩 다 읽었다는 의미이다.(위험)

2. `key`
- `possible_keys` : 이 쿼리에 사용할 수 있다고 고려한 인덱스 목록
- `key` : possible_keys에 있는 여러 인덱스 후보중에 실제로 사용하기로 결정한 인덱스. 만약 이 값이 NULL로 나오면 인덱스를 안타고 풀스캔을 돌림

3. `rows`
    
    DB가 이 쿼리를 해결하기 위해 “물리적으로 읽어야 할 것이라고 예측한 데이터 행의 개수”이다. 숫자가 작으면 작을수록 좋다.
    

4. `Extra`(추가 주요 정보)
- `Using index` : 쿼리에 필요한 모든 컬럼이 이미 인덱스 페이지 안에 다 있어서, 실제 데이터 페이지까지 안 가고 인덱스 선에서 조회를 끝냈다는 의미이다. (커버링 인덱스) → 성능이 미치도록 빨라진다.
- `Using where`: WHERE절 조건으로 데이터 필터링을 수행했다는 의미이다.
- `Using filesort`: ORDER BY정렬 조건을 인덱스로 처리하지 못해서 DB가 조회 결과를 메모리나 디스크 임시 공간에 적재한 뒤 별도로 정렬 연산을 했다는 뜻이다. 엄청난 부하를 준다.
- `Using temporary`: GROUP BY나 DISTICT 등을 처리하기 위해 내부적으로 임시 테이블을 사용할 수 있다. → 큰 목록이나 큰 데이터에서 자주 보이면 인덱스 설계 다시 확인해봐야한다.

## 인덱스가 있어도 못 타는 쿼리

인덱스를 만들어 놨어도, SQL문법을 잘못 쓰면 DB가 인덱스를 무시하고 풀스캔을 탄다. 

#### 묵시적 형변환(타입 불일치)

phone컬럼이 문자열(VARCHAR)타입으로 인덱스가 걸려있는데, 쿼리를 아래처럼 날릴 때 발생한다.

```jsx
-- ❌ 인덱스 안 탐 (형변환 발생)
WHERE phone = 01012345678; 

-- ⭕ 인덱스 정상 작동
WHERE phone = '01012345678';
```

#### 컬럼을 가공하는 경우(함수 사용)

```jsx
-- ❌ 인덱스 안 탐 (컬럼 원본 가공)
WHERE DATE(created_at) = '2026-06-09';

-- ⭕ 인덱스 정상 작동 (컬럼은 그대로 두고 범위를 조절)
WHERE created_at BETWEEN '2026-06-09 00:00:00' AND '2026-06-09 23:59:59';
```

컬럼값 자체를 DATA()같은 함수로 감싸서 변형해 버리면, 정렬되어 있던 B+Tree 키값들이 뒤틀리기 때문에 대조를 할 수 없다.

#### 와일드카드(%)를 앞에 붙인 LIKE문

```jsx
-- ⭕ 인덱스 탐 (앞 글자가 고정되어 있어서 B+Tree가 범위를 좁힘)
WHERE name LIKE 'User_%';

-- ❌ 인덱스 안 탐 (어디서부터 찾을지 몰라 결국 풀스캔)
WHERE name LIKE '%9996';
WHERE name LIKE '%ser%';
```

앞글자가 고정이 안 되어 있으며 트리의 왼쪽으로 갈지 오른쪽으로 갈지 루트 노드에서 판단을 내릴 수 없다. → 인덱스가 아니라 Fulltext Index를 쓰거나 ElasticSearch같은 전문 검색 엔진으로 넘겨야 한다.

## 인덱스 효율을 결정하는 지표 : 카디널리티 & 셀렉티비티

#### 카디널리티

- 해당 컬럼에 저장된 값의 고유성(중복도가 얼마나 낮은가)을 의미한다.
- 카디널리티 높음(중복 없음) : 주민등록번호, 이메일 주소, 주문번호 등 → 인덱스 걸기 최적의 조건이다. 이메일로 인덱스를 타면 1억명 중 정확히 내 정보 1건으로 범위가 즉시 압축되어 효울성이 커진다.
- 카디널리티 낮음(중복 많음) : 성별(남/여), 상태값(활성/비활성) 등 → 인덱스를 걸면 손해이다. 성별에 인덱스를 걸고 ‘남’을 검색해 봤자 여전히 전체 데이터 절반이 남는다. 이럴 때 DB옵티마이저는 인덱스 페이지 갔다가 데이터 페이지 왔다 갔다 하느니 그냥 인덱스 무시하고 쌩으로 테이블 처음부터 끝까지 훓는게(풀스캔) 낫다고 판단한다.

## 복합 인덱스 설계 기법

실무에서는 하나의 컬럼이 아니라 여러 컬럼을 묶어서 하나의 인덱스로 만드는 복합 인덱스를 주로 사용한다.

```jsx
CREATE INDEX idx_location_name_id ON users(Location, Name, ID);
```

이렇게 인덱스를 설정하면 B+Tree상에서 1차로 Location정렬 → 그 안에서 2차로 Name 정렬 → 같은 Name 안에서 3차로 ID순으로 정렬이 차례대로 묶어서 빌드된다.

#### 이 인덱스를 완벽히 타는 쿼리 패턴

- `WHERE Location = '서울'`
- `WHERE Location = '서울' AND Name = '토르'`
- `WHERE Location = '서울' AND Name = '토르' AND ID = 123`

### 이 인덱스를 전혀 타지 못하는 쿼리 패턴

- `WHERE Name = '김철수'`
- `WHERE ID = 123`
- **이유:** 가장 대전제 분기점인 1차 기준(`Location`)이 조건절에 누락되었기 때문에, DB 입장에서는 정렬 구조를 써먹지 못하고 인덱스를 무시하게 된다.

복합 인덱스는 무조건 자주 쓰는 걸 다 때려 넣는 게 아니라 아래 규칙에 맞춰 컬럼 배치 순서를 잡아야 한다.(동등 → 범위 → 정렬)

특정 유저의 최근 주문 목록을 가져오는 쿼리
`WHERE user_id = 10 AND created_at >= '2026-06-01' ORDER BY created_at DESC, id DESC LIMIT 20`

- **후보 인덱스 구성 순서:** `(user_id, created_at, id)`로 순서를 배치한다.
- **이유:** 먼저 `=`(동등 조건)인 `user_id = 10`으로 대상을 좁혀놓은 뒤, 그 좁혀진 범위 안에서 `created_at` 범위(Range)를 필터링하고 마지막 정렬 및 LIMIT 조건으로 이어져야 인덱스 체인을 끝까지 유지할 수 있다. 만약 범위 조건인 `created_at`을 맨 앞에 둬버리면 전체 유저의 날짜 데이터를 크게 먼저 훑어야 해서 효율이 좋지 않다. 범위 조건을 만난 뒤의 컬럼은 정렬 연속성이 깨지기 쉽다.

## 커버링 인덱스

요청한 SELECT절, WHERE절, ORDER BY절 등 쿼리 전체에 필요한 모든 컬럼이 이미 인덱스 페이지 안에 전부 다 포함되어 있는 상태

`users` 테이블에 `(name, age)` 순서로 복합 인덱스가 걸려있다.

```jsx
SELECT name, age FROM users WHERE name = '토르';
```

이 쿼리가 원하는 것은 오직 name과 age정보다. 근데 인덱스 페이지에는 이미 키 값으로 name과 age가 같이 정렬되어있다.

→ 디비는 굳이 실제 데이터 페이지 주소를 찾아가서 뒤지는 과정을 생략하고 가볍고 빠른 인덱스 페이지만 읽어서 결과를 즉시 반환한다.

```jsx
-- ❌ 커버링 인덱스 불가능 예시 (아스테리스크)
SELECT * FROM users WHERE name = '토르';
```

인덱스 페이지에는 name과 age 정보만 있지. 다른 주소나 상세 정보들까지 다 들어있지는 않다. 결국 *때문에 어쩔 수 없이 실제 데이터 블록으로 가서 조회해야 하므로 커버링 인덱스의 이점이 깨져버린다.

## 정리

- 인덱스는 디스크 I/O횟수를 줄이기 위해 트리 높이를 낮추고 범위 검색 포인터를 포함하고 있는 B+Tree자료구조이다.
- EXPLAIN분석 시 tpye이 ref나 range인지, rows 수가 최소화되었는지, Extra에 성능 저하 주범인 filesort나 temporary가 뜨지 않는지 확인해야 한다.
