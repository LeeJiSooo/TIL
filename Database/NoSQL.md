## 가져가야 할 핵심

1. NoSQL은 ‘No SQL’이 아니라 ‘Not noly SQL’의 약자이다. RDBMS를 완전히 대체하는 더 우월한 기술이 아니라, 데이터의 형태와 접근 방식에 따라 상호 보완적으로 선택하는 저장소이다.
2. NoSQL은 다양한 데이터 모델을 제공하므로, 단순히 저장 방식의 편리함보다는 우리의 서비스가 데이터에 어떻게 접근(조회)하는가에 맞춰 제품을 골라야 한다.
3. NoSQL은 유연성과 수평적 확장성(Scale-out)을 얻는 대가로, 일관성 완화, 트랜잭션의 제한, 운영 복잡도 증가라는 트레이드 오프를 감당해야 한다.

## RDBMS의 한계와 NoSQL의 등장

RDBMS는 정해진 스키마(테이블 구조)에 맞춰 데이터를 엄격하게 저장한다. 회원 테이블에 `ID`, `이름`, `이메일` 컬럼이 있으면 무조건 여기에 맞춰야 하며, 구조를 바꾸려면 `ALTER TABLE` 마이그레이션을 거쳐야 한다.

**커머스 백엔드에서 RDBMS 단독 운영 시 발생하는 한계**

종합 커머스 서비스를 구축할 때 티셔츠(색상, 사이즈, 소재), 노트북(CPU, RAM, SSD), 화장품(성분, 용량, 유통기한)처럼 카테고리별로 필요한 속성이 완전히 다르다.

- **RDB 상품 테이블 하나로 몰아넣을 때** : 어떤 카테고리에서는 쓰이지 않는 수십 개의 Nullable(NULL 허용)컬럼이 생기며, 새 카테고리가 추가될 때마다 대규모 스키마 마이그레이션을 해야 한다.
- **보조 테이블로 쪼갤 때** : 테이블 분리 후 속성을 가져오기 위해 수많은 JOIN연산이 필요해져 쿼리가 극도로 복잡해지고 성능이 저하된다.
- 여기에 채팅 메시지, 사용장 행동 로그, 추천 시스템 관계망까지 모두 RDB의 테이블 구조에 억지로 맞추려하면 정합성과 성능이 모두 무너진다. 이 구조적 경직성을 해결하기 위해 유연한 스키마를 가진 NoSQL을 선택한다.

#### 스키마 유연성

NoSQL은 염격한 제약에서 벗어나 자유로운 구조로 데이터를 처리한다. 예를들어 어떤 유저 데이터는 이름과 이메일만 저장하고, 다른 유저 데이터에는 관심 기술, 포트폴리오 링크, 선호 지역 필드를 동적으로 추가하여 저장하는 것이 가능하다.

주의 : 설계를 안 해도 된다는 의미가 아니라, NoSQL환경에서도 어떤 Key로 데이터를 읽을지, 무엇을 함께 묶어서 저장할지, 일관성을 어디까지 보장할지에 대한 설계 능력이 요구된다.

## NoSQL을 사용하는 이유 : 수평적 확장성(Scale-out)

- 수직 확장(Scale-up) : 기존 데이터베이스 서버 자체의 하드웨어 사양(CPU, RAM 등)을 업그레이드하는 방식이다. 성능을 올릴수록 비용이 기하급수적으로 증가하여, 물리적인 하드웨어 한계에 부딪힌다. RDBMS가 주로 이 방식을 취한다.
- 수평 확장(Scale-out) : 비슷한 사양의 저렴한 서버 여러 대를 추가하여 데이터를 분산 저장하고 요청을 쪼개어 처리하는 방식이다. NoSQL제품군 대부분은 처음부터 이 수평적 확장에 유리하도록 아키텍처가 설계되어 있다.

→ 백엔드 개발자는 단순히 NoSQL툴을 쓸 줄 아는 것을 넘어, 분산 환경을 통제하는 레벨(레디스의 클러스터 구축 및 데이터 분산 샤딩 전략)까지 깊이 이해하고 제어할 수 있어야 한다.

## NoSQL의 데이터 모델

#### Key - Value 모델 : “키를 알면 즉시 값을 찾는다”

NoSQL 중 가장 단순하고 직관적인 형태이다. 자바의 `HashMap` 자료구조를 네트워크 데이터베이스 레벨로 확장한 구조이다.

내부적으로 해시 구조를 사용하여 특정 Key를 물리적 위치에 매핑한다. 복잡한 연산이나 조건 조회가 불가능한 대신, 단순한 Key 기반 조회와 저장은 가장 빠르다.

- 캐시 서버 : 복잡한 RDB JOIN이나 무거운 연산을 거쳐 나온 결과물을 매번 재계산하는 것은 비효율적이다. 한 번 계산한 결과를 `Key(결과 ID) - Value(연산 결과)` 형태로 저장해 두고, 다음 요청 때 DB까지 가지 않고 메모리 레벨에서 즉시 반환한다.
- 세션 및 인증 관리 : 사용자 로그인 토큰이나 세션 정보는 매 요청마다 유효성을 검증해야 하므로 속도가 절대적이다. `sessionId : 유저정보` 를 묶어 관리한다.
- 단기성 데이터 : 몇 분 뒤 사라져야 하는 비밀번호 재설정 인증코드(reset-code:email), 실시간 카운팅 시스템 등에서 사용된다.

Redis(레디스), Memcached Valkey 등이 있다. Redis의 경우 단순 Key-Value를 넘어 List, Set, Hash 등 다양한 인메모리 자료구조를 지원한다. 기본적으로 인메모리(RAM)기반이므로 RDB 수준의 영구 지속성을 보장하지 않는다는 특징이 있다.

#### 문서 지향 모델(Document DB) : “객체 덩어리로 함께 읽는다”

Key-Value 모델에서 한 단계 진화한 형태로, Value 자리에 단순 문자열이 아닌 JSON, BSON, XML 같은 구조화된 문서(Document)가 통째로 들어가는 모델이다.

- 각 도큐먼트는 고유한 `_id`를 가진다. ID 중심의 초고속 탐색과, 문서 내부에 포함된 특정 필드 속성값에 개별적으로 인덱스(Index)를 걸어 효율적인 조건 검색을 수행할 수도 있다. 다만 정합성 보장 룰과 인덱스 제어 한계는 RDB보다 약하다.
- RDB와의 차이(JOIN의 제거) :
    - RDB : 정규화 원칙에 따라 게시글과 댓글을 독립된 테이블로 쪼개 저장한 뒤, 조회 시점에 `JOIN`으로 합친다.
    - Document DB : 하나의 게시글 도큐먼트 내부에 댓글들을 배열 형태로 임베딩하여 통째로 저장한다.

```jsx
{
  "_id": "post123",
  "title": "NoSQL에 대하여",
  "author": "김개발",
  "comments": [
    { "author": "박기획", "text": "좋은 글이네요!" },
    { "author": "이디자인", "text": "잘 읽었습니다." }
  ]
}
```

- **효과:** 인스타그램 피드나 블로그 포스트처럼 화면에 본문과 댓글 2~3개가 한 세트로 묶여서 같이 읽히는 데이터는 JOIN 연산 없이 단 1번의 I/O 읽기 작업만으로 모든 데이터를 긁어올 수 있어 조회 효율이 극대화된다.
- **주의점:** 만약 댓글이 수만 개로 늘어나거나, 댓글 데이터만 개별적으로 수정/삭제가 자주 일어나는 비즈니스 패턴이라면 문서 크기가 비대해져 오히려 쓰기 비용이 폭락한다. 즉, **함께 읽히는 빈도가 압도적이고 크기가 어느 정도 제한되는 도메인**에만 맞추어 설계해야 한다. 필요시 문서 간 참조(Reference) 링크를 걸기도 한다.

대표적으로 **MongoDB**가 있다.

#### Wide-Column DB : “대량으로 쌓고 키 범위로 읽는다”

행마다 가질 수 있는 컬럼의 이름과 개수가 동적으로 변하는 구조로, 대규모 데이터 집계와 시계열 분석을 위해 사용하는 모델이다.

- 고정된 소수의 컬럼 구조에 얽매이지 않고, 데이터 특성에 따라 수백 수천 개의 컬럼을 동적으로 늘릴 수 있다. 테이블 전체를 읽는 게 아니라 같은 행 안에서 필요한 특정 컬럼 묶음만 정밀하게 정렬,적재하여 처리한다.
- 대규모 서비스의 서버 로그 수집, 사용자 행동 이벤트 기록, 시간의 축 흐름에 따라 데이터가 쏟아지는 시계열 데이터 처리에 최적화되어 있다.
- 쓰기 처리량이 매우 높고 대용량 데이터를 받아내야 하므로 “어떤 쿼리를 날릴지 먼저 완벽하게 확정한 뒤, 그 조회 조건(특정 사용자, 특정 시간대, 특정 범위 키)에 맞춰 거꾸로 데이터의 키 구조를 정밀하게 설계”하는 것이 핵심이다.

대표적으로 **Cassandra, HBase**가 있다.

#### Graph DB : “관계를 따라 추적한다”

데이터 자체보다 데이터와 데이터 간의 관계에 모든 초점을 맞춘 특수 데이터베이스이다. 데이터를 노드와 간선 구조로 표현하여 저장한다.

- RDB도 외래키로 관계를 표현할 순 있지만, 관계의 깊이가 깊어지면 힘들다. (예 : 내 친구의 친구를 찾으려면 2중 JOIN, 친구의 친구의 친구를 찾으려면 3중 JPON이 걸리며 테이블 전체를 다 뒤지느라 시스템이 뻗는다) 반면 그래프 DB는 각 노드가 옆방의 연결된 이웃 노드의 물리적 주소를 직접 쥐고 추적하도록 설계되어 있어 복잡한 관계망을 타고 들어가는 탐색 연산 속도가 압도적으로 빠르다.
- 소셜 네트워크 서비스(SNS)의 인맥 지도, 친구 추천 시스템, 공통 관심사 매핑, 이커머스의 지능형 상품 추천 맵, 금융 사기단 추적, 지식 그래프 구축 등에 특화되어 있따.
- 서비스 전체 데이터를 그래프DB에 넣지는 않는다. 핵심 비즈니스는 RDB에 두고, 추천 시스템이나 복잡한 관계망 추적 연산이 필요한 특정 서브 도메인의 데이터만 그래프 DB로 동기화 해서 격리 운영한다.

**Neo4j**가 대표적이다.

| **비즈니스 요구사항 및 데이터 성격** | **최적의 저장소 / 데이터 모델** | **선택의 핵심 근거** |
| --- | --- | --- |
| **정확한 돈, 트랜잭션, 정합성이 생명인 정형 데이터** | **RDBMS** (MySQL, PostgreSQL) | 엄격한 스키마와 완벽한 전역적 ACID 보장이 절대적이기 때문이다. |
| **로그인 세션, 임시 인증코드, 초고속 캐싱 및 카운터** | **Key-Value NoSQL** (Redis) | 메모리 레벨에서 고유 고정 키 조회를 최우선으로 빠르게 처리해야 하기 때문이다. |
| **다양한 카테고리 상품 상세, 게시글-댓글 묶음 조회** | **Document NoSQL** (MongoDB) | 유연한 스키마로 가공하며, 자주 같이 읽히는 객체 덩어리를 JOIN 없이 퍼와야 하기 때문이다. |
| **대규모 시스템 접속 로그, 시계열 데이터, 유저 행동 이벤트** | **Wide-Column NoSQL** (Cassandra) | 엄청난 쓰기 트래픽을 견디며 수평적으로 서버를 확장해야 하기 때문이다. |
| **SNS 인맥 네트워크, 선호도 기반 가중치 상품 추천 구조** | **Graph NoSQL** (Neo4j) | 3~4단계 이상 꼬여있는 복잡한 인간/사물 간의 연결 고리를 추적해야 하기 때문이다. |
| **본문 키워드 다중 검색, 오타 교정, 다면 필터링 및 통계** | **Search Engine** (ElasticSearch) | 역인덱스 장부와 유연한 형태소 분석으로 검색 품질을 극대화해야 하기 때문이다. |
