## 가져가야 할 핵심

1. 트랜잭션은 비즈니스 로직상 절대로 쪼개지면 안 되는 논리적인 작업 단위이다.
2. 커밋(Commit)과 롤백(Rollback), 그리고 ACID원칙이 데이터 정합성을 어떻게 지키는지 이해해야 한다.
3. 트랜잭션의 범위 설정이 시스템의 성능, 락, 커넥션 풀에 직접적인 영향을 준다는 사실을 인지해야 한다.

## 트랜잭션 정의

데이터베이스 시스템에서 자료를 처리하는 상호작용이자, 절대로 쪼개져서는 안 되는 논리적인 작업의 최소 단위이다. → 비즈니스 규칙에 따라서 개발자가 정의한다.

**트랜잭션이 없을 때(주문 상황)**

사용자가 쇼핑몰에서 주문 버튼을 눌렀는데 아래와 같이 작업이 실행되면 재앙이 발생한다.

1. 상품 재고 차감 → 성공
2. 주문 내역 생성 → 실패(에러 발생)
3. 결제 정보 기록 → 실패

결과적으로 데이터베이스에는 재고만 줄어들고 주문 내역과 결제 정보는 증발한다. 고객은 주문 성공 여부를 알 수 없고, 운영자는 꼬인 데이터를 수동으로 일일이 맞춰야 한다. → 이러한 상태를 차단하기 위해 트랜잭션을 사용한다.

**“전부 성공(Commit)하거나 전부 실패(Rollback)하거나”**

- 재고 차감, 주문 내역 추가, 결제 정보 기록이라는 3가지 작업은 ‘고객의 주문’이라는 하나의 논리적 행위를 구성한다. 비즈니스 논리상 셋 다 성공하거나, 하나라도 실패하면 아예 없었던 일이 되어야 한다.
- **커밋(Commit)** : 모든 SQL작업이 성공적으로 수행되어 변경 사항을 DB에 영구적으로 반영하는 명령이다.
- **롤백(Rollback)** : 작업 도중 하나라도 에러가 발생하면, 트랜잭션 시작 이후 수행한 모든 변경 사항을 통째로 취소하고 이전 상태로 되돌리는 명령이다.
- **Spring에서의 처리** : `@Transactional` 어노테이션을 메소드에 붙여 사용한다. 이 어노테이션은 Spring에게 “이 메소드가 하나의 트랜잭션 경계”임을 알리는 선언이다. 메소드 시작 전에 트랜잭션을 자동으로 열고, 성공적으로 끝나면 커밋을, 중간에 예외가 발생하면 자동으로 롤백을 수행한다.

## 트랜잭션의 상태 흐름

<img width="1436" height="432" alt="image" src="https://github.com/user-attachments/assets/fee225cd-3efd-40bc-9f89-dcdf2239d9cd" />


트랜잭션은 실행 시작부터 종료까지 위와 같은 생명주기 상태를 거치며 제어된다.

- **활동 (Active):** 트랜잭션이 시작된 점이다. 현재 실행되는 SQL 쿼리들은 트랜잭션에 포함되며, DB는 변경 내용을 실제 테이블에 바로 반영하지 않고 로그와 버퍼에서 격리하여 관리한다.
- **실패 (Failed):** 작업 도중 오류가 발생한 상태이다. 오류 발생 시 트랜잭션 전체를 즉시 무효화할지, 진행해도 될지는 DBMS 종류에 따라 다르다.
    - *PostgreSQL:* 중간에 에러가 나면 이후 명령 수행을 엄격히 제한한다.
    - *MySQL:* 앞에서 성공한 쿼리가 버퍼에 남아있다가 개발자의 실수로 커밋될 여지가 있으므로 예외 처리에 주의해야 한다. 즉, 실패 과정의 세부 처리는 DB마다 다를 수 있다.
- **철회 (Aborted):** 실패를 감지하고 `ROLLBACK` 명령을 실행한 상태이다. 모든 변경 사항을 트랜잭션 이전으로 완전히 되돌리고 트랜잭션을 안전하게 종료한다.
- **부분 완료 (Partially Committed):** 마지막 SQL 작업까지 오류 없이 성공적으로 실행되었으나, 아직 최종 확정(`COMMIT`)을 짓기 직전의 단계이다. 모든 준비는 끝났고 확정 명령만 기다리는 상태이며, 만약 이 순간 시스템 장애가 나면 DB 재시작 시 최종 커밋이 없었으므로 롤백 처리된다.
- **완료 (Committed):** `COMMIT` 명령이 성공적으로 실행되어 트랜잭션이 종료된 상태이다. 이때부터 데이터는 비휘발성 저장 장치에 안전하게 저장되어 지속성을 확보한다.

## 트랜잭션의 원자성 지키기

예시1 : 계좌 이체 쿼리

```jsx
START TRANSACTION;

-- 1. void의 계좌에서 10만 원 차감
UPDATE account SET balance = balance - 100000 WHERE username = 'void';

-- 2. yaro의 계좌에 10만 원 추가
UPDATE account SET balance = balance + 100000 WHERE username = 'yaro';

COMMIT;
```

- 1번 쿼리가 성공한 직후 DB 서버 전원이 꺼졌다고 가정한다. 트랜잭션 장치가 없다면 void의 돈 10만 원은 증발하고 yaro에게는 입금되지 않아 데이터 정합성이 완전히 깨진다.
- 두 작업이 하나의 트랜잭션으로 묶여있다면, 서버가 재시작될 때 DB는 로그 파일을 확인하고 "커밋 완료 마크가 없네?"라고 판단하여 1번 작업을 자동 롤백한다. void의 잔고를 원래대로 되돌려 정합성을 지켜낸다.

에시2 : 패키지 여행 예약 (상하위 트랜잭션 전이)

고객은 기차표, 숙소, 항공권이 모두 예약되어야 패키지 여행이 최종 예약되었다고 인식한다.

```jsx
[상위] 패키지 여행 예약 Transaction
   ├── [하위1] 왕복 항공권 구매 Transaction
   ├── [하위2] 왕복 기차권 구매 Transaction
   └── [하위3] 숙소 1박 예약 Transaction
```

- **하위 트랜잭션 수준의 롤백:** [하위1] 항공권 트랜잭션 진행 중 '가는 편'은 예약 성공했으나 '오는 편' 예약 시 에러가 나면, 항공권 트랜잭션 자체적으로 롤백이 발생해 '가는 편' 예약도 취소된다.
- **상위 트랜잭션 수준의 롤백 전이:** 항공권과 기차권 예약은 완벽히 성공했으나, 마지막 [하위3] 숙소 예약 도중 방이 없어서 실패했다면 어떻게 될까? 패키지 여행이라는 큰 비즈니스 상품 기준으로는 전체 실패이다. 따라서 숙소 예약 실패가 상위 트랜잭션으로 전이되고, 이미 성공했던 항공권과 기차권 예약까지 체인처럼 엮여서 전부 롤백(취소)된다.

**현대 아키텍처(MSA)에서의 보상 트랜잭션**

실무에서는 단일 DB 트랜잭션을 넘어 외부 API나 다른 마이크로서비스(MSA)와 연동되는 경우가 많다.주문 서비스 DB는 성공했어도 결제 서비스 DB에서 실패할 수 있다. MSA환경에서는 이를 물리적으로 묶을 수 없기 때문에, 결제 실패 시 앞서 주문했던 주문을 취소하는 ‘보상 트랜잭션’로직을 개발자가 코드로 직접 구현하여 비즈니스 정합성을 맞춰야 한다.

## 트랜잭션의 범위와 성능(커넥션 풀 고갈 위험)

트랜잭션은 개발자가 범위를 자유롭게 설정할 수 있으나, 반드시 필요한 순수DB 작업만 포함하여 최대한 짧고 간결하게 가져가야 한다. 트랜잭션 범위가 불필요하게 길어지면 심각한 시스템 장애로 이어진다.

- 데이터베이스는 다수의 동시 요청을 처리하기 위해 연결 통로를 미리 파놓는 커넥션 풀을 운용한다. 트랜잭션이 시작되는 순간 쓰레드는 이 커넥션 풀에서 커넥션 하나를 쥐고 놓지 않는다.
- Spring의 `@Transactional`메소드 안에서 외부 결제사 API호출이나 알림톡 발송 로직을 결합했다고 하자. 외부 결제사 서버가 일시적으로 무거워져 응답이 3초간 지연되면, 그 3초동안 주문 데이터 행에는 락이 걸리고 DB커넥션도 묶여있게 된다.
- 동시 주문 요청이 수십 개만 몰려도 커넥션 풀의 모든 커넥션이 대기 상태로 장기간 잡혀버리므로 **커넥션 풀 고갈 장애**가 터진다. 뒤이어 들어오는 일반 로그인, 조회 요청 회원들까지 전부 무한 대기를 타다가 서버가 터지게 된다. 외부 연산이나 무거운 비즈니스 로직은 트랜잭션 경계 밖으로 분리해야 한다.

## 트랜잭션의 4대 기본 원칙 : ACID

#### Atomicity(원자성)

All or Nothing

트랜잭션 내 작업은 전부 성공하거나, 하나라도 실패하면 아예 실행 안 된 상태로 돌아가야 한다. 부분 성공은 절대 없다.

#### Consistency(일관성)

상태의 유지

트랜잭션 완료 후에도 DB는 스키마 제약조건(타입, UNIQUE, 무결성 규칙 등)을 완벽히 지킨 올바른 상태여야 한다.

잔고는 0원 이상이어야 한다는 제약조건이 있을 때 잔고가 마이너스가 되는 트랜잭션을 실행하면 DB는 이를 거부하고 롤백한다.

#### Isolation(격리성)

상호 간섭 불가

동시에 실행되는 여러 트랜잭션들은 서로의 중간 작업 프로세스를 훔쳐보거나 영향을 주어서는 안된다.

#### Durability(지속성)

영구적 보존

일단 성공적으로 커밋(COMMIT)된 결과는 이후 어떤 시스템 장애나 하드웨어 다운이 발생해도 영구적으로 기록에 남아 보존되어야 한다. DB는 이를 위해 Write-Ahead Logging(WAL)같은 로그 메커니즘으로 복구를 보장한다.

## 격리성(Isolation)의 한계와 격리수준(Isolation Level)

완벽한 격리성(모든 트랜잭션을 줄 세워서 하나씩 처리)을 달성하면 데이터 정합성은 100% 지켜지지만, 동시 처리 성능은 안 좋아진다. 따라서 동시성(성능)을 챙기기 위해 격리성을 단계별로 완화한 격리수준 옵션을 선택하여 타협한다.

### 격리 수준 완화 시 발생하는 이상 현상(정합성 에러)

#### Dirty Read(더티 리드)

- 트랜잭션 A가 데이터를 수정 중이고 아직 커밋하지 않았는데, 트랜잭션 B가 그 미확정 데이터를 그대로 읽어갈 수 있는 현상이다.
- 만약 트랜잭션 A가 마음을 바꿔 작업을 롤백해버리면, 트랜잭션 B는 세상에 존재하지도 않는 가짜 데이터(Dirty Data)를 자기고 비즈니스 연산을 처리한 셈이 되어 정합성이 완벽히 박살난다.

#### Non-Repeatable Read(반복 불가능 읽기)

- 트랜잭션 A안에서 1번 회원 정보를 조회(결과 : 나이 20)했다. 그런데 트랜잭션 A가 끝나기 전에 트랜잭션 B가 들어와서 1번 회원의 나이를 25세로 바꾸고 커밋을 완료해 버렸다. 이후 트랜잭션 A가 동일한 메소드 안에서 다시 1번 회원을 조회했더니 이번엔 나이가 25세로 바뀐 채 조회되는 현상이다.
- 하나의 트랜잭션 울타리 안에서 조회 명령을 두 번 내렸는데 중간에 값이 바뀌어 버리므로 데이터의 일관된 분석 및 처리가 불가능해진다.

#### Phantom Read(유령 읽기)

- 트랜잭션 A가 WHERE age ≥ 30 조건으로 범위를 조회(결과 : 9건 데이터 노출)했다. 그 사이트에 트랜잭션 B가 나이가 35세인 새로운 회원 1건을 추가하고 커밋을 했다. 이후 트랜잭션 A가 똑같은 범위 쿼리를 다시 날렸더니, 첫 번째 조회 때 없었던 유령같은 데이터 1건이 튀어나와 총 10건이 조회되는 현상이다.
- 값의 수정이 아니라 새로운 데이터의 삽입/삭제로 인해 데이터 목록 건수 자체가 달라지는 정합성 오류이다.

### 4가지 격리수준(Isolation Level)분석

격리 수준이 낮을수록 성능(동시성)은 올라가지만 정합성이 깨지고, 격리수준이 높을수록 병목이 심해지지만 정합성이 완벽해지는 확실한 트레이드 오프 관계를 가진다.

#### Level 0 : Read Uncommitted(미커밋 읽기 허용)

- 다른 트랜잭션이 커밋하지 않은 데이터까지 무조건 읽는 것을 허용한다.
- Dirty Read, Non-Repeatable Read, Phantom Read가 전부 발생한다.
- 실무에서는 정합성이 너무 쉽게 깨지기 때문에 절대 사용하지 않는다. 만약 RDB에서 정합성이 아예 필요 없는 수준의 날것의 작업을 해야 한다면, 비싼 RDB를 쓰는 것 자체가 에러다. 그런 작업은 애초에 NoSQL저장소로 넘겨서 처리한다.

#### Level 1: Read Committed (커밋된 읽기만 허용)

- 오직 COMMIT이 완료된 깨끗한 데이터만 읽도록 보장한다.
- Dirty Read는 확실히 막아주어 최소한의 정합성을 방어하지만, **Non-Repeatable Read와 Phantom Read는 여전히 발생**한다.
- **Oracle, PostgreSQL 등 수많은 RDBMS의 디폴트(기본) 설정**이다. 적절한 동시 처리 성능을 유지하면서 치명적인 가짜 데이터 읽기(Dirty Read)를 방어해 주므로 가장 널리 쓰인다.

#### Level 2: Repeatable Read (반복 읽기 보장)

- 한 트랜잭션이 시작될 때의 데이터 시점을 고정하여, 다른 트랜잭션이 값을 바꾸고 커밋을 백날 치더라도 **내 트랜잭션 안에서는 무조건 처음 조회했던 일관된 스냅샷 데이터만 보여주도록 보장**한다.
- 이론적으로는 Repeatable Read **(유령 읽기)가 발생할 수 있다.**  *(MySQL InnoDB의 강력함):* MySQL의 InnoDB 엔진은 내부적으로 MVCC(Multi-Version Concurrency Control)와 갭 락(Gap Lock) 기술을 쓰기 때문에, Repeatable Read 레벨에서도 Phantom Read가 기본적으로 거의 발생하지 않는 독보적인 안전성을 자랑한다. 그래서 **MySQL의 디폴트 격리 수준은 Repeatable Read**이다.
- 연말 결산 보고서나 통계용 데이터를 뽑을 때처럼, 여러 단계에 걸쳐 데이터를 반복 조회하고 정밀히 계산해야 해서 중간에 데이터가 절대 바뀌면 안되는 금용/회계 도메인이나 배치 작업에 매우 유용하다. 특점 시점 기준의 스냅샷 정합성을 완벽히 챙길 수 있다.

#### Level 3: Serializable (직렬화 수행)

- 가장 엄격하고 높은 격리 수준이다. 여러 트랜잭션이 동시에 들어오면 묶어서 순차적으로 줄을 세운 뒤 무조건 한 번에 한 트랜잭션만 실행되도록 제어한다.
- 어떠한 이상 현상(Dirty, Non-Repeatable, Phantom)도 완벽히 차단된다.
- 하나의 트랜잭션이 단지 SELECT로 데이터 읽기 작업만 하고 있어도 해당 구역 전체에 공유락을 꽉 쥐고 있기 때문에 다른 트랜잭션이 읽기든 쓰기든 절대 끼어들지 못하고 무한 대기를 탄다. 동시 처리 성능이 최악으로 떨어지므로 극심한 병목 현상이 생긴다. 일반적인 상황에선 사용하지 않으며, 돈이 안 맞으면 큰일 나는 초정밀 금융 거래나 단 1건의 오차도 혀용 않는 크리티컬 선착순 재고 시스템 등 극히 제한적인 구간에서만 전략적으로 켠다.
