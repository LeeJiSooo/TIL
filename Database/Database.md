## 데이터베이스의 핵심 개념과 필요성

#### 가져가야 할 핵심 포인트

1. 데이터와 정보의 차이 구분
2. 데이터베이스가 왜 필요한지 설명할 수 있어야함
3. DB, DBMS, RDBMS의 역할 차이를 구분할 수 있어야함
4. 테이블, 기본키, 외래키가 관계형 데이터베이스의 뼈대임을 이해해야함

데이터베이스는 단순히 데이터를 쌓아두는 공간이 아니다. 

체계적인 구조(Schema)를 가지고 데이터를 저장, 관리함으로써 언제든지 의미 있는 정보로 가공하여 활용할 수 있도록 만들어진 데이터의 집합체이다.

→데이터를 테이블 체계로 저장하여, 데이터의 안정성과 신뢰성을 보장하기 위함이다.

### 데이터베이스를 사용해야하는 이유

#### 데이터의 영속성 및 안정성

스프링 등으로 어플리케이션을 개발할 때, 실행 중에 변수나 객체에 저장되는 데이터는 메모리에 위치한다. 메모리는 휘발성이 있어서 어플리케이션이 종료되거나 서버 전원이 꺼지면 모든 데이터가 사라진다.

해결책 : 데이터베이스는 데이터를 HHD, SSD같은 비휘발성 저장 장치에 안전하게 저장하여 언제든지 다시 불러올 수 있도록 보장한다. → 데이터의 생명을 영속적으로 만들어주는 계층이다.

#### 데이터의 무결성 및 신뢰성

데이터베이스는 데이터의 구조와 규칙을 강제할 수 있다.

나이를 저장하는 컬럼에는 ‘숫자’만 들어가게 하거나, 이메일 컬럼에는 반드시 ‘@’기호가 포함되도록 규칙을 설정할 수 있다. 만약 누군가 나이 컬럼에 ‘스무살’이라는 텍스트를 입력하려고 하면, DB가 입력 자체를 거부하고 저장하지 않는다.

→ 이를 통해 데이터가 항상 깨끗하고 정확한 상태로 유지됨을 보장한다.

#### 데이터의 동시 접근 및 공유

쇼핑몰에 동시에 100명의 사용자가 접속해 상품을 구매할 때 상품 재고를 DB없이 엑셀 파일 등으로 관리하면 데이터가 깨지거나 재고가 -1이 되는 등 동시성 문제가 발생한다.

→ DB는 트랜잭션과 락을 활용하여 동시성 문제를 처리한다.

→ 트랜잭션과 락만으로 모든 게 해결되는 것은 아니며, 트랜잭션의 범위, 제약조건, 올바른 쿼리 작성이 이루어져야 여러 사용자가 동시에 접근해도 데이터 일관성을 지킬 수 있다.

#### 단일 테이블 관리의 한계(이상 현상)

상품 정보(상품명, 가격)과 구매자 정보(구매자명, 주소)를 하나의 테이블에 몰아서 관리할 때 다음과 같은 문제가 발생한다. 테이블이 커질수록 이 상품을 설명하는지, 구매를 설명하는지 경계가 흐려진다.

- 데이터 중복 및 갱신 이상 : 같은 상품을 여러 사람이 구매하면 상품명과 가격이 계속 반복 저장된다. 나중에 상품 가격이 변경되면 모든 행을 찾아서 수정해야 하므로 비효율적이며, 하나라도 놓치면 데이터 불일치가 발생한다.
- 데이터 관리의 어려움(삽입 이상) : 구매자가 순수하게 ‘상품 정보’만 먼저 추가하고 싶어도, 테이블 구조상 구매자 칸이 비어있게 된다. 이를 방지하기 위해 임의의 가짜 데이터를 억지로 맞춰 넣어야 하는 문제가 생긴다.
- 데이터 손실(삭제 이상) : 특정 구매자 정보를 삭제했더니, 그 구매자와 함께 한 행에 보관되어 있던(원래 유지되어야 할)상품 정보까지 같이 사라져 버린다.

#### 해결책 : PK와 FK를 토한 관계 형성

RDB는 데이터를 성격에 따라 분리하고 인덱스를 기반으로 관계를 맺어 위 문제를 해결한다.

- 기본키(PK) : 상품마다 고유한 인덱스를 부여한다. 이 상품 인덱스는 각 상품을 구별하는 유일한 값으로, 절대 중복되어서도 안 되고 비어있어서도 안된다.(NOT NULL & UNIQUE)
- 외래키(FK) : 구매자 테이블에는 상품명을 직접 적는 대신, 상품 테이블과 연결하기 위한 ‘상품 인덱스’를 넣는다. 이처럼 다른 테이블의 기본키를 참조하는 값을 외래키라고 한다.

## 스키마(Schema)의 의미

스키마는 단순히 ‘테이블을 어떻게 만들까’에 대한 명세서가 아니라, 어플리케이션(서버 코드)과 데이터베이스 사이의 단단한 ‘계약’이다.

서버 개발자는 “user 테이블의 email컬럼은 문자열이고, 비어있지 않으며(NOT NULL), 중복되지 않는다(UNIQUE)”라는 전제를 신뢰하고 로그인/회원가입 로직을 작성해야한다. 또한 “주문 테이블의 user_id가 외래키로서 실제 회원을 가리킨다”라는 확신이 있어야 주문 목록을 안심하고 조회할 수 있다.

이러한 전제와 규칙들 어플리케이션 코드만으로는 완벽히 지켜지지 않기 때문에, DB레벨에서 규칙(데이터 타입, NOT NULL, UNIQUE, FOREIGN KEY 등)을 강제해야 한다. 그래서 DB스키마는 데이터의 방어선이다.

→ 스키마를 바꾸는 일(컬럼 추가, 타입 변경, 제약조건 추가 등)은 해당 테이블을 사용하는 서버레벨의 코드 수정도 반드시 동반하므로 신중해야 한다.

## DBMS의 역할과 데이터 저장소 선택

MySQL과 같은 DBMS는 데이터 저장/관계 매핑 외에도 SQL을 해석하고 처리하기 위해 내부적으로 많은 역할을 수행한다.

1. SQL 파싱 : 사용자가 SQL을 보내면 문법적으로 맞는지, 어떤 테이블을 읽어야 하는지 판단한다.
2. 옵티마이저 : 데이터를 처음부터 다 찾을지(Full Scan), 인덱스를 사용할지 등 가장 최적의 실행 방법을 고른다.
3. 저장 엔진 : 데이터를 실제로 디스크에 저장하고 읽어온다.
    1. 성능 최적화 : 자주 읽는 데이터는 디스크 대신 메모리에 캐싱하여 성능을 높인다.
    2. 장애 복구 : 데이터가 바뀔 때 바로 고치는 것이 아니라, 장애 시 복구할 수 있도록 로그를 남긴다.
4. 동시성 및 권한 제어 : 여러 사용자가 동시에 데이터를 건드릴 때 트랜잭션과 락으로 충돌을 조정하고 접근 권한을 검사한다.

| **데이터의 성격 및 특징** | **추천 저장소 / 기술** | **예시 및 이유** |
| --- | --- | --- |
| **정확한 관계와 트랜잭션이 중요한 데이터** | **RDBMS** (관계형 DB) | 주문, 결제, 회원 정보 등 (주문은 저장됐는데 결제가 누락되거나, 존재하지 않는 회원의 주문이 생기면 안 되기 때문) |
| **빠르게 읽고 일정 시간 후 사라져도 되는 데이터** | **인메모리 저장소** | 로그인 세션, 임시 인증 코드, 캐시 데이터 등 (대표적으로 **Redis**) |
| **본문이 길거나 복잡한 단어 검색이 필요한 데이터** | **검색 엔진** | 게시글 본문 검색, 상품 설명 키워드 검색 등 (일반 DB는 느리기 때문에 **ElasticSearch** 활용) |
| **로그, 이벤트 등 대량의 데이터가 쌓이는 상황** | **NoSQL** | 대용량 로그 수집, 정형화되지 않은 이벤트 데이터 적재 등 |

## 정리

- 데이터베이스는 구조화된 데이터의 모음이며, 데이터를 가치 있는 정보로 활용하기 위해 사용한다.
- 데이터의 영속성, 무결성, 동시성을 보장하는 것이 DB를 쓰는 본질적인 이유이다.
- RDB는 데이터를 테이블 단위로 저장하고, 기본키와 외래키를 이용해 관계를 맺는다.
- DBMS는 데이터베이스를 관리하는 소프트웨어이며, 우리는 SQL이라는 언어를 통해 DBMS와 소통한다.
