## 가져가야 할 핵심

1. LIKE연산자를 사용할 때 %가 앞에 붙으면 인덱스를 타지 못하고 풀스캔을 수행한다.
2. Full Text Index는 대량의 텍스트 데이터를 빠르게 검색하기 위한 도구이며, 핵심 자료구조는 역인덱스이다.
3. 한국어 텍스트 검색의 한계를 극복하기 위해 ngram파서나 형태소 분석기가 활용된다.
4. 서비스의 규모와 검색의 중요도에 따라 Full Text Index와 독립된 검색 엔진(ElasticSearch 등)의 명확한 트레이드 오프를 이해해야 한다.

## LIKE검색이 성능 저하를 일으키는 이유

검색 빈도가 낮거나 데이터가 적다면 `LIKE`를 쓸 수 있다. 하지만 검색이 서비스 핵심 기능이고 트래픽이 많다면 `LIKE` 조사는 심각한 성능 저하를 유발한다.

- **`LIKE '검색어%'` :** 검색어가 맨 앞에서 시작하므로 일반 B+Tree 인덱스를 정상적으로 활용한다. 인덱스는 글자의 앞부분을 기준으로 정렬되어 있기 때문에 검색 범위가 특정 구간으로 좁혀진다.
- **`LIKE '%검색어'` 또는 `LIKE '%검색어%'` :** 키워드가 문자열의 어디에 위치할지 알 수 없다. B+Tree 인덱스는 정렬 구조가 깨지며, DB는 인덱스를 포기하고 모든 행을 하나씩 대조하는 풀스캔을 수행한다. 데이터가 100만 건이면 100만 번의 비교 연산이 강제된다.

```jsx
[인덱스 내부의 정렬 예시]
- 검색어 정리
- 오늘의 검색어
- 좋은 디비 검색어 예시
```

위 세 데이터는 모두 '검색어'라는 키워드를 포함하지만, 시작 글자가 모두 다르기 때문에 인덱스 내에서 서로 다른 위치에 흩어져 있다. DB 입장에서는 탐색 범위를 좁히는 것이 불가능하다. 사용자가 검색창에 어떤 단어를 입력할지 모르기 때문에 실무에서는 `%검색어%` 형태를 쓸 수밖에 없으며, 이를 해결하기 위해 Full Text Index를 도입한다.

## Full Text Index의 핵심 원리 : 역인덱스

Full Text Index는 컴퓨터에 저장된 문서 또는 전문(Full Text) 데이터베이스를 빠르게 검색하는 기법이다.

일반 인덱스가 책의 맨 앞에 있는 '목차'(페이지를 알면 내용을 찾아감)라면, 역인덱스는 책의 맨 뒤에 있는 '찾아보기/색인'과 같다. 단어를 기준으로 그 단어가 어떤 페이지에 등장하는지 거꾸로 기록해 두는 개념이다.

### DB가 역인덱스를 구축하는 과정

#### 파싱 및 토큰화

- 인덱스로 지정된 텍스트 컬럼의 문장을 단어 단위로 쪼갠다.
- "Eating healthy foods like fruits" →"Eating", "healthy", "foods", "like", "fruits"로 분리한다.

#### 불용어 처리

- 검색 결과에 별다른 의미가 없거나 너무 자주 등장하는 단어들을 제거하여 인덱스의 크기를 줄이고 효율을 높인다.
- *영어:* 'a', 'is', 'the' 등을 제거한다.
- *한국어:* '은', '는', '이', '가' 등의 조사를 제거한다. MySQL에는 기본 불용어 목록이 내장되어 있으며 개발자가 커스터마이징할 수 있다.

#### 역인덱스 구축

정제된 단어들을 키로 삼고, 해당 단어가 어떤 문서에 위치하는지 지도로 매핑하여 일종의 표 형태로 만든다.

| **단어 (Token)** | **문서 ID (Row ID)** |
| --- | --- |
| healthy | 1, 3, 4, 5, 7, 8, 9 |
| computer | 2 |
| exercise | 5, 8 |

사용자가 "healthy"를 검색하면 DB는 테이블 전체를 뒤지는 대신, 역인덱스 표에서 "healthy"를 찾아내어 이에 매핑된 `1, 3, 4, 5, 7, 8, 9`번 Row ID 목록만 즉시 반환한다.

## Full Text Index의 사용 방법

```jsx
-- 기본 검색 (자연어 검색 모드)
SELECT * FROM 테이블명 WHERE MATCH(컬럼명) AGAINST('검색하고 싶은 단어');

-- 정밀 검색 (불리언 검색 모드)
SELECT * FROM 테이블명 WHERE MATCH(컬럼명) AGAINST('검색하고 싶은 단어' IN BOOLEAN MODE);
```

- **자연어 검색 :** 구글 검색처럼 사용자가 찾고 싶은 단어를 입력하면, MySQL이 자체 알고리즘으로 관련성이 높다고 판단한 행을 연관도 순서대로 정렬하여 보여준다.
- **불리언 검색 :** 개발자가 훨씬 정교한 검색 규칙을 쿼리에 적용한다. 특정 단어를 반드시 포함(`+`)하거나 특정 단어는 검색 결과에서 제외()하는 식의 제어가 가능하다. (예: 'health'는 포함하되 'nutrition'은 제외)

## ngram파서

#### 한국어 도메인에서의 문제점

한국어는 조사와 어미가 발달한 교착어이기 때문에 단어의 형태가 계속 변한다.

- ‘데이터베이스’라는 핵심 단어가 문장 속에서는 ‘데이터베이스는’, ‘데이터베이스를’, ‘데이터베이스의’형태로 결합한다.
- 일반적인 공백 기준 파서는 이를 모두 서로 다른 단어로 인식하여 ‘데이터베이스’로 검색했을 때 정상적으로 필터링하지 못한다.

#### ngram파서의 동작 방식

ngram은 텍스트를 공백이나 의미 단위가 아닌, 의도적으로 지정한 n개의 글자 단위로 쪼개어 인덱싱하는 방식이다. 연속된 글자 수에 따라 나누며, 실무에서는 2글자 단위인 2-gram이 자주 쓰인다.

- **'데이터베이스'를 2-gram으로 쪼갤 시:** '데이', '이터', '터베', '베이', '이스'로 토큰이 분리된다.
- 문장 속에 '데이터베이스는'이 있다면 '스는'이라는 토큰이 추가로 생성된다.
- **효과:** 사용자가 '데이터베이스'로 검색하면 공통된 토큰들('데이', '이터', '터베' 등)의 매칭률을 기반으로 데이터를 찾아낸다. 조사 결합이나 미세한 오타가 발생해도 어느 정도 검색 성공률을 보장한다.

## MySQL Full Text Index의 한계와 검색 엔진 등장

MySQL의 내장 전문 검색은 훌륭한 기능이지만, 실무 비즈니스가 커지면 한계가 명확하여 최종적으로는 ElasticSearch나 OpenSearch 같은 전문 검색 엔진 솔루션을 별도로 구축하게 된다.

#### Full Text Index의 한계점

- **정교한 언어 처리(형태소 분석)의 부족:** ngram은 단순 글자 수 자르기 방식이므로 정밀도가 떨어진다. 현대적인 검색 엔진은 '먹었다', '먹는다'의 어근인 '먹다'를 추출하거나, 'computer'와 'computing'을 'comput'로 묶어 분석하는 고급 어간 추출 기능이 강력하지만 MySQL은 이 부분이 약하다.
- **동의어/유의어 처리 불가:** '노트북'과 '랩탑'을 같은 의미의 카테고리로 묶어 처리하는 동의어 사전 세팅 기능이 부실하다.
- **랭킹 모델 제어의 한계:** 특정 상품이나 광고 스폰서의 게시글에 의도적으로 검색 가중치를 높게 부여하는 등 결과의 우선순위(정확도 점수 계산)를 비즈니스 로직에 맞춰 복잡하게 커스텀하기 어렵다.
- **DB 인프라의 확장성 한계:** 대규모 검색 트래픽이 몰리면 검색 연산 때문에 RDBMS 본연의 업무(주문, 결제, 트랜잭션) 처리 역량까지 함께 마비된다. 검색 기능이 메인 데이터베이스에 직접적인 부하를 주게 된다.

#### 전용 검색 엔진(ElasticSearch)의 강점

- 전문적인 형태소 분석 및 유연한 랭킹(정확도) 스코어 제어 기능을 제공한다.
- '브랜드별 상품 수', '가격대별 상품 건수' 같은 복잡한 다면 검색 및 통계 집계 연산 성능이 압도적이다.
- 처음부터 분산 아키텍처로 설계되어 트래픽과 데이터양이 늘어나면 서버를 옆으로 늘리는 수평적 확장(Scale-out)이 매우 용이하다.

## 기술 선택의 기준(Full Text Index vs 검색 엔진)

#### MySQL Full Text Index를 선택하는 경우

- 검색 기능이 서비스의 핵심이 아닌 **부가적인 기능**일 때 사용한다.
- 별도의 인프라 서버를 띄우고 운영할 만한 예산이나 전문 개발 인력의 여유가 부족할 때 적합하다.
- 단순 키워드 매칭만으로 충분하며, 복잡한 형태소 분석이나 정교한 가중치 정렬 규칙이 필요 없을 때 도입한다.

#### 전문 검색 엔진(ElasticSearch 등)을 도입하는 경우

- 이커머스(쿠팡 등), 채용 사이트, 포털 등 **검색 품질이 서비스의 매출과 직결되는 핵심 기능**일 때 필수적이다. (예: 특정 시간에 선착순 핫딜을 진행할 때 사용자가 검색창에 몰리는 경우, 전용 엔진이 방어해 주어야 한다.)
- 사용자에게 오타 교정, 유의어 추천 등 정확도 높고 관련성 깊은 고품질의 결과를 제공해야 할 때 사용한다.
- 대용량 대규모 트래픽이 집중될 것으로 예상되는 아키텍처 구조일 때 선택한다.
- 검색 결과 화면에 다채로운 필터링 가이드와 통계 분석 데이터를 동시에 가공하여 뿌려주어야 할 때 도입한다.

## 정리

- 문자열 검색 시 `LIKE '%검색어%'` 패턴은 **인덱스를 타지 못하고 풀스캔을 유발하는 안티 패턴**이다.
- **Full Text Index**는 단어를 기준으로 데이터의 위치를 기록하는 **역인덱스** 구조를 사용하여 문장 내 중간 단어도 빠르게 찾아낸다.
- 조사 변형이 심한 한국어 도메인에서는 글자 수 단위로 쪼개어 매칭하는 **ngram 파서**를 연동하여 검색 성공률을 방어한다.
- 내장 전문 검색은 인프라 비용이 없다는 장점이 있으나 정교한 형태소 분석, 동의어 처리, 랭킹 제어가 어렵기 때문에, 검색이 메인 비즈니스인 대형 서비스에서는 **ElasticSearch 같은 전용 검색 엔진으로 고도화**하는 아키텍처가 표준이다.
