## JOIN : 분리된 테이블을 다시 의미 있게 연결하는 도구

정규화를 거치며 성격에 따라 잘게 쪼개진 테이블들을 공통된 키를 기반으로 결합하여, 하나의 큰 테이블처럼 결과를 조회하는 수평 결합 기술이다.

학생 정보 명단(학번, 이름, 학과)과 도서관의 대출 기록 명단(도서명, 대출자, 학번, 대출일)이 따로 있다고 가정해보자

→ “컴공과 학생들의 도서 대출 목록을 알려주세요”라는 요청이 오면 다음과 같이 움직인다.

1. 학생 명단에서 ‘컴공과’학생들을 필터링한다.
2. 그 학생들의 ‘학번’을 키로 삼아 대출 기록 명단으로 가서 일치하는 데이터를 대조해 빌린 책을 찾아내다. 

→ 이 과정을 자동화해 주는 것이 JOIN이다.

#### 쇼핑몰 데이터를 통한 JOIN쿼리 분석

`user`(회원), `product`(상품), `purchase`(구매 이력) 테이블이 존재하며, `purchase` 테이블이 `user_id`와 `product_id`를 통해 둘을 연결한다.

“1번 유저가 구매한 상품들의 이름과 가격, 구매일을 알려줘”

```jsx
SELECT p.name AS product_name, p.price AS product_price, pr.created_at AS purchase_date
FROM purchase pr
JOIN product p ON pr.product_id = p.id
WHERE pr.user_id = 1;
```

1. `FROM purchase pr` : “일단 purchase 테이블에서 시작할게. 그리고 이제부터 purchase를 줄여서 pr이라고 부를거야”
2. `JOIN product p` : “product 테이블 연결할거야 그리고 이제 줄여서 p라고 부를게”
3. `ON pr.product_id = p.id` : “연결을 할건데 조건은 pr테이블의 product_id와 p 테이블의 id가 같은것들이야”
    1. 이게 제가 아까 이야기했던 열쇠. 구매기록에 있는 상품 ID와 상품 정보에 있는 상품 ID가 일치하는것들만 짝지어준다
4. `WHERE pr.user_id = 1;` : 그렇게 연결된 결과들 중에서 user_id가 1인것만 필터링해줘 (특정 유저를 지칭)
5. `SELECT p.name AS product_name, p.price AS product_price, pr.created_at AS purchase_date`  : 최종적으로 테이블에서 조회해올 값들을 지칭.

#### INNER JOIN

- 종류를 명시하지 않고 그냥 `JOIN`이라고만 쓰면 모두 `INNER JOIN`을 의미한다.
- 두 테이블에 모두 연결고리(조건)가 존재하는 데이터만 결과에 포함시킨다.
- 구매 기록이 전혀 없는 회원이나, 등록만 해두고 한 번도 팔린 적 없는 상품은 결과 화면에 절대 나타나지 않는다.

#### LEFT JOIN

- 왼쪽 테이블(`FROM`절에 먼저 작성한 테이블)의 데이터는 **조건 성립 여부와 상관없이 무조건 전체를 출력**하고, 오른쪽 테이블은 연결고리가 매칭되는 데이터만 옆에 붙여준다.
- 오른쪽 테이블에 매칭되는 짝이 없다면 그 자리는 `NULL`로 채워진다.
- “모든 회원 목록과 그 회원의 구매 이력을 보여줘. 구매 이력이 없는 회원도 목록에 누락 시키면 안돼” → `user`테이블을 왼쪽에 두고 `purchas`테이블을 `LEFT JOIN`하면, 구매 이력이 없는 유저도 구매 관련 컬럼들이 `NULL`로 표시되면서 명단에 정상 출력된다.

#### RRIGHT JOIN

- LEFT JOIN의 반대 개념이다. 오른쪽 테이블의 데이터를 무조건 모두 보여주고, 왼쪽 테이블은 연결고리가 있는 데이터만 매칭하여 붙여준다.
- “모든 상품 목록과 그 상품이 팔린 기록을 보여줘. 한 번도 팔리지 않은 상품도 상품 목록에 나와야해”
- →`product`테이블을 오른쪽에 두고 `RIGHT JOIN`을 걸면 된다.

## 잘못 사용된 JOIN과 성능 위험

JOIN은 DB성능 저하를 일으키는 가장 주된 원인 중 하나이므로, 무작정 쿼리를 쪼개기보다 명확한 근거(실행 계획)를 가지고 접근해야 한다.

- **ON 절 컬럼에 인덱스가 없는 경우** : `ON pr.product_id = p.id`로 짝을 맞출 때 해당 컬럼들에 인덱스가 없으면, DB는 두 테이블의 모든 데이터를 한 줄씩 일일이 대조하는 Full Table Scan(풀스캔)을 수행하여 서버가 마비될 수 있다. 쿼리 작성 후 반드시 `EXPLAIN`(실행 계획)을 확인해야 한다.
- **과도하게 많은 테이블 결합:** 3~4개의 테이블 JOIN은 실무에서 매우 흔하고 자연스럽다. 하지만 한 쿼리에 10개가 넘는 테이블을 JOIN 하고 여기에 서브쿼리까지 뒤섞여 있다면 설계 자체가 잘못되었을 확률이 높다. 쿼리가 과하게 복잡해지면 DBMS의 옵티마이저가 최적의 실행 계획을 세우지 못해 **성능이 극도로 저하된다.**

## VIEW : 복잡한 조회를 포장하는 가상 테이블

JOIN이 테이블을 결합하는 기술이라면, VIEW는 복잡한 SQL쿼리 자체를 데이터베이스에 하나의 ‘이름’으로 저장해 두고 테이블처럼 꺼내 쓰는 가상 테이블 기술이다. 실제 물리적인 데이터를 저장하고 있지는 않다.

→ 데이터 조회의 재사용성을 높이고, 복잡한 내부 구조를 숨겨 유지보수를 용이하게 만든다.

매번 유저의 이메일과 구매한 상품 목록을 보려면 아래와 같이 3개의 테이블을 엮는 복잡한 JOIN쿼리를 작성해야한다.

```jsx
-- 이 복잡한 쿼리 자체를 'purchased_product'라는 이름의 VIEW로 등록
CREATE VIEW purchased_product AS
SELECT u.email AS user_email, p.name AS product_name, p.price AS product_price
FROM user u
JOIN purchase pr ON u.id = pr.user_id
JOIN product p ON pr.product_id = p.id;
```

이렇게 한 번 등록해 두면, 여러 화면이나 다른 기능에서 필요할 때 마치 실제 테이블이 존재하는 것처럼 아주 단순하게 조회가 가능하다.

```jsx
SELECT * FROM purchased_product WHERE user_email = 'test@example.com';
```

복잡한 JOIN로직을 VIEW뒤로 깔끔하게 캡슐화하여 가옥성과 재사용성을 극대화한다.

#### VIEW의 성능적 한계와 대안

일반적인 VIEW는 쿼리 성능을 직접적으로 향상시켜 주지 않는다.

`SELECT * FROM purchased_product`를 실행하면 DB 내부적으로는 숨겨져 있던 복잡한 3중 JOIN 쿼리가 그대로 다시 처음부터 실행된다. 

따라서 단순히 SQL을 편하게 쓰는 이점만 있을 뿐이다.

Materialized View (구체화 뷰)

VIEW가 쿼리 문장만 저장하는 '가상 테이블'이라면, **Materialized View는 복잡한 JOIN이나 집계 연산의 결과를 실제 물리 디스크 공간에 테이블처럼 저장해 두는 개념이다.**

- 이 뷰를 조회하면 매번 수백만 건을 다시 계산(JOIN) 하는 것이 아니라, 이미 계산되어 저장된 결과 테이블에서 값만 빼오므로 **조회 속도가 빨라진다.**
- 실제 물리 데이터를 중복 저장하므로 추가적인 디스크 공간이 필요하며, 원본 테이블의 데이터가 변경되었을 때 구체화 뷰도 동기화해주어야 하는 비용이 듭니다.
- **데이터 갱신 빈도가 매우 적은 통계성 데이터**에 적합하다. (예: 매일 하루에 딱 한 번 전날의 매출 통계를 집계하는 대시보드 화면. 볼 때마다 수백만 건을 JOIN 하고 `SUM`하면 대형 장애가 나기 때문에 Materialized View가 유리하다.)

→ 하지만 실제 아키텍처 상에서는 MV를 복잡하게 세팅하기보다, 백엔드단에서 레디스같은 인메모리 저장소를 활용해 데이터를 캐싱하여 처리하는 경우가 훨씬 많다.

## UNION : 결과를 아래로 이어 붙이는 수직 결합

- JOIN : 테이블을 옆으로 붙이는 수평 결합
- UNION : 테이블을 밑으로 쌓아 올리는 수직 결합(두 개 이상 SELECT문의 결과를 하나의 깔끔한 리스트 결과물로 합쳐서 보여주는 연산자)

**사례 1: 기간·연도별 데이터 분할 테이블 조회**

서비스가 오래되어 데이터가 수억 건 쌓이면 성능을 위해 테이블을 연도별로 쪼개어 관리하는 경우가 있다.(`posts_2023`, `posts_2024`). 이때 사용자가 "2023년과 2024년 전체 게시글을 통합해서 보여달라"고 요구하면, 어플리케이션(Java)단에서 각자 조회해 합치는 것보다 DB 레벨에서 `UNION`으로 묶어 한 번에 땡겨오는 것이 효율적이다.

```jsx
SELECT title, content, created_at FROM posts_2023
UNION
SELECT title, content, created_at FROM posts_2024;
```

**사례 2: 성격이 유사한 이종데이터 통합 검색**

쇼핑몰에서 일반 상품은 `product` 테이블에, 중고 상품은 `used_product` 테이블에 분리 저장되어 있다고 하자. 두 테이블의 스키마 구조가 거의 동일할 때, 사용자가 통합 검색창에 "노트북"을 검색하면 두 테이블의 결과를 수직으로 결합하여 하나의 화면에 섞어서 뿌려줄 수 있다.

#### UNION 사용 시 주의사항

위아래로 엮이는 모든 SELECT문의 컬럼 개수, 컬럼의 배치 순서, 그리고 각 컬럼의 데이터 타입이 완벽하게 일치해야 한다.

→첫 번째 쿼리는 3개의 컬럼을 반환하는데 두 번째 쿼리는 2개의 컬럼만 반환하거나 데이터 타입이 다르면, DB 입장에서는 기준점이 깨지기 때문에 하나의 표로 결합하지 못하고 에러를 뱉는다.

## 정리

- JOIN은 쪼개진 데이터를 수평 결합하는 기술이며, 공통 분모만 가져오는 INNER, 왼쪽을 보존하는 LEFT, 오른쪽을 보존하는 RIGHT조인이 있다. ON절에 인덱스가 없으면 대형 성능 장애가 발생한다.
- VIEW는 복잡한 쿼리를 감싸 안는 가상의 별명 테이블이며, 실질적인 조회 성능을 높이려면 물리 디스크에 결과를 저장하는 Materialized View를 쓰거나, 레디스 캐싱을 사용해야 한다.
- UNION은 서로 다른 조회 결과를 수직 결합하는 연산자이며, 합쳐지는 쿼리들 간의 컬럼 개수와 타입 순서가 반드시 일치해야 가능하다.
