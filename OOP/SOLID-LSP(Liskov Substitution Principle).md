## OCP의 한계와 LSP 등장

- OCP(개방-폐쇄 원칙)
    - 기존 코드를 건드리지 않고 새로운 기능을 덧붙일 수 있는 확장성을 의미한다.
    - `if-else`로 복잡해지는 코드를 인터페이스와 다형성을 이용해서 해결할 수 있다.
- 문제 발생(무분별한 상속의 위험
    - OCP를 지키기 위해 인터페이스를 만들고, 이를 구현하는 자식 클래스들을 추가하다 보면 한도 끝도 없이 계속 추가하는 상황이 발생한다.
- LSP(리스코프 치환 원칙)의 목적
    - 상속과 다형성이 안전하게 작동하도록 지키는 원칙이다.
    - 무분별하게 상속하는 것을 경계하고, 다형성의 의도를 올바르게 유지하기 위해 사용한다.

## 가져가야 할 핵심

1. 상속이나 구현을 했다고 해서 다형성이 완성되는 것은 아니다. 자식이 부모의 계약을 준수해야만함을 이해해야 한다.
2. 코드에 `instanceof`로 타입을 검사하거나, 특정 자식 **클래스 전용 if분기문이** 등장하면 “LSP가 깨진 것이 아닌가?”를 반드시 의심해봐야 한다.
3. LSP를 지킬 수 없는 잘못된 상속 관계를 만났을 때, 억지로 상속을 유지하지 않고 **컴포지션 구조로 리팩토링하는 판단력**이 필요하다.

## LSP(리스코프 치환 원칙)

**자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다는 객체지향 설계 원칙**

### "언제나 대체할 수 있어야 한다"의 진짜 의미

- 다형성을 사용할 때 우리는 보통 `Parent p = new Child();`와 같이 변수의 타입은 부모로 선언하고 실제 인스턴스는 자식 객체를 할당한다.
- 객체를 사용하는 클라이언트 코드는 저 객체가 진짜 부모인지, 아니면 수많은 자식 중 누구인지 신경 쓰지 않고 부모 타입에 정의된 메소드를 호출한다.
- **즉, LSP가 성립한다는 것은 메소드를 호출한 곳(클라이언트)에서 내가 지금 호출한 대상이 자식 객체라는 사실을 전혀 몰라야 한다**는 뜻이다.

### 실생활 비유 : 카페의 아메리카노 계약

- **정상적인 상황 (기대와 계약)**
    - 카페에서 바리스타(부모 타입)에게 "아아 하나 주세요"라고 요청한다.
    - 손님이 기대하는 규칙(계약)은 '원두를 갈고, 에스프레소를 내리고, 얼음과 물을 섞어준다'이다.
- **LSP가 깨진 상황 (비정상적인 자식의 등장)**
    - 새로 입사한 알바(자식 타입)가 카운터를 보게 되어 아아를 주문했다.
    - 그런데 이 알바가 "저는 아아를 제 방식대로 재정의(Override) 했습니다. 물 대신 우유를 넣고 따뜻하게 데워드립니다"라며 라떼를 준다.
    - 알바의 주장: *"음료를 만든다는 행위 자체는 수행했으니 문법적 에러는 없습니다."*
    - **결과:** 문법은 맞을지 몰라도, 부모 타입으로 메소드를 호출했던 손님의 기대(아아라는 계약)는 산산조각이 났다.
- **소프트웨어에서의 문제점 (분기문의 등장)**
    - 만약 손님이 "아, 오늘 카운터에 있는 사람 새로운 알바에요? 그럼 아아 말고 생수나 한 병 주세요"라고 한다면?
    - 이것 자체가 **코드에 분기문(`if`)을 넣게 되는 것**이고, 이 순간 다형성은 무너진다.
    - 특정 자식 클래스에서만 예외 처리나 분기 로직이 필요해진다면, 부모와 자식 간의 다형성이 성립하기 어렵다고 판단해야 한다.

### 행위 규칙의 준수 (입력/출력/예외 계약)

자식 클래스가 부모 클래스의 메소드를 오버라이딩할 수는 있지만, 부모 클래스를 호출했던 곳에서 약속했던 **행위의 규칙을 훼손해서는 안 된다.** 상속은 부모의 역할을 물려받아 충실히 수행하거나 더 구체화하는 도구여야 한다.

- **입력 조건:** 파라미터로 1~10 사이의 숫자만 받기로 약속함.
- **출력 조건:** 항상 양수를 반환하기로 약속함.
- **예외 조건:** DB 연결이 끊기면 예외를 던지기로 약속함.
- **LSP 위반:** 자식 클래스가 이 계약을 마음대로 파기하여 갑자기 음수를 반환하거나, 파라미터 허용 값을 5~10으로 좁혀버리는 행위 → **신뢰가 깨짐.**

## LSP를 지켜야 하는 이유

OCP를 달성하기 위해 `Service`클래스가 인터페이스나 추상 클래스에 의존하도록 만들고, DI를 통해 외부에서 구현체를 주입받도록 설계할 수 있다. 이때 **주입받은 구현체(자식 클래스)가 LSP를 위반하여 엉뚱한 짓을 한다면 OCP가 주는 이점이 완전히 사라진다.**

### 결제 시스템으로 보는 OCP 붕괴 예시

- 주문 처리하는 `OrderService`가 있고, 결제를 담당하는 `PaymentClient` 인터페이스가 존재한다.
- 인터페이스의 계약: `pay()` 메소드는 결제 성공 시 **결제 영수증 ID(String)를 반환**하고, 통신 실패 시 **Exception을 던진다.**
- 기존의 `KakaoPaymentClient`는 이 계약을 잘 따르고 있어서 `OrderService`가 안정적으로 동작했다.
- 어느 날 새로운 결제 수단인 `NaverPaymentClient`(자식 구현체)를 추가했다. 그런데 개발자가 "예외 던지면 서버 로그가 지저분해지니까 조용히 null을 남기자"라며 **예외 대신 빈 문자열이나 `null`을 반환하도록 오버라이딩**했다.

`OrderService`는 인터페이스의 계약을 믿고 코드를 작성했다.

```java
String receiptId = paymentClient.pay(); // 인터페이스의 메소드 계약을 믿고 수행
```

- `NaverPaymentClient`가 주입되면 `receiptId`에 `null`이 들어온다.
- `OrderService`는 성공한 줄 알고 이 값을 DB에 저장하려다가 `NullPointerException`이 발생하거나 DB 제약조건 에러가 터진다.
- 결국 이를 수습하기 위해 `OrderService`에 다음과 같은 **지저분한 분기 코드**가 추가된다.

```java
String receiptId = paymentClient.pay();

// LSP가 깨져서 어쩔 수 없이 추가되는 코드 (OCP 완전 붕괴)
if (paymentClient instanceof NaverPayClient) {
    if (receiptId == null || receiptId.isEmpty()) {
        // ... 네이버페이 전용 예외 처리 로직 ...
    }
}
```

자식 클래스가 무슨 짓을 할지 모르니 호출하는 쪽에서 검사하고 처리하는 로직이 추가되었고, 새로운 결제 수단이 늘어날 때마다 기존 코드를 고쳐야 하므로 **OCP가 완전히 붕괴**된다.

### LSP 위반 예시

```java
public class Car {
    public void drive() {
        System.out.println("차가 도로를 달립니다.");
    }
}

// 플라잉카가 Car를 상속 (문법적으로는 컴파일 에러 없음)
public class FlyingCar extends Car {
    @Override
    public void drive() {
        // 부모의 계약(도로를 달린다)을 파기하고 예외를 던짐
        throw new UnsupportedOperationException("플라잉카는 도로를 달리지 않고 하늘을 납니다.");
    }
}
```

클라이언트 코드에서의 문제 :

```java
// 시스템에 있는 모든 자동차를 순회하면서 전진시키는 로직
List<Car> cars = Arrays.asList(new Car(), new FlyingCar());
for (Car car : cars) {
    car.drive(); // 루프를 돌다가 FlyingCar를 만나는 순간 예외가 발생하며 시스템이 뻗어버림
}
```

*“Car 타입이니까 당연히 달릴 줄 알고 drive()를 불렀는데 왜 터지지?”* → 자식이 부모의 기능을 마음대로 바꾸어 규칙을 위반했기 때문이다.

### LSP 준수 예시

```java
public class Car {
    public void drive() {
        System.out.println("차가 도로를 달립니다.");
    }
}

public class ElectricCar extends Car {
    @Override
    public void drive() {
        // 부모의 의도(도로 주행)를 유지하면서 내부 구현만 구체화함
        System.out.println("전기차가 조용히 도로를 달립니다.");
    }
}
```

세부 구현은 다르게 정의할 수 있어도, 반환하는 약속된 값과 행위의 규칙을 지켜주었기 때문에 안전하게 대체 가능하다.

## 해결책 : 상속 대신 조합(Composition)사용하기

LSP를 지킬 수 없는 상속 관계라면, **그 상속 자체가 잘못된 설계**이다. 상속을 유지하기보다 조합(Composition)이나 인터페이스로 구조를 바꾸는 것이 좋다.

- **잘못된 상속의 원인:** 기획팀에서 플라잉카를 계속 만들어달라고 한다. 플라잉카는 도로를 달릴 수 없고(`drive()` 불가), 하늘을 날아야 한다(`fly()`).
- 이때 억지로 `Car` 클래스를 상속받으려다 보니 쓸데없는 `drive()`를 물려받게 되고, 처리할 방법이 없으니 예외를 던지는 이상한 코드가 만들어진 것이다.
- `Car`가 '도로를 달릴 수 있는 타입'이라는 계약을 가졌다면, 도로 주행을 못 하는 `FlyingCar`를 하위 타입으로 두는 상속 관계 자체가 모순이다.
- **주의:** 부모 클래스에 있는 변수나 메소드 몇 줄을 **재사용하겠다고 상속을 쓰면 안 된다.** 상속은 부모가 약속한 행동을 자식이 그대로 지킬 수 있을 때만 써야 한다.

### 조합(Composition)을 이용한 리팩토링 코드

상속 관계를 끊어내고, 필요한 기능을 클래스 내부의 멤버 변수(부품)로 포함시켜서 사용한다.

```java
// 1. 필요한 기능을 담당하는 독립된 부품 객체를 만든다.
public class Battery {
    public void powerOn() {
        System.out.println("배터리 전원을 켭니다.");
    }
}

// 2. 상속을 제거하고, 내부에 배터리라는 부품을 조합(인스턴스화)시킨다.
public class FlyingCar {
    private Battery battery = new Battery(); // 조합(Composition)

    // 부모의 무의미한 drive() 대신 본인만의 명확한 메소드 정의
    public void fly() {
        battery.powerOn();
        System.out.println("플라잉카가 하늘로 이륙합니다!");
    }
}
```

Q. 리스코프 치환에 대해서 설명해주세요

LSP는 부모타입만 알고 호출하더라도 실제 주입된 자식 객체가 부모의 행위적 계약을 깨지 않고 예측 가능하게 동작해야한다는 원칙이다.

## 정리

- LSP는 자식 객체가 부모 객체의 의도와 계약을 그대로 준수하면서 호출하는 곳에서 아무런 의심 없이 부모타입으로 자식을 다룰 수 있게 해야한다는 원칙이다.
- 호출하는 곳 코드에서 조건문이나 instancOf같은 타입 체크가 생기면 치환 가능성이 깨질 수 있다.→ OCP까지 흔들린다.
- 만약 부모의 메소드 계약을 지킬 수 없는 상황이면 설계 자체가 잘못된건지 확인하고 상속보다 조합을 쓰는 것을 고려해볼 수 있다.
