## AI 시대, 왜 여전히 OOP를 알아야 하는가?

OOP는 코드를 작성하는 단순한 방법론을 넘어, AI의 한계를 통제하고 대규모 시스템을 유지보수할 수 있게 돕는 개발자의 안전장치이다.

- AI와 컨텍스트 윈도우의 한계
    - AI는 독립된 코드 조각이나 기능 구현은 인간보다 잘한다.
    - 하지만 프로젝트 규모가 커질수록 AI는 전체 구조를 기억하지 못하고, 앞서 제시한 규칙을 잊거나 엉뚱한 코드를 짜는 컨텍스트 윈도우의 한계(한 번에 처리할 수 있는 기억 용량의 한계)를 드러낸다.
- 개발자의 영역 : 아키텍처 설계와 통제
    - 잘 설계된 OOP구조는 거대한 시스템을 인간이 통제할 수 있는 크기의 모듈과 객체 단위로 쪼개준다.
    - 이를 통해 프롬포트를 단순화할 수 있고 AI가 할루시네이션이나 오작동을 일으키지 않도록 컨섹스트의 범위를 제한할 수 있다.
- 스파게티 코드 방지
    - 개발자에게 객체지향적 기본기가 없으면 AI에게 “그냥 구현해줘”라고 요청하게 된다. AI는 당장 작동은 하지만 수정하기가 불가능한 스파게티 코드를 만들어낸다.
    - AI덕분에 코드 생성 속도는 빨라졌지만 그 코드가 무너지지 않도록 시스템을 유지보수하고 관리하려면 개발자 머릿속에 OOP적 사고가 있어야한다.

## 객체지향  정의

“역할과 책임, 그리고 협력”

객체는 자신이 맡은 역할과 책임을 수행하는 독립적인 단위이다.

객체를 설계할 때 “어떤 데이터를 저장해야 하지?” = “이 객체가 무엇을 알고 있는가(데이터)”보다 “무엇을 보장하고 책임져야 하는가(행동)”이 더 중요하다.

쇼핑몰의 ‘주문(Order) 객체” 예시

- 나쁜 설계 : 주문번호, 총 결제 금액 등을 담아두고 외부 서비스 클래스에서 이 데이터를 꺼내어 검증하고 계산하는 구조
- 좋은 설계 : 주문 객체 스스로 비즈니스 규칙을 책임진다.
    - 주문 금액이 0원 이하인지 검증하는 책임
    - 주문 상태를 변경하는 책임
    - 취소가 가능한 상태인지 판단하는 책임

→ 외부에선 주문 객체에 “주문을 취소해라”고 요청만 하면 된다. 내부의 복잡한 검증 로직은 알 필요가 없다.

## OOP의 핵심 메커니즘과 장점

- 객체간 상호작용과 메세지 : 프로그램의 흐름은 절차적인 순서가 아니라 객체들이 서로 메서드를 호출하며 주고받는 메시지와 협력으로 만들어진다.
- 캡슐화 : 외부 클래스가 객체의 내부 구현을 직접 다루지 않고 제공된 인터페이스(메서드)를 통해서만 상호작용하는 것이다. 내부 코드가 바뀌어도 호출하는 외부 코드가 영향을 받지 않으므로 변경 비용이 줄어든다.

→ 자동차 엔진을 가솔린에서 전기 엔진으로 바꿔도 운전자는 기존 인터페이스인 엑셀만 밟으면 차가 앞으로 나가는 것과 같다.

- 모듈화 : 프로그램을 하나의 거대한 덩어리가 아니라 역할과 책임이 분명한 작은 단위(결제, 배송, 재고 등)로 나누는 방식이다. 결제 로직을 수정할 때 배송이나 재고 코드를 건드릴 필요가 없어야 좋은 설계이다.

## 스프링 프레임워크와 DI(의존성 주입)

DI(Dependency Injection)란? 필요한 객체를 개발자가 코드 내에서 직접 new로 생성하지 않고 프레임 워크가 대신 만들어서 넣어주는 방식이다.

왜 쓰냐 ? 코드 내에 new 카카오페이결제()같은걸 사용하면 나중에 네이버페이로 바꿀 때 기존 코드를 뜯어 고쳐야 한다.(강한 결합)

인터페이스 기반 설계 : “나는 결제 기능 인터페이스를 가진 객체면 뭐든 상권없으니 주문 처리만 하겠다”고 설계하고 프레임워크가 주입해주면 구현은 숨기고 역할만 연결되어 결합도가 낮아지고 확장성이 극대화된다. 스프링은 이런 객체지향을 돕는다.

**OOP가 적용되지 않은 예시**

```jsx
public class Car {
    private int speed = 0;

    // 엔진 시동 책임
    public void start() {
        System.out.println("엔진 시동");
    }

    // 속도 조절 책임
    public void accelerate() {
        speed += 10;
        System.out.println("속도 증가: " + speed);
    }

    // 제동 책임
    public void stop() {
        speed = 0;
        System.out.println("브레이크 작동");
    }
}
```

문제점 : 단순해 보이지만 여기에 ‘시동시 배터리 체크’, ‘가속 시 연비 계산’ 등의 로직이 추가되면 이 클래스는 수천 줄 짜리가 된다. 강한 결합으로 인해 하나의 변경이 예상치 못한 부작용을 일으킨다.

**OOP를 적용한 예시(책임 분리와 위임)**

```jsx
// 엔진은 오직 시동이라는 책임만 갖는다
public class Engine {
    public void start() {
        System.out.println("엔진 시동");
    }
}

// 브레이크는 오직 제동이라는 책임만 갖는다
public class Brake {
    public void apply() {
        System.out.println("브레이크 작동");
    }
}

// Car는 흐름만 제어하고 구체적인 일은 각 객체에 위임(협력)한다
public class Car {
    private final Engine engine;
    private final Brake brake;

    // 조립 (의존성 주입)
    public Car(Engine engine, Brake brake) {
        this.engine = engine;
        this.brake = brake;
    }

    public void start() {
        engine.start(); // 엔진 객체에게 메시지를 보냄
    }

    public void stop() {
        brake.apply();  // 브레이크 객체에게 메시지를 보냄
    }
}
```

브레이크 패드 마찰 방식을 바꾸고 싶다면 Brake클래스만 수정하면 된다. Car나 Engine은 변경의 영향을 받지 않는다. 또한 각 객체가 자신의 일만 하기 때문에 테스트 코드를 작성하기도 쉽다.

## 객체지향 vs 절차지향

|  | 절차지향 프로그래밍 | 객체지향 프로그래밍(OOP) |
| --- | --- | --- |
| 핵심 집중 대상 | “어떤 순서로 함수를 실행하고 로직이 흘러가는가?” | 데이터와 데이터를 처리하는 함수를 ‘객체’라는 캡슐 안에 함께 묶음 |
| 규모에 따른 장단점 | 규모가 작을 때는 시간 순서와 같아 직관적이고 편하지만 대규모 시스템에서는 데이터 구조 하나 변경 시 수백 개의 함수가 깨짐 | 초기 설계 비용이 들 수 있으나 시스템이 거대해져도 변경의 영향이 객체 내부로만 제한되어 유지보수와 확장에 유리하다.  |
| 구조적 특징 | 데이터는 중앙에 있고 여러 함수가 이 데이터를 가져다 사용 | 데이터와 데이터를 처리하는 함수를 ‘객체’라는 캡슐 안에 함께 묶음 |

## 정리

- 클래스는 데이터를 저장하는 보관소에서 벗어나서 객체를 ‘역할과 책임을 다하고 다른 객체와 협력하는 존재’이다.
- 이렇게 책임을 분리하는 이유는 코드 수정할 때 편리하게 확장성 좋게 하기 위해서이다. 새로운 기능을 붙일 때 기존 구조를 망가뜨리지 않고 유연한 시스템을 만들기 위해서이다.
