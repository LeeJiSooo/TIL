## SOLID원칙의 흐름과 ISP의 등장

- SRP(단일 책임 원칙) : 클래스의 책임을 하나로 좁힌다.(책임 = 변경해야 하는 이유)
- OCP(개방-폐쇄 원칙) : 변경에는 닫혀 있고 확장에는 열려 있게 만든다.
- LSP(리스코프 치환 원칙) : 다형성을 안전하게 보장하기 위해 자식이 부모의 계약을 지켜야 한다.

**인터페이스 설계 시 흔히 하는 실수(만능 인터페이스)**

- 설계하다 보면 “나중에 혹시 필요할지도 모르니까”, 또는 “비슷한 도메인이니까”라는 이유로 하나의 거대한 인터페이스에 수십 개의 메소드를 떄려 넣는 실수를 하곤한다.
- 이러한 ‘만능 인터페이스’는 처음엔 편할지 몰라도, 실제 필요한 곳에서만 골라 쓰려고 하면 불필요하고 불편하다.
- 결과적으로 호출하는 입장(클라이언트)에서 사용하지 않는 기능까지 강제로 떠안아야 하는 문제가 생긴다.

**ISP(인터페이스 분리 원칙)의 핵심**

- 인터페이스를 거대하게 만들지 말고, 인터페이스를 사용하는 곳(클라이언트)에 목적에 맞게 잘 쪼개라는 원칙이다.

## 가져가야 할 핵심

1. ISP의 중심은 구현 클래스가 아니라, 그 인터페이스를 사용하는 클라이언트(호출하는 곳)이다.
2. 사용하지 않는 메소드에 의존할 때 발생하는 불필요한 결합과 변경의 연쇄 반응을 구조적으로 파악해야 한다.
3. 하나의 거대한 서비스를 목적에 따라서 여러 개의 작은 인터페이스로 분리하는 기준을 배운다.

## ISP(인터페이스 분리 원칙)

**클라이언트는 자신이 사용하지 않는 메서드에 의존하지 않아야 한다는 객체지향 설계 원칙**

### 흔히 하는 실수 : 구현체 중심의 사고

- 개발자들이 인터페이스를 만들 때 가장 많이 하는 실수 중 하나는 **구현체 입장에서 생각하는 것**이다.
- “`UserServiceImpl`이라는 클래스를 만들 건데, 여기에 로그인, 회원가입, 비밀번호 변경 기능이 다 들어가니까 이걸 묶어서 하나의 `UserService`라는 인터페이스를 만들어야지”라고 생각하는 것이 대표적이다.
- 하지만 ISP에서 이야기하는 인터페이스는 구현체를 위한 설계가 아니다. **사용하는 쪽(클라이언트)이 어떤 기능을 필요로 하는지 명확히 드러내는 계약**으로 해석해야 한다.

### 불필요한 결합과 변경 전파 예시

일반 사용자가 쓰는 `UserController`가 있고, 관리자가 쓰는 `AdminController`가 있다고 가정해 보자.

1. `UserController`는 "관리자용 사용자 통계 추출"이라는 메소드가 전혀 필요 없다.
2. 하지만 두 컨트롤러가 모두 하나의 거대한 `UserService`를 주입받고 있다면, 실제로는 일부 기능만 사용함에도 불구하고 **필요 없는 메소드까지 함께 의존하게 되는 구조적 불필요한 결합**이 발생한다.
3. 이 상황에서 관리자의 요구사항이 바뀌어 "관리자용 사용자 통계 추출" 메소드의 파라미터가 변경되면, `UserService` 인터페이스의 명세도 바뀌어야 한다.
4. 결국 **통계 기능과 전혀 상관없는 `UserController`도 자기가 의존하고 있던 인터페이스가 바뀌었기 때문에 변경 영향권에 들어오게 된다.**
5. 이는 빌드나 테스트 범위가 불필요하게 넓어지는 원인이 되며, 코드를 읽는 사람에게도 *"이 컨트롤러가 통계 기능이랑 상관이 있나?"* 하는 혼란을 준다.

**결론:** 사용 맥락이 다른 기능은 인터페이스 차원에서 철저히 분리되어야 한다.

## ISP를 지켜야 하는 이유(Spring Boot 관점)

- Spring Boot 개발 시 각 계층의 클라이언트가 자신에게 필요한 기능에만 의존하도록 만들어 불필요한 결합과 변경 전파를 구조적으로 차단하기 위해서이다.

### 커머스 서비스의 성장과 인터페이스 비대화

- 처음에는 `OrderService` 인터페이스에 `주문 생성`, `주문 취소` 두 개만 있었다.
- 시간이 지나면서 `배송 상태 업데이트`, `일일 매출 통계 추출`, `환불 처리` 같은 기능들이 전부 이 하나의 인터페이스에 추가된다.
- **이 상태에서 테스트 코드를 짜거나 유지보수를 하려고 하면:**
    - 인터페이스가 가진 계약(메소드)이 너무 많으면 테스트의 의도가 흐려진다.
    - "고객은 주문 생성만 하면 되는구나"를 확인하고 싶은데, 타입만 보면 배송, 정산, 환불 인터페이스에 다 연결되어 있는 것처럼 보이기 때문에 코드 분석 시 혼란에 빠진다.
- **해결책 (목적에 따른 분리):**
    - `OrderCommandUseCase`, `DeliveryManageUseCase` 등으로 인터페이스를 나눈다.
    - 구현체 클래스는 이 인터페이스들을 다중 구현(implements)할지언정, **바라보는 클라이언트(컨트롤러 등)의 시선에서는 거대한 서비스가 아니라 자신에게 명확한 인터페이스만 가져다 쓸 수 있게 되므로 변경 전파가 차단**된다.

## 코드 예시로 보는 ISP와 원칙 간의 연결성

**ISP 위반 예시**

```java
public interface Car {
    void drive(); // 도로를 달리는 기능
    void fly();   // 하늘을 나는 기능
    void sail();  // 물 위를 떠다니는 기능
}

// 일반 스포츠카 구현
public class SportsCar implements Car {
    @Override
    public void drive() {
        System.out.println("도로를 맹렬하게 달립니다.");
    }

    // 스포츠카는 날 수도 없고 물에 뜨지도 않는데 자바 문법상 강제로 구현해야 함
    @Override
    public void fly() {
        // LSP(리스코프 치환 원칙) 위반으로 이어진다.
        throw new UnsupportedOperationException("자동차는 날 수 없습니다.");
    }

    @Override
    public void sail() {
        throw new UnsupportedOperationException("자동차는 물 위를 달릴 수 없습니다.");
    }
}
```

- 누군가 탈것을 통합하겠다고 `Car` 인터페이스에 주행, 비행, 항해를 다 넣는 순간 문제가 일어난다.
- 자바 문법상 인터페이스의 메소드는 무조건 오버라이딩해야 하므로, `SportsCar`는 **자신이 할 수 없는 일까지 억지로 빈 깡통으로 만들거나 예외를 던지며 떠안게 된다.**
- **SOLID 원칙의 연결성:** ISP를 위반해서 거대한 인터페이스를 만들면 구현체가 쓸데없는 기능을 떠안으며 **LSP 위반**으로 직결된다. 원칙들은 독립된 것이 아니라 서로 긴밀히 연결되어 있다.

ISP 준수 예시(역할별 분리)

```java
public interface Drivable {
    void drive();
}

public interface Flyable {
    void fly();
}

public interface Sailable {
    void sail();
}

// 스포츠카는 자신이 진짜 할 수 있는 역할(Drivable)만 선택해서 구현
public class SportsCar implements Drivable {
    @Override
    public void drive() {
        System.out.println("도로를 빠르게 달립니다.");
    }
}

// 수륙양용차처럼 여러 역할을 할 수 있다면 다중 구현으로 해결
public class AmphibiousCar implements Drivable, Sailable {
    @Override
    public void drive() {
        System.out.println("도로를 달립니다.");
    }

    @Override
    public void sail() {
        System.out.println("물 위를 떠다닙니다.");
    }
}
```

- 자바 API의 `Runnable`, `Closeable` 같은 인터페이스들이 `~able` 형태로 작게 쪼개져 있는 이유가 바로 이 때문이다. **특정 역할이나 능력만 명확히 부여**하기 위함이다.
- 자바는 클래스 다중 상속은 제한하지만 **인터페이스 다중 구현은 허용**하므로, 원하는 능력(인터페이스)만 조합해서 객체를 완성할 수 있다.

## SRP vs ISP

Q. 단일 책임 원칙(SRP)과 인터페이스 분리 원칙(ISP)은 결국 둘 다 잘게 쪼개라는 뜻 아닌가요? 둘의 차이점이 무엇인가요?

두 원칙 모두 응집도를 높이고 결합도를 낮추기 위해 분리방향을 제시하지만, 바라보는 관점이 다르다.

- SRP(단일 책임 원칙)는 구현체인 클래스 관점이다. 클래스가 단 하나의 변경 이유(책임)만을 가져야함을 강조한다.
- ISP(인터페이스 분리 원칙)는 인터페이스를 사용하는 클라이언트(호출하는 곳)관점이다. 클라이언트가 자신이 사용하지 않는 메서드에 불필요한 의존성을 갖지 않도록 하고, 그로 인한 변경의 피해를 입지 않도록 차단하는 것에 초점을 맞춘다.

## 정리

- ISP는 클라이언트가 사용하지 않는 메소드에 의존하도록 강제해서는 안 된다는 원칙이다.
- 인터페이스가 비대해지면 구현 클래스가 빈 깡통 메서드를 만들거나 예외를 던지게 되어 LSP위반으로 이어진다.
- 이를 해결하게 위해 인터페이스를 역할과 목적에 맞게 분리해야 한다.
- 단, 처음부터 기계적으로 다 쪼개기보다는 시스템이 복잡해지고 사용 맥락이 분리되어 변경에 영향을 받기 시작하는 시점에 적용하는 것이 현명하다.
