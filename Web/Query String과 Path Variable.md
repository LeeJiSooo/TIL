## 자원의 식별 vs 자원의 정렬/필터

Path Variable : 자원의 주소 그 자체. 수많은 자원 중 “단 하나의 특정 자원을 명확하게 식별”할 때 사용한다.

Query String : 자원을 보여주는 옵션. 자원 목록을 “검색, 필터링, 정렬, 페이징”하여 결과를 다르게 받아보고 싶을 때 사용한다.

#### Path Variable(경로 변수)

- 이 값이 없으면 자원 전체를 특정할 수 없는가? → YES
- URL 경로의 일부로 포함되어 자원의 고유한 정체성을 규정한다.
- GET /users/123/profile
    - 여기서 유저 ID인 123이 없다면 도대체 누구의 프로필을 보여줘야할지 시스템이 알 수 없다. → 이 값은 자원을 구성하는 필수적인 식별자이다.

#### Query String(쿼리 스트링)

- 이 파라미터가 없어도 이 URL의 본질(목록 조회)이 변하지 않는가? → YES
- URL 끝에 ? 로 시작하며, 필수 정보라기보다는 결과를 어떻게 예쁘게 필터링해서 보여줄지에 대한 부가적인 옵션이다.

- **필터링:** `GET /products?category=shoes&size=270` (신발 중 270 사이즈만 필터)
- **정렬:** `GET /posts?sortBy=like&order=desc` (게시글을 좋아요 순으로 정렬)
- **페이지네이션:** `GET /posts?page=10&size=20` (10번째 페이지 데이터를 20개씩 끊어서)
- **검색:** `GET /search?q=웹+개발` (웹 개발이라는 키워드가 포함된 자원 검색)

| **구분** | **Path Variable (경로 변수)** | **Query String (쿼리 스트링)** |
| --- | --- | --- |
| **목적** | 특정 리소스를 **식별** | 리소스 목록을 **필터링, 정렬, 검색, 페이징** |
| **자원 관계** | **자원의 주소 그 자체** (정체성 규정) | **자원의 옵션** (결과를 보여주는 방식 변형) |
| **생략 가능 여부** | 생략 불가 (생략 시 다른 URL이 되거나 에러) | 생략 가능 (생략 시 기본 전체 목록 제공) |
| **URL 형태** | `/posts/123` | `/posts?category=tech` |

#### 좋은 RESTful 설계의 예

똑같은 /posts/123이라는 자원의 고유 주소를 두고, HTTP Method(행위)만 변경하여 완벽한 CRUD를 성립시킨다.

- `GET /posts/123` ➔ 123번 게시글을 **조회**
- `PUT /posts/123` ➔ 123번 게시글을 **전체 수정**
- `DELETE /posts/123` ➔ 123번 게시글을 **삭제**

#### 흔히 하는 설계 실수

123번 회원을 조회할 때 GET /users?id=123 형태로 설계해도 기술적으로 서버는 동작한다. 하지만 이는 RESTful한 설계 원칙에 어긋난다.

id=123은 유저 목록을 필터링하는 부가 옵션이 아니라 유저 한 명을 지정하는 고유 식별자이기 때문에 GET /users/123으로 작성하는 것이 자원의 성격에 훨씬 적합하다.

## 정리

Path Variable과 QueryString의 가장 큰 차이는 ‘자원의 식별’이냐 ‘자원의 옵션’이냐의 차이이다.

Path Variable은 해당 값이 없으면 리소스 자체를 특정할 수 없는 필수적인 식별자로 자원의 주소 역할을 하고 Query String은 자원의 본질을 바꾸지 않으면서 정렬, 필터링, 검색, 페이지네이션 등 결과를 제한하는 부가적인 옵션을 전달할 때 사용한다.
