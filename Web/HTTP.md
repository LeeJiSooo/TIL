HTTP는 현대 웹에서 HTML, CSS, JS, 이미지, 영상, JSON 등 모든 형태의 웹 자원을 주고받는 전송 규약이다.

#### 요청과 응답의 단 방향성

- HTTP통신은 항상 클라이언트의 요청으로 시작해서 서버의 응답으로 끝난다.
- 서버가 먼저 클라이언트에게 말을 거는 경우는 절대 없다. 서버는 오직 요청을 받았을 때만 그에 대한 처리 결과를 반환한다.

#### 무상태성(Stateless)

장점

- 서버 구조의 단순화와 확장성 : 서버가 클라이언트의 상태를 보관하지 않기 때문에 트래픽이 몰려 서버 인스턴스를 10대 늘려도(로드밸런싱) 사용자는 1번 서버, 5번 서버, 10번 서버 아무 곳에나 요청을 던져도 똑같은 응답을 받을 수 있다. → 서버 늘리기가 매우 쉬워진다.
- 우지보수성 향상 : 요청 간의 의존성이 없으므로 API 설계가 단순해지고 복잡성이 낮아진다.

단점

- 상태를 기억하지 못하므로 페이지를 이동할 때마다 계속 로그인을 다시 해야하는 불편함이 생긴다. → 하지만 단점을 해결하기 위해 쿠키, 세션, JWT같은 기술을 도입하여 클라이언트가 요청을 보낼 때마다 자신의 신분증을 첨부하게 만들어 로그인 상태를 유지한다.

#### URL구조

우리가 주소창에 치는 URL은 서버에 있는 자원의 구체적인 위치와 조건을 의미한다.

https://example.com/products?id=123

- https(프로토콜) : 어떤 규칙으로 통신할 것인가? (보안 봉투를 쓴 암호화 HTTP)
- example.com(호스트/도메인) : 어떤 서버 컴퓨터로 찾아갈 것인가? (인터넷 세상의 건물 주소)
- /products(자원의 경로) : 그 서버 안에 있는 어떤 자원(데이터)를 원하는가?
- ?id=123(쿼리 스트링/파라미터) : 자원을 보낼 때 어떤 추가 조건을 붙일 것인가?(”products 자원 중에서 id가 123번인 상품 데이터만 골라서 보여줘”)

#### HTTP 버전별 발전사

| **버전** | **기반 프로토콜** | **핵심 특징 및 통신 방식** | **비유** |
| --- | --- | --- | --- |
| **HTTP/1.1** | **TCP** | **한 연결당 하나의 요청**만 순서대로 처리.
HTML, CSS, JS, 이미지 등 수십 개 파일을 하나씩 줄 서서 다운로드하므로 앞의 파일이 밀리면 뒤도 다 밀림 (**HOLB** 현상). | 편도 1차선 도로
(앞차 안 가면 뒤차 다 대기) |
| **HTTP/2** | **TCP** | **멀티플렉싱(Multiplexing) 도입**.
하나의 연결(TCP 통로)을 뚫어놓고 여러 개의 요청과 응답을 동시에 병렬로 주고받을 수 있어 속도가 획기적으로 빨라짐. | 왕복 다차선 고속도로
(동시에 여러 차 이동 가능) |
| **HTTP/3** | **UDP (QUIC)** | **TCP의 근본적인 한계 극복**.
TCP가 연결을 맺고 끊을 때 걸리는 지연 시간(Handshake)을 줄이기 위해, 신뢰성 연결 레이어를 자체 구현한 **UDP(QUIC) 기반**으로 전환하여 연결 속도와 전송 효율을 극대화함. | 목적지까지 멈추지 않는
초고속 직행 자기부상열차 |

## 정리

- HTTP는 요청과 응답 구조를 가지며 서버의 확장성을 극대화하기 위해 무상태성을 지향한다.
- 오랫동안 쓰인 HTTP/1.1의 순차적 전송 한계를 극복하기 위해 HTTP/2에서 하나의 연결로 동시에 여러 데이터를 주고받는 멀티플렉싱이 도입되었고 최근 HTTP/3에서는 TCP의 연결 지향적 지연 시간을 완전히 해결하기 위해 UDP(QUIC)기반으로 진화하여 더 빠른 웹 환경을 제공하고 있다.
