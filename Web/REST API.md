## REST의 정의

“자원의 상태를 특정 형식(JSON, XML 등)으로 표현하여 전송한다”는 의미이다.

특정 기술, 언어, 프레임워크가 아니라 웹의 장점을 살리기 위한 소프트웨어 아키텍처 스타일이다.

REST API : 이 REST 아키텍처 가이드라인을 따라 개발자들끼리 이해하기 쉽고 예측 가능하도록 설계한 API이다.

## REST를 구성하는 아키텍처 원칙

#### 1. 일관된 인터페이스

모든 자원에 접근하는 방식을 통일한다. 구조와 데이터 형식이 일관되므로 개발자가 API 사용법을 고민하는 시간을 줄여준다.

**HATEOAS와 Self-Descriptiveness**

- **Self-Descriptiveness (자체 표현 구조):** 메시지 본문만 보고도 의미를 다 알 수 있어야 한다. `{ "result": 25 }` 대신 `{ "temperature": 25, "unit": "celsius" }`처럼 키값을 명확히 하고, 올바른 HTTP Status Code를 사용하는 것이 일부분이다.
- **HATEOAS:** 응답 데이터에 '다음에 할 수 있는 행위(예: 마이페이지 이동, 게시글 상세 보기)'의 **URL 링크를 포함**하여 반환하는 규칙이다. (보통 응답 헤더의 `Location` 등을 활용)

#### 2. 클라이언트 - 서버 분리

UI/화면 담당하는 클라이언트와 데이터/비즈니스 로직을 처리하는 서버의 역할을 명확히 분리하여 독립적인 발전을 도모한다. → 분리 덕분에 서버 코드가 바뀌어도 앱 화면을 새로 빌드할 필요가 없음

#### 3. 무상태성(Stateless)

서버는 클라이언트의 이전 상태를 기억하지 않는다. JWT토큰 등을 활용해 클라이언트가 매 요청마다 모든 맥락(신분증)을 담아 보낸다. → 1번 서버가 터져서 2번 서버로 요청이 가도 맥락(JWT)이 있으므로 문제 없이 처리됨(수평확장/Scale-out)

#### 4. 캐시 가능성(Cacheable)

자주 변경되지 않는 데이터는 클라이언트나 중간 서버에 임시 저장(캐시)하도록 응답 헤더(Cache-Control)에 유효 기간을 명시한다. → 매번 서버 DB까지 가지 않고 브라우저 캐시에서 바로 꺼내쓰므로 속도 향상, 서버 부하 감소

#### 5. 계층화 시스템

클라이언트는 최종 서버와 직접 통신하는지, 중간의 로드밸런서, API게이트웨이, 프록시 서버와 통신하는지 알 필요가 없어야 한다. → 게이트웨이 뒤에 서버가 몇대가 있든 클라이언트는 단일 엔드포인트만 바라봄

## RESTful URI설계 규칙

REST API 설계의 핵심 원칙은 “URL은 자원(명사)만 나타내고 행위는 HTTP Method(동사)로 표현한다.

#### 1) 정보의 자원은 반드시 명사로 표현한다 (동사 금지)

- ❌ `GET /getUsers`, `POST /createUser`, `POST /getUserPostsById/{userId}`
- ⭕ **`GET /users`**, **`POST /users`**, **`GET /users/{userId}/posts`**

#### 2) 행위는 HTTP Method로 지시한다

- ❌ 유저 생성: `GET /createUser` | 유저 조회: `POST /getUserById`
- ⭕ 유저 생성: **`POST /users`** | 유저 조회: **`GET /users/{userId}`**

#### 3) 슬래시(`/`)는 계층 관계를 나타내며, 마지막에는 포함하지 않는다

슬래시는 디렉터리 구조처럼 하위 계층을 나눌 때만 쓴다. 마지막에 슬래시가 붙으면 또 다른 하위 계층이 있는 것으로 오해를 부를 수 있다.

- `GET /users/{userId}/posts/`
- ⭕ **`GET /users/{userId}/posts`** (특정 유저가 작성한 게시글 목록)

#### 4) 가독성을 위해 소문자만 사용하고, 밑줄(`_`) 대신 하이픈()을 쓴다

대문자는 혼선을 줄 수 있으므로 소문자가 표준이다. 또한 밑줄(`_`)은 브라우저에서 링크 밑줄과 겹쳐 안 보일 수 있으므로 단어 연결이 필요할 땐 하이픈을 쓴다.

- ❌ `/Users/get_profile_image`
- ⭕ **`/users/{userId}/profile-image`**

#### 5) URL 경로에 파일 확장자를 포함하지 않는다

자원 그 자체의 이름만 URL에 명시해야 한다. 클라이언트가 JSON을 원하는지, PNG 이미지를 원하는지는 URL이 아니라 HTTP **`Accept` 헤더**를 통해 소통하는 것이 원칙이다.

- ❌ `GET /users/123/profile.png`
- ⭕ **`GET /users/123/profile`** + 헤더에 데이터 타입 명시

## URI vs URL

#### URI(Uniform Resource Identifier)

통합 자원 식별자

인터넷상에 있는 자원을 나타내는 가장 고유한 식별 번호나 주소 전체를 의미한다. 

#### URL(Uniform Resouce Locator)

통합 자원 위치 정보

흔히 말하는 ‘웹사이트 주소’다. 자원이 컴퓨터 네트워크상의 ‘어디’에 컴퓨터가 저장되어 있는지 그 경로를 나타낸다.

[https://example.com/products?id=123](https://example.com/products?id=123은)

이 전체 주소는 URI일까, URL일까?

→ URI이다. 왜냐하면 제일 끝에 붙은 ?id=123(쿼리 스트링)때문이다. 

?id=123은 자원의 위치가 아니라 “products중에서 id가 123번인 자원을 식별해줘”라는 조건이다.
