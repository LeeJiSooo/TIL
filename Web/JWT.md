## JWT(JSON Web Token)

인증에 필요한 정보들을 JSON포맷으로 토큰 내부에 직접 담아두고 서버의 비밀키로 서명한 자체 포함형 토큰이다.

## JWT가 서버 확장(Scale-out)과 MSA에 필수인 이유

- 상태를 안 적음(무상태 인증) : 서버는 로그인 상태를 기억하지 않는다. 토큰 안에 유저 ID, 닉네임, 권한이 다 들어있기 때문에 서버는 “이 토큰이 위조되었나”만 검증하면 된다.
- 수평 확장 최적화 : 1번 서버가 발행한 JWT를 100번 서버가 받아도 서버의 비밀키만 같으면 검증이 가능하다.
- 인증 서버에서 JWT 한 장만 발급받으면 회원, 상품, 결제 등 수많은 마이크로 서비스들이 DB를 각각 공유하지 않고도 토큰만 읽어서 유저를 식별할 수 있다.

## 보안과 데이터 무결성 : 서명의 원리

토큰을 클라이언트에 저장하면 해커가 위변조하기 쉽지 않을까? → 서명하면된다.

- 서버는 토큰을 만들 때 내부 내용과 서버만 알고 있는 비밀키를 조합하여 암호화된 서명값을 토큰 뒤에 붙여서 준다.
- 해커가 마음대로 토큰 안에 값을 수정하는 순간 서버에서 계산한 서명과 매칭이 안되어서 들통난다. 서버는 서명이 다르면 즉시 요청거부(401 Unauthorized)하므로 데이터 변조가 불가능하다.

## 엑세스 토큰, 리프레시 토큰

JWT는 한 번 발급하면 서버가 강제로 뺏을 수 없다. 그래서 실무에서는 보안과 사용자 편의성을 모두 잡기 위해 두 가지 토큰을 나누어서 쓴다.

#### 액세스 토큰

- 실제 서버의 데이터나 api에 접근할 수 있는 실질적인 권한을 가진 토큰
    
    Authorization : Bearer <토큰> 헤더에 담겨 온다.
    
- 유효기간 이 10~15분 내외로 매우 짧음 → 탈취당하더라도 해커가 쓸 수 있는 유효기간을 줄여 보안 리스크를 막기 위함이다. 하지만 이것만 사용하면 사용자는 15분마다 계속 새로 로그인을 해야한다.

#### 리프레시 토큰

- 오직 액세스 토큰을 재발급 받기 위한 용도로만 사용되는 토큰
- 유효기간이 7일~14일 정도로 매우 길다. 액세스 토큰이 만료되었을 때만 딱 한번 전송된다.

1. 서버는 유효기간이 짧은 액세스 토큰과 긴 리프레시 토큰을 둘 다 발급한다. 이때 서버는 리프레시 토큰의 원본을 레디스나 디비에 기록해둔다.
2. 클라이언트는 가벼운 엑세스 토큰만 헤더에 실어서 api를 요청한다.
3. 15분이 지나 액세스 토큰이 만료되면 서버는 401 Unauthorized응답을 던진다.
4. 클라이언트는 당황하지 않고 조용히 보관 중이던 리프레시 토큰을 들고 서버의 ‘토큰 재발급 api’를 요청한다.
5. 서버는 들어온 리프레시 토큰의 서명을 검증하고 내 레디스에 저장된 값과 일치하는지 확인한다. 검증이 통과되면 새로운 액세스 토큰을 발급해서 돌려준다.
6. 클라이언트는 새로 받아온 액세스 토큰을 가지고 아까 실패했던 api요청을 다시 보낸다.
    
    → 사용자는 로그인이 풀린 줄도 모르고 서비스를 다시 누리게됨
    

#### RTR(Refresh Token Rotation)전략 : 탈취 차단

리프레시 토큰은 유효기간이 길어서 이것마저 탈취당하면 보안에 문제가 생기는데 이걸 막는 기술이 RTR이다.

클라이언트가 리프레시 토큰을 들고 와서 새로운 액세스 토큰을 요구할 때 액세스 토큰만 주는게 아니라 리프레시 토큰도 다시 생성해주는 방식이다.

## JWT + 세션 혼합

JWT의 단점은 “한 번 발급하면 서버가 마음대로 제어(강제 로그아웃)할 수 없다는 무상태성이다.

- 로그인을 하면 서버는 유효기간 매우 짧은 액세스토큰과 장기 세션 퀴를 둘 다 준다.
- 평소 api 요청은 빠른 속도와 확장성을 위해 짧은 JWT만 보고 검증한다. 그러다 토큰이 만료되면 세션 쿠키를 이용해 서버에 새 JWT를 요청한다.

만약 관리자가 블랙리스트 유저를 발견해서 강제 로그아웃 시키고 싶다면? 

→ 서버에서 해당 유저의 세션만 삭제해버리면 된다. 그럼 토큰이 만료되었을 때 재발급이 안 되므로 동시 접속 제어와 강제 로그아웃이 가능해진다.

| 구분 | **세션(Session)기반 인증** | **JWT(Token)기반 인증** |
| --- | --- | --- |
| 상태 저장소 | 서버 측(Redis 등 세션 스토어) | 없음(클라이언트가 토큰 보관) |
| 서버 확장성 | 세션 클러스터링 등 추가 인프라 세팅 필요 | 매우 용이(비밀키만 있으면 어디서나 검증 가능) |
| 중앙 제어성 | 강제 로그아웃, 동시 접속 제어가 매우 쉬움 | 한 번 발급하면 통제 불가능(RTR이나 혼합 방식으로 보완) |
| 네트워크 부하 | 가벼운 세션ID만 전송하므로 적음 | 토큰 내부에 정보를 많이 담으면 패킷이 무거워짐 |

#### 다른 기기에서 로그인하면 기존 기기가 풀리게 하고싶으면?

1. 사용자 아이디/비밀번호를 들고 로그인 API를 요청한다.
2. 서버는 레디스에서 유저ID로 저장된 기존 리프레시 토큰이 있는지 확인한다.
3. 만약에 있다면 ? 서버는 고민하지 않고 Redis에 있던 기존 리프레시 토큰을 완전히 지워버린다.
4. 그러고 나서 방금 로그인한 기기를 위한 새로운 리프레시 토큰을 저장하고 응답으로 내려준다.
5. 기존 기기는 액세스 토큰 만료시 재발급이 막혀서 자연스럽게 로그아웃된다.
