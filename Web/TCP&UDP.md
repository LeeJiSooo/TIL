## TCP(Transmission Control Protocol)

두 개의 호스트를 연결하고 데이터 스트림을 교환하게 해주는 중요한 네트워크 프로토콜

OSI 7계층 중 4계층(전송 계층)에 속하며, 출발지에서 목적지까지 데이터가 온전히 전달되는 것을 책임진다. 우리가 브라우저 주소창에 주소를 치고 엔터틀 누르는 순간, 컴퓨터와 서버 사이에는 이 TCP 연결이 맺어진다.

- **연결 지향형 프로토콜:** 데이터를 보내기 전에 반드시 상대방과 연결을 설정한다. 전화 걸 때 상대방이 전화를 받아야 "여보세요" 하고 통화(통신)를 시작하는 것과 같다. 이 과정을 통해 양쪽 기기가 준비됐음을 확인하고 통신 동안 연결을 유지한다.
- **데이터 스트림(Stream):** 1MB 파일을 보낼 때 한 덩어리로 던지지 않는다. 잘게 쪼개서 세그먼트(Segment)라는 작은 단위로 만들고, 각 조각에 번호표(Sequence Number)를 붙여 차례대로 보낸다. 받는 쪽은 이 번호표를 보고 원래 순서대로 재조립해 완전한 파일로 복원한다. 이 과정 덕분에 데이터가 연속된 흐름(Stream)처럼 끊김 없이 전달되어 보인다.

## TCP의 동작 방식

### 1. 연결 설정 (3-Way Handshake)

데이터를 보내기 전, 클라이언트와 서버가 3번의 통신을 통해 서로의 상태를 확인하는 과정이다.

1. **SYN (클라이언트 ➔ 서버):** "나 전화 걸었어. 데이터 받을 수 있어? 들리면 응답해줘."
    - 클라이언트가 서버에게 보내는 첫 메시지. 패킷에는 "이제부터 데이터 번호표 1000번부터 시작한다"라는 **초기 순서 번호(시퀀스 넘버)** 정보가 담겨 있다.
2. **SYN-ACK (서버 ➔ 클라이언트):** "전화 잘 받았어! 내 목소리도 들려? 들리면 응답해줘."
    - 서버의 응답. "아, 1000번 잘 받았고 다음엔 1001번 보내줘(**ACK**). 그리고 내 번호표는 5000번부터 시작할게(**SYN**)."
3. **ACK (클라이언트 ➔ 서버):** "잘 들린다! 이제 본론(데이터 주고받기) 이야기하자."
    - 클라이언트가 서버에게 확답을 준다. "5000번 잘 알았고, 다음부턴 5001번 보내줘."
    - 이 과정이 끝나야 비로소 양쪽이 통신할 준비를 마친다.

### 2. 데이터 전송 및 신뢰성 확보

- **재조립:** 세그먼트로 쪼개져 온 데이터들을 수신 측이 시퀀스 넘버를 보고 순서대로 조립해 어플리케이션 레벨에 전달한다. 계층이 나뉘어 있기 때문에 개발자는 이 조립 과정을 전혀 신경 쓸 필요가 없다.
- **ACK 메커니즘:** 송신 측이 데이터를 보내면 수신 측은 잘 받았다는 영수증으로 ACK 패킷을 돌려준다. 송신 측이 일정 시간 동안 ACK를 받지 못하면 "데이터가 날아갔구나" 판단하고 데이터를 다시 보낸다(**재전송**). 그래서 유실이 거의 없다.

### 3. 흐름 제어 (Flow Control) vs 혼잡 제어 (Congestion Control)

- **흐름 제어 (1:1 속도 조절):** 받는 쪽이 한 번에 처리할 수 있는 양에는 한계가 있다. 송신 측이 마구 퍼부으면 수신 측 버퍼가 꽉 차서 데이터를 버리게 된다. 이를 막기 위해 수신 측은 "나 지금 8KB는 더 받을 수 있어" 같은 상태 정보를 슬라이딩 윈도우(Sliding Window)라는 값에 담아 송신 측에 계속 알려준다. 송신 측은 이 윈도우 크기를 보고 데이터 양을 조절한다.
- **혼잡 제어 (네트워크 전체 조절):** 흐름 제어가 둘 사이의 문제라면, 혼잡 제어는 데이터가 지나가는 네트워크 상황을 고려하는 것이다. 인터넷망이 막히기 시작하면 TCP는 데이터 전송 속도를 스스로 줄여서 혼잡을 가중시키지 않는다. **Slow Start**, **혼잡 회피** 같은 알고리즘을 사용해 동작한다.

### 4. 연결 종료 (4-Way Handshake)

- "나 이제 보낼 데이터 없으니 연결 끊을게"라는 **FIN** 패킷을 주고받으며 4단계에 걸쳐 안전하게 연결을 마무리 짓는다.

## UDP(User Datagram Protocol)

빠르고 간단한 통신을 위해 연결 설정 없이 데이터를 전송하는 네트워크 프로토콜이다.

- **비연결성 및 비신뢰성:** TCP처럼 "지금부터 통신 시작하자" 하는 3-Way Handshake 같은 사전 약속이 아예 없다. 그냥 보내고 싶은 데이터가 있으면 주소 붙여서 바로 네트워크로 던져버린다.
- **데이터그램(Datagram) 단위:** TCP의 세그먼트와 달리 UDP는 **데이터그램**이라는 단위로 데이터를 보낸다. 여기에는 시퀀스 넘버(번호표) 같은 개념이 없다. 그래서 먼저 보낸 데이터가 나중에 도착할 수도 있고, 중간에 유실될 수도 있다. UDP는 이에 대해 일절 책임지거나 보장하지 않는다. 순서가 바뀌거나 유실된 걸 해결해야 한다면 개발자가 상위 레벨에서 직접 코드로 구현해야 한다.
- **압도적인 속도 (가벼운 헤더):** 온갖 확인 절차를 생략하기 때문에 프로토콜 자체의 오버헤드(성능 저하)가 적어서 속도가 엄청나게 빠르다. TCP 헤더는 신뢰성을 챙기느라 이것저것 정보가 많아 최소 **20바이트** 정보를 먹지만, UDP 헤더는 그런 거 다 빼서 딱 **8바이트**밖에 안 된다. 처리할 정보도 적고 절차도 없으니 빠를 수밖에 없다.
- **주요 사용처:** 유튜브 라이브 스트리밍, 온라인 게임, 실시간 영상 통화. 영상 프레임 하나 놓치거나 캐릭터 위치 정보 하나 사라져서 잠시 화면이 깨지거나 순간 이동(렉)하는 것이, 게임이나 방송이 아예 멈추는 것보다 낫기 때문이다.
