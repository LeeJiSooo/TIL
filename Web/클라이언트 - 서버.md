## 웹의 기본 구조 : 클라이언트와 서버

웹은 요청하는 주체와 이를 처리하는 주체 간의 끊임없는 대화로 이루어진다.

- 클라이언트 : 자원이나 서비스를 요청하는 주체(ex. 웹 브라우저, 모바일 앱, 인베디드 기기 등)
- 웹 서버 : 클라이언트의 요청을 받아 처리하고 응답하는 주체이다. 프로그램이 실행되고 있는 물리적 컴퓨터나 가상머신(VM)을 의미한다.
- 데이터베이스 : 사용자 정보, 상품 목록, 게시글 등 모든 데이터를 체계적으로 저장하고 관리하는 데이터 저장소이다.
- 프론트엔드와 백엔드의 역할 분담
    - Client-Side Language(HTML, CSS, JavaScript) : 클라이언트 측에서 요청을 생성하고 화면을 그리는 역할이다. 사용자가 버튼을 클릭하면 자바스크립트가 서버에 데이터를 요청하는 명령을 내린다.
    - Server-Side Languate(Java, Python, Node.js 등) : 서버 내부에서 핵심 비즈니스 로직을 수행하고 데이터베이스와 상호작용하며 실질적인 작업을 처리한다.

## 네트워크 및 주소 체계

### IP와 도메인 & DNS

- IP주소 : 컴퓨터가 네트워크 상에서 서로를 식별하기 위해 사용하는 고유한 숫자 주소이다.
- 도메인(Domain) : 숫자로 된 IP주소를 사람이 기억하기 쉽도록 문자로 표현한 주소이다.
- DNS(Domain Name System) : 도메인 이름과 실제 IP주소를 매칭해 주는 인터넷의 주소록이다.
    - 브라우저에 도메인을 입력하면 컴퓨터는 가장 가까운 DNS서버에 해당 도메인의 IP를 조회한다. DNS가 IP 주소를 찾아 반환해주면 브라우저는 그 IP를 가지고 실제 서버에 접속을 시도한다.

### 포트

하나의 서버(IP) 안에는 웹 서비스, 이메일, 파일 전송 등 여러 개의 프로그램이 동시에 실행될 수 있다. 이때 어떤 프로그램으로 들어가야 하는지 구분하는 문(통로)의 번호가 포트이다.

- 80 : HTTP통신(기본 웹 서비스)
- 443 : HTTPS통신(보안이 적용된 웹 서비스)
- 22 : SSH통신(개발자가 서버에 원격 접속할 때 사용)

→ 서버 운영 시 보안을 위해 반드시 필요한 포트만 열어두고 나머지는 닫아서 관리해야 한다.

## 통신 규약과 상태 관리

### 프로토콜

클라이언트와 서버가 서로 데이터를 주고 받기 위해 정의한 공통의 통신 규약(약속/규칙)이다. 웹에서는 주로 HTTP를 기반으로 요청과 응답 메시지를 주고받는다.

### HTTP의 특징 : 무상태성(Stateless)

서버는 클라이언트의 이전 요청 상태를 기억하지 못하며 모든 요청을 독립적인 것으로 취급한다. → 만약 HTTP본연의 기능만 쓴다면 쇼핑몰에서 로그인 했어도 다음 페이지(마이페이지)로 이동할 때 서버가 로그인 사실을 기억하지 못한다.

해결책 : 우리가 로그인 상태를 유지하며 서비스를 이용할 수 있는 이유는 쿠키, 세션, JWT같은 기술을 사용하여 Stateless한 HTTP의 한계를 보완하기 때문이다.

## 시스템 보호와 권한 관리

### 방화벽(Firewall)

외부의 비정상적인 접근이나 악성 트래픽, 해킹 시도로부터 내부 서버를 보호하는 네트워크 보안 시스템이다. 허용되지 않은 접근을 입구에서 차단한다.

### 인증과 인가의 차이

보안의 핵심은 ‘내가 신뢰할 수 있는 사람인지(인증)’확인하고 ‘그 사람이 어디까지 할 수 있는지(인가)’제한하는 것이다.

- 인증(Authentication) : 세스템에 등록된 사용자가 맞는지 신원을 확인하는 절차
    - 아이디/비밀번호 로그인, 지문 인식, OAuth
- 인가(Authorization) : 인증된 사용자가 특정 자원에 접근하거나 작업을 수행할 수 있는지 권한을 검증하는 절차
    - 일반 사용자는 본인 글만 수정 가능, 관리자는 전체 게시글 삭제 및 회원 관리 가능

→ 시스템은 인증되지 않은 사용자의 접근을 원천 차단해야 하며, 인증된 사용자라 하더라도 본인에게 인가되지 않은 영역의 작업은 엄격히 제한해야 한다.

## 프로토콜 관점으로 보는 HTTPS 통신 흐름

#### 주소 찾기와 출발 준비

1. 도메인 이름 입력 및 자동 리다이렉트(HTTP → HTTPS)
    - 사용자가 http://google.com으로 접속을 시도하더라도 서버나 브라우저가 보안을 위해 안전한 https:// 주소로 자동으로 방향을 틀어준다(리다이렉트)
2. 주소록 찾기(DNS조회)
    - 컴퓨터는 텍스트 주소를 모르기 때문에 IP주소가 필요하다.
    - 우선순위 :  내 컴퓨터 내부의 DNS캐시(기록)확인 → 기록이 없다면 통신사(ISP) DNS나 상위 루트 DNS서버까지 단계별로 찾아가 최종 IP주소를 알아낸다.
3. 회사/기관의 출입문 통과(Foward Proxy 통과 - 선태적)
    - 인트라넷(내부망)환경이라면 외부 인터넷으로 나가기 전 Foward Proxy서버를 거친다. 이곳에서 트래픽을 감시하거나 유해 사이트를 차단해주고 자주 가는 사이트를 캐싱하여 속도를 높인다.
        - Foward Proxy : 클라이언트(사용자) 앞단에서 클라이언트를 대리(보안, 사이트 차단용)
        - Reverse Proxy: 웹 서버 앞단에서 서버를 대리(로드밸런싱, SSL암호화 처리용 - Nginx, Apache 등)

#### 신뢰할 수 있는 길 연결하기

1. 신뢰성 있는 통신로 확보(TCP 3-Way Handshake)
    - IP주소를 알아냈으니 이제 구글 서버와 통신을 시작한다. 본격적인 데이터를 주고받기 전 서로 주고 받을 준비가 됐는지 확인하는 신뢰성 구축 단계이다.
    1. SYN : 클라이언트  → 서버 (” 나 너랑 연결하고 싶은데 괜찮아?”)
    2. SYN-ACK : 서버 → 클라이언트 (”어 나 준비됐어. 너도 내 요청 맞는지 확인해봐”)
    3. ACK : 클라이언트 → 서버(”확인했어. 이제 진짜 데이터 주고받자”)
    
    → 이 세번의 통신이 끝나야 HTTP요청을 보낼 수 있는 기본 물리적 통로가 열린다.
    

#### 안전한 보안 봉투 만들기

TCP연결은 되었지만 아직 데이터는 가공되지 않은 평문 상태이다. 중간에 누군가 훔쳐볼 수 없도록 보안 봉투를 씌우는 단계(TLS Handshake)가 진행된다. 목적은 1. 진짜 구글 서버가 맞는지 신원 확인, 2. 데이터를 암호화할 ‘비밀키’ 공유

5~6. “너 진짜 구글 서버 맞아?” 신원 검증 및 인증서 요청

- 브라우저가 서버에 SSL/TLS인증서와 공개키(자물쇠)를 달라고 요청한다.
- 서버는 도메인 주소, 인증서 발급 기관(CA), 서버의 공개키가 담긴 인증서를 브라우저에 보내준다.

1. “믿을 만한 기관이군” (CA 인증서 검증)
    - 브라우저는 전 세계쩍으로 신뢰받는 기관들(CA)의 공개키를 이미 가지고 있다.
    - 이 CA공개키로 서버가 보낸 인증서의 서명을 풀어본다. 제대로 풀리면 위조되지 않은 진짜 구글 서버임이 증명된다.

1. 둘만의 비밀 암호 만들기(대칭키/세션키 생성)
    - 신원이 확인되면 클라이언트와 서버는 앞으로 데이터를 암호화할 ‘비밀키(대칭키)’를 안전하게 나누어 가진다.
    - 이 과정에서 비대칭키(공개키)알고리즘을 사용해 키를 안전하게 전달하며 한 번 키가 공유되면 모든 준비가 끝난다.

#### 안전한 데이터 교환

1. 안전한 HTTPS통신 시작
    - 이제 모든 데이터는 8단계에서 만든 비밀키로 암호화되어 오고간다. 중간에 해커가 패킷을 훔쳐 가더라도 열어볼 수 없는 안전한 통신 상태가 유지된다.

→사용자가 주소를 입력하면 먼저 DNS를 통해 IP주소를 찾고 TCP 3-Way HandShake를 통해 신뢰성 있는 연결 통로를 만든 뒤, TLS Handshake를 통해 상대방의 신원을 검증하고 데이터를 암호화할 비밀키를 안전하게 공유한다. 이 과정이 끝나야 안전한 HTTPS데이터 통신이 이루어진다.

## 모놀리식, MSA

### 모놀리식 아키텍처

모든 기능(회원, 상품, 주문, 결제)의 코드가 하나의 거대한 프로젝트에 묶여, 하나의 서버(단일 구동 환경)에서 실행되는 구조

장점

- 빠른 개발 속도 : 하나의 코드 저장소에서 개발하므로 구조가 단순하고 속도가 빠르다.
- 쉬운 테스트와 배포 : 서버 하나만 빌드해서 올리면 끝이기 때문에 배포 파이프라인이 아주 단순하다.
- 통합 관리의 용이성 : 트랜잭션 처리(ex. 주문 시 결제가 동시에 성공하거나 실패해야 하는 처리)를 데이터베이스 수준에서 쉽게 묶을 수 있다.

단점

- 단일 장애점 : 결제 기능의 사소한 코드 한 줄 때문에 에러가 나면 회원 조회나 상품 검색을 포함한 서버 전체가 다운된다.
- 비효율적인 스케일 아웃 : 블랙프라이데이로 인해 ‘상품 조회’트래픽만 폭주하더라도 트래픽과 무관한 회원, 결제 로직까지 포함한 거대한 서버 통째로 복해야 하므로 서버 자원 낭비가 심하다.
- 배포 부담 증가 : 시스템이 커질수록 코드 한 줄 고치고 재시작하는 데 수십 분이 걸린다.

### 마이크로서비스 아키텍처(MSA)

모놀리식의 한계를 극복하기 위해 비즈니스 기능별로 서버를 잘게 쪼개어 각각 독립적인 애플리케이션으로 운영하는 구조

장점

- 장애 격리 : 결제 서버가 다운되더라도 상품 서버나 회원 서버는 살아있다. 결제만 잠시 안될 뿐 고객은 여전히 상품을 둘러보고 장바구니에 담을 수 있어 최악의 사용자 경험을 피한다.
- 유연하고 효율적인 확장 : 특정 서비스에만 트래픽이 몰리면 다른 서버는 그대로 두고 상품 서버만 10대, 20대로 늘려서 자원을 효율적으로 사용할 수 있다.
- 팀별 독립성 및 기술 자유도 : 팀별로 서브를 나누어 관리하므로 배포가 자유롭고 속도가 빠르다. 기능에 맞는 최적의 기술 스택을 선택할 수 있다.

단점

- 데이터 일관성 유지의 어려움 : 모놀리식에서는 DB하나에서 처리하면 끝났던 작업이 MSA에서는 서버가 쪼개지면서 각자 자기 DB를 가진다. “주문은 성공했는데 결제가 실패했다”면 멀리 있는 주문 서버의 데이터를 어떻게 취소해야 할지 코드로 아주 복잡하게 구현해야 한다.
- 네트워크 통신 비용과 지연 : 모놀리식에서는 메모리 내의 함수 호출로 끝났던 작업이 MSA에서는 서버 간에 네트워크를 타고 데이터를 주고받아야 하므로 통신 속도가 느려지고 네트워크 에러 리스크가 생긴다.
- 인프라 관리 포인트와 비용 증가 : 서버가 1개에서 10개로 늘어나면 모니터링 시스템, ci/cd 배포 파이프라인, 로그 수집 장비도 10배로 복잡해진다. 이를 관리할 데브옵스 엔지니와 비용이 추가로 필요하다.
