## 정의

HTTP캐시는 웹 페이지, 이미지, 기타 미디어 파일 등 웹 리소스들을 사용자의 브라우저나 서버에 임시로 저장해두는 기술이다.

## 전체 아키텍처의 캐시 레이어

캐시는 단순히 브라우저에만 존재하는 게 아니다. 전체 시스템 아키텍처 곳곳에 레이어 형태로 존재하며, **클라이언트에 가까울수록 개인화된 데이터 처리**에 유리하고, **서버에 가까울수록 다수의 사용자를 위한 공통 데이터 처리**에 유리하다.

### Private Cache (개인 캐시)

- 사용자 개개인의 크롬, 사파리 같은 웹 브라우저 로컬 환경이다.
- 오직 **나만을 위한 캐시**다. 1GB짜리 대용량 교재 파일을 처음 조회할 때는 서버에서 통째로 받아오지만, 잠시 후 다시 열 때는 브라우저가 "전에 받아둔 거네?" 판단하고 로컬 캐시에서 즉시 꺼내 쓴다. 사용자는 대기 시간이 없고, 서비스 운영자는 트래픽 비용을 획기적으로 아낄 수 있다.

### Shared Cache (공유 캐시)

- 회사, 학교, 혹은 전 세계 거점에 위치한 공통 캐시 서버이다.
- **프록시 캐시:** 회사 네트워크의 프록시 서버가 대표적이다. A 직원이 특정 뉴스 기사를 보면 기사 이미지가 프록시 서버에 캐싱된다. 이후 B 직원이 같은 기사를 볼 때 언론사 서버까지 안 가고 회사 내 프록시 서버에서 바로 이미지를 긁어와 속도가 빠르다.
- **CDN (Content Delivery Network):** Shared Cache 개념을 전 세계 규모로 확장한 것이다. (예: AWS CloudFront, Cloudflare) 전 세계 주요 거점(Edge)에 캐시 서버를 두고, 사용자와 가장 가까운 서버에서 콘텐츠(이미지, 영상 등)를 서빙하여 물리적 거리를 줄이고 속도를 극한으로 끌어올린다.

### Managed Cache (어플리케이션/백엔드 캐시)

- 백엔드 서버 로컬 메모리나 별도의 인메모리 DB
- 백엔드 개발자가 **직접 코드 레벨에서 통제하고 관리하는 캐시 전략**이다.
    - 데이터 분산 환경이 아니라면 **Caffeine Cache** 같은 로컬 캐싱을 쓰고, 서버가 여러 대인 분산 환경(MSA 등)이라면 **Redis** 같은 인메모리 DB를 중앙 캐시 저장소로 두고 사용한다.
    - 복잡한 연산 결과(랭킹 계산 등)나 변하지 않는 데이터(상품 카테고리 목록 등)를 DB까지 가서 무겁게 쿼리하지 않고, Redis에서 바로 꺼내 응답 속도를 높인다.

## 캐시 대상 선정과 무효화 전략

- 캐싱의 적격 대상: 조회가 빈번하게 일어나지만, 데이터 변경은 거의 일어나지 않는 값 (예: 상품 카테고리 목록, 공지사항 등)
- 캐싱하면 위험한 대상: 자주 변경되고 실시간성이 중요한 데이터. 만약 잦은 변경 데이터를 무작정 캐싱하면 사용자는 옛날 데이터를 보게 되어 시스템에 심각한 오류(잘못된 정보 표기 등)가 생긴다.
- 캐시 무효화 정책 : 데이터가 바뀔 때 캐시를 어떻게 지우거나 갱신할 것인가에 대한 전략이다. 캐시를 도입할 때는 무조건 이 무효화 정책이 세트로 고려되어야만 한다.

### 백엔드 캐시 무효화 전략

#### TTL (Time To Live) 기반 무효화

- **방식:** 캐시를 저장할 때 아예 "유통기한(만료 시간)"을 같이 설정하는 방법이다. 예를 들어 Redis에 카테고리 목록을 넣으면서 "유효기간은 1시간"이라고 세팅한다.
- **동작:** 1시간이 지나면 Redis가 알아서 그 데이터를 삭제한다. 그럼 다음 사용자가 요청했을 때 캐시가 없으니(`Cache Miss`), 자동으로 DB에서 최신 데이터를 긁어와 다시 캐시에 채우게 된다.
- **장점:** 관리가 매우 쉽고 단순하다.
- **단점:** 실시간성이 떨어진다. 만약 10분에 데이터가 바뀌어도, 캐시는 1시간이 지나야만 사라지기 때문에 최대 50분 동안 사용자는 옛날 데이터를 보게 된다.

#### Write-Through (동시 저장)

- **방식:** 데이터를 수정하거나 저장할 때, **DB와 캐시 저장소에 동시에 데이터를 업데이트**하는 전략이다.
- **동작:** 사용자가 프로필 이름을 'soo'에서 'jay'로 변경하면, 백엔드 서버가 DB의 name도 'jay'로 바꾸고, Redis 캐시에 들어있는 name도 동시에 'jay'로 덮어써 버린다.
- **장점:** 캐시가 항상 최신 상태를 유지하므로 데이터가 꼬일 일이 없다.
- **단점:** 데이터를 쓸 때마다 캐시와 DB 두 곳에 모두 접근해야 하므로 저장/수정 속도가 조금 느려질 수 있다.

#### Cache Eviction / Delete (명시적 삭제 )

- **방식:** 데이터가 수정되거나 삭제되는 시점에, **기존 캐시를 통째로 날려버리는(Delete) 전략**이다. 스프링 부트 개발할 때 `@CacheEvict` 라는 어노테이션으로 정말 많이 구현하는 방식이다.
- **동작:** 상품의 가격이 변경되면, 가격 캐시를 최신 값으로 업데이트하는 게 아니라 그냥 **캐시 자체를 삭제**해 버린다. 그리고 나중에 누군가 그 상품을 조회할 때 캐시가 비어있으니까 DB에서 최신 가격을 긁어다가 캐시에 새로 넣어준다.
- **장점:** 안 쓰는 데이터까지 미리 캐시에 다 업데이트해 둘 필요가 없어서 메모리가 효율적이다.
- **단점:** 캐시가 지워진 직후 첫 번째 조회 요청을 보낸 사용자는 DB까지 갔다 와야 하므로 살짝 느릴 수 있다.

## HTTP 캐시의 표준 동작 방식

브라우저가 `logo.png` 이미지를 서버에 요청하고 캐싱하는 구체적인 라이프사이클 흐름이다.

1. **요청 및 캐시 검사:** 브라우저가 서버에 패킷을 쏘기 전, 자기 컴퓨터 내부에 `logo.png`가 있는지 확인한다.
2. **만료 검사 (유통기한 확인):** 캐시에 파일이 있다면 신선도(유효기간)를 검사한다. 서버는 최초 응답 시 헤더에 "이 파일은 10분 동안만 유효하다" 같은 **유통기한**을 적어 보낸다. 기한 내라면 서버에 요청을 아예 안 보내고 로컬 파일을 쓴다.
3. **조건부 요청 (If-Modified-Since):** 유통기한이 지났다면 파일이 바뀌었을지 모르니 서버에 물어봐야 한다. 무작정 새 파일을 달라고 하는 게 아니라, 효율을 위해 **조건부 요청**을 보낸다.
    - **Last-Modified:** 최초 응답 시 서버가 헤더에 `Last-Modified: <파일의 최종 수정 시간>`을 담아 보낸다.
    - 브라우저는 유통기한이 지나면 요청 헤더에 `If-Modified-Since: <저장해둔 최종 수정 시간>`을 적어서 서버에 던진다. *"내가 이 시간에 받아둔 파일이 있는데, 그 이후로 서버에서 바뀐 적 있어?"* 라고 물어보는 것이다.
4. **서버의 판단 및 응답:**
    - **바뀐 게 없다면:** 서버는 **`304 Not Modified`** 라는 상태 코드를 반환한다. 이 응답은 무거운 데이터(Body)가 비어있어 매우 가볍다. 브라우저는 이를 받고 캐시의 유통기한을 새로 갱신해서 기존 로컬 파일을 그대로 쓴다.
    - **바뀌었다면:** 서버는 **`200 OK`** 상태 코드와 함께 새로 업데이트된 이미지 파일을 응답 Body에 실어서 보내고, 브라우저는 캐시를 새 파일로 교체한다.

## SWR(Stale-While-Revalidate)전략

`304 Not Modified` 응답이 아무리 가벼워도, 유통기한이 지나면 결국 서버랑 통신을 한 번 주고받아야 하므로 네트워크 왕복 시간(RTT)만큼의 지연(대기 시간)이 무조건 발생한다. 이 찰나의 대기 시간조차 없애기 위해 나온 영리한 기법이 바로 **SWR 전략**이다.

"일단 낡은 데이터라도 화면에 바로 보여주고, 백그라운드에서 조용히 재검증해서 캐시를 업데이트해라”

#### `Cache-Control: max-age=1, stale-while-revalidate=60` 동작 메커니즘

- **0 ~ 1초 구간 (신선함):** `max-age=1` 규칙에 따라 캐시가 아주 신선한 상태다. 브라우저는 서버에 묻지도 않고 캐시 데이터를 즉시 반환한다.
- **1초 ~ 61초 구간 (SWR 작동):** 유통기한(1초)은 지났지만 SWR 허용 기간(60초) 이내다. 이때 사용자가 페이지에 진입하면, 브라우저는 **일단 보관 중이던 낡은 캐시 데이터(Stale)를 화면에 0초 만에 바로 보여준다.** 덕분에 사용자는 대기 시간을 전혀 느끼지 못한다.
    - **그와 동시에:** 브라우저는 백그라운드(화면 뒤편)에서 서버로 "새 데이터 있냐?" 하고 재검증 요청을 보낸다. 만약 서버에 새 데이터가 있다면 화면 깜빡임 없이 조용히 캐시를 최신 버전으로 업데이트해 둔다. (다음에 사용자가 방문하면 이 업데이트된 화면을 보게 된다.)
- **61초 이후 구간 (완전 만료):** SWR 허용 기간마저 지나버렸기 때문에, 이때 요청이 오면 기존 방식대로 서버에 재검증 요청을 보내고 응답이 올 때까지 화면이 대기한다.

### ex) SNS 피드

데이터가 실시간으로 100% 완벽히 정확할 필요는 없지만, 사용자에게 끊김 없이 무조건 빠른 응답을 보여주는 게 최우선인 서비스(인스타그램, 트위터 피드 등)에서 유용하다.

나중에 프론트엔드 진영에서 **SWR 라이브러리**나 TanStack Query (React Query)를 쓰게 되면 내부적으로 다 SWR 전략을 기반으로 굴러간다. 이때 설정하는 `staleTime`(데이터가 신선하다고 인정하는 시간)과 `gcTime/cacheTime`(메모리에 보관할 캐시 유지 시간) 개념이 바로 이 이론에서 파생된 옵션들이다.

| **캐시 종류 / 전략** | **관리 주체** | **핵심 목적 및 효과** | **실무 키워드** |
| --- | --- | --- | --- |
| **Private Cache** | 사용자 브라우저 | 나만을 위한 웹 자원 임시 저장, 트래픽 비용 절감 | 로컬 스토리지, HTTP 캐시 |
| **Shared Cache** | 네트워크 / CDN | 다수를 위한 공통 자원 중계, 전 세계 초고속 전송 | AWS CloudFront, Edge |
| **Managed Cache** | 백엔드 개발자 | 무거운 DB 조회 최소화, 비즈니스 연산 결과 저장 | **Redis**, Caffeine, 캐시 무효화 |
| **SWR 전략** | 프론트엔드 아키텍처 | RTT 지연 시간 0으로 제거, 사용자 경험(UX) 극대화 | TanStack Query, `staleTime` |
