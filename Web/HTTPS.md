## 키의 두가지 종류 : 대칭키 vs 비대칭키

#### 대칭키

암호화할 때와 복호화(암호를 풀 때)할 때 **동일한 하나의 키**를 사용하는 방식이다.

장점 : 컴퓨터가 계산하기에 아주 단순해서 **속도가 매우 빠르다.**

단점 : 상대방과 통신을 하려면 이 대칭키를 어떻게든 전달해 줘야 하는데, 네트워크로 키를 보내는 과정에서 해커가 가로채면 암호화가 소용이 없어진다.

#### 비대칭키

공개키와 비밀키라는 서로 다른 한 쌍의 열쇠를 사용하는 방식이다. 두 키는 수학적으로 긴밀하게 연결되어 있다.

- 공개키 : 전 세계 누구에게나 보여줘도 되는 열려있는 자물쇠이며 주로 데이터를 암호화할 때 사용한다.
- 비밀키 : 단 하나의 키이다. 공개키로 잠근 데이터를 풀 때(복호화) 사용한다.

장점 : 공개키를 대놓고 전달해도 되기 때문에 키 교환 과정이 안전하다.

단점 : 대칭키에 비해 속도가 상대적으로 많이 느리다.

HTTPS : 느리지만 안전한 비대칭키로 딱 한 번만 대칭키를 안전하게 공유하고 그 이후의 진짜 데이터 통신은 대빠른 대칭키로 주고받자
## 서버의 신원 확인과 CA인증서 발급

기존 HTTP는 데이터를 평문으로 보내서 중간 다 탈취되는 스니핑 공격에 취약했다. HTTPS는 암호화를 기본으로 하는데 통신을 하기 전에 브라우저가 지금 접속하려는 사이트가 위장 가짜 사이트가 아니고 진짜 사이트인지 검증하는 과정이 필요하다. 이때 신분증을 끊어주는 공인 기관을 CA라고 한다.

#### 서버 관리자가 신분증(인증서)를 발급받는 과정

1. 키 쌍 생성 : 서버 관리자는 서버 공개키와 서버 비밀키 한 쌍을 만들어 비밀키는 서버 내부에 숨긴다.
2. CSR(인증서 발급 신청서)작성 : “우리는 진짜 00서버이고 이게 우리 서버 공개키야”라는 정보가 담긴 신청서를 작성하고 위조 방지를 위해 서버의 비밀키로 서명한다.
3. CA에 CSR제출 : Let’s Encrypt나 GoDaddy같은 공인 CA기관에 신청서를 보낸다. CA는 진짜 서버가 맞는지 확인하기 위해 “너네 웹 서버 DNS 레코드에 이 값 추가해봐”같은 방식으로 도메인 소유권을 엄격하게 검증한다.
4. 인증서 발급 : 검증이 끝나면 CA는 “이 공개키는 진짜 00서버의 것이 맞음을 보증함”이라는 내용이 담긴 디지털 인증서(신분증)을 발급해준다. 이때 CA는 이 신분증이 진짜임을 보장하기 위해 CA 자신의 비밀키로 서명을 해서 넘겨준다.
5. 인증서 설치 : 서버 관리자는 이 발급받은 인증서를 웹서버에 장착한다.

#### TLS 핸드셰이크와 대칭키 교환

이제 클라이언트가 서버에 접속해서 진짜 안전한 통신을 하기 위해 준비하는 과정이다. 이 과정을 TLS핸드셰이크라고 부른다.

1. 클라이언트 요청 & 서버 응답
    
    브라우저가 서버에 “연결하자”하고 요청을 보내면 서버는 공인 기관에서 받은 인증서를 보내준다.
    
2. 클라이언트의 인증서 검증
    - 브라우저는 이 인증서가 가짜가 아닌지 확인해야 한다. 현대 브라우저 안에는 전 세계 신뢰할 수 있는 CA목록과 그 CA들의 공개키가 내장되어 있다.
    - 브라우저는 내장된 CA의 공개키를 꺼내서 서버가 준 인증서의 서명을 복호화한다. 서명이 성공적으로 풀리면 인증서를 보고 서버를 신뢰하게 된다. 동시에 인증서 안에 있는 서버의 공개키를 획득하게 된다.
3. 대칭키 생성 및 안전한 전달
    - 이제 본격적인 대화에 쓸 임시 비밀번호(대칭키)를 브라우저가 스스로 만든다.
    - 이 대칭키를 그냥 인터넷으로 보내면 해커가 가로채니까 브라우저는 아까 인증서에서 꺼낸 서버의 공개키로 암호화해서 전송한다.
4. 데이터 복호화 및 대칭키 공유 완료
    
    암호화된 대칭키를 받은 서버는 오직 자신만의 비밀키로 복호화해서 대칭키를 꺼낸다. → 오직 브라우저와 서버 둘만 같은 대칭키가 안전하게 공유됐다.
    
5. 대칭키를 사용한 고속 암호화 통신
    
    키 교환이라는 어렵고 느린 작업이 끝났으니 이제부터 주고받는 로그인 정보, 마이페이지 등 모든 HTTP패킷은 이 대칭키로 암호화/복호화하면서 통신을 진행한다.
    

## HTTP vs HTTPS

HTTP : 아무런 암호화 장치 없이 데이터를 평문 그대로 전송하는 규약이다. 해커가 중간에 가로채서 열어보면 비밀번호나 개인정보가 생으로 노출된다.

HTTPS : HTTP에 안전함을 더한 프로토콜이다. CA를 통해 서로의 신원을 엄격하게 확인하고 비대칭키 방식으로 대칭키를 안전하게 나눠 가진 뒤 둘 만 아는 암호키로 변환된 데이터를 주고받기 때문에 안전하다.

## 정리

HTTPS는 보안성과 성능을 모두 잡기 위해 대칭키와 비대칭키 방식을 조합하여 동작한다.

먼저 CA의 비밀키로 서명된 인증서를 브라우저가 내장된 CA공개키로 검증하여 서버 신원을 확인하고

서버의 공개키를 얻는다. 이후 속도가 느린 비대칭키 방식으로 이번 세션에 사용할 대칭키를 안전하게 암호화하여 교환한 뒤 실제 데이터 통신은 속도가 빠른 대칭키를 활용해 초고속 암호화 통신을 수행한다.
