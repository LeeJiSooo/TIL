## HTTP Request Method : 명확한 ‘동사’의 역할

클라이언트가 서버에게 무작정 요청한다는 것만 보내면 서버는 무엇을 해야할지 알 수 없다. HTTP Method는 자원에 대해 내가 지금 어떤 행동을 하려는 것인지 의도를 명확히 밝히는 규칙이자 약속이다.

- GET(조회) : “자원 주세요” - > 데이터 읽기(Read)
- POST(생성) : “새 자원 구해서 보냅니다” → 데이터 쓰기(Create)
- PUT(전체 수정) : “새것으로 아예 갈아 끼워 주세요” → 데이터 전체 교체(Update)
- PATCH(부분 수정) : “이 부분만 수정해 주세요” → 데이터 일부 변경(Update)
- DELETE(삭제) : “이 자원 없애 주세요” → 데이터 삭제(Delete)

URL에는 오직 자원(명사)만 표현하고 그 자원에 대한 행위(동사)는 HTTP Method로 표현한다.

## 사용 이유

Request Method를 사용하면 요청 본문 열어보기 전에 GET, POST메서드만 갖고 어떤 작업인지 빠르게 판단할 수 있다.  → 서버 작업 효율 높이고 요청의 의도를 명확하게 정의해서 문제를 줄인다.

→/users라는 API가 있다면 유저 자원에 GET으로 요청을 하면 사용자 목록을 조회하는 것, POST요청을 보내면 새로운 사용자를 생성하는 것 → 약속된 방식으로 API를 설계하면 협업이 수월하고 명확하다.

## 멱등성

연산을 한 번 수행하든 여러 번 수행하든 상관없이 서버의 최종 상태가 항상 똑같이 유지되는 성질이다.

왜 네트워크에서 멱등성이 중요할까?

모바일 환경이나 네트워크는 언제든 끊길 수 있다. 클라이언트가 요청을 보냈는데 터널에 진입해 응답을 못 받았다면 클라이언트는 “내 요청이 서버에 가다 깨진 건지, 서버는 처리 했는데 돌아오는 길에 깨진 건지” 알 방법이 없다.

- 멱등한 요청(GET, PUT) : 서버 상태가 망가지지 않으므로 실패 했을 때 마음 편히 재시도를 보낼 수 있다.
- 멱등하지 않은 요청(POST) : 함부로 다시 보냈다간 결제가 중복 처리되거나 똑같은 글이 두 번 올라가는 참사가 발생한다.

## HTTP Request Method

| **Method** | **CRUD** | **멱등성 (Idempotent)** | **안전성 (Safe)** | **실무 핵심 포인트 및 주의점** |
| --- | --- | --- | --- | --- |
| **GET** | **R**ead | **O** | **O** | **조회 전용.** 서버 데이터를 변경하면 안 됨. 검색엔진 봇이 마구 긁어가도 안전해야 함. |
| **POST** | **C**reate | **X** | **X** | **자원 생성.** 보낼 때마다 데이터가 계속 추가되므로 중복 제출(Retry)을 가장 주의해야 함. |
| **PUT** | **U**pdate | **O** | **X** | **자원 전체 교체.** 누락된 필드가 있으면 기존 데이터가 지워질(null) 수 있으므로 전체 Row를 보내야 함. |
| **PATCH** | **U**pdate | **X** (가변) | **X** | **자원 일부 수정.** 단, '좋아요 +1' 같은 상대적인 연산을 보낼 경우 실행 횟수마다 결과가 바뀌므로 멱등하지 않을 수 있음. |
| **DELETE** | **D**elete | **O** | **X** | **자원 삭제.** 첫 요청에만 200, 이후엔 404를 뱉더라도 '삭제된 상태 자체'는 동일하므로 멱등함. |

안전성? 호출해도 서버의 자원 상태를 절대 변경하지 않는(읽기 전용)성질

#### PUT vs PATCH

PUT : 리소스의 전체 교체이다. 만약 {name : “토르” , age : 11}에서 나이만 바꾸고 싶다고 PUT으로 {age:12}만 보내면 이름 필드가 통째로 날아가거나 에러가 날 수 있따.

PATCH : 리소스의 부분 수정이다. 변경을 원하는 {agd: 12}만 보내면 다른 필드는 안전하게 유지된다.

#### 실무에서의 데이터 삭제 : Soft Delete

실제 현업에서는 DELETE요청이 들어왔다고 해서 데이터베이스에서 데이터를 완전히 지워버리는 Hard Delete를 거의 하지 않는다.

사용자가 실수로 지웠다고 복구를 요청하거나 탈퇴한 회원의 데이터를 분석해야 하는 등 데이터 보존이 필수적이기 때문이다.

테이블에 is_deleted 또는 delete_at같은 컬럼을 두고 DELETE요청이 오면 이 값을 true나 현재 시간으로 변경한다. 조회할 때는 이 삭제 플래그가 없는 데이터만 걸러서 보여준다.

## HTTP 상태코드

상태 코드는 클라이언트가 보낸 요청에 대해 서버가 어떻게 처리를 완료했는지 직관적으로 알려주는 3자리 숫자 약속이다. 

상태코드의 첫 번째 자릿수는 응답의 분류를 나타낸다.

- **1xx (Informational):** "요청을 받았으니 **처리 중**이야. 조금만 기다려봐." (실무에서 자주 보기는 힘듦)
- **2xx (Successful):** "요청을 정상적으로 **성공**적으로 처리했어."
- **3xx (Redirection):** "네가 찾는 자원이 다른 곳으로 이사 갔어. **그쪽 주소로 다시 가봐.**"
- **4xx (Client Error):** "**클라이언트 너한테 문제가 있어.** 요청을 잘못 보냈으니 확인해봐."
- **5xx (Server Error):** "너의 요청은 멀쩡한데, **우리(서버) 창고에 불이 났어.** 미안해."

#### 401 Unauthorized(인증 실패)

클라이언트가 누구인지 증명하지 못했을 때 발생한다. 로그인이 필요한 페이지에 비로그인 상태로 접근하거나 토큰을 누락하고 요청을 보냈을 때 발생한다.

#### 403 Forbidden(인가 실패)

신원은 확인(로그인 성공)되었지만 해당 자원에 접근할 수 있는 권한이 없을 때 발생한다. 일반 회원이 ‘관리자 페이지’를 조회하려고 요청할 때 발생한다.

#### 404 Not Found(찾을 수 없음)

요청한 URL에 해당하는 리소스가 서버에 존재하지 않을 때 발생한다. 주소를 잘못 타이핑했거나 이미 삭제된 데이터를 조회하려고 할 때 주로 발생한다.

#### 500 Internal Server Error(서버 내부 오류)

서버가 요청을 처리하는 과정에서 예상치 못한 예외(NullPointerException, DB연결 끊김 등)나 버그가 발생하여 정상적인 응답을 주지 못하는 상태이다.

#### 개발자가 상태 코드를 제대로 써야 하는 이유

- 프론트엔드와 백엔드의 원활한 소통 및 흐름 제어

로그인 api를 호출했을 때 프론트엔드 개발자는 서버가 보내주는 상태 코드를 이정표 삼아 화면을 분기 처리한다.

200을 받으면 로그인 성공! → 메인 페이지로 이동 및 JWT토큰 저장

401을 받으면 인증 실패 → 화면에 아이디나 비밀번호를 다시 확인해주세요 경고창 노출

- 디버깅 효율 극대화

에러가 터졌을 때 앞자리 숫자만 봐도 누가 코드를 고쳐야 하는지 알 수 있다.

4xx대 에러 발생 : 프론트엔드 책임

5xx대 에러 발생 : 백엔드 책임

## 정리

- HTTP Request Method는 자원에 대한 행위를 명확히 규정하는 동사 역할을 한다. 특히 네트워크 장애 시 재시도 가능 여부를 결정하는 멱등성이 중요한데 GET, PUT, DELETE는 여러 번 호출해도 최종 상태가 같아 멱등성을 보장하지만 새로운 리소스를 생성하는 POST나 상태 변화를 줄 수 있는 PATCH는 멱등하지 않는다.
- HTTP 상태 코드는 서버의 처리 결과를 나타내는 3자리 숫자로, 개발자 간의 소통과 디버깅 효율을 높여준다. 특히 4xx에러는 클라이언트의 잘못된 요청을, 5xx에러는 서버 내부 로직 에러를 의미하여 일차적인 책임 소재를 파악하게 해준다. 또한 신분증이 없어 누구인지 모르는 상태인 401(인증실패)와 누군지는 알지만 접근 권한이 없는 403(인가 실패)을 명확히 구분하여 API를 설계하는 것이 중요하다.
