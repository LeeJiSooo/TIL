## 3-Way Handshake의 정의와 필요성

TCP/IP 네트워크 프로토콜에서 본격적인 데이터 전송 전에 두 호스트 간의 연결을 초기화(수립)하는 3단계 절차이다.

- **연결 지향성:** 데이터를 보내기 전에 보내는 쪽과 받는 쪽이 서로 통신할 준비가 되었는지 확인하고 공식적인 연결을 수립한다. 마치 전화를 걸었을 때 상대방이 수화기를 들고 서로 "여보세요" 확인한 뒤 통화를 시작하는 것과 같다.
- **신뢰성 보장:** 이 연결 과정을 통해 양쪽 기기가 준비 상태를 확답받으므로, 통신 중에 데이터가 중간에 유실되거나 순서가 뒤바뀌지 않고 목적지에 정확하게 도달하는 것을 보장할 수 있다.

## 동작 방식

### 1단계: SYN (클라이언트 ➔ 서버)

- **패킷 정보:** `SYN (SEQ = 1000)`
- **상황:** 클라이언트(브라우저)가 서버에 접속을 시도하는 첫 번째 단계로, **SYN(Synchronize) 패킷**을 보낸다.
- **SYN 플래그:** TCP 헤더에는 여러 제어 비트(플래그)가 있다. 이 패킷이 연결을 요청하는 용도임을 명시하기 위해 **SYN 플래그 비트를 1**로 설정한다.
- **시퀀스 넘버 (SEQ):** TCP의 순서 보장 기능을 위해 세그먼트에 고유 번호를 붙인다. "나 이제부터 데이터 보낼 건데 1000번부터 시작할게"라는 의미로 초기 시퀀스 번호(ISN)를 임의의 숫자로 생성해서 보낸다.
- **상태 변화:** 클라이언트의 연결 상태가 `CLOSED`에서 **`SYN_SENT`**(응답 기다리는 중)로 바뀐다.

### 2단계: SYN-ACK (서버 ➔ 클라이언트)

- **패킷 정보:** `SYN-ACK (SEQ = 2000, ACK = 1001)`
- **상황:** 서버가 SYN 패킷을 받고, 접속을 허용할 수 있는 상태인지 확인한 뒤 문제가 없으면 응답 패킷을 보낸다.
- **SYN + ACK 플래그:** "나도 연결할 준비가 됐다. 내 데이터는 2000번부터 시작할게"라는 의미로 **SYN 플래그를 1**로 세팅하고 자신의 초기 시퀀스 번호(2000)를 담는다. 동시에 클라이언트의 요청을 잘 받았다는 의미로 **ACK 플래그도 1**로 설정한다.
- **승인 번호 (ACK Number):** "네가 보낸 1000번 잘 받았고, 난 이제 너한테 1001번 데이터를 받을 준비가 됐어"라는 의미를 담는다. 즉, **다음에 받아야 할 데이터 번호**이며 `클라이언트의 SEQ + 1`이 된다.
- **상태 변화:** 서버의 연결 상태가 `LISTEN`(대기 중)에서 **`SYN_RCVD`**(클라이언트의 최종 확답을 기다리는 중)로 바뀐다.

### 3단계: ACK (클라이언트 ➔ 서버)

- **패킷 정보:** `ACK (SEQ = 1001, ACK = 2001)`
- **상황:** 클라이언트가 SYN-ACK 패킷을 받으면 통신이 가능하다는 것을 인지하고, 마지막으로 "응답 잘 받았으니 이제 통신 시작하자"라는 확인 패킷을 보낸다.
- **ACK 플래그:** **1로 설정**해서 보낸다. 이제 연결 수립이 끝났으므로 더 이상 새로운 연결 요청(SYN) 플래그는 없다.
- **시퀀스 넘버 (SEQ):** 2단계에서 서버가 1001번을 기대한다고 알려줬으므로 `SEQ = 1001`로 맞춰서 보낸다.
- **승인 번호 (ACK Number):** "네가 보낸 SYN(SEQ=2000) 잘 받았고, 난 이제 너한테 2001번 데이터를 받을 준비가 됐어"라는 의미로 `서버의 SEQ + 1`인 2001로 설정해서 보낸다.
- **상태 변화:** 이 시점부터 클라이언트와 서버 모두 연결 상태가 `ESTABLISHED`로 변하며, 비로소 HTTP 메시지(데이터)를 마음껏 주고받을 수 있게 된다.

## SYN Flooding 공격

3-Way Handshake의 구조적 취약점을 노린 대표적인 디도스(DDoS) 공격이다.

- 공격 방식: 공격자가 대량의 SYN 요청만 마구 보내고, 서버가 보낸 SYN-ACK 응답에 대해 마지막 ACK를 보내지 않고 잠수를 타버리는 수법이다.
- 결과: 서버는 마지막 확인 응답(ACK)을 기다리며 연결을 완전히 맺지도, 끊지도 못하는 반쯤 열린 상태가 된다. 이런 유령 요청이 수만, 수십만 개 쌓이면 서버의 자원(Backlog Queue)이 통째로 소진되어 정상적인 사용자의 요청을 처리하지 못하게 된다.

새로운 서버를 세팅하고 애플리케이션(스프링 부트)을 정상적으로 배포했는데, 외부에서 전혀 접속이 안 되는 상황이 실무에서 정말 자주 발생한다.

이때 가장 먼저 의심해야 하는 것이 **방화벽(Firewall) 설정**이다. 클라우드(AWS Security Group 등)나 OS 방화벽이 특정 포트로 들어오는 TCP 요청, 즉 **첫 번째 단계인 SYN 패킷 자체를 차단**하고 있을 확률이 높다.
