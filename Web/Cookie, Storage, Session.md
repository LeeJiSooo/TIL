## 쿠키의 정의와 필요성

HTTP의 무상태성(Stateless)이라는 약점을 보완하기 위해, 서버가 웹 브라우저(클라이언트)에 저장해 두는 작은 기록 정보 파일

HTTP 통신은 상태가 없어서 방금 로그인 요청을 성공했어도 다음 마이페이지 요청이 들어오면 서버는 이 사용자가 누구인지 기억하지 못한다. 이 불편함을 해결하는 게 쿠키다.

동작 과정

- 클라이언트가 요청을 보내면, 서버가 응답할 때 헤더에 **`Set-Cookie: ID=soo`** 같은 형태로 "이것 좀 기억해 둬" 하고 쿠키를 보낸다.
- 브라우저는 이 쿠키를 자신의 로컬 저장 공간에 보관한다.
- 이후 브라우저는 **해당 서버(URL)에 요청을 보낼 때마다 자동으로 헤더에 이 쿠키를 실어서 함께 전송**한다.
- 서버는 들어온 쿠키를 읽고 사용자를 식별해 로그인 상태를 유지하거나 맞춤 서비스를 제공한다.

### 퍼스트파티 vs 서드파티 쿠키

#### 퍼스트파티 쿠키

현재 내가 방문 중인 웹사이트의 도메인과 쿠키에 설정된 도메인이 일치하는 쿠키다.

`example.com`에 접속했을 때, `example.com` 서버가 직접 발행한 쿠키다. (로그인 상태 유지, 장바구니 기억 등에 사용)

#### 서드파티 쿠키

내가 방문한 사이트의 도메인과 다른, 제3의 도메인(서버)에서 발행한 쿠키다.

`news.com`(뉴스 사이트)에 방문했는데, 그 페이지 안에 구글 광고 배너(`google-ads.com`)가 붙어 있는 경우다. 이때 구글 광고 서버가 내 브라우저에 몰래 쿠키를 심을 수 있는데, 이게 바로 서드파티 쿠키다.

**사용자 행동 추적 및 맞춤형 광고 →** 사용자가 `news.com`을 떠나 스포츠, IT 등 다른 사이트를 돌아다녀도, 그곳에 구글 광고가 있다면 구글은 이 서드파티 쿠키를 읽어 "아, 이 사람이 요즘 이런 거에 관심 있구나" 하고 정보를 수집해서 타겟 광고를 띄운다.

### 쿠키의 근본적인 한계와 단점

- **보안 취약성:** 쿠키는 파일 형태로 **클라이언트(브라우저)에 직접 저장**된다. 만약 여기에 비밀번호나 민감한 개인정보를 그대로 담아두면 해커에게 쉽게 탈취(XSS 공격 등)당할 수 있다.
    - *해결책:* 민감한 정보는 서버 측 세션에 저장하고, 쿠키에는 암호화된 난수(Session ID 등)만 담아야 한다.
- **브라우저 간 공유 불가:** 쿠키는 특정 브라우저에 종속된다. 집 컴퓨터 크롬에서 로그인을 해놨어도, 회사 컴퓨터 사파리로 접속하면 다시 로그인을 해야 하는 본질적인 한계가 있다.
- **네트워크 부하 증가:** 쿠키는 조건이 맞으면 **모든 HTTP 요청마다 자동으로 전송**된다. 만약 쿠키에 4KB의 데이터를 꽉 채워두면, 아주 가벼운 CSS 파일 하나를 요청할 때도 무조건 4KB의 쿠키 데이터가 같이 날아가므로 네트워크 대역폭 낭비가 생긴다.
    - *해결책:* 서버가 굳이 알 필요가 없는 단순 브라우저용 데이터는 쿠키가 아니라 로컬 스토리지(Local Storage)나 세션 스토리지(Session Storage)에 나눠 담아야 한다.

### 쿠키의 보안 속성

쿠키의 보안 취약점을 막고 안전하게 통신하기 위해 개발자가 반드시 설정해야 하는 핵심 속성들이다.

#### Secure (HTTPS 전송 강제)

- **쿠키가 오직 HTTPS(암호화 통신) 프로토콜을 통해서만 전송**되도록 제한하는 속성이다. 일반 HTTP 연결에서는 브라우저가 이 쿠키를 절대 서버로 보내지 않는다.
- 중간에서 패킷을 가로채는 MITM(Man-In-The-Middle, 중간자 공격)을 방지하기 위해 사용한다.
- `Set-Cookie: sessionid=xyz; Secure`

#### SameSite (CSRF 공격 방어 정책)

내가 의도하지 않았는데 다른 사이트에서 악의적인 요청을 날려 내 계정을 탈취하는 **CSRF(사이트 간 요청 위조) 공격**을 막기 위해, "다른 도메인에서 우리 사이트로 요청을 보낼 때 쿠키를 실어 보낼 것인가"를 결정하는 정책이다. 옵션은 크게 3가지가 있다.

- 1) SameSite = Strict
    - 가장 엄격한 통제. **오직 퍼스트파티 상황에서만 쿠키가 전송**된다. 타 도메인에서 링크를 타고 들어오거나 이메일에 적힌 주소를 클릭해 들어오는 경우, 무조건 쿠키를 전송하지 않는다. (보안은 최고지만 사용자가 링크 타고 올 때마다 로그인이 풀려있어 불편함)
- 2) SameSite = Lax (최신 크롬 브라우저의 기본값)
    - 보안과 사용자 편의성의 균형을 맞춘 정책이다. 기본적으로 서드파티 쿠키 전송을 막지만, **사용자가 직접 링크를 클릭하여 이동하는 안전한 상황(주로 GET 요청, `<a href>`, `<link rel="prerender">` 등)에는 예외적으로 쿠키 전송을 허용**한다.
    - 조건: 1)도메인이 완전히 동일하거나, 2)안전한 GET 요청으로 페이지를 이동할 때 동작한다.
    - `Set-Cookie: sessionid=xyz; SameSite=Lax`
- 3) SameSite = None
    - 과거 크롬의 기본값이었으며, **어떤 도메인이든 상관없이 모든 서드파티 상황에서 쿠키를 전송**할 수 있게 다 열어두는 정책이다. CSRF 공격에 가장 취약하다.
    - 이 옵션을 쓰려면 **반드시 앞서 배운 `Secure` 속성(HTTPS)이 세트로 설정되어 있어야만 브라우저가 인정**해 준다.

## Storage

서버로 보낼 필요 없이, 오직 클라이언트(브라우저)에만 안전하게 데이터를 저장하기 위해 만들어진 브라우저 전용 저장 공간

예전에는 브라우저에 데이터를 저장할 수 있는 기술이 오직 쿠키 밖에 없었다.

그래서 서버가 굳이 알 필요 없는 데이터(ex.사용자가 켜놓은 다크모드 설정, 화면 언어 설정 등)까지 쿠기에 다 때려 넣곤 했따.

이렇게 하면 HTTP 요청을 보낼 때마다 그 불필요한 데이터들이 네트워크를 타고 서버로 계속 오고 가기 때문에 네트워크 대역폭 낭비와 성능 저하를 일으켰다.

웹 스토리지

- 스토리지에 저장된 데이터는 쿠키처럼 서버로 자동으로 전송되지 않고 오직 브라우저에만 머문다. 넉분에 네트워크 부하가 완전히 사라진다.
- 쿠키는 4KB 정도만 저장할 수 있지만 웹 스토리지는 브라우저마다 다르지만 보통 5MB에서 10MB까지 훨신 여유롭게 데이터를 담을 수 있다.

### 로컬 스토리지 vs 세션 스토리지

웹 스토리지는 데이터의 생명주기(유지 기간)에 따라 두 가지로 명확하게 나뉜다.

#### 로컬 스토리지

- 사용자가 브라우저 탭을 닫거나, 크롬을 완전히 종료하거나, 심지어 컴퓨터를 껐다 켜도 데이터가 사라지지 않고 그대로 유지된다.
- 사용자가 직접 브라우저 쿠키/캐시를 지우거나, 개발자가 자바스크립트 코드(`localStorage.clear()`)로 명시적으로 삭제하기 전까지는 평생 남아있다.
    - ex. 다크 테마 설정값, “이 창 다시 보지 않기” 체크 여부, 다국어 설정

#### 세선 스토리지

- 데이터가 '세션(Session)' 동안만 유지된다. 여기서 말하는 웹의 세션은 **브라우저의 탭이나 창 하나**를 의미한다.
- 사용자가 해당 **브라우저 탭이나 창을 닫는 순간** 세션 스토리지에 저장된 데이터는 흔적도 없이 깨끗하게 사라진다.
- 같은 웹사이트에 접속하더라도, **다른 탭이나 다른 창에서 열면 서로 완전히 다른 독립된 세션 스토리지 영역**을 가지게 된다. 다만, 같은 탭 안에서 새로고침을 하는 것으로는 데이터가 날아가지 않는다.
    - ex. 쇼핑몰 주문서나 입력 폼 작성 중, 사용자가 실수로 새로고침을 했을 때 입력 데이터 날림 방지용 복구 데이터

## Session

사용자의 민감한 상태 정보를 클라이언트가 아닌 서버 측 안전한 저장소에 보관하고 관리하는 기술

“쿠키와 세션은 경쟁 관계가 아니라 완벽한 협력 관계”이다.

중요한 알맹이(유저정보)는 서버(세션)에 숨겨두고 사용자는 그 알맹이를 찾을 수 있는 열쇠(세션ID)만 쿠키에 담아 매 요청마다 서버에 보여주는 식이다.

### 세션 기반 로그인 흐름

1. 로그인 요청 (Client → Server): 사용자가 ID/PW를 입력해서 서버로 던진다.
2. 세션 생성 및 저장 (Server): 서버가 DB를 확인해서 인증에 성공하면, 단 하나뿐인 무작위 문자열인 고유 세션 ID(Session ID)를 만든다. 그리고 서버 측 세션 저장소(Redis 등)에 `Key = 세션ID`, `Value = {이메일, 닉네임, 회원권한}` 형태로 저장한다.
3. 세션 ID 반환 (Server → Client): 서버는 응답할 때 헤더에 `Set-Cookie: SESSION_ID=xyz123` 형태로 세션 ID만 실어서 브라우저에게 보낸다.
4. 쿠키 저장 (Client): 브라우저는 응답을 받고, 내 컴퓨터의 쿠키 저장소에 `SESSION_ID=xyz123`을 저장한다.
5. 이후 요청 시 자동 전송 (Client → Server): 사용자가 마이페이지 버튼을 누르면, 브라우저가 알아서 요청 헤더에 `Cookie: SESSION_ID=xyz123`을 실어서 서버로 보낸다.
6. 세션 조회 및 인증 (Server): 서버는 요청 헤더의 세션 ID(`xyz123`)를 읽고, 즉시 세션 저장소로 가서 "이 ID랑 매칭되는 서류 유저정보 꺼내봐" 하고 뒤진다. 정보가 확인되면 로그인된 사용자로 인정하고 마이페이지 데이터를 응답한다. 만약 만료되었거나 없는 ID라면 로그인 페이지로 리다이렉트 시킨다.
