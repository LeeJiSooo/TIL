- `@Id`와 `@GeneratedValue`
    - DB테이블에 PK가 있듯 JPA 엔티티에도 식별자가 필요하며, 이를 매핑하는 게 `@Id`이다.
    - 개발자가 ID값을 직접 지정하지 않고, DB의 `AUTO_INCREMENT` 처럼 알아서 값을 생성하도록 돕는 어노테이션이 `@GeneratedValue` 이다.
- 기본키 생성 전략이란?
    - JPA에게 “기본키 알아서 만들어줘”라고 했을 때 숫자를 1씩 증가시키는 방식, 별도의 키 생성용 테이블을 쓰는 방식 등을 결정하는 전략이다.
    - 단순히 기본키를 어떻게 만들 것인가에 따라 영속성 컨텍스트의 핵심 기능(쓰기 지연)활용 여부가 달라지며, 이는 곧 심각한 성능 차이로 이어진다.
- JPA의 추상화
    - 테이블 설계 시 MySQL은 `AUTO_INCREMENT`, Oracle은 `SEQUENCE`를 주로 쓴다.
    - JPA는 다양한 데이터베이스를 지원해야 하므로, DB마다 조금씩 다른 키 생성 방식을 추상화하여 4가지 전략(`TABLE`, `SEQUENCE`, `IDENTITY`, `AUTO`)으로 제공한다.

## 가져가야 할 핵심

1. 기본 키 생성 전략들의 동작 원리와 물리적인 차이점을 명확히 이해한다.
2. **`IDENTITY` 전략**이 영속성 컨텍스트의 핵심인 **'쓰기 지연(Write-Behind)'에 어떤 치명적인 영향**을 미치는지 파악한다.
3. **`SEQUENCE` 전략**의 **`allocationSize`** 옵션이 어떻게 네트워크 비용을 줄이고 성능을 최적화하는지 이해한다.
4. 분산 환경(MSA)에서 `UUID`를 PK로 쓰는 이유와 장단점, 그리고 최신 개선 체계(`UUID v7`, `TSID`)를 인지한다.

## 4가지 기본키 생성 전략

### 1. TABLE 전략(실무에서 안 씀)

- 어떤 DB를 쓴든 상관없이 시퀀스 동작을 흉내 내기 위해, 데이터베이스 내부에 키 생성 전용 테이블을 하나 만들어 두고 관리하는 방식
- MySQL처럼 시퀀스 오브젝트가 없는 DB에서도 종류 상관없이 시퀀스처럼 PK를 채번하고 싶을 때 사용한다.

```java
@Entity
@TableGenerator(
    name = "user_gen",
    table = "id_generator", // 키 생성 전용 테이블 이름
    pkColumnName = "entity_name",
    valueColumnName = "next_id",
    pkColumnValue = "user",
    allocationSize = 1
)
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.TABLE, generator = "user_gen")
    private Long id;
    private String email;
}
```

- `id_generator` 테이블 구조 예시

| **entity_name** | **next_id** |
| --- | --- |
| user | 1 |
| post | 1 |
- 동작 방식 : `user` 엔티티를 저장할 때 `entity_name`이 ‘user’인 로우를 찾아서 `next_id` 컬럼 값을 읽어와 PK로 쓰고, 다음 생성을 위해 그 값을 1 증가시켜 업데이트 해둔다.
- 장점 : 이식성이 좋다. 오라클, MySQL, PostgreSQL 등 모든 DB에서 똑같이 테이블을 만들어 쓸 수 있다.
- 단점 :
    - 엔티티 객체 하나를 DB에 밀어 넣으려면 아래 과정을 매번 거쳐야 한다.
        1. 키 생성 테이블을 조회해서 현재 값 가져오기(SELECT)
        2. 가져온 값을 1 증가시켜서 다시 키 생성 테이블에 반영(UPDATE)
        3. 실제 내 엔티티 데이터를 저장(INSERT)
    - 엔티티 하나 저장할 때마다 무조건 SELECT + UPDATE 쿼리가 한 쌍으로 추가 발생한다.
    - 여러 스레드가 동시에 회원가입을 시도하면 키 생성 테이블의 똑같은 로우를 동시에 업데이트 하려고 하므로 DB에 심각한 락이 걸려 성능이 안좋아진다.

### 2. SEQUENCE전략(오라클, PostgreSQL 등)

- DB 자체에서 순차적으로 숫자를 빠르고 안전하게 만들어주는 특별한 객체인 ‘시퀀스 오브젝트’를 사용하는 방식이다.

```java
@Entity
@SequenceGenerator(
    name = "post_seq",
    sequenceName = "post_seq", // DB에 있는 실제 시퀀스와 매핑
    allocationSize = 50        // 성능 최적화의 핵심 옵션
)
public class Post {
    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "post_seq")
    private Long id;
    private String title;
}
```

- 동작 방식 (`em.persist(post)` 호출 시):
    1. JPA 영속성 컨텍스트에 엔티티를 보관하려면 식별자(pk)가 필수적이다.
    2. 아직 DB에 저장하기 전이라 엔티티에 ID값이 없다.
    3. 이에 JPA는 먼저 DB로 가서 시퀀스에게 “다음 PK 값 뭐가 될지 알려줘”라고 요청한다.
    4. DB가 값(예 :1)을 주면, JPA는 이를 엔티티의 ID로 세팅한 다음 영속성 컨텍스트에 안정적으로 저장한다.
- 쓰기 지연과의 관계:
    - 이 과정에서 `INSERT`쿼리는 아직 DB로 날라가지 않았다. 오직 PK 값만 시퀀스에서 먼저 조회해 온 것이며, 실제`INSERT`쿼리는 영속성 컨텍스트 안의 ‘쓰기 지연 SQL 저장소’에 차곡차곡 쌓인다.
    - 이후 트랜잭션이 커밋될 때 쌓여있던 `INSERT` 쿼리들이 한 번에 DB로 방출된다.

#### 네트워크 비용을 줄이는 `allocationSize`

- 시퀀스 방식은 엔티티를 영속할 때마다 DB시퀀스를 조회해야 하므로, 만약 100명을 저장하면 DB를 100번 다녀와야 해서 성능이 떨어질 수 있다. 이를 최적화 하는 옵션이 allocationSize이다.
- 작동 원리(설정값이 50일 때):
    1. JPA가 처음 시퀀스 값을 달라고 DB에 요청한다.
    2. DB시퀀스는 값을 1만 올리는 게 아니라, 한번에 **`allocationSize(50)`만큼 증가**시켜서 시퀀스 번호를 1에서 바로 51로 점프시킨다.
    3. DB가 JPA에게 *"너 이제 메모리에서 1부터 50까지 알아서 마음대로 꺼내 써라"* 하고 허락을 준 셈이다.
    4. JPA는 이후 50번의 저장 동안은 **DB에 가지 않고 애플리케이션 메모리 상에서 1, 2, 3, 4... 순차적으로 ID를 할당**한다. (네트워크 통신 X)
    5. 50개를 다 쓰면(51번을 채번할 때) 다시 DB에 가서 다음 세트(51~100)를 받아온다.
- `allocationSize` 덕분에 DB 왕복 통신을 획기적으로 줄일 수 있으며, **쓰기 지연을 통한 최적화를 완벽하게 누릴 수 있는 전략**이다.

### 3. IDENTITY 전략 (MySQL, MariaDB 등)

- DB의 **`AUTO_INCREMENT`** 기능을 활용하는 방식이다. 데이터를 테이블에 넣을 때 PK 컬럼을 비워두고 `INSERT`하면, DB가 알아서 1, 2, 3... 순서대로 값을 채워준다. 설정이 간단하고 편리하다.

```java
@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String email;
}
```

- 한계 : 쓰기 지연과의 충돌
    - JPA는 영속성 컨텍스트에 엔티티를 저장(`persist`)할 때 **반드시 PK 값이 있어야 관리(식별)가 가능**하다.
    - 그런데 `IDENTITY` 전략은 **PK 값을 오직 DB가 직접 데이터를 집어넣어야만 만들어준다.** 쿼리를 날리기 전까지는 PK 값을 알 수 없다.
- JPA의 예외적 동작 (즉시 쿼리 실행):
    - 이 문제를 해결하기 위해 JPA는 `IDENTITY` 전략에 한해서만 특별 예외 동작을 취한다.
    - `em.persist(user)`를 호출하는 순간, 쿼리를 모아두는 쓰기 지연을 하지 않고 **그 즉시 DB에 `INSERT` 쿼리를 날려버린다.**
    - 쿼리가 날아간 직후, JDBC의 `statement.getGeneratedKeys()`라는 기능을 통해 DB가 방금 생성한 PK 값을  빼와서 영속성 컨텍스트에 엔티티를 등록한다.
- `em.persist()` 시점에 즉시 쿼리가 나가야 하므로, **모아두었다가 한 번에 쏘는 '쓰기 지연을 통한 최적화(Batch INSERT)'를 전혀 사용할 수 없다.**

#### 그럼 MySQL + JPA를 쓰면 대량(Bulk)저장은 망한건가?

- **아니다.** 만약 10,000명의 회원 데이터를 엑셀 업로드로 한 번에 저장하는 기능을 만든다고 치자. `IDENTITY` 전략을 쓰면 10,000번의 개별 쿼리가 각각 네트워크를 타고 흘러가므로 대단히 비효율적이다.
- **실무 우회책:** 이럴 때는 JPA의 한계를 명확히 인지하고 기능을 잠시 우회한다. **Spring Data JDBC**를 가져와서 `jdbcTemplate.batchUpdate()`를 사용하거나, 별도의 **네이티브 쿼리**를 직접 작성해서 벌크 연산을 처리하면 성능 문제를 깔끔하게 해결할 수 있다.

## UUID전략

### MSA와 분산 환경에서의 PK 충돌 문제

- 최근에는 하나의 거대한 서버가 아니라, 서비스를 쪼개서 여러 대의 서버와 여러 대의 DB를 쓰는 분산 환경을 많이 구축한다.
- 만약 DB 여러 대가 쪼개져 있는 상태에서 `AUTO_INCREMENT`를 쓰면, 1번 DB에서도 ID 1번이 나오고 2번 DB에서도 ID 1번이 나올 수 있다.
- 나중에 데이터를 하나로 합치거나 정산하려고 하면 식별자가 중복되어 충돌하는 문제가 발생한다.

### UUID의 도입

- 이 문제를 해결하기 위해 유일함이 거의 완벽히 보장되는 128비트 난수 문자열인 `UUID`를 PK로 사용한다. (예: `550c2433-e29b-a122...`)
- **장점:** 전 세계 어떤 DB 시스템에서 생성해도 충돌할 걱정이 없다.
    - **DB에 가기 전에 자바 애플리케이션 단에서 미리 ID를 생성할 수 있다.** 즉, `IDENTITY` 전략이 가졌던 **쓰기 지연 불가능 문제를 완벽히 해결**하며 최적화가 가능해진다.
- **단점:** UUID는 기본적으로 매우 긴 문자열이기 때문에 정수형(`Long`)보다 메모리를 많이 먹고 **인덱스 성능이 떨어진다.**
    - 특히 MySQL의 경우 PK 순서대로 데이터를 물리적으로 정렬(클러스티드 인덱스)하는데, 완전 랜덤한 UUID가 계속 들어오면 데이터가 중간중간 무작위로 끼어들면서 물리적 재정렬이 일어나 성능 저하(Page Split)가 생기기도 한다.
- **실무 보완책:** 이러한 단점을 극복하기 위해, 앞부분에 '타임스탬프(시간)'를 섞어서 시간 순서대로 정렬이 가능하도록 설계된 `UUID v7`이나 **`TSID`**, **`ULID`** 같은 최신 식별자 체계를 도입해서 실무 성능을 방어한다.

| **전략 명칭** | **주 사용 DB** | **키 채번(ID 획득) 타이밍** | **쓰기 지연 최적화** | **특징 및 실무 요약** |
| --- | --- | --- | --- | --- |
| **TABLE** | 모든 DB 지원 | `em.persist()` 시 키 테이블 조회 | **가능** | `SELECT` + `UPDATE`가 항상 동반되어 DB 동시성 락 문제로 **실무에서 안 씀** |
| **SEQUENCE** | Oracle, PostgreSQL | `em.persist()` 시 DB 시퀀스 조회 | **완벽히 가능** | **`allocationSize`** 옵션으로 메모리에 키를 선할당 받아 네트워크 왕복 비용을 획기적으로 줄임 |
| **IDENTITY** | MySQL, MariaDB | **`em.persist()` 호출 즉시 `INSERT` 실행** | **불가능** | 설계가 간편해 널리 쓰이나, 대량 데이터 저장 시에는 **`JdbcTemplate` 벌크 연산으로 우회** 필요 |
| **UUID** | 분산 환경 / MSA | **`em.persist()` 전 자바단에서 즉시 생성** | **완벽히 가능** | DB 다중화 환경에서 충돌을 방지함. 인덱스 성능 저하를 막기 위해 시간 정렬형(`UUID v7`, `TSID`)을 권장 |
| **AUTO** | DB 방언에 따름 | 선택된 실제 DB 전략에 따름 | DB에 따라 다름 | JPA의 기본값이며, 설정된 DB 종류(Dialect)에 맞춰 위 전략 중 하나를 자동으로 선택함 |
