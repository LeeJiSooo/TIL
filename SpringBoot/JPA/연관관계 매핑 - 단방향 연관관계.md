## 가져가야 할 핵심

1. DB는 외래키 하나로 양방향 통행이 가능하지만, 객체는 참조 필드가 있어야만 갈 수 있는 단방향 통행구조임을 인지한다.
2. 실제 DB테이블에 외래키를 들고 있는 진짜 ‘연관관계의 주인’이 누구인지 판별한다.
3. @OneToMany 단방향 매핑이 왜 사용하면 안 되는 안티 패턴인지 이해한다.

## 왜 ‘단방향’으로 시작해야 하는가?

- 단방향 연관관계
    - 한 엔티티에서만 다른 엔티티를 참조해 일방통행으로 관계를 맺는 매핑 방식이다.(A는 B의 주소를 아는데, B는 A의 존재를 모르는 상태)
- 엔티티를 설계할 때는 특별한 이유가 없다면 ‘단방향 연관관계’로 설계를 시작해야 한다.
- 처음부터 양쪽 길을 다 뚫어놓는 양방향 참조를 한다면, 영속성 컨텍스트 내에서 객체 상태가 꼬이거나 무한 루프에 빠져 코드 복잡성이 상승한다.
- 나중에 진짜 반대 방향으로 객체 탐색이 필요해지는 시점에 양방향을 열어주면 된다. 양방향을 추가한다고 해서 물리적인 DB스키마(외래키 구조)가 바귀는 것이 아니며, 단지 자바 객체에서 참조를 하나 더 열어주는 것뿐이기 때문이다.

**양방향 참조의 문제(업데이트 포인트 분산):**

- 자바 객체 세상을 양방향으로 만들면 데이터를 생성/수정할 때마다 **개발자가 양쪽 객체의 상태를 매번 똑같이 맞춰주는 추가 코드를 직접 짜야 한다.**
- 만약 한쪽(유저 게시글 리스트)에 추가해 주는 것을 깜박하면, DB에는 외래키 덕분에 데이터가 잘 들어갔을지 몰라도 자바 메모리(영속성 컨텍스트)안에서 유저는 글을 썼다는 사실을 모르기 때문에 **메모리 - DB** 간 정합성이 어긋난다.

**단방향 참조의 이점(입데이트 포인트 단일화) :**

- 단방향은 애초에 유저 객체 안에 게시글 리스트 필드가 없다.
- 따라서 게시글을 새로 작성할 때 `Post` 객체에만 유저를 세팅(`post.setUser(user)`)해 주면 끝이 나기 때문에, 코드가 단순해지고 버그가 생길 여지가 차단된다.

## 안티패턴 : 일대다(@OneToMany)단방향 매핑

연관관계의 매핑을 설계할 때 가장 중요한 질문은 “누가 외래키를 가질 것인가?”이다. DB구조상 1:N관계에서 외래키는 무조건 N쪽에 존재한다.

이때, 물리 외래키를 가진 쪽의 엔티티가 ‘연관관계의 주인’이 되어야 코드가 자연스러워지는데, 이걸 **1쪽에 주인을 수동 설정하는** `@OneToMany` 단방향 매핑을 쓰면 성능이 안 좋아진다.

**User(1)가 주인이 되어 Post(N)를 관리하는 코드**

```java
@Entity
public class User {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @OneToMany
    @JoinColumn(name = "user_id") // 외래키는 Post 테이블에 생기는데, 주인은 User가 됨
    private List<Post> posts = new ArrayList<>();
}

@Entity
public class Post {
    @Id @GeneratedValue
    private Long id;
    private String title;
    // Post 안에는 User에 대한 필드(참조)가 아예 없는 상황!
}
```

**이 상태에서 신규 유저와 게시글 2개를 저장하는 비즈니스 로직**

```java
// 1. 유저 생성 및 저장
User user = new User();
user.setName("Adapterz");
em.persist(user);

// 2. 게시글 생성 및 저장
Post post1 = new Post(); post1.setTitle("첫 번째 글"); em.persist(post1);
Post post2 = new Post(); post2.setTitle("두 번째 글"); em.persist(post2);

// 3. 연관관계 맺기 (User가 주인이므로 User의 리스트에 직접 추가)
user.getPosts().add(post1);
user.getPosts().add(post2);

em.flush();
```

**실제 콘솔에 찍히는 SQL로그(의도치 않은 쿼리 생김)**

```sql
-- 1. User INSERT (정상)
insert into user (name, id) values ('Adapterz', 1);

-- 2. Post INSERT (이때 post 테이블의 user_id 외래키는 뭔지 몰라서 null로 박힘)
insert into post (title, id) values ('첫 번째 글', 1); -- user_id = null
insert into post (title, id) values ('두 번째 글', 2); -- user_id = null

-- 3. 개발자가 지시한 적 없는 기괴한 수동 UPDATE 쿼리 2번 발새
update post set user_id=1 where id=1;
update post set user_id=1 where id=2;
```

**예상치 못한 UPDATE쿼리가 발생한 이유?**

1. em.persist(post1)을 할 때 , Post엔티티 안에는 user필드가 없기 때문에 내가 어떤 유저가 쓴 글인지 모른다. 그래서 일단 DB에 post테이블 안에 user_id를 null로 채워서 INSERT를 날린다.
2. 이후 로직 마지막에 user.getPosts().add(post1)을 한 뒤 트랜잭션이 끝날 때, JPA영속성 컨텍스드가 post테이블에 있는 외래키에 userID를 추가함 → UPDATE 쿼리 추가 방출

→ 만약 가입 시 게시글이 100개라면, INSERT 100번으로 끝날 일이, INSERT 100번 + 수동 UPDATE 100번 = 총 200번의 쿼리가 나가는 성능 저하가 발생한다.

**Q. JPA에서 연관관계의 주인이 무엇인지 설명하고, 왜 항상 N쪽 엔티티가 주인이 되어야 하는지 이유를 쿼리 발생 관점에서 설명해주세요.**

연관관계의 주인이란 데이터베이스의 외래키를 관리하는 주체를 의미하며, 외래키의 제어 권한(등록, 수정)을 가진다. 관계형 데이터베이스의 구조상 1:N관계에서는 항상 N쪽에 외래키 컬럼이 물리적으로 존재한다.

객체 모델에서도 이 물리 구조와 일치시켜 N쪽에 엔티티를 연관관계의 주인으로 설정해야만, 객체의 상태 변경이 해당 테이블에 대한 자연스러운 INSERT나 UPDATE 쿼리로 매끄럽게 이어진다.

만약 1쪽으로 주인을 설정하는 @OneToMany 단방향 매핑을 사용하게 되면, N쪽 객체를 저장할 때 외래키를 알지 못해 null로 먼저 INSERT한 뒤, 나중에 1쪽 객체의 리스트 변경을 감지하여 다른 테이블에 UPDATE쿼리를 추가로 날리는 비효율이 발생한다.

이는 대량 데이터 저장 많은 쿼리로 인해 성능 저하를 유발하므로 외래키가 있는 N쪽을 주인으로 삼아야 한다.

## 정리

- 단방향 연관관계는 두 객체 중 한쪽 방향으로만 점을 찍어 탐색할 수 있도록 참조 필드를 열어둔 안전하고 직관적인 구조이다.
- 불필요하게 처음부터 양방향 설계를 도입하면 비즈니스 로직마다 양쪽 객체 정합성을 수동으로 맞춰야 하므로 단방향으로 시작하는 것이 표준이다.
- 연관관계의 주인은 물리 테이블상 외래키를 쥐고 있는 N쪽 엔티티여야 한다.
- 1쪽 엔티티가 리스트를 쥐고 주인이 되는 @OneToMany 단방향 매핑은 의도치 않은 추가 UPDATE쿼리를 유발하는 안티패턴이므로, 다대일(@ManyToOne) 단방향 구조를 기본적으로 가져가야 한다.
