## 가져가야 할 핵심

1. remove()를 호출하면 즉시 DELETE 쿼리가 나가지 않고, 영속성 컨텍스트에 마킹된 후 플러시 시점에 방출된다.
2. JPA 관리망 안에 들어와 있는 객체만 지울 수 있으므로 반드시 조회 후 영속 상태로 만든 뒤 삭제해야 한다.
3. 데이터를 진짜 날려버리지 않고 살려두는 논리 삭제 기법과, 이를 지원하는 JPA의 특수 어노테이션 메커니즘을 이해한다.

## 데이터의 생명주기 마감 remove()

- JPA 데이터 조작의 흐름
    - 데이터를 생성하고 수정하기 위해 `persist()`와 변경 감지를 배웠고, 데이터를 읽어오기 위해 `find()`를 배웠다.
    - 엔티티 생명주기의 마지막 단계로서, 데이터베이스에 존재하는 데이터를 안전하게 제거하기 위해 사용하는 메서드가 `remove()`이다.
- `remove()`의 본질
    - persist()가 즉시 INSERT가 아니었듯, remove() 역시 즉시 DB에서 데이터를 지워버리는 명렁어가 아니다.
    - 영속성 컨텍스트에 안전하게 흔적을 남겨두고 나중에 한방에 처리하는 ‘삭제 예약’이 본질이다.

영속 상태의 엔티티를 ‘삭제 예정’ 상태로 표시하여, 플러시나 트랜잭션 커밋 시점에 데이터베이스에 `DELETE` 쿼리가 반영되도록 예야하는 메서드

### 내부 동작 프로세스(삭제 예약 메커니즘)

개발자가 코드에서 `entityManager.remove(user)`를 호출하면 영속성 컨텍스트 내부에서는 다음과 같은 연쇄 반응이 일어난다.

1. 삭제 마킹(1차 캐시) : 1차 캐시에 들어있는 해당 엔티티 객체 옆에 ‘삭제 예정’이라는 플래그를 붙여둔다.
2. 쓰기 지연 : 쓰기 지연 SQL 저장소에 `DELETE FROM users WHERE user_id = ?` 쿼리 문자열을 생성해서 킵해둔다. (아직 DB 전송 X)
3. 최종 방출: 트랜잭션이 무사히 끝나서 커밋되거나 flush()가 호출되는 순간, 큐에 있던 `DELETE`쿼리가 DB로 날아가 실행되면서 물리적인 삭제가 완료된다.

### 무조건 영속 상태의 객체여야 한다.

대상 엔티티는 반드시 JPA가 관리하고 있는 영속 상태여야만 `remove()`를 쓸 수 있다.

```java
// 못된 삭제 방법
User user = new User();
user.setId(5L); // "DB에 5번 유저 있으니까 지워주겠지?"

entityManager.remove(user); // 에러 발생 (IllegalArgumentException)
```

- 이유 : new 키워드로 방금 만든 객체는 JPA 입장에서 영속성 컨텍스트에 존재하지도 않는 비영속 객체이다. 1차 캐시에 들어있지도 않은 객체를 지워달라고 하면 JPA는 에러를 던진다.
- 데이터베이스에 있는 5번 유저를 지우고 싶다면, `em.find()`로 먼저 조회해서 1차 캐시에 올려놓은(영속 상태)다음에 `remove()`로 넘겨줘야 한다.

## 예약 방식을 사용하는 이유

### 트랜잭션 롤백 시 안전한 데이터 복구를 위해

- 만약 remove()를 하자마자 DB에서 데이터가 즉시 지워진다고 가정해보자.
- 그런데 삭제 직후 다음 자바 로직을 수행하다가 에러가 터져서 트랜잭션을 롤백 해야 하는 상황이 오면 이미 DB에서 데이터가 물리적으로 지워졌다면 이를 다시 원상 복구하고 되돌리는 연산은 까다롭다.
- 하지만 JPA는 자바 메모리에 예약만 걸어두기 때문에, 중간에 에러가 나면 그냥 영속성 컨텍스트 메모리를 통째로 리셋해버리면 된다. DB에는 삭제 쿼리가 발생하지 않아 데이터가 안전해진다.

### 1차 캐시와 ‘데이터 정합성’일치

- remove(user)를 호출해 삭제 예약을 걸어둔 직후, 똑같은 트랜잭션 안에서 5번 유저를 또 조회(em.find())하는 상황이 생길 수 있다.
- 이때 JPA는 무겁게 DB까지 가지 않고 1차 캐시를 먼저 확인한다. 그런데 방금 지우겠다고 ‘삭제 예정 마킹’을 달아놨다.
- JPA는 이것을 보고 아직 DB에 지워지진 않았지만 이번 작업 내에서 삭제될 데이터라고 판단해서 null을 반환해준다. → 자바 메모리 내에서 정합성이 유지된다.

만약 개발자가 JPA remove를 쓰지 않고, 네이티브 쿼리나 JDBC를 써서 `DELETE FROM users WHERE id = 5` 를 직접 DB에 날리면 어떻게 될까?

→DB에서 5번 유저는 지워졌지만 JPA 1차 캐시 내부에는 5번 유저 객체가 ‘좀비 데이터’로 존재한다. 이후 JPA로 5번 유저를 찾으면 캐시에서 옛날 데이터를 꺼내오는 데이터 정합성 장애가 터진다.

## 소프트 딜리트(논리 삭제)

- 하드 딜리트 : DB에 DELETE 쿼리를 날려서 물리적으로 데이터를 완전히 파쇄하는 기법이다.
- 소프트 딜리트 : DB에서 데이터를 진짜 지우지 않고, 삭제 여부를 나타내는 컬럼 값만 수정하는 논리적인 삭제 기법

**왜 진짜 데이터를 안 지우고 남겨둘까?**

우리가 쓴 글을 지웠다가 복구해달라고 요청할 수도 있고, 탈퇴한 회원의 구매 이력을 바탕으로 연말에 데이터 통계 분석을 해야할 수도 있다. 또는 전자상거래법 등 법적으로 결제/주문 데이터는 5년간 보관해야 하는 의무가 있을 수도 있다.

보통 테이블에 `deleted_at`(삭제 일시)같은 컬럼을 두고, 삭제 요청이 오면 이 값을 채워 넣는 `UPDATE`쿼리로 대체한다.

```sql
-- 5번 회원이 탈퇴할 때 진짜 지우지 않고 탈퇴 시각을 마킹 (논리 삭제)
UPDATE users SET deleted_at = CURRENT_TIMESTAMP WHERE user_id = 5;

-- 앞으로 조회할 때는 탈퇴 안 한(살아있는) 회원만 걸러서 조회해야 함
SELECT * FROM users WHERE user_id = 5 AND deleted_at IS NULL;
```

### JPA의 remove()는 소프트 딜리트할 때 무용지물인가?

아니다. 개발자가 평소처럼 remove()를 호출해도 내부적으로는 물리 삭제 대신 소프트 딜리트용 UPDATE쿼리가 나가도록 어노테이션을 제공한다.

```java
@Entity
@Table(name = "users")
// 1. remove()가 호출될 때 진짜 DELETE 대신 실행할 SQL을 강제로 덮어쓴다
@SQLDelete(sql = "UPDATE users SET deleted_at = CURRENT_TIMESTAMP WHERE user_id = ?")
// 2. 앞으로 이 엔티티를 조회(SELECT)하는 모든 순간에 자동으로 이 WHERE 조건을 강제 장착한다
@SQLRestriction("deleted_at IS NULL")
public class User {

    @Id @GeneratedValue
    @Column(name = "user_id")
    private Long id;

    private LocalDateTime deletedAt; // 삭제 시각을 담을 필드
}
```

- 개발자의 코드 편의성 : 이렇게 세팅해 두면 개발자는 복잡하게 생각할 필요 없이 em.remove(user)만 호출하면 된다. 그러면 JPA가 알아서 지정된 소프트 딜리트 UPDATE쿼리를 날려준다.

`@SQLRestriction`은 모든 조회에 항상 적용되기 때문에, 만약 관리자 페이지에서 "탈퇴한 회원 목록까지 다 포함해서 보여주세요"라거나, "탈퇴한 회원을 다시 복구해 주세요"라는 특수 요건이 들어오면 필터에 걸려 조회가 안 되는 문제가 생긴다.

- 그럴 때는 어노테이션에 의존하지 않고 엔티티 내부에 `restore()` 같은 소프트 딜리트 해제 메소드를 직접 구현해서 필드 값을 수동 조작하거나, 상황에 따라 껐다 켤 수 있는 하이버네이트의 **`@Filter`** 기능을 활용해 정교하게 제어한다.

Q. JPA에서 em.remove()를 호출했을 때의 메커니즘과, 데이터가 실제로 삭제되는 시점에 대해서 설명해주세요.

em.remove()는 호출 즉시 데이터베이스에 DELETE 쿼리를 날리는 메서드가 아니라, 영속성 컨텍스트 내에서 해당 엔티티를 ‘삭제 예정’상태로 마킹하는 일종의 삭제 예약 메서드이다.

호출 시점에는 1차 캐시에 삭제 플래그가 들어가고 쓰기 저장소에 DELETE 쿼리가 생성되어 보관된다.

이후 트랜잭션이 안전하게 종료되어 커밋되거나 flush()가 호출되는 시점에 모아둔 DELETE쿼리가 데이터베이스로 전송되어 실제 물리적 삭제가 진행된다.

단, 삭제를 진행하려는 대상 엔티티는 JPA가 식별할 수 있도록 반드시 사전에 조회된 ‘영속 상태’여야만 정상적으로 삭제 예약 처리가 가능하다.

## 정리

- remove()는 즉각적인 데이터 파쇄가 아니라, 영속성 컨텍스트의 쓰기 지연 시스템을 활용한 안전한 삭제 예약 메서드이다.
- 비영속 객체를 지울 수 없으며, em.find() 등으로 먼저 가져와 1차 캐시에 넣어둔 후 영속 상태의 엔티티만 삭제의 대상이 될 수 있다.
- 삭제 예약을 해두면 트랜잭션 롤백 시 데이터 복구가 매우 깔끔하고, 같은 트랜잭션 내 재조회 시 null을 즉각 반환해주어 메모리 정합성을 방어한다.
- 데이터를 영구 소멸시키는 위험을 막기 위해 실무에서는 `deleted_at` 컬럼을 다루는 소프트 딜리트(논리 삭제)를 주로 사용하며, JPA의 `@SQLDelete`와 `@SQLRestriction`을 조합하면 `remove()` 메서드의 표준 격식을 그대로 유지하면서 안전하게 소프트 딜리트를 구현할 수 있다.
