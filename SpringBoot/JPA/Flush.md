## 가져가야 할 핵심

1. 플러시는 커밋이 아니다. 영속성 컨텍스트의 변경 내용을 DB에 반영하는 ‘동기화’작업일 뿐이다.
2. 플러시를 사용하는 시나리오 3가지에 대해서 이해

## 영속성 컨텍스트와 DB의 연결고리, 플러시(Flush)

영속성 컨텍스트의 변경 내용을 즉시 데이터베이스에 반영하여 자바 메모리와 DB상태를 동기화하는 JPA의 메서드

- JPA 쓰기 전송 타이밍의 수동 조작
    - 앞서 persist()와 쓰기 지연을 배우면서, 엔티티를 저장해도 즉시 DB에 쿼리가 날아가지 않고 트랜잭션이 커밋되는 시점에 모아서 날아간다고 배웠다.
    - 하지만 코드를 짜다 보면, 트랜잭션이 최종 커밋될 때까지 기다리지 않고 “지금 당장” DB에 쿼리를 보내서 상태를 동기화하고 싶은 순간이 생긴다.
    - 그럴 때 영속성 컨텍스트의 쓰기 지연 저장소를 강제로 비워내는 버튼이 플러시이다.
- entityManager.flush()가 호출되면 JPA는 우선 변경감지(더티체킹)를 가동하여 바뀐 엔티티를 찾아 UPDATE쿼리를 생성한 뒤, 쌓여있던 모든 쿼리를 DB로 전송한다.

### 플러시 호출 시 오해하면 안 되는 주의사항

- **데이터가 영구 저장된 것이 아니다(커밋이 아님)** : 쿼리가 DB로 날아갔다고 해서 트랜잭션이 완전히 끝난 것이 아니다. 아직 커밋 전이기 때문에, 이후에 예외가 터져서 롤백이 되면 플러시로 날아간 쿼리들도 DB단에서 전부 깔끔하게 취소된다.
- **1차 캐시가 비워지는 것이 아니다** : 플러시는 쓰기 지연 SQL저장소의 쿼리들을 DB로 비워내는 작업이다. 1차 캐시에 들어있는 엔티티 객체들은 지워지지 않고 그대로 유지된다.

## 굳이 수동으로 Flush를 쓰는 이유

어차피 트랜잭션 커밋할 때 알아서 자동으로 플러시가 호출되는데, 왜 개발자가 코드로 굳이 수동 flush를 해야 할까?

### DB 제약조건 예외를 조기에 당겨서 처리하기 위해서

- for문을 돌며 회원 10명을 연달아 가입(`persist`)시킨다. DB의 이메일 컬럼에는 유일성 제약조건이 걸려있다. 그런데 하필 5번째 회원의 이메일이 중복 데이터였다.
- 수동 Flush 미사용 시 : 10명 모두 1차 캐시에만 담긴 채 로직이 쭉 흘러갔다. 그러다 맨 마지막 트랜잭션 커밋 시점에 10개의 쿼리가 한 방에 날라가다가 5번째 쿼리에서 뒤늦게 DB 제약조건 에러가 더진다. 결국 10명 전체가 통째로 롤백된다.
- 수동 Flush 사용 시 : `persist()`직후에 명시적으로 `flush()`를 호출하면 즉시 DB에 쿼리가 날아가므로 5번쟤 유저가 들어오는 순간 그 즉시 에러를 낚아챌 수 있다. → 이를 `try-catch`문으로 감싸서 에러를 잡은 뒤, 해당 중복 데이터만 스킵하거나 따로 예외 처리를 해주는 식의 제약 조건 제어가 가능해진다.

### 네이티브 SQL 쿼리 실행 전 데이터 동기화를 위해서

- 복잡한 통계나 대형 레거시 쿼리를 처리하기 위해 JPA 엔티티 조작과 네이티브 SQL을 섞어서 사용한다.
- 문제 발생 : 영속 상태인 엔티티의 닉네임을 자바 코드로 바꿨다. 하지만 아직 커밋 전이라 DB에는 `UPDATE`쿼리가 안 나갔다. 이 상태에서 네이티브 쿼리로 DB를 직접 조회한다면 어떻게 될까?
- 네이트브 쿼리는 JPA 1차 캐시에 뭐가 들어있는지 아예 모른다. 오직 영속성 컨텍스트 바깥의 물리 DB 상태만 보고 읽어오기 때문에 수정되기 전의 옛날 데이터를 읽어오는 ‘데이터 불일치’장애가 터진다.

→ 네이티브 SQL을 실행하기 직전에 반드시 `entityManager.flush()` 를 명시적으로 호출해서 변경 사항을 DB에 먼저 밀어 넣어야 한다. 그래야 네이티브 쿼리가 최신 데이터를 안전하게 읽을 수 있다.

### 대용량 데이터 배치 처리시 OOM 방지

- 엑셀 업로드 기능을 통해 10만 건의 데이터를 한 번에 DB에 밀어 넣어야 한다. 만약 아무 대책 없이 10만번 persist()만 연달아 호출하면, 1차 캐시와 쓰기 지연 저장소에 10만 개의 자바 객체와 10만 개의 쿼리가 메모리에 계속 쌓이게 된다. 결국 서버 메모리가 버티지 못하고 OOM(메모리 고갈)에러로 서버가 다운된다.

→ 해결책 : 이럴 때 1,000건 단위로 끊어서 다음과 같이 처리한다.

```java
for (int i = 0; i < 100000; i++) {
    em.persist(largeData);

    if (i % 1000 == 0) {
        em.flush(); // 1. 1,000개의 쿼리를 DB로 보내 쓰기 지연 저장소를 비운다.
        em.clear(); // 2. 1차 캐시를 싹 비워서 자바 메모리(OOM)를 방어한다.
    }
}
```

Q. JPA에서 플러시와 커밋의 결정적인 차이점이 무엇인가?

가장 결정적인 차이는 데이터의 확정 여부이다.

플러시는 영속성 컨텍스트의 쓰기 지연 저장소에 모인 쿼리들을 데이터베이스에 전송하여 메모리와 DB의 상태를 동기화하는 작업이다. 플러시가 일어나도 트랜잭션이 종료된 것이 아니기 때문에 언제든 롤백이 가능하며, 1차 캐시의 엔티티들도 그대로 유지된다.

커밋은 해당 트랜잭션을 종료하면서 플러시로 날아간 데이터 조작들을 데이터베이스에 영구적으로 완전히 반영되는 최종 확정 작업이다. 커밋이 완료된 이후에는 롤백이 불가능하다.

## 정리

- Flush는 JPA가 가지고 있던 자동 쿼리 전송 타이밍을 개발자가 원할 때 수동으로 제어하는 ‘동기화 버튼’이다.
- 플러시는 영속성 컨텍스트를 비우는 것이 아니라 쓰기 지연 큐만 DB로 내보내는 것이므로 1차 캐시는 보존되며, 커밋이 아니기 때문에 예외 발생 시 안전하게 롤백이 가능하다.
- 예외를 조기에 발견하거나 네이티브 쿼리 전에 동기화 하기 위해서 사용할 수 있다.
