- 영속성 컨텍스트와 트랜잭션의 복습
    - 영속성 컨텍스트는 JPA가 엔티티를 관리하는 일종의 메모리 공간이며, 이 공간 안에 객체가 들어가야 1차 캐시, 변경 감지 등의 기능이 동작한다.
    - 트랜잭션은 이 공간에 모인 변경 사항들을 DB에 최종 반영하는 작업 단위이다.
- `persist()`의 진짜 목적
    - 자바 코드에서 new 키워드로 생성한 순수 자바 객체를 영속성 컨텍스트라는 공간으로 넣기 위해 사용하는 메서드가 persist()이다.
    - 단순히 DB에 SQL을 넣는 명령어라기보다는, 객체를 영속성 컨텍스트의 관리 대상으로 만드는 ‘영속 상태로의 전환’이 본질이다.

## 가져가야 할 핵심

- persist()는 즉시 DB에 저장하는 것이 아니라 영속성 컨텍스트가 이 객체를 관리하기 시작했다는 ‘등록’의 의미이다.
- 엔티티에 설정된 기본키 생성 전략에 따라 `persist()`가 호출되는 순간의 동작(쿼리 방출 여부)이 달라진다.

## persist()

- 새로 생성한 엔티티를 영속성 컨텍스트에 등록하여, 최종적으로 데이터베이스에 저장되도록 만드는 메서드

예시 : 쇼핑몰의 장바구니

- 쇼핑몰에서 마음에 드는 티셔츠를 장바구니에 담는다(persist)고 해서, 내 통장에서 돈이 결제되고 택배가 바로 출발하지ㅏ 않는다. 장바구니는 단지 “나 나중에 이거 살 거야”하고 내 목록에 등록해 두는 공간일 뿐이다. 실제 결제는 주문서에서 결제하기 버튼(commit)을 누를 때 이루어진다.
- JPA도 마찬가지이다. new User()로 만든 객체는 아직 JPA가 존재도 모르는 ‘비영속 상태’다. 이 객체를 장바구니에 담듯 em.persist(user)를 해줘야 영속성 컨텍스트에 들어가며 ‘영속상태’가 된다.

## 기본키 전략에 따른 persist()의 동작 차이

영속성 컨텍스트는 내부적으로 `Map<Key, Value>` 형태로 생겼고, **Key에는 엔티티의 `@Id(PK)`**, **Value에는 엔티티 객체 인스턴스**가 들어간다. 즉, **영속성 컨텍스트가 객체를 관리하려면 무조건 PK(식별자)를 알아야 한다.**

### SEQUENCE전략(쓰기 지연을 활용하는 케이스)

- **동작:** `em.persist(user)`를 호출하면, JPA는 DB의 '시퀀스 오브젝트'로 가서 번호표(`allocationSize` 만큼 뭉텅이로 혹은 다음 번호)를 먼저 받아온다.
- **결과:** 아직 DB에 데이터를 넣지 않았지만, **PK 값은 확보했기 때문에** 그 번호를 Key 삼아 영속성 컨텍스트(1차 캐시)에 객체를 저장할 수 있다.
- 따라서 실제 `INSERT` SQL은 즉시 날아가지 않고 **쓰기 지연 저장소에 차곡차곡 모아두었다가, 트랜잭션이 커밋(`commit`)되는 순간에 한방에 실행**된다.

### IDENTITY전략(쓰기 지연을 활용하지 못하는 케이스)

- **동작:** MySQL의 `AUTO_INCREMENT` 같은 `IDENTITY` 전략은 **실제로 DB에 데이터를 `INSERT`해서 넣기 전까지는 PK 값을 알 수 없는 구조**다.
- **결과:** JPA 입장에서는 1차 캐시에 Key로 등록해야 할 PK를 알 방법이 없으므로, 예외적으로 **`em.persist(user)`가 호출되는 그 시점에 쓰기 지연을 포기하고 즉시 DB로 `INSERT` 쿼리를 날려버린다.**
- 쿼리가 실행된 직후 DB가 발행해 준 PK 값을 받아와서 그제야 영속성 컨텍스트(1차 캐시)에 정식 등록을 마친다.

## 왜 쓰는가 ? 편리함과 트랜잭션 일관성

엔티티를 JPA가 관리하도록 위임함으로써 중복 코드 없이 안정적으로 저장 로직을 처리하고, 트랜잭션 일관성을 유지하기 위해서이다.

- 개발자 생산성 증가 : 과거에 수동으로 문자열 SQL을 작성하고 파라미터를 바인딩하던 작업을 `persist()` 메서드 한 줄로 편하게 처리한다. JPA가 현재 사용하는 DB(MySQL, Oracle 등) 방언에 맞춰 SQL을 동적으로 알아서 생성해준다.
- 안전한 롤백 : 쓰기 지연이 작동하는 환경(Sequence 등)에서는 여러 엔티티를 연달아 저장하다가 중간에 예외가 발생하더라도, DB에 쿼리가 날라가기 전이므로 트랜잭션 롤백 시 메모리에만 있던 변경 사항들이 깔끔하게 모효화되어 안전하다.
- 일단 persist()를 통해 객체가 영속 상태(1차 캐시 안)로 들어오면, 나중에 엔티티 값이 바뀔 때 우리가 아무 것도 안 해도 변경감지(더티체킹)가 발동하여 자동으로 UPDATE쿼리까지 보장받는다.

## 알아두면 좋은 정보 : persist()와 find()

**`EntityManager.find()`의 메커니즘:**

- `find(User.class, 1L)`은 PK로 엔티티를 조회하는 메서드이다.
- 이때 JPA는 무조건 DB로 가지 않고, **영속성 컨텍스트(1차 캐시)에 `1L`이라는 Key가 있는지 먼저 찾는다.**
- 만약 방금 내가 `em.persist(user1)`(ID가 1L이라고 가정)을 호출해서 1차 캐시에 올려둔 상태라면, 아직 트랜잭션 커밋 전이라 DB에 데이터가 없더라도 `em.find(User.class, 1L)`을 했을 때 **1차 캐시에 있는 객체를 즉시 찾아서 반환**해 준다. 영속성 컨텍스트 덕분에 메모리 내에서 데이터 정합성이 유지되는 것이다.

Q. em.persist()메서드를 호출하면 데이터베이스에 INSERT쿼리가 실행되는 시점은 언제인가요?

기본적으로 em.persist()는 객체를 영속성 컨텍스트에 ‘등록’하는 메서드이므로, 호출 즉시 쿼리가 나가지 않고 쓰기 지연 저장소에 보관되었다가 트랜잭션이 커밋되는 플러시 시점에 실행된다. 대표적으로 SEQUENCE전략이 쓰기 지원을 활용한다.

하지만 기본키 생성 전략이 IDENTITY인 경우는 예외이다. 영속성 컨텍스트에 엔티티를 적재하려면 식별자(pk)가 필수적인데, IDENTITY전략은 DB에 실제 데이터를 넣어야만 PK를 얻을 수 있다. 따라서 이 경우에는 영속성 컨텍스트로 등록에 필요한 PK값을 확보하기 위해 예외적으로 persist()호출 시점에 즉시 INSERT쿼리가 실행된다.

## 정리

- persist()는 단순한 SQL 실행 명령어가 아니라, 자바 객체를 영속성 컨텍스트에 담아 ‘영속상태’로 등록하는 메서드이다.
- 쓰기 지연 덕분에 원래는 커밋 시점에 쿼리가 모여서 나가는 것이 맞지만, IDENTITY전략을 쓸 때는 식별자를 미리 알 수 없다는 한계 때문에 persist()즉시 INSERT쿼리가 방출된다.
