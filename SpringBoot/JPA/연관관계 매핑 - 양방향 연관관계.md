## 가져가야 할 핵심

1. 객체의 양방향이 ‘서로 다른 단방향 2개’가 합쳐진 구조라는 본질을 이해한다.
2. 외래키를 관리하는 진짜 권한을 가진 연관관게 주인과, 읽기 전용 ‘mappedBy’의 의미를 파악한다.
3. 메모리 상의 객체 정합성을 맞추는 연관관계 편의 메서드와 장애를 유발하는 무한 루프(순환참조)해결을 이해한다.

## 왜 양방향 연관관계가 필요한가?

- 요구사항의 변화
    - 게시글 기준으로 작성자 이름 보여주기는 단방향(@ManyToOne)으로 해결되었다.
    - 하지만 서비스를 만들다 보면 “마이페이지에서 내가 작성한 게시글 목록을 전부 보여주세요”같은 반대 방향의 요청이 들어오게된다.
- 단방향의 한계:
    - 객체에 참조가 없으면 점을 찍어 갈 수 없기 때문에, User 객체만 쥐고 있는 상태에서는 유저가 쓴 글 목록을 꺼낼 방법이 없다.
    - 결국 데이터베이스 구조(외래키)를 활용하기 위해 User 객체 안에도 List<Post> posts필드를 추가해서 양쪽 객체가 서로를 참조하게 만드는 ‘양방향 연관관계’가 필요하다.

## 외래키는 1개, 객체 참조는 2개

데이터베이스 테이블 세계와 객체 세계의 구조적 차이 때문에 JPA는 혼란을 느낀다.

- 데이터베이스 : post 테이블에 있는 user_id 외래키 단 1개로 양방향 조인을 한다.
- 자바 객체 : Post.users 참조와 User.posts참조까지 총 2개의 변경 포인트가 생긴다.

#### JPA의 혼란

만약 개발자가 게시글의 작성자를 바꾸고 싶어서 post.setUser(user2)도 호출하고, 동시에 기존 유저의 리스트에서 user1.getPosts().remove(post)도 했다고 하자.

JPA입장에서는 “결국 내가 DB에 날릴 쿼리는 post.user_id 외래키를 바꾸는 UPDATE 쿼리 하나인데, 자바 코드에 참조가 2개나 있어서 누구를 기준으로 삼아서 DB 외래키를 업데이트해야하는가? 라는 혼란이 생긴다.

## 해결책 : 연관관계의 주인과 `mappedBy`

둘 중 딱 한명에게 데이터베이스 외래키를 관리(등록, 수정)할 수 있는 권한을 준다. 권한을 가진 객체는 ‘연관관계의 주인’이고 권한이 없는 반대편은 오직 읽기(조회)만 가능한 ‘읽기 전용’으로 둔다.

### 누가 주인이 되어야 하는가?

- 실제 물리 데이터베이스 테이블에 외래키가 위치한 곳의 엔티티 필드가 주인이 된다.
- post테이블이 user_id외래키를 쥐고 있으므로, Post 엔티티의 User user 필드가 연관관계의 주인이 된다.
- 주인이 되는 쪽에 @ManyToOne과 @JoinColumn(name = “user_id”)을 사용한다.

### 주인이 아닌 쪽 : mappedBy

- 읽기 전용으로 설정된 반대편 클래스는 mappedBy라는 옵션을 걸어준다.
- @OneToMany(mappedBy = "user")
- 주인이 아니기 때문에 user.getPosts().add(post)같은 코드를 짜서 리스트를 아무리 수정해봤자, JPA는 이를 무시하고 DB에 아무런 쿼리를 날리지 않는다. 오직 조회할 때만 이 리스트를 활용한다.

## 연관관계 편의 메소드

주인이 아닌쪽은 읽기 전용이라 값을 변경해봤자 DB에 반영이 안 된다면 귀찮게 양쪽에 값을 다 안 넣고 진짜 주인인 Post에만 유저를 세팅하면 되는거 아닌가?

**주인에게만 값을 세팅했을 때의 문제점(객체 정합성 파괴)**

```java
User user = new User();
user.setName("Adapterz");

Post post = new Post();
post.setTitle("JPA");

// 주인인 Post에만 User를 세팅합니다.
post.setUser(user);

// User 입장에서 게시글 목록의 개수는 0이 나옵니다.
System.out.println("작성한 게시글 수: " + user.getPosts().size());
```

만약 하나의 트랜잭션 안에서 방금 쓴 글을 포함해 유저의 글 목록을 다시 조회하거나 비즈니스 로직을 이어 나간다면 DB에는 글이 써졌는데 자바 메모리상에는 글 개수가 0으로 나오는 버그가 터진다.

### 해결책 : 연관관계 편의 메소드 사용

개발자가 매번 양쪽 객체에 일일이 값을 넣어주는 걸 깜박하지 않도록 엔티티 내부 메서드 하나에 양방향 동기화 로직을 묶어버린다.

```java
@Entity
public class Post {
    @ManyToOne
    @JoinColumn(name = "user_id")
    private User user;

    // 연관관계 편의 메서드
    public void setUser(User user) {
        this.user = user; // 내쪽에 세팅(DB업데이트하기 위해서)
        user.getPosts().add(this); // 반대편 리스트에서 나를 추가합니다. (객체 동기화용)
    }
}
```

이제 서비스 로직에서 양쪽을 다 챙길 필요 없이 post.changeUser(user);만 호출하면 DB외래키 전송과 자바 메모리 객체 정합성이 동기화된다.

## 양방향 연관관계의 무한루프(순환 참조)

### Controller에서 엔티티를 직접 반환할 때

- 컨트롤러에서 return user; 라고 엔티티를 그대로 반환하면, 스프링 내부의 Jackson라이브러리가 이 객체를 JSON문자열로 변환(직렬화)한다.
- User을 열어보니 posts리스트가 있네? → Post객체로 진입해서 JSON으로 바꿈
- Post를 열어보니 내부 표현 식에 user필드가 또 있네? → 다시 User 객체로 진입해서 JSON으로 바꿈
- User → Post → User → Post → User …끝없이 서로를 참조하며 돌다가 결국 메모리가 터지며 500에러가 발생한다.

→ 컨트롤러 영역에는 절대로 엔티티 객체 날것 그대로 노출하면 안된다. 

→ 순수한 데이터 덩어리인 DTO로 변환해서 반환하면 참조를 차단할 수 있다.

## 정리

- DB는 외래키 하나로 양방향 조인을 하지만, 객체는 단방향 2개를 결합해야 하므로 변경 포인트가 2개 생기는 패러다임 불일치가 생긴다.
- 연관관계의 주인 : 혼란을 막기 위해 물리 테이블 상 외래키를 가진 엔티티 필드를 주인으로 선언해 제어권을 주고 반대편은 mappedBy를 붙여 읽기 전용으로 설정한다.
- 주인이 아닌 쪽은 DB수정을 못 하지만, 메모리 상의 객체 정합성을 위해 연관관계 편의 메소드를 작성해 양쪽 객체 상태를 항상 맞춰주어야 안전하다.
