## 가져가야 할 핵심

1. 영속성 컨텍스트의 개념과 객체가 겪게 되는 4가지 생명주기를 이해한다.
2. 1차 캐시와 동일성 보장 : 같은 데이터를 두 번 조회해도 쿼리는 왜 한 번만 나가는지 원리를 파악한다.
3. 쓰기 지연과 변경 감지(더티 체킹) : 쿼리를 모아서 보내는 이유와 `update()`메서드 없이도 데이터가 수정되는 객체지향적 원리를 이해한다.

## 영속성 컨텍스트

- 영속성 컨텍스트의 본질
    - JPA가 엔티티 객체를 관리하며, 동일 트랜잭션 내에서 변경 사항을 추적하고 자동으로 데이터베이스와 동기화하는 메모리 상의 임시 저장소이다.
    - 애플리케이션(Spring)과 DB 사이에서 데이터를 직접 주고받지 않고, 둘 사이에 영속성 컨텍스트라는 ‘다리’를 두는 것이다.
    - 개발자가 객체를 만들어 저장해 달라고 하면 DB로 직행하지 않고 이 컨텍스트에 먼저 담기며, 트랜잭션이 끝날 때 한방에 처리된다.
- 논리적 개념과 `EntityManager`
    - 영속성 컨텍스트는 코드로 직접 new해서 눈으로 볼 수 있는 객체가 아니라 철저히 논리적인 개념이다.
    - 개발자는 `EntityManager`라는 객체를 매개체로 삼아 `persist()`, `find()` 같은 명령을 내려 이 영속성 컨텍스트에 접근하고 제어한다.
    - 스프링 환경에서는 사용자의 요청이 들어와 트랜잭션이 시작되면 내부적으로 `EntityManager`와 그에 종속된 하나의 영속성 컨텍스트가 논리적으로 생성되어 작동한다.

## 엔티티의 4가지 생명주기

JPA가 제공하는 1차 캐시, 변경 감지 등의 모든 기능들은 오직 ‘영속 상태’인 엔티티에만 적용된다. 

#### 비영속(New / Transient)

- 영속성 컨텍스트와 전혀 관계가 없는 상태이다.
- `User user = new User();` 처럼 방금 생성한 순수한 자바 객체가 이에 해당한다. JPA는 이 객체가 메모리에 존재하는지조차 모르며, 영속성 컨텍스트의 어떤 도움도 받지 못한다.

### 영속(Managed)

- 엔티티가 영속성 컨텍스트에 등록되어 JPA의 관리 대상이 된 상태이다.
- 비영속 객체를 만들고 `em.persist(user)` 를 호출하여 엔티티 매니저에게 넘겨주거나, DB에서 조회(`em.find()`)해온 엔티티가 이 상태에 놓인다. 트랜잭션 커밋 시점에 DB와 동기화된다.

### 준영속(Detached)

- 한 번 영속 상태였던 엔티티가 어떤 이유로 인해 영속성 컨텍스트에서 완전히 분리된 상태이다.
- `em.detach(user)`를 호출해 특정 객체만 끄집어내거나, `em.clear()`로 통째로 비우거나, 트랜잭션이 종료되어 영속성 컨텍스트 자체가 닫히면 모든 객체는 준영속 상태가 된다. **JPA의 관리 범위를 벗어났기 때문에 이 객체의 값을 아무리 바꿔도 DB에 전혀 반영되지 않는다.**

### 삭제(Removed)

- 엔티티를 영속성 컨텍스트와 데이터베이스에서 아예 지우기로 삭제를 예약한 상태이다.
- `em.remove(user)`를 호출하면 영속성 컨텍스트 안에서 해당 객체에 '삭제 플래그'가 꽂힌다. 트랜잭션이 커밋될 때 실제 DB에서도 데이터가 삭제(`DELETE`)된다.

## 영속성 컨텍스트의 핵심 메커니즘

### 1. 1차 캐시와 동일성 보장

<img width="964" height="744" alt="image" src="https://github.com/user-attachments/assets/72e43a68-45f5-4c9c-8237-864e70ec80f4" />


- 영속성 컨텍스트 내부에 존재하는 엔티티 저장소로, **`Map<Key, Value>` 스타일**로 동작한다.
    - Key : 엔티티 식별자(`@Id` 값)
    - Value : 엔티티의 객체 인스턴스 주소값

```java
// 트랜잭션 시작
System.out.println("=== 첫 번째 조회 ===");
User user1 = em.find(User.class, 1L); // 1. 캐시에 없으므로 DB에서 SELECT 후 1차 캐시에 저장

System.out.println("=== 두 번째 조회 ===");
User user2 = em.find(User.class, 1L); // 2. 1차 캐시에서 즉시 반환 (DB SELECT 안 날아감)

System.out.println("user1 == user2 : " + (user1 == user2)); // 3. 결과: true (동일성 보장)
// 트랜잭션 커밋/종료
```

- **동일성(`==`)과 동등성(`equals`)의 차이:**
    - `==` 비교는 주소값 자체가 같은지 비교하는 **동일성 비교**이다.
    - `equals()`는 주소가 달라도 내부 논리적 값이 같은지 비교하는 **동등성 비교**이다.
    - **JPA 1차 캐시가 보장하는 것은 완벽한 주소값 일치, 즉 동일성(`==`)이다.**
- **유의사항 :**
    - 1차 캐시는 전역 캐시(Redis 같은 거 거나 2차 캐시)가 아니다. 오직 **하나의 트랜잭션 범위 안에서만 생존하는 아주 수명이 짧은 캐시**이다.
    - 따라서 대단한 성능 개선 효과를 드라마틱하게 주진 않는다. 성능을 비약적으로 올리려면 Redis 같은 별도의 캐시 레이어를 써야 한다. 1차 캐시의 진정한 가치는 **객체의 동일성을 완전히 보장해 주어 자바 컬렉션처럼 데이터를 다루게 해준다**는 패러다임 일치에 있다.

### 2. 쓰기 지연

<img width="1732" height="1158" alt="image" src="https://github.com/user-attachments/assets/20c4c271-da93-4458-997e-cd1735e200b5" />


- `persist()` 를 부를 때마다 `INSERT` SQL을 즉시 실행하지 않고, 영속성 컨텍스트 내부의 ‘**쓰기 지연 SQL 저장소(큐)’에 쿼리를 차곡차곡 모아두는 기능이다.**

```java
tx.begin(); // 트랜잭션 시작

User user1 = new User(1L, "A");
User user2 = new User(2L, "B");

System.out.println("=== persist 호출 전 ===");
em.persist(user1); // 이 시점에는 INSERT 쿼리가 절대로 실행되지 않는다
em.persist(user2); // 쓰기 지연 SQL 저장소에 쿼리만 차곡차곡 쌓임
System.out.println("=== persist 호출 후 ===");

System.out.println("=== 트랜잭션 커밋 전 ===");
tx.commit(); // 이 순간 내부적으로 [플러시(Flush)]가 발동하면서 모아둔 쿼리가 DB로 한방에 전송
System.out.println("=== 트랜잭션 커밋 후 ===");
```

- **플러시(Flush) 단계의 정의:** 영속성 컨텍스트의 변경 내용(SQL 저장소에 쌓인 쿼리들)을 **실제 데이터베이스에 동기화(밀어 넣기)하는 작업**이다. 트랜잭션이 커밋되기 직전에 JPA가 자동으로 호출해 준다. (필요 시 `em.flush()`로 수동 호출도 가능)
- **쓰기 지연이 주는 성능적/구조적 이점:**
    1. **네트워크 비용 감소:** 쿼리를 모아서 한 번에 배치 실행(Batch Insert)할 수 있어 효율적이다.
    2. **DB 테이블 락(Lock) 유지 시간 최소화:** 로직 중간에 `UPDATE`나 `INSERT`가 바로 날아가면, 그 순간 해당 로우에 락이 걸려 트랜잭션이 끝날 때까지 다른 사용자가 데이터 수정을 못 하고 멈춘다. 쓰기 지연을 쓰면 로직을 다 수행하고 **커밋 직전에만 잠깐 락이 걸리기 때문에 동시 처리량에서 이점**을 본다.
    - *(단, MySQL `IDENTITY` 전략은 PK 확보를 위해 `persist()` 즉시 `INSERT`가 나가므로 쓰기 지연이 예외적으로 적용되지 않는다.)*

### 3. 변경 감지(Dirty Checking)

<img width="1322" height="984" alt="image" src="https://github.com/user-attachments/assets/7cb81a17-88a0-44c3-a1ca-10d68997ae53" />


- 엔티티를 조회하거나 영속화할 때 최초 로딩 시점의 상태를 ‘스냅샷’으로 1차 캐시에 저장해둔다. 그리고 커밋 시점에 현재 엔티티의 값과 스냅샷을 비교하여, 변경된 부분이 있으면 자동으로 UPDATE SQL을 생성해주는 기능이다.

```java
tx.begin(); // 트랜잭션 시작

// 1. 엔티티 조회 (가져오는 순간 1차 캐시에 들어가고 최초 상태가 '스냅샷'으로 구워짐)
User user = em.find(User.class, 1L);

// 2. 엔티티 데이터 수정
user.setNickname("새로운닉네임");
user.setPassword("새로운비밀번호");

// 🚫 em.update(user); 혹은 em.persist(user); 같은 수정 메소드는 존재하지 않는다

tx.commit(); // 3. 트랜잭션 커밋 -> 자동으로 플러시가 돌며 현재 값과 스냅샷 비교 후 UPDATE SQL을 날림
```

- 왜 이렇게 설계했을까? (객체중심 사고)
    - 자바의 List 컬렉션을 쓸 때를 생각해보자. 리스트에서 객체를 하나 꺼내서 필드 값을 수정했다고 해서, 우리가 리스트에 대고 `list.add(user)`를 다시 하거나 `list.update(user)`같은 명령을 내리지 않는다. 그냥 값을 바꾸면 끝이다.
    - JPA는 영속성 컨텍스트를 ‘자바 컬렉션’처럼 생각하고 다루도록 설계되었다. 개발자가 SQL수정문에 얽매이지 않고 순수 자바 객체의 상태를 바꾸는 비즈니스 로직에만 몰입할 수 있도록 돕는다.

## 정리

- 영속성 컨텍스트는 애플리케이션과 데이터베이스 사이에서 엔티티를 관리해주는 논리적인 메모리 저장소이다.
- 영속성 컨텍스트가 존재하기 때문에 객체지향과 RDB간의 간극인 패러다임 불일치가 해결되며, 1차 캐시(동일성 보장), 쓰기 지연(락 시간 최소화), 변경 감지(자동 수정)같은 기능이 가능하다.
- 이 모든 기능이 가능하게 하려면 반드시 ‘영속’상태여야 하므로, 엔티티의 생명주기를 정확히 이해해서 처리해야 한다.
