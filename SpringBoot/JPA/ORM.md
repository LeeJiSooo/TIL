## 패러다임의 충돌과 ORM의 등장

- 자바와 DB의 차이
    - 자바 : 철저한 객체지향 언어. 모든 데이터와 값을 객체 중심으로 다루고 설계한다.
    - DB(관계형 데이터베이스, RDB) : 테이블 중심. 행과 열로 데이터로 다루며 정규화 규칙을 따른다.
- 현실적인 개발 환경의 문제점
    - 우리가 하는 백엔드 개발은 결국 Java/Spring으로 DB에 있는 데이터를 가져와 가공하고 저장하는 일이다.
    - 태생이 다른 두 기술이 만나다 보니 구조적 충돌이 일어난다. 과거에는 이 충돌을 메우기 위해 개발자가 자바 코드 안에서 직접 SQL쿼리를 한 줄 한 줄 치고, DB에서 꺼내온 데이터를 객체 필드에 일일이 매핑하는 단순 반복 작업을 해야 했다.
- 해결 : ORM
    - 이 반복적인 작업을 자동화하여 개발자를 편하게 만들어 준 기술적 개념이 바로 ORM이다.

## 가져가야 할 핵심

1. ORM의 정확한 의미와 자바와 데이터베이스 사이에서 중간 계층으로서 어떤 역할을 하는지 이해한다.
2. 객체지향 모델과 관계형 데이터베이스 모델 간의 근본적인 간극인 **패러다임 불일치**의 실체를 파악한다.
3. SQL 중심 개발의 한계를 깨닫고, ORM을 도입했을 때 얻을 수 있는 설계의 일관성과 코드 생산성 향상을 이해한다.

## ORM(Object-Relational Mapping)

자바의 객체와 데이터베이스 테이블을 자동으로 연결하고 반환해, SQL 중심이 아닌 객체지향 코드로 데이터를 조작할 수 있게 해주는 기술이다.

### 중간 계층으로서의 역할

- ORM은 객체와 테이블 간의 구조적 차이를 중간 계층에서 스스로 해소해 준다.
- 덕분에 백엔드 개발자는 복잡한 SQL을 직접 작성하지 않고도, 익숙한 객체지향 기반 코드로 데이터베이스를 다룰 수 있게 된다.
    - 객체 : 클래스와 필드 중심의로 설계
    - DB : 테이블과 컬럼 중심으로 정규화에 맞춰 설계
    
    → 두 세계의 모양이 다르지만, 중간에 ORM이 매끄러운 번역기 역할을 하며 연결한다.
    

### 회원가입(저장 및 조회)

- 과거(SQL 중심 개발)
    - 자바에서 new User()로 객체를 생성해 이름, 이메일 등을 세팅한 뒤, 저장하려면 자바 코드 안에 `"INSERT INTO users (name, email...) VALUES (?, ?)”` 같은 복잡한 SQL문자열을 직접 작성하고 파라미터를 하나씩 바인딩해야 했다.
    - 조회할 때도 `SELECT`쿼리를 실행한 뒤, 결과셋(ResultSet)에서 데이터를 하나씩 꺼내 자바 객체에 담아주는 포장 코드를 일일이 짜야 했다.
- 현재(ORM 사용 개발)
    - 개발자는 ORM이 제공하는 메소드(예: `persist()`)를 호출하여 “이 객체좀 저장해줘”라고 요청만 하면 끝난다.
    - 그러면 ORM이 자바 객체의 클래스 이름과 필드를 자동으로 분석해, 매핑된 users테이블 구조에 맞는 INSERT SQL을 스스로 생성하고 실행까지 대신해 준다.
    - 조회할 때도 ORM이 테이블 데이터를 가져와 자동으로 `User` 객체에 포장해서 개발자에게 전달한다.

→ SQL은 자바와 완전히 다른 또 하나의 언어이다. 자바 개발을 하다가 데이터를 만지기 위해 수시로 다른 언어(SQL)의 사소한 문법까지 신경 쓰는 것은 복잡하고 비효율적이다. → ORM은 개발자가 오직 객체지향 프로그래밍 자체와 비즈니스 로직에만 집중할 수 있도록 돕는다.

## 왜 쓰는가? 패러다임 불일치(간극)

객체지향과 관계형 데이터베이스 간의 패러다임 불일치를 해결하여, 설계의 일관성을 유지하고 개발 생산성을 극대화하기 위해 사용한다.

### 상속의 차이

- **객체지향:** 상속이라는 개념이 매우 자연스럽고 강력하게 쓰인다. 부모 클래스 `Item`이 있으면 이를 상속받는 자식 클래스 `Album`, `Movie`, `Book`을 쉽게 만들 수 있다.
- **RDB:** 테이블에는 **상속이라는 개념이 아예 존재하지 않는다.** (유사한 기법으로 슈퍼타입-서브타입 관계를 설계하긴 하지만, 객체의 상속 메커니즘과는 완전히 다르다.)
- **문제점:** 자바에서 `Album` 객체 하나를 저장하려고 하면, 개발자가 SQL 중심일 때는 `Item` 테이블에 한 번 INSERT 하고, `Album` 테이블에 또 한 번 INSERT 하는 쿼리를 각각 직접 다 짜야 하므로 번역하기 번거롭고 복잡하다.

### 연관 관계(Association)와 방향성

- **객체지향:** 객체는 '참조(Reference)'를 통해 관계를 맺는다. `member.getTeam()`처럼 내부에 다른 객체의 주소값을 필드로 두고 찾아간다. 이 관계는 **단방향**이다. 회원이 팀 참조를 갖고 있어도, 팀 객체 내에 회원 리스트 참조가 없다면 팀에서 회원을 찾을 수 없다.
- **RDB:** 테이블은 '외래키(FK, Foreign Key)'를 사용해 `JOIN`으로 관계를 맺는다. `MEMBER` 테이블에 `TEAM_ID`라는 외래키 하나만 있으면, 회원에서 팀을 조인할 수도 있고 팀에서 회원을 조인할 수도 있는 **양방향** 구조이다.
- **문제점:** 객체 모델링을 살려 `Member` 클래스 안에 `Team team`이라는 참조 필드를 두면, DB에 저장할 때는 매번 `member.getTeam().getId()`를 호출해 외래키 값으로 수동 변환해 줘야 한다. 반대로 조회할 때도 `TEAM_ID`를 먼저 뽑아낸 뒤, 그걸로 다시 `Team` 객체를 조회해서 `Member` 안에 셋팅해 줘야 하므로 자바와 DB 간의 호환이 심각하게 불편하다.

### 데이터 탐색(Navigation)의 한계

- **객체지향:** 참조만 올바르게 걸려있으면 점(`.`)을 찍어서 끊임없이 객체 그래프를 탐색할 수 있다.
    - *ex) `member.getTeam().getCompany().getName()`*
- **RDB:** SQL을 실행하는 순간 **탐색할 수 있는 데이터의 범위가 딱 고정**된다.
- **문제점:** 만약 처음에 회원 테이블과 팀 테이블만 `JOIN`해서 데이터를 가져왔다면, `member.getTeam()`까지는 조회가 잘 된다. 하지만 이 상태에서 무심코 `member.getTeam().getCompany()`를 호출하는 순간 데이터가 채워져 있지 않아 `NullPointerException(NPE)`이 터진다. 결국 개발자는 "이 코드가 어디까지 안전하게 점을 찍고 탐색할 수 있지?"를 확인하기 위해 **매번 작성된 SQL 쿼리문을 직접 확인해야만 하는 강한 종속성**이 발생한다.

### 데이터 식별 방법(Identity)의 모순

- **RDB:** 테이블은 기본키(PK) 값이 같으면 무조건 같은 행으로 식별한다.
- **객체지향:** 자바는 두 객체의 변수가 가리키는 메모리 주소값(`==`)으로 동일성을 식별한다.

```java
// 순수 JDBC / DAO 환경에서의 두 번 조회
Member member1 = memberDAO.getMember(100);
Member member2 = memberDAO.getMember(100);

System.out.println(member1 == member2); // 결과: false
```

**문제점:** `getMember(100)`을 호출할 때마다 매번 내부에서 `new Member()`로 새로운 객체를 생성해 반환하므로 주소값이 다르다. 데이터베이스에서는 분명 아이디가 100인 '같은 회원'인데, **자바 코드 상에서는 '다른 객체'로 취급받는 모순**이 생긴다.

## ORM의 이점

### 순수 JDBC를 사용한 SQL중심 코드(ORM 미사용)

개발자가 SQL 문자열을 직접 다뤄야 하므로 오타에 취약하고 반복 코드가 많다.

```java
public void save(User user) {
    // 문자열 쿼리 직접 작성 (오타 나면 컴파일 시점에 못 잡고 런타임 에러 발생)
    String sql = "INSERT INTO users (id, name, email) VALUES (?, ?, ?)";

    try (Connection conn = dataSource.getConnection();
         PreparedStatement pstmt = conn.prepareStatement(sql)) {

        // 일일이 객체에서 값을 꺼내서 쿼리에 바인딩해 주는 노가다성 코드
        pstmt.setLong(1, user.getId());
        pstmt.setString(2, user.getName());
        pstmt.setString(3, user.getEmail());

        pstmt.executeUpdate(); // SQL 직접 실행

    } catch (SQLException e) {
        e.printStackTrace();
    }
}
```

### JPA(ORM)를 사용한 객체 중심 코드

자바 객체를 데이터베이스 인터페이스에 밀어 넣듯, 단 한줄로 깔끔하게 처리가 가능하다.

```java
public void save(User user) {
    // 객체를 영속성 컨텍스트에 저장하겠다고 선언하면 끝 (SQL은 ORM이 알아서 생성)
    entityManager.persist(user);
}
```

Q. ORM이 무엇이고, 이를 사용하는 핵심 이유는 무엇인가요?

ORM은 자바 객체와 관계형 데이터베이스 테이블 사이의 번역기 역할을 해주는 중간 계층 기술이다.

이를 사용하는 근본적인 이유는 객체지향 모델과 관계형 데이터베이스 모델이 가진’패러다임 불일치’를 해결하기 위함이다. 

구체적으로는 상속 구조의 부재, 참조와 외래키의 연관관계 차이, 객체 그래프 탐색의 제한, 그리고 동일성 식별 모순(동일한 PK 조회시 == 비교 실패)등의 문제를 ORM이 중간에서 자동으로 매핑하고 해결해 준다.

결과적으로 개발자가 SQL 반복 작성할 필요 없이 오직 비지니스 로직 구현과 객체지향적 설계에만 집중할 수 있도록 생산성을 높여주는 도구이다.

## 정리

- ORM은 자바 객체와 관계형 데이터베이스 사이를 연결해 주는 개념이자 기술이다.
- 두 기술 간의 구조적 차이인 ‘패러다임 불일치’를 중간에서 완벽히 해결한다.
- 개발자가 SQL에 종속되지 않고 비즈니스 로직에만 집중할 수 있는 환경을 제공한다.
- ORM은 거대한 ‘개념’이며 자바 진영에서는 이 ORM이라는 개념을 구현하기 위해 JPA라는 표준 인터페이스 규격을 만들어 명시해 놓은 것이다.
