## 가져가야 할 핵심

1. 서비스 계층에서 JPA 기술을 격리시키는 관심사의 분리 목적을 명확히 이해한다.
2. 자바 인터페이스일 뿐인 리포지토리가 어떻게 구현체 없이 작동하는지 런타임 동적 프록시 생성 원리를 파악한다.
3. 리포지토리의 실제 알맹이인`SimpleJpaRepository`와 내장된 트랜잭션 메커니즘을 이해한다.

## 사용 이유 : 관심사의 분리와 도메인 단위 분리

만약 서비스 계층 코드에 `em.createQuery("select u from User u", User.class)` 같은 JPA 코드가 날것 그대로 섞여 있다면 어떻게 될까?

### 관심사의 분리가 깨진 나쁜 아키텍처

- **서비스 계층의 본분:** "사용자가 회원가입을 하면 환영 이메일을 보낸다" 같은 순수한 비즈니스 로직(도메인 규칙)만 담당해야 하는 곳이다.
- **문제점:** 여기에 "테이블명이 무엇이고, 쿼리를 어떻게 짜고, 트랜잭션을 언제 커밋할지" 같은 **구체적인 DB 기술**이 들어가는 순간 코드가 지저분해지고 결합도가 높아진다. 나중에 데이터베이스를 바꾸거나 JPA에서 다른 기술로 전환할 때 서비스 코드를 전부 변경해야 한다.

### Repository 계층 분리

데이터 접근 로직을 도메인 단위로 묶어 `Repository` 계층으로 독립시키면 다음과 같은 장점이 생긴다.

- **기술의 격리:** 서비스 계층은 이제 DB가 MySQL인지, Oracle인지, 영속성 도구로 JPA를 쓰는지 MyBatis를 쓰는지 전혀 몰라도 된다.
- 서비스는 그냥 `userRepository.findByEmail(email)`을 호출하며 **"이메일로 사용자 좀 찾아줘"라는 비즈니스적인 의도**만 던지면 끝이다. DB 구조가 바뀌거나 쿼리 튜닝을 해야 할 때 서비스 코드는 바꾸지 않아도 오직 리포지토리 계층만 수정하면 되므로 **유지보수성과 재사용성이 상승**한다.

## 구현체 없는 인터페이스가 작동하는 ‘동적 프록시’

자바 문법상 인터페이스는 스스로 객체가 될 수 없어 구현클래스가 필요하다. 그런데 우리는 아래처럼 선언만 해놓고 어떻게 바로 가져다 쓸까?

```java
@Repository
public interface UserRepository extends JpaRepository<User, Long> {
    // 아무것도 안 짰는데 save(), findById()가 그냥 동작한다.
}
```

### 스프링 컴포넌트 스캔 시점

1. 애플리케이션이 구동될 때, 스프링 데이터 JPA는 프로젝트 내부를 훑으며 `JpaRepository`를 상속받은 리포지토리 인터페이스들을 찾는다.
2. 그리고 자바의 **동적 프록시(JDK Dynamic Proxy) 기술**을 발동시켜서, 런타임(메모리)에 해당 인터페이스를 구현한 가짜 객체(프록시 구현체)를 만든다.
3. 이 동적으로 만들어진 가짜 구현체 객체를 **스프링 컨테이너의 '빈(Bean)'으로 자동 등록**해 준다.
4. 덕분에 서비스 계층에서 `@RequiredArgsConstructor`로 의존성을 주입받을 때, 우리가 직접 만들지도 않은 구현 객체가 주입되어 코드가 돌아간다.

## SimpleJpaRepository

*"스프링이 가짜 프록시 객체를 동적으로 만들어주는 건 알겠는데, 그럼 그 안에서 실행되는 진짜 `save()`나 `delete()` 소스코드는 어디에 적는가?"*

스프링 데이터 JPA 내부에는 진짜 클래스인 `SimpleJpaRepository`가 미리 구현되어 준비되어있다.

```java
// 스프링 데이터 JPA가 내장하고 있는 실제 알맹이 클래스의 대략적인 모습
@Repository
@Transactional(readOnly = true) -- 3. 기본적으로 조회용 트랜잭션 장착
public class SimpleJpaRepository<T, ID> implements JpaRepositoryImplementation<T, ID> {

    private final EntityManager em; // 1. 결국 내부선 엔티티 매니저를 씀

    @Transactional -- 4. 쓰기 작업이 필요한 메서드에는 커밋 권한을 다시 부여
    public <S extends T> S save(S entity) {
        if (entityInformation.isNew(entity)) {
            em.persist(entity); // 2. 결국 순수 JPA로 복귀
            return entity;
        } else {
            return em.merge(entity);
        }
    }
}
```

우리가 `userRepository.save(user)`를 호출하면, 프록시 가짜 객체가 이 요청을 받아서 진짜 알맹이인 `SimpleJpaRepository`의 `save()` 메서드로 위임해 주는 구조다. 거기서 식별자 유무 판별식에 따라 `em.persist()`나 `em.merge()`가 실행된다.

**Q. 서비스 계층에 `@Transactional`을 깜빡하고 안 적었는데, `userRepository.save()`를 호출하면 데이터가 DB에 저장이 될까요? 아니면 에러가 날까요?**

**에러 없이 완벽하게 저장된다.**

- 공통 구현체 클래스 `SimpleJpaRepository` 내부를 보면, 클래스 자체와 `save()`, `delete()` 같은 데이터 변경 메서드 위에 **`@Transactional`이 이미 자체적으로 붙어있기 때문이다.**
- 서비스 레이어에 트랜잭션이 없더라도, 리포지토리의 `save()` 메서드가 실행되는 그 찰나에 **메서드 단위로 새로운 트랜잭션이 시작되고 메서드가 끝날 때 커밋까지 처리**하고 나오기 때문에 에러 없이 정상 저장이 된다.

## 정리

- **리포지토리의 역할:** 영속성 작업의 보일러플레이트 코드를 추상화하여 없애주는 **데이터 접근 계층의 표준 인터페이스**이다.
- **설계 목적:** 서비스 계층에서 지저분한 DB 연결 기술이나 SQL/JPQL 문법을 완벽히 격리시켜 **관심사의 분리**를 달성하고, 아키텍처의 유연성을 극대화한다.
- **동적 프록시 기술:** 런타임에 스프링이 인터페이스를 상속받는 **가짜 구현체(프록시)를 메모리에 동적 생성하여 스프링 빈으로 등록**하므로, 개발자는 구현 클래스를 직접 짤 필요가 없다.
- **SimpleJpaRepository:** 모든 공통 리포지토리 메서드의 진짜 알맹이 코드가 작성되어 있는 실체 클래스이며, **내부에 자체적인 `@Transactional` 방어벽**을 치고 있어 서비스단에 트랜잭션이 누락되더라도 안전하게 독립적인 CRUD를 할 수 있다.
