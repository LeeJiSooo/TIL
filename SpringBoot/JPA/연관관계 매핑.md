## 가져가야 할 핵심

1. 관계형 DB의 외래키 방식과 객체지향의 참조방식의 차이점과 이로 인한 패러다임 불일치를 명확하게 이해한다.
2. 객체 그래프 탐색이 무엇이며, 이것이 왜 우리 코드를 데이터베이스 종속적인 코드에서 순수한 객체지향 코드로 바꾸어 주는지 파악한다.

- 우리는 테이블에서 외래키를 두고, 필요할 때 JOIN을 쿼리로 날려서 데이터를 엮어 가져오는 방식을 배웠다.
- 하지만 우리가 실제로 비즈니스 로직을 짜는 것은 자바 언어 기반의 객체지향 프로그래밍 세계이다.

**패러다임의 불일치**

- 데이터베이스의 세계  : 오직 외래키 하나만을 중심으로 관계를 맺는다.
- 객체의 세계 : 외래키라는 개념 자체가 없으며, 오직 참조를 통해서만 다른 객체와 관계를 맺는다.

→ 이 방향성과 표현 방식의 차이를 매꿔주는 기술이 연관관계 매핑이다.

## 데이터베이스 vs 자바 객체의 결정적 차이

### 데이터베이스 : 외래키 하나로 양방향 자유롭게 가능

- users 테이블과 post 테이블이 있다면, 다(N)에 해당하는 post테이블에 user_id라는 외래키 컬럼을 단 하나만 파서 연결한다.
- 데이터베이스는 이 외래키 하나만 있으면 회원 게시글을 찾든(`users JOIN post`), 게시글에서 회원을 찾든(`post JOIN users`) **조회 관점에서 양방향 넘나들기**가 가능하다.

```sql
-- 시나리오 A: 게시글(1번) 기준으로 작성자 이름 가져오기
SELECT p.title, u.name FROM post p JOIN users u ON p.user_id = u.id WHERE p.id = 1;

-- 시나리오 B: 특정 회원(1번)이 작성한 게시글 목록 가져오기
SELECT p.title, u.name FROM users u JOIN post p ON p.user_id = u.id WHERE u.id = 1;
```

### 자바 객체 : 참조가 있는 쪽으로만 갈 수 있음

- 자바 프로그래밍 언어에는 JOIN이라는 문법이 존재하지 않는다. A객체가 B객체를 알려면, A객체 내부에 B객체의 주소값(참조)을 필드로 쥐고 있어야만 한다.

```java
public class Post {
    private User user; // Post에서 User를 참조
}
```

- Post에 User user 필드만 있으면, 우리는 게시글을 통해 유저는 찾을 수 있어도 유저 객체만 쥔 상태에서 그 유저가 쓴 게시글 목록으로 가는 것은 불가능하다. 그러고 싶다면 User클래스 안에서도 List<Post>필드로 추가해야한다.

## 연관관계 매핑이 없으면?

만약 연관관계 매핑을 모른다면, 개발자는 데이터베이스 테이블 구조를 자바 클래스에 그대로 복사한다. 객체 참조가 아니라 식별자 숫자를 그대로 들고 있는 구조다.

**매핑 없이 실별자(Long userId)를 직접 다루는 코드**

```java
@Entity
public class Post {
    @Id @GeneratedValue
    private Long id;
    private String title;

    @Column(name = "user_id")
    private Long userId; // 객체지향적이지 못하고 DB 테이블을 흉내 낸 식별자 숫자 필드
}
```

**저장할 때의 문제**

```java
Post post = new Post();
post.setTitle("안녕하세요");
post.setUserId(user.getId()); // 객체를 주입하는 게 아니라, 유저 객체에서 굳이 '숫자 ID'를 꺼내서 주입해야 함
em.persist(post);
```

**조회할 때의  문제**

```java
Post findPost = em.find(Post.class, 1L);

// 작성자의 이름을 출력하고 싶을 뿐인데...
Long authorId = findPost.getUserId();          // 1. 게시글에서 숫자 ID를 추출한다.
User author = em.find(User.class, authorId);   // 2. 그 숫자를 가지고 DB나 1차 캐시를 대조해 유저를 또 조회한다. (쿼리/메서드 추가 호출)
System.out.println(author.getName());          // 3. 이제야 이름을 본다.
```

자바 코드가 데이터 베이스 구조에 종속되어 버린다. 개발자는 객체를 다루는 것이 아니라, 식별자 숫자를 뽑아서 조립하는 노동을 반복하게 된다.

## 연관관계 매핑 적용

JPA에게 어노테이션을 통해 객체 간의 관계성을 일러두면, 자바 코드는 완벽하게 객체지향적인 모습을 되찾는다.

```java
@Entity
public class Post {
    @Id @GeneratedValue
    private Long id;
    private String title;

    @ManyToOne // 여러 개의 게시글(Many)은 하나의 회원(One)에 속한다
    @JoinColumn(name = "user_id") // 실제 DB에 저장될 때는 user_id 외래키 컬럼에 매핑해준다.
    private User user; // 식별자 숫자가 아닌 '객체 자체의 참조'를 필드로 선언
}
```

- 자바 코드에서는 Post 객체가 User객체를 직접 참조하게 둔다. 하지만 이걸 실제 DB테이블에 저장할 때는 User객체의 PK를 빼서 post테이블의 user_id외래키 컬럼에 넣는다. 반대로 DB에서 읽어올 때는 외래키를 기준으로 연관된 User 객체를 조인해서 찾아와 자바 필드에 채워준다.

저장할 때 좋음

```java
Post post = new Post();
post.setTitle("안녕하세요");
post.setUser(user); // 유저 객체 자체를 통째로 주입(JPA가 알아서 FK로 변환해 저장함)
em.persist(post);
```

조회할 때 좋음(객체 그래프 탐색)

```java
Post findPost = em.find(Post.class, 1L);
System.out.println(findPost.getUser().getName()); // 점(.) 하나 찍어서 단 한 줄로 끝
```

## 객체 그래프 탐색

우리가 위에서 연관관계 매핑을 잘 해놓은 덕분에, 게시글 가져온 뒤 작성자의 이름을 알고 싶다면 자바 코드 상에서 점만 찍어서 선을 타고 이동할 수 있다.

- post.getUser().getName();

→ 이걸 객체 그래프 탐색이라고 한다. 개발자가 복잡한 SQL JOIN문을 직접 타이핑하지 않아도, 연관 관계 매핑 정보가 잘 되어 있다면 자바 코드에서 객체를 따라 점만 찍어도 JPA가 알아서 조인 쿼리를 생성 및 실행해서 연관 객체를 준다.

→ 개발자는 오직 순수한 비즈니스 로직 설계와 객체지향 프로그래밍 자체에만 몰입할 수 있다.

## 사용 이유 : 데이터 정합성과 비즈니스 의도의 명확화

1. 비즈니스 도메인의 규칙 표현 : post.setUser(user)라는 코드는 단순히 데이터베이스 컬럼에 숫자 형태의 데이터 코드를 채워넣으라는 것이 아니다.
    
    “이 게시글의 저자는 이 회원이다”라는 도메인의 제약과 비즈니스적인 관계의 정합성을 자바 언어의 특성을 살려 명확하게 표현하는 행위이다.
    
2. 참조 무결성 에러 차단 : 식별자 숫자를 직접 다루다 보면 실수로 DB에 존재하지 않는 식별자를 주입해서 데이터베이스의 참조 무결성 제약조건이 깨지는 사고가날 수 있다. 하지만 객체 참조 방식을 쓰면 반드시 영속 상태인 실제 객체만 주입할 수 있으므로, 플러시 시점에 JPA가 안전하게 최신 상태를 읽어 쿼리를 날려주어 데이터 정합성이 방어된다.

## 정리

- 데이터베이스는 외래키로 관계를 맺고 양방향 소통하지만, 자바는 참조로 관계를 맺고 일방 통행한다. 이 간극을 채우는 것이 연관관계 매핑이다.
- 매핑이 완벽하게 세팅되려면 우리는 SQL에 종속되지 않고 점만 찍어서 연관 데이터를 자유롭게 넘나들며 가져오는 객체 그래프 탐색을 할 수 있다.
- 이를 통해 개발자는 단순 SQL 반복 작성하지 않고 순수한 객체지향 비즈니스 도메인 설계에만 집중할 수 있고, 데이터 정합성까지 보장받는다.
