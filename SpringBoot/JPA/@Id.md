- **`@Entity`의 복습**
    - 자바 클래스 위에 `@Entity`를 붙여주면 JPA가 "아, 해당 클래스는 데이터베이스에 있는 어떤 테이블과 연결되는 클래스구나"라고 관리 대상으로 인식한다.
- **DB 테이블 설계의 필수 요소: PK (기본키)**
    - 데이터 행이 아무리 많이 쌓여있어도 "이 데이터가 바로 그 데이터다"라고 유일하게 식별할 수 있는 기준이 되며, 절대 중복을 허용하지 않는다.
    - 데이터베이스에서는 DDL을 통해 테이블 생성 시 PK(Primary Key) 제약조건을 붙여준다.
- **JPA의 식별 방식 해결**
    - **자바 객체:** 메모리 주소(참조값)를 기준으로 객체를 식별한다.
    - **데이터베이스:** 기본키(PK) 값으로 행을 식별한다.
    - 이 서로 다른 **두 식별 방식을  연결시켜 주는 매핑 도구**가 바로 `@Id` 어노테이션이다.

## 가져가야 할 핵심

1. JPA가 엔티티를 인식하고 식별하는 유일한 기준으로서 `@Id` 가 가진 역할과 중요성을 이해한다.
2. 비즈니스 의미가 있는 ‘자연키’와 인위적인 ‘대리키’의 차이점을 파악하고, 실무에서 왜 무조건 대리키를 사용해야 하는지 알아야 한다.
3. @Id가 단순히 DB의 PK컬럼 매핑을 넘어, 영속성 컨텍스트(1차 캐시, 변경 감지 등)가 엔티티를 제어하는 키 역할을 수행함을 이해한다.

## @Id

엔티티의 식별자를 지정해 주는 기본 키 매핑 어노테이션

### 식별자 할당 방식

1. 직접 할당 : 개발자가 자바 코드에서 `user.setId("user_123")`와 같이 **PK 값을 직접 세팅**해서 DB에 저장하는 방식이다.
2. **자동 생성 (`@GeneratedValue`):** 데이터베이스의 `AUTO_INCREMENT` 기능이나 `SEQUENCE` 객체를 활용하여, 개발자가 직접 값을 넣지 않아도 **데이터베이스가 알아서 1, 2, 3, 4... 순차적으로 번호(PK)를 매겨주는 방식**이다.

## 자연키 vs 대리키(실무에서 대리키를 써야 하는 이유)

회원 테이블을 설계할 때 어떤 컬럼을 @Id로 지정해야 할까? 이때 키의 성격에 따라 자연키와 대리키로 나뉜다.

- 자연키 : 주민등록번호, 이메일, 전화번호처럼 비즈니스적인 의미를 가지고 있으면서 중복되지 않는(유일성을 만족하는)실제 데이터 컬럼
- 대리키 : 비즈니스적인 의미는 전혀 없으나, 오직 식별만을 위해 인위적으로 만들어낸 키/ 1,2,3…처럼 자동으로 순차 증가하는 숫자형 값이나 복잡한 난수 문자열인 UUID등이 해당된다.

### 자연키를 썼을 때 발생하는 시나리오

#### 이메일을 PK로 썼을 때

- 기획팀에서 유저들의 요구로 인해 "이메일 주소 변경 기능을 추가해 주세요"라는 요청이 들어왔다.
- 이메일이 기본키(PK)이므로, 사용자의 이메일을 수정하는 것은 **DB 관점에서 엄청난 물리적 비용을 수반**한다.
- 단순히 `User` 테이블의 값 하나만 바꾸고 끝나는 게 아니다. 회원 테이블의 PK(이메일)는 이미 게시글, 주문, 결제, 장바구니 등 **수십 개의 다른 연관 테이블에서 외래키(FK)로  잡고 인덱싱되어 있다.**
- 결국 이메일 딱 하나 변경하겠다고 연관된 수십 개 테이블의 데이터를 전부 찾아서 대규모 `UPDATE` 쿼리를 날려야 한다. 이 과정에서 **DB에 심각한 락(Lock)이 걸려 서비스가 마비되거나 데이터 정합성이 깨지는 대형 장애**가 발생한다.

#### 주민등록번호를 PK로 썼을 때

- 주민등록번호는 변하지 않는 고유한 값이니 안전할 줄 알고 PK로 삼았다.
- 그런데 국가의 **개인정보보호법이 강화**되어 "이제 시스템 DB에 주민등록번호를 원본 그대로 저장하거나 식별자로 쓰면 안 됩니다"라고 법이 바뀐다.
- 그 법령이 시행되는 순간, **우리 서비스의 핵심 기본키(PK) 구조 전체가 불법**이 되어 시스템을 통째로 갈아엎어야 한다.

### 올바른 해결책

비즈니스 환경과 요구사항이 어떻게 변하든 ‘절대 변하지 않는 안전한 값’은 비즈니스 의미가 없는 대리키이다.

- 이메일이나 전화번호 같은 유일한 값들은 DB에 유니크 제약조건만 걸어서 비즈니스적인 중복 가입을 막는 용도로 쓴다.
- 진자 시스템 식별자는 아무 의미 없는 대리키에게 맡겨서 처리해야 안전하다.

## 왜 쓰는가?

`@Id` 로 지정한 기본키는 영속성 컨텍스트 안에서 엔티티를 찾고, 비교하고, 관리하기 위한 유일한 키이기 때문에 필수적이다.

JPA가 엔티티를 자동 관리해 주는데, 이 중심에 `@Id`가 있어야 아래와 같은 핵심 기능들이 정상적으로 돌아간다.

### 1차 캐시와 `@Id`

- 1차 캐시는 같은 트랜잭션 안에서 한 번 꺼내온 데이터를 자바 메모리에 잠깐 들고 있는 임시 보관함이다.
- 영속성 컨텍스트는 내부적으로 마치 자바의 Map 자료구조처럼 데이터를 보관한다.
    - Key : 엔티티의 `@Id` 값
    - Value : 엔티티 객체의 주소값
- 우리가 `em.find(User.class, 1L)`을 할 때, 영속성 컨텍스트가 *"얘가 지금 몇 번 회원을 찾는 거지?"* 하고 식별할 수 있는 기준이 바로 `@Id`에 매핑된 값이다. 이 key가 존재하기 때문에 두 번째 조회 시 DB에 가지 않고 메모리에서 똑같은 객체를 그대로 반환할 수 있는 것이다.

### 변경 감지(Dirty Checking)와 `@Id`

- JPA는 조회해 온 엔티티의 값을 바꾸기만 해도, 개발자가 수정 명령(`update()`)을 내리지 않아도 트랜잭션이 끝날 때 알아서 DB에 `UPDATE` 쿼리를 날려준다.
- `user.setNickname("새이름")`만 호출해도 영속성 컨텍스트는 최초 조회 시점의 스냅샷과 비교하여 변경 사항을 감지한다.
- 이때 **어떤 행(Row)의 닉네임을 업데이트해야 하는지 알아야 하므로**, 영속성 컨텍스트는 엔티티가 가진 `@Id` 값을 파싱해 `WHERE id = ?` 조건문을 완성하여 안전하게 DB에 반영한다.

## 정리

- @Id는 JPA가 엔티티 객체를 식별하기 위해 사용하는 유일하고 절대적인 기준이며, 데이터베이스의 PK와 1:1로 매핑된다.
- 실무 환경과 기획 정책은 언제든 변할 수 있으므로, 비즈니스 의미가 있는 자연키(이메일, 주민번호 등)대신 의미 없는 순차 번호나 난수 같은 대리키를 무조건 PK로 선택해야 연쇄 변경 문제를 막을 수 있다.
- @Id는 단순한 DB매핑 어노테이션의 역할을 넘어, JPA 핵심 엔진인 영속성 컨텍스트에서 1차 캐시 저장/ 조회 및 변경 감지를 수행할 때 데이터를 식별하는 유일한 열쇠로 작동한다.
