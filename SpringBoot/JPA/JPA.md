- ORM(객체 관계 매핑)은 객체지향과 RDB 사이의 패러다임 불일치를 해결해 주는 ‘개념’이자 ‘이론’이다.
- 이 개념을 자바 언어에 맞게 공식적으로 정립한 기술이자, 자바 ORM 생태계의 시작점이 바로 JPA이다.

## 가져가야 할 핵심

JPA의 명확한 개념을 파악하고, 개발자가 마주하는 JDBC, 하이버네이트와의 유기적인 연결 관계를 완벽히 이해한다.

## JPA(Java Persistence API)

자바 객체와 관계형 데이터베이스를 매핑하고, 객체 단위로 데이터를 저장 및 조회할 수 있게 해주는 자바 진영의 공식 표준 ORM 인터페이스이다.

### 인터페이스와 구현체의 관계(JPA vs 하이버네이트)

- JPA는 표준 명세서(인터페이스)다.
    - JPA는 실제 스스로 움직이는 코드가 아니다. 자바에서 ORM 기술을 만드려면 “이런 이름의 메소드를 만들고, 이런 규칙을 지켜서 작동하게 해라”라고 정해둔 공식 규칙이자 가이트라인(인터페이스 모음)일 뿐이다.
- 하이버네이트는 실제 동작하는 구현체이다.
    - JPA라는 인터페이스를 바탕으로, 실제 내부에 살을 붙여서 돌아가게 만든 구체적인 소프트웨어가 하이버네이트이다.
    - 자바 ORM진영에서 하이버네이트의 점유율은 압도적으로 높기 때문에, 실무에서는 다른 구현체를 고민할 필요가 없다.

→ 우리가 자바에서 ORM을 쓴다고 할 때, 개념적으로는 ‘JPA 표준’을 따르는 것이고 실제 물리적으로는 그 안에서 동작하는 ‘하이버네이트’라는 도구를 가져다 쓰는 것이다.

## JDBC와의 관계

JPA를 사용하면 개발자가 JDBC를 직접 다루지 않고도 객체를 DB에 넣고 뺼 수 있다. 

### JDBC(Java Database Connectivity)의 역할

- 자바 어플리케이션이 데이터베이스와 연결되기 위한 가장 밑바닥에 존재하는 표준API이다.
- 어떻게 DB와 커넥션을 맺고, 쿼리를 전달하고, 결과를 받아 처리할 것인가에 대한 원초적인 도구이다.
- MySQL을 쓰든, 오라클을 쓰든, 어떤 DB를 쓰더라도 자바 프로그램은 결국 이 JDBC를 거쳐서만 데이터베이스와 통신할 수 있따.

- 과거 : 개발자가 JDBC를 직접 다루며 쿼리를 짜는 것은 엄청난 노동이고 피곤하며 번거로운 일이었다.(커넥션 열고, 닫고, 예외 처리하고, 파라미터 맵핑..등)
- 현재(JPA 도입) : 개발자가 JPA를 사용하면 JDBC를 직접 호출할 필요가 없다. JPA(하이버네이트)가 알아서 객체의 구조를 분석해 적절한 SQL을 만들고, 내부적으로 알아서 JDBC API를 호출해 처리를 끝내준다.

개발자가 작성한 **Spring 애플리케이션**이 규칙인 **JPA 인터페이스**를 호출하면 →실제 핵심 엔진인 **하이버네이트 구현체**가 돌면서 SQL을 생성하고 → 자바 최하단 통신 도구인 **JDBC**를 찔러서 → 최종적으로 데이터베이스(DB)와 통신하게 된다.

## 1차 캐시

JPA를 이해할 때 절대 빼놓을 수 없는 임시 메모리 공간이 **1차 캐시**이다.

### 1차 캐시의 작동 원리(동일 트랜잭션 기준)

1. 같은 트랜잭션 안에서 개발자가 ID가 1번인 회원을 최초로 조회(`em.find(Member.class, 1L)`)했다고 가정하자.
2. JPA는 먼저 데이터베이스로 가서 데이터를 가져온 뒤, 자바 객체로 변환한다.
3. 이때, 객체를 그냥 주는 게 아니라 J**PA만의 임시 메모리 공간인 ‘1차 캐시’에 올려둔다.**
4. 그러고 나서 똑같은 트랜잭션 안에서 ID가 1번인 회원을 **한 번 더 조회**하면?
5. JPA는 이번에 DB로 네트워크 통신을 하러 가지 않는다. 방금 전 채워둔 **1차 캐시를 먼저 확인하고**, **아까 조회해둔 1번 회원이 있으니까 메모리에서 그걸 그대로 꺼내서 반환**해 준다.

### 1차 캐시가 주는 이점

- DB 접근 횟수 축소: 불필요하게 똑같은 `SELECT` 쿼리를 여러 번 날리지 않으므로 성능상 이점이 있다.
- 객체 동일성 보장 : 메모리에 있는 동일한 객체 주소값을 그대로 던져주기 때문에, 앞서 배운 `user1 == user2` 같은 동일성 비교 시 `true`가 완벽히 보장된다.

Q. JPA와 하이버네이트(Hibernate), 그리고 JDBC의 관계와 차이점을 설명해 주세요.

JPA는 공식 인터페이스 명세이고, 하이버네이트는 이를 실제로 구현한 동작 구현체이다. JDBC는 데이터베이스와 물리적으로 통신하는 최하단의 표준 AP이다.

개발자가 스프링 환경에서 JPA를 사용해 객체 중심으로 코드를 짜면, 내부적으로 하이버네이트가 객체를 분석해 SQL을 생성하고, 자바의 가장 밑바닥 통신 도구인 JDBC API를 대신 호출하여 DB와 통신해 준다. 

덕분에 개발자는 SQL과 JDBC API를 직접 다루지 않고도 객체지향적인 비즈니스 로직에만 몰두할 수 있게 된다.

## 정리

- JPA는 자바의 공식 ORM 표준 인터페이스 명세서이다.
- 이를 실제로 움직이게 만드는 대표적인 구현체가 하이버네이트이다.
- JPA를 도입함으로써 개발자는 복잡하고 피로한 **JDBC API나 SQL을 직접 다루지 않고**, 완벽히 객체지향적인 코드 작성에만 집중할 수 있다.
- 내부적으로 작동하는 **1차 캐시**라는 메커니즘 덕분에, 불필요한 DB 접근을 줄여 성능을 높이고 조회된 객체의 **동일성(`==`)을 보장**받는다.
