## Entity

- JPA 코드 레벨의 시작
    - ORM이 무엇인지, 그리고 패러다임 불일치를 해결하기 위해 왜 필요한지 배웠다.
    - 자바 진영의 공식 ORM 표준 인터페이스인 JPA를 실제로 사용할 때, 그 중심에 있는 첫 번째 개념이 바로 엔티티(Entity)이다.
- 과거 (SQL 중심 개발)의 데이터 반환 방식
    - 과거에는 회원 목록을 조회하기 위해 `SELECT * FROM users`라는 쿼리를 날렸다.
    - 이 쿼리의 결과로 데이터베이스는 표(테이블) 형태의 결과물인 `ResultSet`을 반환한다.
    - 자바는 객체지향 언어이므로 표 형태의 데이터를 그대로 쓸 수 없다. 따라서 `while`문 루프를 돌면서 `new User()`로 객체를 매번 생성하고, 컬럼이 20개면 20번씩 값을 하나하나 수동으로 꽂아 넣어야 했다. (무척 불편하고 번거로운 작업)
- **JPA 방식: `@Entity` 어노테이션의 의미**
    - 자바 클래스를 하나 만들어두고 그 위에 `@Entity`를 붙이면 상황이 완전히 달라진다.
    - 이 어노테이션의 진짜 의미는 "이 클래스는 데이터베이스의 `users` 테이블과 똑같이 생긴 클래스다"라고 JPA에게 알려주는 것이다.
    - 스프링이 실행될 때 어노테이션 스캔을 통해 이를 인식하며, "이 클래스는 내가 관리해야 할 대상이니까 클래스 필드를 보고 DB 테이블 컬럼과 매핑해야겠다"라고 판단한다.
    - 이렇게 JPA의 관리 대상이 된 객체들을 영속성 컨텍스트(Persistence Context)라는 특수한 공간에서 관리하게 되며, 이 상태의 객체를 '영속 상태의 엔티티'라고 부른다.

## 가져가야 할 핵심

1. 단순한 데이터 전송용 객체(DTO)나 일반 순수 객체와, J**PA 영속성 컨텍스트가 직접 추적하고 관리하는 ‘엔티티’의 차이점**을 이해한다.
2. 엔티티를 만들 때 무조건 지켜야 하는 JPA표준 규칙(기본 생성자의 필수성, `final` 키워드 사용 불가 원인)을 리프렉션/프록시 매커니즘과 엮어서 파악한다.
3. 객체의 안전성을 깨뜨리는 무분별한 `@Setter` 사용이 왜 실무에서 안티 패턴이 되는지 이해하고 대체 방법을 배운다.

## 왜 쓰는가? 데이터와 비즈니스 로직의 응집

데이터를 하나의 객체로 묶어 구조와 규칙을 함께 관리함으로써, 코드의 오류를 줄이고 설계를 일관되게 유지하기 위해서이다.

### 데이터베이스의 한계와 로직의 파편화

- 데이터베이스는 훌륭한 데이터 저장소이지만, 복잡한 비즈니스 규칙(기획 요건)을 담을 수 없다는 단점이 있다.
- 물론 DB에 제약조건을 걸거나 트리거를 심어서 일부 처리할 수는 있지만, 복잡한 비즈니스 로직을 DB 엔진에 때려 넣는 것은 유지보수 측면에서 좋지 않다.
    - *기획 예시: "VIP 회원은 상품 구매 시 10% 할인을 받는다", "비밀번호를 변경한 지 3개월이 지나면 변경 권고 상태가 된다"*  → 이런 세세한 규칙을 DB 자체에 녹이기는 어렵다.
- 엔티티가 없다면(단순히 데이터만 담는 껍데기라면)생기는 문제 :
    - 위와 같은 로직들이 전부 **Service 계층**으로 흩어지며 **로직의 파편화**가 발생한다.
    - A 개발자는 `UserService`에 비밀번호 변경 로직을 짜고, B 개발자는 `AdminService`에 관리자가 비밀번호를 초기화하는 로직을 각자 따로 짠다.
    - 결과적으로 코드가 중복되고, 나중에 정책이 바뀌면 두 곳의 코드를 다 찾아서 고쳐야 한다. 하나라도 빼먹으면 고스란히 서비스 버그와 장애로 이어진다.

### 엔티티 중심의 해결책(높은 응집도와 객체의 자율성)

- 로직의 파편화를 막기 위해, 데이터를 쥐고 있는 User 엔티티 클래스 내부에 `changePassword()`같은 비즈니스 메서드를 직접 만들어 두는 것이 좋다.
- 그 메소드 내부에 *"이전 비밀번호와 같으면 안 된다"*, *"특수문자가 포함되어야 한다"* 같은 업무 규칙을 적어둔다.
- 이렇게 설계하면 비밀번호를 변경하려는 애플리케이션 내의 모든 서비스 로직은 **오직 User 엔티티의 `changePassword()` 메소드를 호출하는 방법으로만 변경**해야 하므로 로직의 파편화가 사라지고 책임이 명확해진다.

객체지향의 대원칙 : 어떤 클래스가 특정 데이터를 갖고 있다면, 그 데이터 자체를 조작하고 다루는 행위(메서드) 역시 해당 클래스 내부에 모여있는 것이 응집도가 높다. → 객체가 남에 의해 수정당하는 것이 아니라, 스스로의 상태(필드)를 직접 책임지고 관리하는 주체적인 객체가 되도록 설계해야 한다.

## 엔티티 선언 규칙과 제약사항

```java
import jakarta.persistence.*;

@Entity
public class User {

    @Id @GeneratedValue
    private Long id;
    
    private String email;

    // 1. 기본 생성자가 반드시 존재해야 함 (최소 protected)
    protected User() {}
}
```

### 1. 파라미터가 없는 기본 생성자가 반드시 존재해야 한다.

이유(리플렉션 메커니즘) : JPA 구현체(하이버네이트)는 데이터베이스에서 데이터를 읽어와 자바 객체로 변환할 때 리플렉션이라는 기술을 사용한다.

- 리플렉션은 클래스의 이름만 문자열로 알고 있으며 실행중(런타임)에 해당 클래스의 인스턴스를 동적으로 생성해낼 수 있는 기술이다.
- 만약 클래스에 파라미터가 3~4개 있는 복잡한 생성자가 존재한다면, 하이버네이트 입장에서는 런타임에 “첫 번째 파라미터에는 뭘 넣어야 하지? 두 번째는 뭐지”하고 판단할 수 없어 객체를 생성하지 못하고 에러를 던진다.
- 따라서 하이버네이트가 일을 할 수 있도록 객체를 일단 메모리에 올릴 수 있게 해주는 ‘빈 기본 생성자’가 필수적이다. 하이버네이트는 먼저 이 기본 생성자로 빈 객체를 만든 뒤, 리플렉션을 이용해 데이터베이스에서 가져온 값들을 필드에 꽂아 넣는 방식으로 동작한다.

### 2. 클래스 필드에 `final` 키워드를 사용할 수 없다.

- 클래스에 final을 못 쓰는 이유(프록시 상속 구조) :
    - JPA는 성능을 아끼기 위해 진짜 객체 대신 껍데기만 있는 가짜 객체(프록시)를 만들어 사용하는데, 이 가짜 객체(프록시)가 원본 엔티티 클래스를 ‘상속’받아서 만들어진다.
    - 하지만 클래스에 `final`이 붙어있으면 자바 문법상 상속이 원천 차단된다. 상속이 안 되면 프록시를 만ㄷ르 수 없고, 프록시를 못 만들면 JPA의 핵심인 지연 로딩을 쓸 수 없게 되기 때문에 클래스에 `final`을 붙이면 안된다.
- 필드에 `final`을 못 쓰는 이유:
    - 앞서 설명했듯 하이버네이트는 기본 생성자로 빈 객체를 먼저 생성한 후, 리플렉션으로 필드 값을 채워 넣는다. 그러나 필드에 `final`이 붙어있으면 생성자 시점 이후에 값을 절대 바꿀 수가 없으므로, 하이버네이트가 DB데이터를 필드에 채워 넣는 과정 자체가 막히기 때문이다.

## 무분별한 @Setter는 왜 사용하면 안될까?

### Setter의 문제점 : 의도 상실과 변경 추적 불가

클래스 레벨에 세터를 열어두면 애플리케이션 전역 어디서나 `ser.setPassword()`, `user.setPoint()` 같은 메소드로 객체의 값을 마음대로 변환할 수 있다.

- **예시 시나리오:** 서비스 운영 중 특정 회원의 등급이 갑자기 VIP로 변경되는 심각한 버그가 리포팅되었다. 원인을 찾기 위해 코드를 뒤져보는데, `User` 엔티티에 세터가 무분별하게 열려있어 프로젝트 내에 `setUserRole()`을 호출하는 곳이 50군데가 넘는 상황이다.
- **결과:** 도대체 어느 파일의 어떤 비즈니스 로직을 타고 들어와 값이 망가졌는지 코드를 역추적하는 것 자체가 불가능에 가까워진다. 가독성도 최악으로 떨어진다.

**나쁜 예 : 무분별하게 세터를 열어둔 코드**

```java
@Entity
@Getter @Setter // 클래스 레벨에 Setter를 열어둠
public class User {
    @Id @GeneratedValue
    private Long id;
    private String nickname;
    private int point;
}

// 서비스 로직 어딘가에서...
user.setPoint(user.getPoint() + 100); // 도대체 왜 100을 더하는지 코드를 보는 사람은 의도를 전혀 알 수 없음
user.setNickname("새이름");
```

**좋은 예 : Setter를 닫고 의도가 명확한 비즈니스 메서드 제공**

```java
@Entity
@Getter
//  @Setter는 클래스 레벨에서 철저히 제거
public class User {
    @Id @GeneratedValue
    private Long id;
    private String nickname;
    private int point;

    protected User() {}

    // 의도가 명확하고 규칙이 살아있는 비즈니스 메서드 1
    public void addOrderPoint(int price) {
        // 포인트 적립 규칙을 엔티티 내부에 응집: 구매 금액의 1%
        int earnedPoint = price / 100;
        this.point += earnedPoint;
    }

    // 의도가 명확하고 규칙이 살아있는 비즈니스 메서드 2
    public void changeNickname(String newNickname) {
        // 엔티티 내부에서 검증(Validation)까지 스스로 수행
        if (newNickname == null || newNickname.trim().isEmpty()) {
            throw new IllegalArgumentException("닉네임은 비어있을 수 없습니다.");
        }
        this.nickname = newNickname;
    }
}

// 서비스 로직에서 사용할 때...
user.addOrderPoint(10000); //  "아, 이 유저가 주문을 해서 포인트가 적립되는 중이구나!"라고 의도가 한눈에 파악됨
user.changeNickname("새이름");
```

- **객체 중심 설계의 장점:** 코드를 읽는 동료 개발자나 미래의 내가 서비스 로직만 슥 읽어도 비즈니스 흐름을 100% 이해할 수 있다. 나중에 포인트 적립에 버그가 생기면 여기저기 헤맬 필요 없이 `User` 엔티티 내부의 `addOrderPoint` 메소드 딱 한 곳만 열어서 확인하면 된다.
- **도메인 워딩의 중요성:** 무분별한 Setter는 객체의 일관성을 해치고 변경 추적을 방해한다. 반드시 의미 있는 비즈니스 언어(도메인 워딩)를 기반으로 메소드를 구현해야 한다.

## 정리

- Entity는 데이터베이스 테이블과 1:1로 매핑되는 특수한 자바 클래스이며, 데이터 보관을 넘어 비즈니스 로직까지 한곳에 응집하여 처리하는 객체지향 설계의 핵심 단위이다.
- JPA가 런타임에 리플렉션을 통해 객체를 동적으로 생성할 수 있도록 파라미터가 없는 기본 생성자가 반드시 필요하다.
- 성능 최적화(지연 로딩)를 위해 원본을 상속한 프록시 객체를 만들어야 하므로, 클래스 필드에 `final` 키워드는 절대 사용할 수 없다.
- 객체의 안전성과 유지보수성을 확보하고 변경 추적을 수월하게 만들기 위해 무분별한 Setter 사용을 지양하고, 의도가 명확히 드러나는 비즈니스 메서드를 설계하여 객체의 상태를 변경해야 한다.
