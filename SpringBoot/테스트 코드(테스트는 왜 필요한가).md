## 가져가야 할 핵심

- 테스트 코드가 무엇인가? → 내가 만든 기능이 내 의도대로 동작하는지를 사람의 눈이 아니라 또 다른 코드로 확인하는 게 테스트 코드
- 테스트 코드를 왜 쓰는가? → 테스트는 버그를 잡기 위한 도구가 아니라 코드 고칠 때 문제를 덜어주는 안전망 같은 역할을 하고, 기능이 어떻게 동작해야 하는지를 보여주는 문서와 같은 역할
- 테스트 코드가 어떻게 동작하는가? → 모든 테스트는 준비하고, 실행하고, 검증하는 세 단계로 흘러간다.

## 테스트 코드의 정의와 AI 시대의 가치

- 소프트웨어의 특정 기능이나 비즈니스 로직이 개발자의 의도대로 올바르게 작동하는지 검증하기 위해 작성된 또 다른 코드이다.
- 사람이 눈으로 일일이 마우스 클릭해가며 화면을 확인하는 것이 아니라, 코드가 코드를 자동으로 검증하게 만드는 시스템이다.
- 이제 단순한 기능 구현 코드는 AI에 프롬포트를 넣으면 그럴싸하게 만들어낸다. “AI가 짠 코드가 우리 회사의 복잡한 비즈니스 요구사항에 맞춰 정확하게 동작하는가?”를 판별하고 최종 검증하는 안전장치가 바로 테스트 코드이다.
    
    비즈니스 시나리오의 경계 조건을 설계하고 개입하는 것은 결국 개발자의 몫이기 때문에, 좋은 테스트 코드를 알아보고 작성해야한다.
    

## 테스트 코드를 짜야 하는 이유

#### 안전망이 된다.

- 코드가 쌓이다 보면 “내가 방금 고친 이 코드 때문에 기존 기능이 안 돌아가면 어쩌지?”라는 불안감이 생긴다.
- 에러 위치 조기 확보 : 문제가 터졌을 때 수만 줄의 코드 중 어디가 고장 났는지 찾는 데 시간이 가장 오래 걸린다.  테스트 코드가 있으면 어느 지점의 어떤 단위 테스트가 실패했는지 바로 알려주기 때문에 버그 발생 범위를 좁힐 수 있다.
- ex) 배송비 계산 로직을 고치다가 실수로 주문 금액의 쿠폰 할인 적용 순서를 건드렸다고 가정해보자.
    
    화면만 대충 봐서는 티가 안 나서 나중에 사용자가 결제한 뒤에야 “금액이 이상하다”며 장애로 이어지기 쉽다. 
    
    하지만 할인 검증 테스트 코드가 하나라도 있었다면, 코드를 고치고 저장하자마자 테스트가 깨지면서 실수를 잡아낼 수 있다.
    

#### 리팩토링을 마음 편하게 할 수 있다.

- 리팩토링은 기능은 그대로 둔 채 코드 구조를 개선하거나 가독성을 높이는 작업이다.
- 테스트 코드가 없으면 “멀쩡히 돌아가는 코드 괜히 건드렸다가 깨질까 봐”무서워서 코드 개선을 포기하게된다. 테스트 코드가 있으면 구조를 바꾼 뒤에 테스트가 다 수행되는 것을 보고 기능이 완벽히 유지되었구나를 확신할 수 있다.

#### 문서화와 협업

- 노션이나 위키에 적어둔 기능 명세서는 코드가 바뀔 때 개발자가 수동으로 업데이트를 안 해주면 순식간에 낡아서 쓸모없는 문서가 된다.
- 하지만 테스트 코드는 요구사항이 바뀌어 코드가 달라지면 테스트 자체가 실패하기 때문에, 개발자가 테스트 코드를 수정할 수 밖에 없다. → 항상 최신 명세를 반영하는 가장 정확한 제품 설명서가 도니다.
- 새로 합류한 개발자가 프로젝트를 빠르게 파악하는 좋은 방법도 구현 코드를 다 뜯어보는 게 아니라 잘 짜인 테스트 코드를 읽으며 “아, 이 클래스는 이런 상황에서 이런 책임을 지고 동작하는구나”를 파악한다.

#### 좋은 객체지향 설계 유도

- 테스트하기 어려운 코드는 좋은 코드가 아니다. 여러 기능이 한 덩어리로 뭉쳐 있고 의존성이 복잡하게 꼬여있기 때문에 테스트가 안 짜지는 것이다.
- 어떻게든 테스트가 가능한 구조로 코드를 짜려고 고민하다 보면, 자연스럽게 기능을 잘게 나누고 객체가 하나의 책임만 갖도록 분리(단일 책임 원칙)하게 되면서 전체적인 소프트웨어 아키텍처가 깔끔해진다.

## 테스트 코드의 동작 방식

준비 → 실행 → 검증

1. 준비(Given) : 테스트할 기능을 실행하기 위해 필요한 가짜 환경, 입력값, 객체 등을 미리 세팅하는 단계
2. 실행(When) : 우리가 진짜로 검증하고 싶은 핵심 기능(메서드)을 호출하는 단계
3. 검증(Then) : 실행해서 나온 실제 결과값이 내가 기대했던 예상값과 일치하는지 확인하는 단계 → 다르면 테스트 실패

검증 대상 프로덕션 코드 (`Calculator.java`)

```java
public class Calculator {
    public int add(int a, int b) {
        return a + b;
    }
}
```

검증을 위한 테스트 코드 (`CalculatorTest.java`)

```java
import org.junit.jupiter.api.Assertions;
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;

public class CalculatorTest {

    @Test // 이 메소드는 일반 메소드가 아니라 '테스트 실행용'임을 JUnit 프레임워크에 알리는 어노테이션
    @DisplayName("두 정수를 더했을 때 정확한 합을 반환해야 한다")
    void testAdd() {
        // 1. 준비 (Given) : 덧셈을 하기 위해 필요한 환경과 데이터를 세팅
        Calculator calculator = new Calculator();
        int a = 5;
        int b = 3;
        int expectedResult = 8; // 우리가 머릿속으로 기대하는 결과값

        // 2. 실행 (When) : 실제 검증 대상인 add 메소드를 호출
        int actualResult = calculator.add(a, b);

        // 3. 검증 (Then) : 기대값(8)과 실제 로직 결과가 일치하는지 자동 비교
        Assertions.assertEquals(expectedResult, actualResult);
    }
}
```

- `assertEquals(A, B)`: A와 B가 같으면 테스트 성공, 다르면 즉시 에러를 내며 테스트 실패를 선한다.
- 테스트가 실패했다는 것은 "우리가 의도하고 정의한 요구사항과 실제 코드의 동작이 어긋났다"라는 신호다.

**"이 테스트가 통과했다고 해서 `add` 메소드가 모든 상황에서 완벽하다고 보장할 수 있을까?"**

→ 아니다. 이 테스트는 오직 5 + 3 = 8인 상황 하나만 검증한 것이다. 음수를 넣었을 때, 혹은 int범위를 넘어가는 아주 큰 수를 넣었을 때 어떻게 될지는 알 수 없다.

→ 하나의 기능에 대해서도 성공하는 케이스뿐만 아니라 예외가 발생하는 다양한 경계조건들을 쪼개어 여러 개의 테스트 코드로 작성한다.

## 정리

- 테스트 코드는 소프트웨어가 우리 의도대로 동작하는지를 사람이 아닌 코드로 검증하는 도구이다.
- 목적은 단순히 버그를 잡는 게 아니라 두려움 없이 코드를 고칠 수 있게 해주는 리팩토링 안전망이며 기능이 어떻게 동작해야하는지를 보여주는 문서 역할도 한다.
- 동작 방식은 세 단계로 나눠진다. 준비하고, 실행하고, 검증한다. 내가 설정한 기대값과 실제값을 비교해서 같으면 성공, 다르면 실패. 그리고 테스트가 실패했다는 것은 코드가 내가 생각했던 의도랑 어긋났다라는 신호이다.
