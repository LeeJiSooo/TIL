## 가져가야 할 핵심

- 요청은 컨트롤러로 바로 가지 않고 서블릿 필터 체인 먼저 통과하고 시큐리티 보안 검사 필터 단계에서 일어난다.
- 필터 구조는 크게 세가지 부품으로 나뉜다. DelegatingFilterProxy는 서브릿 컨테이너랑 스프링을 잇는 다리같은 역할을 하고, FilterChainProxy는 요청에 맞는 보안 체인을 골라주는 역할을 하고, 시큐리티 필터 체인은 실제 보안 필터들이 숮서대로 줄지어있는 집합이다.
- 인증에 성공한 사용자 정보는 시큐리티 컨텍스트 홀더를 통해서 다룰 수 있다.

#### DelegatingFilterProxy (서블릿과 스프링을 잇는 다리)

- 서블릿 필터는 서블릿 컨테이너(톰캣)가 관리하는 영역이고, 우리가 만드는 보안 설정이나 빈은 스프링 컨테이너가 관리한다.
    
    → 서로 다른 세계이기 때문에 서블릿 필터에서는 스프링 빈을 직접 주입받아 사용할 수 없다.
    
- 서블릿 필터 체인에 자신이 필터로 등록되어 있다가, 요청이 들어오면 직접 보안 검사를 하지 않고 스프링 컨테이너 내부의 실제 보안 처리 빈에게 요청을 넘겨주는 프록시 역할을 수행한다.
    
    → 스프링의 의존성 주입 등 모든 기능을 활용해 보안 필터를 유연하게 구성할 수 있다.
    

#### FilterChainProxy (보안 체인을 골라주는 라우터)

- DelegatingFilterProxy로부터 일을 넘겨받는 스프링 빈이며, 스프링 시큐리티의 실질적인 시작점이자 관리자이다.
- 라우터 역할 : 스프링 시큐리티는 여러 개의 보안 필터 체인을 가질 수 있다. FilterChainProxy는 들어온 요청의 URL패턴을 확인하고, 어떤 SecurityFilterChain을 적용할지 결정을하여 요청을 보낸다. (/api/와 /admin/ 요청에 서로 다른 보안 규칙을 적용할 수 있는 이유가 이 컴포넌트 덕분)

#### SecurityFilterChain (실제 보안 필터들의 집합)

- FilterChainProxy에 의해 선택된, 실제 보안 로직을 수행하는 필터들의 집합니다.
- CSRF 방어 필터(`CsrfFilter`), 헤더 처리 필터(`HeaderWriterFilter`), 폼 로그인 처리 필터(`UsernamePasswordAuthenticationFilter`), 권한 검사 필터(`AuthorizationFilter`) 등 책임이 다른 필터들이 정해진 순서대로 줄지어 있음
- 요청은 이 필터들을 위에서 아래로순서대로 통과한다. 중간에 하나의 필터라도 인증/인가에 실패하면 흐름이 중단되고 예외처리로 넘어간다.
    
    → 모든 관문을 완벽히 통과해야만 DispatcherServlet을 거쳐 컨트롤러에 도달할 수 있다.
    

## 인증된 사용자 정보의 보관 : SecurityContextHolder & SecurityContext

인증 필터가 사용자의 신원을 확인하면, 사용자의 신원, 자격, 증명, 권한을 담은 `Authentication`객체를 생성한다.

1. SecurityContext : `Authentication`객체를 담아두는 객체(지금 요청을 처리하는 인증 정보 저장소)
2. SecurityContextHolder : `SecurityContext`를 보관하고 관리하는 래퍼 클래스
3. ThreadLocal전략 : `SecurityContextHolder` 는 기본적으로 쓰레드 로컬 방식으로 동작한다. 
    
    → 요청을 처리하는 현재 스레드 안에서만 공유되는 독립된 저장 공간이다.
    
    → 컨트롤러, 서비스, 리포지토리 등 같은 요청을 처리하는 동안이라면 코드 어디에서든 로그인한 사용자 정보를 꺼내 쓸 수 있음. (파라미터로 유저 정보를 계속 들고 다닐 필요가 없음)
    
4. Stateless(JWT)환경에서의 동작 : 세션을 쓰지 않는 JWT 환경에서도 똑같이 동작한다. 세션에 영구 저장하지 않을 뿐, 요청이 들어왔을 때 토큰을 검증하고 그 요청이 처리되는 동안만 정보를 SecurityContextHolder에 올려서 쓰고 요청이 끝나면 삭제한다. (다음 요청 때는 다시 토큰을 통해 인증해야 함)

## 필터 vs 인터셉터 vs AOP 비교

#### 필터

- 웹 애플리케이션의 최전방. DispatcherServlet에 요청이 도달하기 전, 서블릿 컨테이너 수준에서 가장 먼저 실행된다.
- `HttpServletRequest`, `HttpServletResponse` 같은 HTTP 원시 정보(저수준 데이터)에 직접 접근할 수 있음. 이 시점에는 아직 어떤 컨트롤러가 요청을 처리할지 알 수 없으며, 스프링context 외부이기 때문에 빈 주입이 번거러움 →`DelegatingFilterProxy`가 필요한 이유
- 인증, 인가 , CORS 설정, 인코딩 변환, 로깅 등 애플리케이션 전역에 공통으로 적용해야 하는 대규모 보안/웹 규칙. 부적절한 요청을 스프링 MVC컨텍스트 내부로 진입하기 전에 원천 차단하는데 가장 적합하다.

### 인터셉터

- 스프링 MVC영역 내부이다. DispatcherServlet과 Controller 사이에서 실행된다.
- 디스패처 서블릿을 거친 후에 실행되므로 이미 스프링 여역 안이다.
    
    → 어떤 핸들러(컨트롤러 메스드)가 호출될지 정보를 알 수 있고, 스프링 빈에서도 자유롭게 접근할 수 있음. 
    
- 로그인 여부에 따른 세밀한 페이지 리다이렉트 처리, 컨트롤러 실행 후 뷰에 공통 데이터 주입, 컨트롤러 호출 전후의 세부적인 인터페이스 로그 기록 등 스프링 MVC 및 핸들러 정보가 필요한 웹 관련 작업이다.

#### AOP

- 비즈니스 로직의 가장 안쪽이며, 메서드 호출 전후나 예외 발생시 동작한다.
- 웹 요청의 개념을 넘어선 비즈니스 계층의 영역이다. 웹 정보 보다는 호출되는 메서드의 이름, 파라미터, 어노테이션, 반환값 같은 세밀한 자바 객체 레벨의 정보를 다룬다.
- 선언적 트랜잭션 관리, 특정 비즈니스 메서드 단위로 정밀한 로깅, "관리자가 상품 삭제 메소드를 호출할 때 로그를 남긴다"와 같은 **비즈니스 로직 중심의 세밀한 메소드 단위를 제어한다.**

| **구분** | **필터 (Filter)** | **인터셉터 (Interceptor)** | **AOP** |
| --- | --- | --- | --- |
| **관리 컨테이너** | 서블릿 컨테이너 (Tomcat) | 스프링 컨테이너 (Spring MVC) | 스프링 컨테이너 (Spring Context) |
| **실행 시점** | `DispatcherServlet` 실행 전/후 | `DispatcherServlet` ↔ `Controller` 사이 | 비즈니스 로직 `Method` 실행 전/후 |
| **접근 가능한 정보** | HTTP 원시 요청/응답 객체 | HTTP 요청/응답 + 선택된 컨트롤러(Handler) 정보 | 메소드 파라미터, 반환값, 어노테이션 등 객체 정보 |
| **주요 역할** | 전역적 보안(인증/인가), 로깅, 인코딩 | 웹 기술 중심의 공통 체크, 데이터 가공 | 비즈니스 로직 세부 제어, 트랜잭션 관리 |

## 정리

- 요청은 클라이언트에서 출발해서 바로 컨트롤러로 가는 게 아니라 서블릿 필터 체인 먼저 지나고 그 안에서 DelegatingFilterproxy가 바깥 서블릿 세계랑 안쪽 스프링 세계를 잇는 다리 역할을 한다.
- **웹 요청 흐름:** 클라이언트 ➔ 서블릿 필터 체인(시큐리티) ➔ 디스패처 서블릿 ➔ 인터셉터 ➔ 컨트롤러(AOP)
- 시큐리티 : `DelegatingFilterProxy`(서블릿-스프링 연결 다리) ➔ `FilterChainProxy`(URL 매핑 라우터) ➔ `SecurityFilterChain`(실제 보안 필터 집합)
- 인증 완료 시 `Authentication` 객체가 `SecurityContext`에 담겨 **스레드당 독립된 저장소**인 `SecurityContextHolder(ThreadLocal)`에 안전하게 보관된다.
- 가장 바깥의 전역 보안은 **필터**, 스프링 MVC 핸들러 정보가 필요하면 **인셉터**, 세밀한 메소드 단위 비즈니스 제어는 **AOP**가 담당한다.
