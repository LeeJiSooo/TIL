## 가져가야 할 핵심

- 스프링 시큐리티가 무엇이고, 무엇을 해결해주는 도구인지 말할 수 있어야 한다. 그냥 로그인을 만들어주는 게 아니라 인증/인가 개념 안에서 보안 전반을 책임지는 프레임워크이다.
- 인증이랑 인가가 어떻게 다른지 - 인증 흐름과 인가 흐름에서 순서를 이해하기 위해서 필요하다.
- 웹에서 요청이 오면 Controller에 도달하기 전에 어디를 지나가는지, 그 앞단의 필터 체인에서 스프링 시큐리티가 보안 검사를 끝낸다는 큰그림

## 스프링 시큐리티의 정의와 역할

- 스프링 기반 애플리케이션의 인증과 인가를 포함한 보안 기능을 처리하는 하위 프레임워크
- 단순히 로그인 기능을 만들어 주는 도구가 아니라, 보안 전반을 책임지는 프레임워크
- 도입 이유 : 개발자가 매번 세션 관리, 비밀번호 암호화 및 저장, 보안 공격 방어(CSRF 등)와 같은 복잡한 로직을 직접 구현하지 않고, 표준화된 구조 위에서 안전하게 애플리케이션을 보호하기 위함

## HTTP의 Stateless(무상태)특성과 시큐리티

- HTTP는 상태가 없는(Stateless)프로토콜 : 서버는 클라이언트의 이전 요청을 전혀 기억하지 못한다. 로그인을 하고 다른 페이지로 이동해도, HTTP의 독립적 특성 때문에 서버는 방금 로그인한 사람이 누구인지 스스로 알 수 없다.
    
    → 쿠키, 세션, JWT 등의 장치를 사용해 상태를 따로 유지한다.
    
- 인증에 성공한 사용자를 어떻게 기억하고, 다음 요청에서 어떻게 알아볼 것인가?
    
    → 스프링 시큐리티가 다루는 핵심 영역이 바로 이 상태 유지와 맞닿아 있음
    

## 인증(Authentication) vs 인가(Authorization)

보안 흐름의 순서를 이해하기 위해 두 개념의 차이를 반드시 구분해야 한다. 

항상 인증이 인가보다 먼저 진행된다.

→ 누구인지 모르는 사람에게 권한을 따지는 것은 의미가 없기 때문에

| **구분** | **인증 (Authentication)** | **인가 (Authorization)** |
| --- | --- | --- |
| **개념** | **"당신은 누구십니까?"** (신원 확인) | **"이 작업을 할 권한이 있습니까?"** (권한 부여) |
| **예시** | 아이디/비밀번호 입력하여 로그인 성공 후 시스템이 등록된 회원임을 확인하는 단계 | 로그인된 사용자가 관리자 페이지에 접근하거나, 남의 글을 수정할 수 있는지 검사하는 단계 |
| **비유** | 콘서트장 입구에서 **신분증과 티켓을 확인**하는 단계 | 입장한 관객 중 **VIP 티켓을 가진 사람만 VIP 라운지에 진입**하도록 허용하는 단계 |

## 서블릿(Servlet)생태계와 필터의 이해

스프링 시큐리티가 동작하는 위치를 이해하기 위한 백엔드/인프라 기초 개념이다.

- 서블릿(Servlet) : 자바에서 웹 요청 하나를 받아서 응답을 만들어내는 표준 규격 객체이다. 브라우저의 HTTP 요청을 자바 코드로 받아 처리하는 표준 단위이며, `DispatcherServlet`이 대표적인 서블릿의 한 종류이다.
- 서블릿 컨테이너 : 톰캣과 같은 WAS를 의미한다. 네트워크 통신을 자바 객체(HttpServletRequest/Response)로 변환해주고, 서블릿의 생성, 실행, 소멸을 관리한다. → 스프링부트는 톰캣을 내장하고 있어 별도 설치 없이 구동된다.
- 서블릿 필터 : 서블릿보다 앞단에 서서, 요청이 서블릿에 닿기 전에 먼저 가로채는 장치이다.
    - 서블릿 컨테이너가 관리하며, `DispatcherServlet` 을 포함한 그 어떤 서블릿보다 먼저 실행된다.
    - 여러 필터가 체인 형태로 연결되어 요청을 순서대로 처리한다.(필터체인)

## 웹 요청의 흐름과 시큐리티의 동작 시점 : Filter에서 Controller까지

1. 클라이언트 요청 발생
2. WAS(톰캣)도착
3. 서블릿 필터 체인 → 스프링 시큐리티가 보안 검사를 수행하고 끝내는 시점
4. 디스패처 서블릿 / 인터셉터
5. 컨트롤러 도달(비즈니스 로직 처리)

**왜 인터셉터가 아닌 필터에서 보안 검사를 할까?**

- 동작 시점의 차이 : 필터는 디스패처 서블릿보다 먼저 컨테이너 레벨에서 동작하고, 인터셉터는 스프링 MVC 내부(디스패처 서블릿과 함께)에서 동작한다.
- 종속성 분리 : 스프링 시큐리티는 애플리케이션 전체의 보안을 책임져야 하므로, 특정 웹 기술(Spring MVC)에 종속되지 않고 가장 최전방에서 모든 요청을 검사할 수 있는 필터 기반으로 설계된다.

## 스프링 시큐리티 내부 동작 원리

<img width="1308" height="1050" alt="image" src="https://github.com/user-attachments/assets/0d20eaf0-95df-46f3-92a4-ccee76351033" />


**DelegatingFilterProxy (다리 역할)**

- 서블릿 필터는 서블릿 컨테이너(톰캣)가 관리하고, 시큐리티 설정이나 Bean은 스프링 컨테이너가 관리하므로 서로 다른 세계에 존재한다.
- `DelegatingFilterProxy`는 서블릿 필터 체인에서 요청을 받아, 스프링 빈으로 등록된 보안구조로 요청을 넘겨주는 다리 역할을 한다.
- 실제 보안 처리는 `FilterChainProxy`(스프링 빈)가 위임받아 실행한다.

**SecurityFilterChain (보안 필터 목록)**

- 현재 요청에 맞는 보안 필터들의 묶음이다. 서로 다른 역할을 하는 필터들이 순서대로 정렬되어 있다.
- ex) CSRF공격 방어 필터, 폼 로그인 처리 필터, URL 권한 검사 필터, 보안 예외 처리 필터 등
- 요청은 필터 체인을 순서대로 통과하며, 중간에 단 하나의 필터라도 인증/인가에 실패하면 흐름이 중단되고 예외 처리로 넘어간다.
    
    → 모든 필터를 완벽히 통과해야만 컨트롤러에 도달할 수 있다.
    

## 스프링 시큐리티 설정 방식

과거에는 `WebSecurityConfigurerAdapter`를 상속받아 구현했으나 현재는 **권장되지 는다.** 지금은 설정 클래스 내에 `SecurityFilterChain`을 **Bean으로 직접 등록**하여 컴포넌트 기반으로 설정한다.

```java
@Configuration
@EnableWebSecurity // 스프링 시큐리티 활성화
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // 1. URL 패턴별 권한 설정 (인가)
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**", "/login").permitAll() // 누구나 접근 가능
                .requestMatchers("/admin/**").hasRole("ADMIN")       // ADMIN 역할 필수
                .anyRequest().authenticated()                        // 그 외 모든 요청은 인증 필요
            )
            // 2. 폼 로그인 기능 설정
            .formLogin(form -> form
                .loginPage("/login") // 커스텀 로그인 페이지 경로
                .defaultSuccessUrl("/home", true) // 로그인 성공 시 이동할 URL
                .permitAll()
            )
            // 3. 로그아웃 기능 설정
            .logout(logout -> logout
                .logoutUrl("/logout")
                .logoutSuccessUrl("/login?logout")
            );

        return http.build();
    }
}
```

`authorizeHttpRequests`: URL 패턴별로 누가 접근할 수 있는지(인가 규칙)를 세밀하게 정의하는 핵심 메서드

## 정리

- 스프링 시큐리티는 인증과 인가를 포함한 보안 처리를 해주는 하위 프레임워크이다.
- 인증은 “누구인가(신원 확인)”, 인가는 “무엇을 할 수 있는가(권한 확인)” 항상 인증이 먼저다.
- 스프링 시큐리티를 통한 보안 처리가 컨트롤러 안이 아니라, 그 앞단의 서블릿 필터 체인에서 일어난다.
- 요청은 여러 필터를 순서대로 통과하고 하나라도 실패하면 거기서 멈추고 전부 통과해야 컨트롤러로 온다.
- 서블릿 세계와 스프링 세계를 연결해주는 다리는 `DelegatingFilterProxy`이다.
