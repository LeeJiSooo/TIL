## MVC패턴의 정의와 근본적인 목적

애플리케이션의 구성 요소를 데이터와 비즈니스 로직(Model), 화면 표현(View), 요청 처리와 흐름제어(Controller)라는 세 가지 역할로 명확하게 분리하는 아케틱처 패턴이다.

근본적인 이유 : 관심사의 분리

MVC의 목적은 단순히 기능을 나열하듯 쪼개는 것이 아니다. 변경의 이유가 다른 코드들을 구조적 차원에서 아예 격리하는 것이다.

화면 표현(디자인)이 바뀐다고 해서 비즈니스 규칙(계산 로직 등)이 바뀔 필요가 없고, 흐름이 바뀐다고 해서 데이터 구조가 변할 필요가 없도록 만들기 위함이다.

### 왜 관심사를 분리해야 하는가?

과거 JSP나 PHP로 개발하던 시절에는 하나의 파일 안에 사용자의 요청을 받는 코드, DB에 SQL을 날리는 코드, HTML/CSS로 화면을 그리는 코드가 한데 섞여 있었다.

- 유지보수 문제 : 기획팀에서 “페이지 색상만 파란색에서 빨간색으로 바꿔주세요”라고 요청했을 뿐인데 개발자는 수백 줄의 뒤섞인 코드 속에서 HTML태그를 고쳐야 했다. 이 과정에서 화면 색상을 바꾸려다 결제 로직이나 DB코드를 건드려 시스템이 마비되는 문제가 빈번했다.

### MVC의 해결 방식(변경의 영향도 최소화)

회사가 성장하여 기존 웹 서비스 이외에 아이폰, 안드로이드 모바일 앱을 추가로 출시한다고 가정

- 만약 분리하지 않았다면 : Model이 웹의 View/Controller와 떡져 있었다면 모바일 앱용 서비스를 만들 때 상품 조회 및 할인율 계산 로직을 똑같이 처음부터 다시 짜야 한다. 향후 할인율 정책이 바뀌면 웹 코드와 모바일 코드를 각각 찾아다니며 두 번 수정해야 하므로 실수할 확률이 높아진다.
- MVC를 적용했다면 : 상품 조회, 재고 파악, 할인 적용 로직은 오직 Model(비즈니스 영역)에만 존재한다. 요청이 웹에서 오든 모바일 앱에서 오든 비즈니스 로직 자체가 변할 이유가 없으므로 Model은 그대로 둔 채 각 요청에 맞는 Controller와 View만 갈아 끼우면 끝난다.

## SpringBoot에서 ‘Model’의 진짜 의미

스프링부트에서 웹(HTTP)요청과 무관하게 순수 비즈니스 로직과 데이터를 처리하는 모든 영역을 통틀어 Model이라고 부른다.

- Service 계층 : 비즈니스 흐름을 관장하고 예외 처리 및 트랜잭션을 제어하는 영역
- Domain 객체 : 순수 비즈니스 규칙과 상태를 가진 객체들
- Repository계층 : 데이터베이스와 직접 통신하며 데이터를 영속화하는 여역

## 현대 REST API환경에서의 View의 진화

과거 전통적인 MVC에서는 서버가 HTML/CSS/JSP 등을 직접 렌더링해서 브라우저로 보냈다. 하지만 현대 백엔드 개발(REST API)환경에서는 상황이 다르다.

- 전통적인 View의 소멸 : 현대 백엔드는 화면을 직접 그리지 않고 클라이언트에게 순수 데이터(JSON)만 전달한다. 실제 화면을 그리고 상태를 관리하는 View의 역할은 프론트엔드 프레임워크가 완전히 독립된 레이어로 가져갔다.
- 직렬화와 Message Converter: 그렇다면 백엔드에선 View가 완전히 사라지는 걸까? 백엔드가 가진 자바 객체 데이터를 외부 세계(브라우저, 앱)가 이해할 수 있는 표준 규격인 JSON데이터로 변환하여 내보내는 ‘표현 계층’의 본질적인 역할은 똑같다.

→ 현대 REST API 환경에서는 전통적인 HTML 렌더링 View 대신, **Spring의 Message Converter가 개입하여 자바 객체를 JSON 형식으로 직렬화(Serialization)하고 HTTP Response Body에 담아주는 데이터 표현 과정이 현대적인 View 계층을 대체**하여 동작한다.

스프링부트의 전형적인 MVC흐름 예시

```jsx
@RestController // @Controller + @ResponseBody (JSON 응답을 의미)
public class UserController {

    private final UserService userService; // Model(Service) 의존성 주입

    // 1. Controller가 클라이언트의 HTTP GET 요청을 가장 먼저 마주함
    @GetMapping("/users/{id}")
    public UserResponse getUser(@PathVariable Long id) {
        
        // 2. Controller는 흐름만 제어하고, 실제 비즈니스 로직은 Model(Service)에 위임
        return userService.findUser(id); 
        
        // 3. 반환된 자바 객체(UserResponse)는 메시지 컨버터에 의해 
        //    JSON으로 직렬화되어 응답(현대적 의미의 View)으로 나감
    }
}
```

## 정리

- MVC의 본질은 관심사의 분리이다. 영역을 명확히 나눠놓아야 기획이 바뀌든 프런트엔드 기술 스택이 바뀌든 백엔드 핵심 비즈니스 로직이 침해받지 않는다.
- Spring Boot의 Model은 단순한 클래스가 아니라 Service, Domain, Repository를 포괄하는 순수 비즈니스 로직 영역 전체를 뜻한다.
- 현대 백엔드 개발에서 View는 HTML파일이 아니라 자바 객체를 JSON 형식으로 변환하여 ResponseBody에 담아 전달하는 직렬화 및 데이터 표현 계층으로 진화했다.
