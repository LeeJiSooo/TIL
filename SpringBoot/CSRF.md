# CSRF

## CSRF공격의 정의와 성립 원인

사용자가 자신의 의지와는 무관하게, 공격자가 의도한 악성 행위(데이터 수정, 등록, 삭제, 송금 등)를 특정 웹사이트에 요청하도록 조작하는 공격기법이다. 

유저는 자기가 어떤 요청을 보냈는지 조차 인지하지 못 한다.

#### 공격이 성립하는 핵심 원인(브라우저의 특성)

CSRF공격이 가능한 근본적인 이유는 “브라우저는 특정 도메인으로 요청을 보낼 때, 그 도메인과 관련된 쿠키(세션ID 등)를 요청 헤더에 자동으로 실어 보낸다”라는 특성 때문이다.

#### CSRF공격 시나리오 예시(송금)

- **정상 로그인:** 유저가 은행 사이트(`my-bank.com`)에 정상 로그인함. 서버는 세션 ID가 담긴 쿠키를 발급하고, 브라우저는 이를 저장함.
- **악성 사이트 유인:** 공격자가 이메일이나 게시글 링크를 통해 유저를 악성 사이트(`evil-site.com`)로 유인함.
- **위조된 요청 발생:** 악성 사이트 내부에는 유저 모르게 1픽셀짜리 투명한 이미지나 숨겨진 폼 태그가 심겨 있음.HTML
    
    ```
    <img src="https://my-bank.com/transfer?to=attacker&amount=1000000" width="1" height="1" />
    ```
    
- **쿠키 자동 전송:** 브라우저는 이미지를 불러오기 위해 `my-bank.com`으로 요청을 날리는데, 이때 **유저가 이전에 저장해 둔 `my-bank.com`의 세션 쿠키를 자동으로 붙여서** 보냄.
- **서버의 오인:** 은행 서버는 요청을 받았을 때 유효한 세션 쿠키가 들어있으니, "로그인한 정당한 유저가 보낸 요청이 맞구나!" 하고 돈을 이체해 버림.

## 서버가 CSRF를 막기 어려운 이유와 CSRF토큰

서버가 요청을 받았을 때 세션 쿠키가 정상이라면 “이 요청을 보낸 주체는 로그인된 유저가 맞다(인증 성공)”은 확실히 알 수 있다. 하지만 서버가 절대 알 수 없는 정보는 “이 사용자가 정말로 이 이체 버튼을 의도해서 눌렀는가?”이다. 

→ 인증은 신원만 증명할 뿐, 이 요청이 우리 사이트의 정상적인 화면에서 출발했는지를 증명하지 못 한다.

#### 해결책 : CSRF토큰

서버는 인증 정보(쿠키)외에 추가적인 증표를 하나 더 요구하는데, 이것이 CSRF토큰이다. 

1. **토큰 발급 및 발행** : 유저가 사이트에 처음 접속하거나 로그인하면, 서버는 예측 불가능한 임의의 문자열(CSRF토큰)을 생성한다. 이 토큰을 서버 세션에 저장하고, 클라이언트로 내려 보낸다. → 서버와 브라우저가 똑같은 증표를 나눠 갖는다.
2. **요청 시 토큰 첨부** : 유저가 상태를 바꾸는 요청(POST, PUT, DELETE 등)을 보낼 때, 정상적인 사이트의 자바스크립트나 폼을 받아두었던 CSRF토큰을 HTTP 요청 헤더나 파라미터에 명시적으로 담아서 보낸다.
3. **CsrfFilter의 검증** : 스프링 시큐리티의 CsrfFilter가 요청을 낚아채서 요청에 담겨온 토큰과 서버 세션에 보관된 토큰을 비교하고, 일치하면 통과시키고 토큰이 없거나 다르면 즉시 요청을 거부한다.

**공격자가 토큰도 위조해서 보내면 안 되나?**

공격자의 사이트(`evil-site.com`)에서는 `my-bank.com`이 발행한 CSRF 토큰 값을 읽거나 훔쳐갈 수 없다. 왜냐하면 브라우저의 **SOP(동일 출처 정책)** 제약 때문에, 다른 도메인의 스크립트는 우리 사이트의 HTML 응답 내용물에 접근할 수 없기 때문이다.

→ 브라우저가 쿠키는 자동으로 붙여주지만, CRRF토큰은 자동으로 붙여주지 않는다. 토큰은 정상적인 화면 내의 자바스크립트가 직접 꺼내서 심어야 해서, 토큰 값을 모르는 공격자는 쿠키만 전송하는 것으로 방어벽을 뚫지 못 한다.

## 설정 및 비활성화 판단 기준

스프링 시큐리티는 CSRF보호가 기본적으로 활성화되어 있다. 

→ 이것을 오히려 끄는 설정을 작성하게 되는데, 무작정 끄면 안 되고 명확한 기준이 있어야 한다.

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            // 전통적인 SSR(Thymeleaf, JSP 등)이 아닌 REST API 서버라면 보통 비활성화
            .csrf(csrf -> csrf.disable()); 

        return http.build();
    }
}
```

| **인증 방식** | **브라우저 자동 전송 여부** | **CSRF 위험성** | **CSRF 보호 활성화 여부** |
| --- | --- | --- | --- |
| **세션 / 쿠키 기반 인증**
(전통적인 MVC, Thymeleaf 등) | **O** (요청 시 세션 쿠키가 자동으로 실려 나감) | **매우 높음** (클릭 한 번으로 공격 성립) | **반드시 활성화 (ON)** |
| **토큰 기반 인증 (헤더 중심)**
(일반적인 REST API + JWT) | **X** (개발자가 자바스크립트로 `Authorization: Bearer <JWT>` 헤더를 명시해야 함) | **없음** (공격자 사이트에서는 이 헤더를 강제로 심을 수 없음) | **비활성화 가능 (DISABLE)** |

JWT 쓰면 CSRF 안전하니까 꺼도 되냐?

→안된다. 중요한 것은 토큰의 종류(JWT냐 세션ID냐)가 아니라 ‘저장 및 전송 방식’때문이다.

- 만약 프런트 개발자가 보안상의 이유(XSS 방지 등)로 JWT를 브라우저의 쿠키에 저장해두고 사용하면?
- 토큰이 JWT여도 결국 쿠키이기 때문에 브라우저가 요청 시 자동으로 실어서 보낸다. → CSRF공격에 똑같이 노출됨

## 정리

- CSRF공격이 성립하는 이유는 브라우저가 쿠키를 요청해 자동으로 실어보내기 때문이다.
- 이런 특성 때문에 로그인 된 상태면 그 사람 권한으로 위조된 요청을 보낼 수도 있다.
- 서버가 이걸 막기 어려운 이유는 유효한 쿠키만 보고는 인증된 사용자라는 것 까지는 알 수 있지만 그 사용자가 진짜 이걸 의도한 요청인지는 모른다.
- 그래서 등장한 게 CSRF토큰이다.
- 쿠키는 브라우저가 자동으로 붙여주지만 CSRF토큰은 그렇지 않아서 SOP 때문에 토큰을 훔칠 수 없으므로 공격자가 이 토큰을 읽을 수 없다.
- 상황에 따라서 CSRF보호를 끌 수도 있는데 중요한 것은 인증 정보가 브라우저에 따라서 자동으로 전송되는가? 여부를 따져야 한다.
    - 인증 정보가 브라우저에 의해 자동으로 실려 나가는 구조(쿠키/세션 기반)라면 CSRF보호를 필수로 켜야 하고, 개발자가 수동으로 헤더에 실어 나르는 구조(LocalStorage + Header JWT)라면 안전하므로 꺼도 무방하다.
