## 가져가야 할 핵심

- 좋은 테스트와 나쁜 테스트를 가르는 기준이 따로 있다. 그 원칙 중 하나가 FIRST원칙
- FIRST원칙이 각각 무엇을 뜻하는지 이해하기. 단어 뜻보다 중요한 것은 각 원칙이 개발자에게 어떤 피드백과 신뢰를 주는가? 빠른 테스트는 무엇을 가능하게 하고 독립적인 테스트는 실패했을 때 무엇을 쉽게 만드는지..연결해서 보기
- 나쁜 테스트는 왜 문제가 되는가?

## 좋은 테스트의 정의와 기준이 필요한 이유

- 좋은 테스트의 정의 : 버그를 안정적으로 찾아내고, 코드 변경에 대한 확실한 신뢰를 주며, 프로덕션 코드가 바뀌어도 유지보수하기 쉬운 테스트
- 기준이 필요한 이유 : 테스트만 짜기만 한다고 다 좋은 것이 아니다. 좋은 테스트와 나쁜 테스트를 가르는 기준이 없으면, 잘못 작성한 테스트 코드가 오히려 코드 수정할 때마다 같이 깨진다.
- 정산이나 할인 정책처럼 복잡한 도메인은 ai가 대충 짜준 테스트로는 예외 상황을 완벽하게 잡을 수 없다. 개발자가 좋은 테스트의 기준을 명확히 알고 있어야 ai의 결과물을 판별하고, 누락된 핵심 비즈니스 시나리오를 직접 짚어낼 수 있다.

## 좋은 테스트의 기준 : FIRST원칙

“이 원칙이 개발자에게 어떤 피드백과 신뢰를 주는가?”에 집중해서 이해해야 한다.

#### F - Fast(빠르게 실행되어야 한다)

- 단순히 성능이 좋다는 뜻이 아니라, 테스트가 빨라야 개발자가 코드를 한 줄 고칠 때마다 부담 없이 실행(피드백 주기 활성화)할 수 있다. 테스트가 몇 분씩 걸리면 아무도 안 돌리게 되고 생산성이 저하된다.
- 느려지는 원인과 해결(외부 의존성 제거) : 주로 실제 DB를 조회하거나 외부 API를 직접 네트워크로 호출할 때 테스트가 느려지고 불안정해진다.

**나쁜 예시(실제 외부 DB 의존)**

```java
public class OrderService {
    // 실제 운영/테스트 DB와 직접 연결되는 고비용 저장소 객체
    private final OrderRepository orderRepository = new RealOrderRepository(); 

    public void placeOrder(Order order) {
        orderRepository.save(order);
    }
}
```

테스트를 돌릴 때마다 진짜 DB서버를 타기 때문에 네트워크 지연이 생기고, 데이터 상태에 따라 흔들리며 속도가 매우 느려진다.

**좋은 예시(가짜 객체 - Test Double 활용)**

```java
@Test
void placeOrder_savesOrder() {
    // Given : 실제 DB 대신 메모리에서 0.001초 만에 동작하는 가짜 저장소 주입
    OrderRepository fakeRepository = new FakeOrderRepository(); 
    OrderService orderService = new OrderService(fakeRepository);
    Order order = new Order(...);
		
    // When
    orderService.placeOrder(order);

    // Then
    assertTrue(fakeRepository.existsById(order.getId()));
}
```

진짜 DB를 안 쓰면 덜 정확한거 아닌가? → 이 테스트의 목적은 “주문 저장 로직의 흐름이 맞는지”빠르게 확인하는 단위 테스트이다. 불필요하고 느린 외부 의존성을 떼어내고 가짜 객체로 검증 하는게 훨씬 이득이다. 만약 DB연동 자체가 목적이라면 그건 단위 테스트가 아니라 통합테스트의 영역에서 따로 다뤄야 한다.

#### I - Independent(독립적이어야 한다)

- 각 테스트는 서로 절대 영향을 주면 안 되고, 어떤 순서로 실행하든 항상 똑같이 통과해야 한다.
- 테스트가 독립적이어야만 에러가 터졌을 때 “이 시나리오 하나가 깨졌구나”하고 문제를 단번에 파악할 수 있다. 얽혀있으면 도미노처럼 무너져서 진짜 원인이 가려진다.

**나쁜 예시(Static 필드로 공유 데이터 사용)**

```java
class UserTest {
    private static User testUser; // 공용 static 변수

    @Test
    void testA_createUser() {
        testUser = new User("test"); // testB가 쓸 데이터를 공유 목적으로 생성
    }

    @Test
    void testB_updateUser() {
        testUser.changeName("updated"); // testA가 먼저 실행된다는 가정에 의존
    }
}
```

- 프레임워크가 testB를 먼저 실행해 버리면 testUser가 null이라 무조건 터진다. testB 자체의 로직 에러가 아닌데도 실패하므로 디버깅이 힘들어진다.

**좋은 예시(@BeforeEach로 격리)**

```java
class UserTest {
    private User testUser;

    @BeforeEach // 각 테스트 메소드가 실행되기 바로 직전에 매번 독립적으로 실행됨
    void setUp() {
        testUser = new User("test"); // 완전히 새로운 객체를 매번 독립 생성
    }

    @Test void createUser() { /* 검증 */ }
    @Test void updateUser() { testUser.changeName("updated"); }
}
```

#### R - Repeatable(반복 가능해야 한다)

- 내 PC(로컬)든, 배포 나가는 CI 서버든, 오늘 돌리든, 주말 한밤중에 돌리든 어떤 환경과 시간에서도 무조건 항상 같은 결과가 나와야 한다.
- Flaky Test(깨지기 쉬운 테스트)의 위험성 : 현재 시간, 랜덤 값, 외부 네트워크 상태에 의존하면 어제는 통과하고 오늘은 실패하는 불안정한 테스트가 된다. 이런 게 한두 번 쌓이면 개발자는 실패가 떠도 “아, 그거 원래 가끔 깨지는 테스트야”라며 무시하게 되고, 진짜 치명적인 버그를 놓치는 문제가 발생한다.

**나쁜 예시(실행하는 당일 날짜에 의존)**

```java
@Test
void addEventPoint_adds100_onWeekends() {
    PointService pointService = new PointService();
    LocalDate today = LocalDate.now(); // 실행하는 시점의 요일에 따라 결과가 계속 바뀜

    int points = pointService.calculateEventPoints();

    if (today.getDayOfWeek() == DayOfWeek.SATURDAY || today.getDayOfWeek() == DayOfWeek.SUNDAY) {
        assertEquals(100, points); // 주말에 테스트 돌리면 여기를 검증
    } else {
        assertEquals(0, points);   // 평일에 테스트 돌리면 여기를 검증
    }
}
```

테스트 코드 안에 if-else 분기문이 들어간 것 자체가 잘못 설계되었다는 것임. 주말 로직을 테스트하려는 건지, 평일 로직을 테스트하려는 건지 명확하지 않음

**좋은 예시(고정된 시간/상태 주입)**

```java
@Test
void addEventPoint_adds100_onWeekends() {
    LocalDate saturday = LocalDate.of(2025, 10, 11); // 특정 '토요일' 날짜로 아예 고정
    // 외부에 의존하던 시간을 생성자 파라미터로 주입할 수 있게 설계 변경
    PointService pointService = new PointService(saturday); 

    int points = pointService.calculateEventPoints();

    assertEquals(100, points); // 언제 돌려도 무조건 주말 할인 100점 성공!
}
```

#### S - Self-Validating(스스로 검증해야 한다)

- 테스트는 그 자체로 성공인지 실패인지 스스로 판단해서 리포팅을 띄워야 한다.
- 개발자가 직접 콘솔 창을 열어서 System.out.println()으로 찍힌 로그 값을 눈으로 보며 “5가 잘 나왔군”하고 확인해야 한다면 나쁜 테스트이다. 테스트가 수천 개로 늘어나면 사람이 절대 수동으로 검증할 수 없기 때문이다.
- 기계가 스스로 판단할 수 있어야 Github Actions같은 CI 서버가 자동으로 테스트를 돌리고 통과 여부를 명확히 결정해 줄 수 있다.

#### T - Timely(동시에 작성되어야 한다)

- 테스트 코드는 프로덕션 코드를 다 구현하고 나중에 붙이는 게 아니라, 기능 구현과 동시에(혹은 직전에)작성되어야 한다.
- 뒤늦게 나중에 테스트를 붙이려고 하면, 이미 코드가 실제 DB나 현재 시스템 기간을 코드 내부에 숨겨두는 “테스트 하기 꼬인 나쁜 구조”로 굳어져 있다.

## 테스트 커버리지가 높으면 무조건 좋은 테스트인가?

- 테스트 커버리지란 : 내 테스트 코드가 실제 프로덕션 코드의 몇 % 줄을 실행하고 지나갔는지를 나타내는 양적인 지표이다.
- 커버리지가 높다고 무조건 좋은 테스트가 아니다. 안에 핵심 비즈니스 로직을 비교하는 검증문을 단 한 줄도 안 적고, 단순히 메서드 호출만 쭉 실행해 주는 무의미한 테스트를 짜도 커버리치 수치는 올라간다.
- 커버리지는 테스트가 아예 누락된 빵꾸 난 구멍이 어디인지 찾아내는 ‘보조 참고 자료’일 뿐, 그 수치 자체가 개발의 최종 목표가 되면 질적으로 떨어지는 이상한 테스트만 양산하게 된다. 진짜 중요한 건 “얼마나 많은 코드를 밟앗냐’가 아니라 “핵심 비즈니스 예외 상황들을 얼마나 의미있게 잘 검증했는가”이다.

## TDD

TDD는 단순히 테스트를 빨리 짜는 기법이 아니라, 테스트를 첫 번째 사용자로 삼아 쓰기 편하고 결합도가 낮은 코드를 뽑아내는 ‘소프트웨어 설계 방법론’이다.

1. Red(실패하는 테스트 작성) : 만들고 싶은 비즈니스 기능의 요구사항을 코드로 정의하고, 구현 코드가 없는 상태에서 먼저 실행해 실패를 확인하는 단계.
    
    ex. “5만원 넘으면 무료 배송”이라는 로직을 짜기 전, 5만원 주문 시 배송비 == 0 검증문부터 적고 시작
    
2. Green(테스트를 통과시키는 코드 작성) : 방금 실패한 그 테스트를 겨우 통과시킬 정도의 가장 빠르고 심플한 최소한의 코드만 작성
3. Refactor(구현 코드 다듬기) : 테스트가 초록불로 기능의 안정성을 받쳐주고 있으니, 안심하고 중복을 제거하고, 가독성을 높이며, 깔끔한 구조로 코드를 정돈함(리팩토링 과정에서 실수로 기능이 깨져도 초록불이 빨간불로 바뀌며 바로 알려줌)

| 원칙 (FIRST) | 핵심 메시지 (개발자에게 주는 가치) | 위반 시 발생하는 대가 (나쁜 테스트) |
| --- | --- | --- |
| **Fast** (신속하게) | 잦은 실행을 유도하여 에러 피드백 주기를 엄청나게 줄여준다. | 테스트가 무거워 안 돌리게 되고 생산성이 바닥침. |
| **Independent** (독립적으로) | 깨진 시나리오 하나만 정확히 골라내어 디버깅 범위를 좁혀준다. | 테스트 순서가 꼬이면 연쇄적으로 터져 범인을 못 찾음. |
| **Repeatable** (반복 가능하게) | 어떤 상황과 환경에서도 100% 신뢰할 수 있는 안정망이 된다. | 평일엔 성공하고 금요일엔 깨지는 Flaky 테스트화 (양치기 소년). |
| **Self-Validating** (자가 검증) | 기계(CI)가 대신 판단하게 하여 자동 배포 파이프라인 구축을 돕는다. | 개발자가 일일이 콘솔 로그 뒤적이며 수동 검사해야 함. |
| **Timely** (적시에) | 테스트하기 좋은 구조(낮은 결합도, 높은 응집도)의 설계를 유도한다. | 다 구현하고 짜려면 이미 코드가 꼬여서 테스트 작성을 포기함. |
