## Lock 정의

Lock은 멀티스레드 환경에서의 안전한 자원 접근을 보장하기 위해 사용하는 동기화 제어 인터페이스이다.

java.util.concurrent.locks 패키지에 포함되어 있으며 공유 자원에 대해 스레드가 명시적으로 락을 획득하고 해제하는 방식으로 동기화를 수행한다.

synchronized는 코드 블록이나 메서드 단위에 락을 자동으로 적용하고 해제하지만 Lock은 락의 적용 시점과 해제 시점, 락의 대상 객체까지도 개발자가 직접 제어할 수 있다.

→ 동기화 범위를 세밀하게 조절할 수 있고 락을 기다릴지 말지 결정하거나, 일정 시간 대기 후 포기하는 방식으로 처리할 수도 있다.

Lock은 단순한 상호 배제를 넘어 공정성 설정, 조건 변수 활용, 비차단 락 시도 등 복잡한 멀티스레드 시나리오를 지원할 수 있는 구조적 유연성을 제공한다.

<aside>
💡 비차단 락(non-blocking lock)은 스레드가 락을 획득하지 못하면 기다리지 않고 바로 다음 로직으로 넘어가는 방식의 락 처리 기법이다.

Lock 인터페이스의 tryLock()메서드를 사용해 구현할 수 있으며 락을 사용할 수 있는지 확인하고 가능하면 락을 얻고 아니면 포기한다.

→ 스레드가 락을 잡기 위해 무한히 대기하는 것을 방지하고, 필요한 경우 대기 시간 없이 다른 작업을 처리하거나 재시도할 수 있게 한다.

</aside>

## 사용 이유

멀티스레드 환경에서도 자원 접근을 정밀하게 제어하고, 안정성과 예측 가능성을 확보할 수 있다.

Lock은 락을 언제 걸고 풀지 개발자가 직접 결정할 수 있다.

락 획득을 기다릴지 포기할지, 일정 시간만 기다릴지 등을 선택할 수 있다.

→ 이러한 제어 능력은 synchronized 의 기본 동기화 방식으로는 구현하기 어렵다.

→synchronized는 블록 진입 시 자동으로 락을 걸고 빠져나갈 때 자동으로 해제되는 구조이기 때문에 예외상황, 대기 조건, 우선순위, 타임아웃 등 다양한 운영 시나리오를 제대로 반영할 수 없다.

Lock은 먼저 대기한 스레드가 먼저 락을 얻로고 순서를 보장해줄 수 있다.

→synchronized는 락이 풀렸을때 누가 락을 먼저 차지할지 알 수 없다. 운이 나쁜 스레드가 있으면 영원히 락을 못얻게 될 수도 있다.

예를 들어 티켓 예매를 만들때 synchronized로 만들 수는 있는데 한명이 좌석에 접근하는 동안 9999명은 스레드가 멈춘 상태로 무한정 대기에 빠지게될 수도 있다.

→ 누군가 이미 좌석을 점유하고 있다면 기다리지 않고 즉시 “이미 예매중인 좌석입니다”라고 메시지를 사용자에게 반환하고 스레드를 해제한다. → 서버 자원을 아낄 수도 있고 사용자에게 빠른 피드백을 줄 수 있다.

## 사용 방법

주로 ReentrantLock 구현체를 통해 사용한다.

<aside>
💡 ReetrantLock은 자바에서 제공하는 Lock 인터페이스의 대표적인 구현체로 한 스레드가 이미 획득한 락을 다시 획득할 수 있도록 허용하는 재진입 가능락이다.

같은 스레드가 여러번 lock()을 호출하면 횟수만큼 락이 중첩되어 적용되고, unlock()도 그에 맞춰 같은 횟수만큼 호출해야 완전히 해제된다.

</aside>

```jsx
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

Lock lock = new ReentrantLock();

lock.lock();
try {
    // 임계 영역
} finally {
    lock.unlock(); // 반드시 해제해야 함
}
```

unlock()을 호출하지 않으면 해당 락이 영구적으로 점유되어 다른 스레드가 접근할 수 없는 교착상태(deadlock)에 빠질 수 있다.

항상 try-finally블록을 사용해 락 해제를 보장해야 한다.

### 일반적인 락 사용

문이 열릴 때까지 앞에서 무조건 하염없이 기다린다. 다른 일을 하거나 도망치지 못한다.

```jsx
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

public class PublicBathroom {
    private final Lock 문고리 = new ReentrantLock();

    public void use(String 사람) {
        // 1. 문이 잠겨있으면 열릴 때까지 여기서 영원히 대기합니다.
        문고리.lock(); 
        
        try {
            // 2. 문이 열려서 들어옴 (임계 영역 시작)
            System.out.println(사람 + " -> 화장실 들어감. 문 잠금!");
            Thread.sleep(1000); // 볼일 보는 중... (1초)
            
        } catch (InterruptedException e) {
            // 예외 처리 생략
        } finally {
            // 3. 아무리 배가 아픈 예외 상황이 터져도 나갈 땐 무조건 문을 열어줍니다.
            System.out.println(사람 + " -> 볼일 끝! 문 열고 나감.");
            문고리.unlock(); 
        }
    }
}
```

- **철수**가 먼저 화장실에 들어가서 `문고리.lock()`을 했다.
- 바로 뒤에 **영희**가 와서 `문고리.lock()`을 만나는 순간, 영희는 **아무것도 못 하고 화장실 문 앞에서 철수가 나올 때까지 강제로 대기**해야 한다. 철수가 10분이 걸리든 1시간이 걸리든 꼼꼼하게 기다리는 게 바로 일반적인 `lock()` 이다.

### 비차단 락 - tryLock()

줄을 서긴 서는데 지정 시간 내에 문이 열리지 않으면 쿨하게 포기한다.

```jsx
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;
import java.util.concurrent.TimeUnit;

public class FamousRestaurant {
    private final Lock 가게문 = new ReentrantLock();

    public void visit(String 손님) {
        try {
            // "가게문이 열렸는지 확인해줘. 만약 만석이면 딱 '2초'만 줄 서서 기다려볼게!"
            // 락을 얻으면 true, 2초 동안 끝내 못 얻으면 false를 반환합니다.
            if (가게문.tryLock(2, TimeUnit.SECONDS)) { 
                
                // [A 코스] 2초 안에 자리가 나서 식당 입장 성공!
                try {
                    System.out.println(손님 + " -> 오! 자리가 났네요. 맛있게 밥 먹는 중...");
                    Thread.sleep(3000); // 3초 동안 식사
                } finally {
                     // 다 먹었으니 다음 사람 쓰라고 자리를 비워줍니다(락 해제).
                    System.out.println(손님 + " -> 잘 먹었습니다. 퇴장!");
                    가게문.unlock();
                }
                
            } else {
                // [B 코스] 2초를 기다렸는데도 자리가 안 나서 대기를 포기할 때 실행되는 영역!
                System.out.println(손님 + " -> 아우, 2초나 기다렸는데 줄이 안 줄어드네? 그냥 딴 데 가자! (우회)");
            }
            
        } catch (InterruptedException e) {
            // 예외 처리 생략
        }
    }
}
```

- **철수**가 먼저 식당에 들어가서 밥을 먹기 시작했다. 철수는 밥 먹는 데 총 **3초**가 걸린다.
- 0.5초 뒤에 **영희**가 식당에 와서 `tryLock(2, 약속시간)`을 외친다. "나 딱 **2초**만 기다려본다!"
- 영희는 2초 동안 문 앞에서 기다린다. 하지만 안에서 먹고 있는 철수는 3초 뒤에야 나올것이다.
- 결국 2초가 지나버려서 `tryLock`은 `false`를 뱉어내고, 영희는 무한정 기다리며 굶는 게 아니라 `else` 블록으로 빠져나가 **"그냥 딴 데 가자!" 하고 제 갈 길을 간다.**

일반 락(lock()) : 앞사람이 안 나오면 뒤에 온 스레드는 무한히 대기

비차단 락(tryLock(시간)) : 일정 시간만 대기해보고 안되면 else로 빠져나와서 다른 유연한 것들을 실행

→ 응답 지연이 허용되지 않은 고속도로 진입, 결제 시스템, 게임 서버 등에서 유용하게 사용된다.

→ 락을 획득하지 못했을 경우 우회하거나 재시도 로직을 명확하게 처리해야 한다.

### 공정성 설정 - 락 요청 순서 보장

```jsx
private final Lock lock = new ReentrantLock(true); // 공정성 설정
```

`ReentrantLock(true)` 와 같이 생성자에 true를 전달하면 락을 먼저 요청한 스레드가 먼저 락을 획득하는 구조로 동작한다. → 스레드가 계속해서 락을 획득하지 못하는 기아 상태(starvation)를 방지한다.

<aside>
💡 기아상태(Starvation)는 멀티스레드 환경에서 어떤 스레드가 자원(락 등)을 계속해서 획득하지 못한 채 무한히 대기하게 되는 상황이다.

다른 스레드들이 우선순위가 높거나 락을 계속 점유하고 있어서 특정 스레드가 계속해서 기회를 얻지 못하는 불공정한 실행 상태가 발생한다.

</aside>

### 조건 변수 사용 - 특정 조건 만족 시까지 대기

상황 : 카페에 의자(공유 자원)가 딱 1개밖에 없다. 자리가 없으면 손님은 진동벨을 받고 대기실에서 대기해야한다.

조건 변수 = 진동벨

```jsx
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

public class Cafe {
    private final Lock 카페문 = new ReentrantLock();
    
    // 진동벨(Condition)을 하나 만듭니다.
    private final Condition 빈자리_진동벨 = 카페문.newCondition(); 
    
    private int 빈자리개수 = 1; // 카페에 있는 총 의자 개수

    public void 커피마시기(String 손님) {
        카페문.lock(); // 1. 일단 카페 안으로 들어와 문을 잠급니다.
        
        try {
            // 2. 자리가 있는지 확인합니다. 자리가 0개면?
            while (빈자리개수 == 0) {
                System.out.println(손님 + " -> 아, 자리가 없네. 진동벨 받고 대기실에서 잘게요... 💤");
                
                // 3. 진동벨을 쥐고 대기실로 갑니다. 
                // ★중요: 대기실로 들어가는 순간, 자기가 쥐고 있던 [카페문 열쇠]는 자동으로 반납됩니다!
                빈자리_진동벨.await(); 
                
                System.out.println(손님 + " -> (지이잉~) 진동벨 울렸다! 잠에서 깨서 자리 비었는지 다시 확인!");
            }

            // 4. 자리가 비어있거나, 깨어났더니 자리가 생겼다면! 자리에 앉습니다.
            빈자리개수--; // 내가 앉았으니 빈자리 감소
            System.out.println(손님 + " -> 빈자리에 앉아서 커피 맛있게 마시는 중... (냠냠)");
            Thread.sleep(1000); // 1초 동안 커피 마심

            // 5. 다 마셨으니 일어납니다.
            빈자리개수++; // 빈자리 다시 늘어남
            System.out.println(손님 + " -> 다 마셨다! 자리 비워주고 갈게요.");

            // 6. 대기실에서 진동벨 받고 자고 있는 다음 사람의 진동벨을 울려줍니다!
            빈자리_진동벨.signal();

        } catch (InterruptedException e) {
            // 예외 처리 생략
        } finally {
            카페문.unlock(); // 7. 나갈 땐 무조건 카페 문을 열어줍니다.
        }
    }
}
```

- **철수 입장:** 철수가 먼저 `커피마시기()`를 호출해 `카페문.lock()`을 하고 들어온다. 빈자리가 1개 있으니 `while`문을 통과하고 의자에 앉아(`빈자리개수 = 0`) 커피를 마신다.
- **영희 입장 및 대기:** 철수가 커피를 마시는 도중에 영희가 들어온다. 철수가 아직 안 끝났으니 카페 문 앞에서 잠깐 기다리다가, 철수가 `sleep`하는 사이 문이 열려 영희가 들어온다.
- **영희 취침 (`await`):** 그런데 들어와 보니 `빈자리개수 == 0` 이다. 영희는 `빈자리_진동벨.await()`을 누른다. 그 순간 영희는 **카페문 열쇠를 바닥에 내려놓고** 대기실로 가서 잠이 든다.
- **철수 퇴장 및 알림 (`signal`):** 철수가 커피를 다 마시고 자리에서 일어난다.(`빈자리개수 = 1`). 그리고 나가기 전에 대기실을 향해 `빈자리_진동벨.signal()`을 날려 진동벨을 울린다. 그리고 `카페문.unlock()`으로 완전히 나간다.
- **영희 기상 및 커피 흡입:** 대기실에서 자던 영희가 진동벨 소리에 눈을 뜬다. 대기실을 나와서 `while`문 조건을 다시 보니 이제 `빈자리개수`가 1이 되어 있다. 영희는 자리에 앉아 커피를 마신다.

공정성 설정(true)는 무슨 역할일까?

- **기본 설정 (공정성 X) :** 철수가 나갈 때 `signal()`을 주면, 자바가 대기실에 자고 있는 사람들 중 **아무나 무작위로** 깨운다. 운이 나쁜 민수는 제일 먼저 와서 기다렸는데도 계속 잠만 자고, 방금 막 온 나비가 깨어나서 자리를 차지할 수도 있다. 민수는 계속 굶주리는 기아 상태(Starvation)에 빠진다.
- **공정성 설정 (`new ReentrantLock(true)`) :** 대기실에 줄 선 순서(Queue)를 도입합니다. 철수가 `signal()`을 보내면 자바가 무조건 **"가장 오래 기다린 순서대로"** 영희를 깨우고, 그다음 민수를 깨운다. 차례를 보장해 주기 때문에 기아 상태가 절대 발생하지 않는다.

## ThreadLocal

`ThreadLocal`은 스레드마다 고유하게 독립된 변수를 저장할 수 있도록 해주는 클래스이다.

동일한 `ThreadLocal` 인스턴스를 여러 스레드에서 참조하더라도, 각 스레드마다 서로 다른 값을 저장하고 읽어올 수 있게 해주는 구조이다.

`ThreadLocal`은 변수 하나를 만들면, 그 변수에 대해 스레드별로 서로 다른 공간을 자동으로 분리해서 관리해준다.

자바에서 하나의 프로그램이 여러 개의 스레드를 동시에 실행하면, 공유되는 변수에 여러 스레드가 접근하면서 동시성 문제(경합, 교착 등)가 발생할 수 있다.

이를 해결하려면 보통 `synchronized`나 `Lock`을 통해 동기화를 사용하지만, 경우에 따라 아예 “스레드마다 따로 쓰는 공간”을 만들어버리는 방식이 더 단순하고 효과적일 수 있다. → ThreadLocal의 목적

스레드마다 독립된 값을 갖기 때문에 동기화 없이도 스레드 안전(thread-safe)하게 데이터를 저장하고 사용할 수 있다.

ThreadLocal = “나만 열 수 있는 개인 금고(지갑)”

일반 멤버 변수가 모든 스레드가 공유하는 '공동 저금통'이라면, `ThreadLocal`은 "스레드별로 독립적인 공간을 가지는 개인 지갑"이다. 아무리 수천 개의 스레드가 동시에 이 변수에 접근해도, 자바가 스레드 ID를 확인해서 딱 자기 지갑만 열어준다.

```jsx
import java.util.Random;

public class WalletService {

    // 스레드 전용 지갑(ThreadLocal)을 생성합니다.
    // 각 스레드가 이 변수를 처음 읽을 때 0원으로 초기화되도록 설정했습니다.
    private static final ThreadLocal<Integer> myWallet = ThreadLocal.withInitial(() -> 0);

    public void manageMoney(String 스레드이름) {
        // 1. 현재 내 지갑에 얼마 있는지 확인 (ThreadLocal.get())
        int currentMoney = myWallet.get();
        System.out.println(스레드이름 + " -> 현재 지갑 잔액: " + currentMoney + "원");

        // 2. 용돈을 무작위로 받아서 지갑에 넣음 (ThreadLocal.set())
        int allowance = new Random().nextInt(10000) + 1000; // 1000원~11000원
        myWallet.set(currentMoney + allowance);
        
        System.out.println(스레드이름 + " -> 용돈 " + allowance + "원 보관함에 넣음. 최종 잔액: " + myWallet.get() + "원");

        // 사용이 끝나면 지갑을 반드시 비워줍니다. (ThreadLocal.remove())
        // (스프링 환경이나 톰캣처럼 스레드를 재사용하는 곳에서는 필수!)
        myWallet.remove();
    }
}
```

- **철수 스레드**와 **영희 스레드**가 동시에 이 `WalletService`를 이용한다.
- `myWallet`이라는 변수는 static으로 선언되어 **단 하나만 존재**하는 것처럼 보인다.
1. **철수 스레드 진입:** `myWallet.get()`을 호출한다. 자바는 " 철수 스레드가 왔군" 하고 **철수 전용 공간**에서 `0`을 꺼내준다. 철수가 무작위로 5,000원을 벌어서 `myWallet.set(5000)`을 한다. 철수 지갑에는 5,000원이 들어있다.
2. **영희 스레드 진입:** 정확히 같은 순간에 영희가 `myWallet.get()`을 호출한다. 일반 변수였다면 철수가 넣어둔 5,000원이 보였겠지만, `ThreadLocal`이기 때문에 자바는 **영희 전용 공간**을 연다. 영희 지갑은 여전히 `0`원이다. 영희가 3,000원을 벌어서 넣으면 영희 지갑만 3,000원이 된다.
3. **결과:** 철수와 영희는 `myWallet`이라는 동일한 변수 이름을 사용해 코드를 짜고 있지만, 서로의 돈을 절대 훔쳐보거나 덮어쓸 수 없다. **락(`lock`)을 전혀 걸지 않았는데도 Race Condition이 발생하지 않는다.**

ThreadLocal에 저장된 값은 해당 스레드가 종료되기 전까지 계속 메모리에 남아있다.

따라서 웹 서버와 같이 스레드가 재사용되는 환경에서는 작업이 끝난 후 반드시 remove()를 호출해 값을 삭제해야 메모리 누수를 방지할 수 있다.

### 실무에서의 활용

#### 1. ThreadLocal의 실무 등장 배경 (문제와 해결)

- **문제점 (파라미터 지옥):** 사용자의 로그인 요청이 들어오면 톰캣은 스레드를 배정하고, 이 스레드는 `Controller -> Service -> Repository` 계층을 차례로 거친다. 이때 로그인한 유저 ID를 모든 메서드의 매개변수로 일일이 넘겨주면 코드가 지저분해진다.
- **해결책 (스레드 전용 공책):** 요청의 시작점(Filter/Interceptor)에서 스레드 전용 보관함(`ThreadLocal`)에 유저 ID를 딱 한 번 저장해 두면 스레드가 어떤 클래스나 메서드로 이동하든, 파라미터 전달 없이 전역에서 자기 공책을 열어 ID를 꺼내 쓸 수 있다.
- **장점:** 스레드 간 공간이 완벽히 분리되어 있어, 수천 명이 동시에 접속해도 **동시성 문제(Race Condition)가 절대 발생하지 않는다.**

#### 2. Spring Boot 실무 연결 핵심 요약

스프링 프레임워크는 우리가 매일 쓰는 핵심 기능 두 가지를 구현할 때 내부적으로 이 `ThreadLocal` 기술을 핵심 기반으로 사용한다.

Spring Security (인증 정보 관리)

- 로그인한 사용자의 정보를 애플리케이션 어디서든 파라미터 없이 꺼내 쓰게 해준다.
- 인증이 완료되면 사용자의 정보(Principal, 권한 등)를 `SecurityContextHolder`라는 저장소에 보관하는데, 이 홀더의 내부 구조가 바로 `ThreadLocal`로 이루어져 있다.
- 덕분에 내 스레드가 실행하는 코드 어디에서든 `SecurityContextHolder.getContext().getAuthentication()`을 호출하면 내 로그인 정보를 안전하게 가져올 수 있다.

스프링 트랜잭션 관리 (`@Transactional`)

- 하나의 요청(트랜잭션) 안에서 여러 Service와 Repository 메서드가 실행되더라도, 모두 동일한 DB 커넥션(Connection)을 사용해 데이터를 일관되게 동기화(Commit/Rollback)하도록 보장한다.
- `@Transactional`이 붙은 메서드가 시작되면, 스프링은 DB 커넥션 풀에서 커넥션을 하나 꺼내 **현재 스레드의 `ThreadLocal`에 저장해둔다.**
- 이후 호출되는 모든 Repository(JPA, MyBatis 등)는 이 `ThreadLocal`에서 커넥션을 꺼내 쓰기 때문에, 매개변수로 커넥션을 넘겨주지 않아도 자연스럽게 하나의 트랜잭션으로 묶이게 된다.

## 정리

- Lock인터페이스는 synchronized 한계를 극복하고 개발자가 명시적으로 락의 획득과 해제를 제어할 수 있게 만들어주는 도구이다.
- tryLock()을 사용하면 락을 얻지 못했을 때 무한정 대기하지 않고 즉시 다른 로직을 수행하는 비차단 처리가 가능해서 응답성을 높일 수 있다.
- 쓰레드 로컬은 락을 걸어서 자원을 공유하는 대신 쓰레드마다 독립적인 공간을 제공하여 동시성 문제를 해결하는 방법이며 스레드풀을 쓰는 환경이면 remove()해줘야한다.
