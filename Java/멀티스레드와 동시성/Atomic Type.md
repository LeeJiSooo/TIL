## Atomic Type 정의

Atomic Type은 멀테스레드 환경에서 동기화 없이도 값을 안전하게 변경할 수 있도록 만든 클래스이다.

자바의 java.util.concurrent.atomic 패키지에 포함되어 있으며 경쟁 조건(race condition)을 막고자 하는 목적이다.

Atomic타입은 내부적으로 CAS(Compare-And-Swap)연산을 사용하기 때문에 스레드 간 경쟁 조건 없이 안전하게 동작한다.

1. 현재 메모리 값(V)
2. 예상 값(E)
3. 새로 바꿀 값(N)

```jsx
if (V == E) {
    V ← N
} else {
    실패 → 다시 시도
}
```

CAS는 “현재 값이 예상 값과 같을 때만 새 값으로 변경”하는 원자적 연산이다.

이 연산은 하드웨어 차원에서 기계어 명령으로 처리되며 CPU가 비교하고 교체하는 순간은 절대 다른 스레드가 끼어들지 못하게 하드웨어가 보장해준다. → 스레드를 멈추게 하는 lock없이도 논블로킹 방식으로 빠른 동시성 제어가 가능하다.

→ 여러 스레드가 동시에 값을 읽고 쓰더라도 충돌 없이 안전하게 변경이 가능하다.

## 사용 이유

Atomic Type이 사용되는 이유는 **멀티스레드 환경에서 변수에 대한 안전한 연산을 락 없이 처리할 수 있기 때문이다.**

전통적으로 synchronized나 Lock을 사용하면 성능 저하나 데드락 가능성이 존재하지만, **Atomic Type은 내부적으로 CAS(Compare-And-Swap) 알고리즘을 통해 원자성(atomicity)을 보장한다.**

이 덕분에 **동기화 비용 없이도 스레드 간 충돌 없이 안전하게 값을 증가, 감소, 설정할 수 있으며**, 주로 카운터, 플래그, 상태 토글 같은 **단일 변수 조작 작업에서 높은 성능과 안정성**을 동시에 확보할 수 있다.

**→ 경쟁 조건(Race Condition)을 피하면서도 가볍고 효율적인 스레드 안전 연산이 필요한 경우에 적합하다.**

**synchronized: 무겁고 안전한 자물쇠(Context Switching 발생) - 비관적 락 방식**

1. 스레드 차단 : 스레드 A가 락을 쥐고 있으면 스레드 B는 대기실로 이동
2. 커널 개입: 자바가 운영체제에게 스레드를 잠재워달라고 요청. OS 커널이 개입하여 스레드 상태를 BLOCKED로 변경
3. 콘텍스트 스위칭 : CPU는 잠든 스레드 B를 치우고 무거운 작업을 함
4. 꺠우기 : 스레드 A가 끝나면 OS가 다시 잠든 스레드 B를 깨움

→ 단순히 count++같은 숫자 하나 올리는 0.000001초짜리 작업을 하겠다고 스레드를 재우고 깨우는 OS레벨의 콘텍스트 스위칭를 하므로 성능 낭비가 심하다.

**Atomic Type(CAS) : 자물쇠 없는 뺑뺑이? - 낙관적 락 방식**

1. 자유로운 진입 : 스레드 B를 잠재우지 않고 CPU 안에 계속 둠
2. 비교 후 교체(CAS) : 메모리에 값을 저장하기 직전 하드웨어(CPU 명령어)도움을 받아 처음에 읽은 값과 실제 값이 일치하는지 확인
3. 실패 시 즉시 재시도 : 맞으면 Swap하고 틀리면 스레드가 잠드는게 아니라 즉시 재연산

스레드가 대기실로 가거나 잠들지 않고 CPU를 잡은 채로 아주 빠르게 재시도하기 때문에 스레드를 깨우고 재우는 OS레벨의 오버헤드가 없어 빠르다.

## 사용 방법

멀티스레드 환경에서 **공유 자원의 값을 안전하게 수정**하려면 경쟁 조건(Race Condition)을 방지해야 한다.

AtomicInteger는 이러한 목적을 위해 제공되는 **원자성(Atomicity)을 보장하는 클래스이다.**
내부적으로 **CAS(Compare-And-Swap)** 방식으로 동작하며, synchronized 키워드 없이도 **성능과 안정성**을 모두 확보할 수 있다.

### AtomicInteger 사용 예시

```jsx
import java.util.concurrent.atomic.AtomicInteger;

public class Car {
    private AtomicInteger fuel = new AtomicInteger(0);

    public void refuel() {
        int newFuel = fuel.incrementAndGet(); // 연료값 1 증가 후 반환
        System.out.println("주유 완료 (연료량: " + newFuel + ")");
    }
}
```

- AtomicInteger는 동기화 블록 없이도 **스레드 안전한 증가 연산**이 가능하다.
- incrementAndGet()은 내부적으로 CAS(Compare-And-Set)를 사용해 원자적 연산을 보장한다.

### 예시 : “새로고침 누르고 취소표 낚아채기”

자물쇠(synchronized, lock) 방식은 내가 수강신청 사이트에 접속하면 내가 나갈때까지 다른 학생들은 나갈때까지 아예 사이트에 들어오지 못하게 문을 잠가버리는 방식이다. → 단순 대기(Blocking)

반면 CAS방식은 아무도 잠그지 않고 모두가 자유롭게 들어오게 하되, “내가 본 화면의 남은 좌석 수가 결제 버튼을 누르는 순간까지 그대로 유지되었는가?”를 확인해서 딱 매치될 때만 결제를 성공시키는 방식이다.

```jsx
public class CourseRegistration {
    // volatile을 붙여서 '현재 잔여석' 전광판이 모든 스레드(학생)에게 실시간으로 보이게 합니다.
    private volatile int remainingSeats = 1; 

    public void register(String 학생이름) {
        int 내가본잔여석;
        int 예약후잔여석;

        // [do-while 루프] 예약에 성공할 때까지 계속 새로고침하며 시도합니다! (무한 루프가 아니라 찰나의 순간)
        do {
            // 1. 현재 전광판에 보이는 잔여석을 내 모니터로 읽어옵니다. (Read)
            내가본잔여석 = getRemainingSeats(); 
            
            // 만약 읽어왔는데 자리가 0개라면 새로고침을 멈추고 끝냅니다.
            if (내가본잔여석 == 0) {
                System.out.println(학생이름 + " -> 실패! 잔여석이 없습니다.");
                return; 
            }

            // 2. 자리가 있으니까 내 모니터에서 '1개 줄여야지' 하고 계산해둡니다. (Modify)
            예약후잔여석 = 내가본잔여석 - 1; 

            // 3. [CAS 핵심] 전광판에 가서 낚아채기를 시도합니다! (Write 시도)
            // "내가 아까 모니터로 본 자리가 1개였는데, 지금 전광판도 여전히 1개 맞니?" 물어봅니다.
            System.out.println(학생이름 + " -> 전광판 확인 중... 내가 본 좌석: " + 내가본잔여석);

        } while (!compareAndSetSeats(내가본잔여석, 예약후잔여석)); 
        // 만약 그새 다른 학생이 채가서 전광판이 바뀌었다면(!false == true), 다시 do 문으로 올라가 새로고침(반복)합니다!

        // 루프를 빠져나왔다는 건 성공했다는 뜻입니다!
        System.out.println(학생이름 + " -> 수강신청 최종 성공!! (남은 좌석: " + 예약후잔여석 + ")");
    }

    public int getRemainingSeats() {
        return remainingSeats;
    }

    // CPU가 하드웨어 레벨에서 원자적으로 처리해주는 "비교 후 교체" 검사대
    public boolean compareAndSetSeats(int 내가봤던좌석, int 바꿀좌석) {
        synchronized (this) {
            // 전광판의 진짜 실시간 좌석이 내가 아까 리딩했던 좌석과 일치하는가?
            if (remainingSeats == 내가봤던좌석) {
                remainingSeats = 바꿀좌석; // 일치하면 재빨리 내 값으로 덮어쓰고 성공(true) 반환!
                return true;
            }
            // 그 사이에 다른 놈이 채가서 전광판이 바뀌어 버렸다면 실패(false) 반환!
            return false;
        }
    }
}
```

현재 취소표(`remainingSeats`)는 딱 **1개** 남아있다.

1. **동시 읽기 (1단계):** 철수와 영희가 동시에 새로고침을 해서 현재 남은 좌석을 읽는다.
    - 철수가 본 좌석 = 1
    - 영희가 본 좌석 = 1
2. **동시 계산 (2단계):** 둘 다 1을 뺀 0을 저장하겠다고 준비한다.
    - 철수의 다음 목표 값 = 0
    - 영희의 다음 목표 값 = 0
3. **철수의 선타 (3단계 CAS):** 철수가 아주 미세하게 빨라서 `compareAndSetSeats(1, 0)`을 먼저 호출했다.
    - "지금 전광판 보니까 1개 맞네? 내가 본 거랑 똑같다!" → 전광판을 **0**으로 바꾸고 `true`를 얻어 루프를 탈출한다. (철수 수강신청 성공)
4. **영희의 후타 (3단계 CAS):** 뒤이어 영희가 `compareAndSetSeats(1, 0)`을 호출한다.
    - "지금 전광판 보니까... 어? 철수가 채가서 **0**이네? 내가 아까 봤던 건 **1**인데 서로 다르잖아!" → `false`를 반환한다.
5. **영희의 재시도:** `while(!false)`가 되면서 영희는 포기하지 않고 `do` 문으로 다시 튕겨 올라갑니다. (자동 새로고침)
6. **영희의 최종 종료:** 다시 올라가서 좌석을 읽어보니(`getRemainingSeats()`), 이미 철수가 0으로 만들어버렸다. `if (내가본잔여석 == 0)`에 걸려서 영희는 아쉽게 실패 문구를 출력하며 종료한다.

- 원자성(Atomicity) : 내가 읽고 수정하고 쓰는 모든 행동이 끝날 때까지 아무도 끼어들지 못하게 줄을 세운다. → 해결책 - synchronized, Lock, AtomicInteger
- 가시성(Visibility) : 내가 바꾼 값이 다른 스레드의 캐시에 가려지지 않고 실시간으로 메인 메모리에 보이게 한다. → 해결책 - volatile, synchronized

## 생각해보기 - 실시간 조회수 구현

만약 BTS나 블랙핑크의 신곡 뮤직비디오가 나왔다고 가정해보자. 전 세계에서 1초에 10만명이 동시에 영상을 클릭한다.

만약 실시간으로 DB에 +1을 반영한다면? (비관적 락/동기화 방식)

- 1초에 10만 번 데이터베이스의 동일한 row에 락을 걸고 업데이트해야한다.
- 1초에 10만 개의 스레드가 줄을 서고 콘텍스트 스위칭이 발생하면 DB서버는 즉시 터져버리고 저너 세계 유튜브 서비스가 마비될 것이다.

만약 메모리(redis)나 CAS방식(낙관적 락)으로 실시간 반영한다면? 

- 자바 서버 내부의 AtomicInteger나 분산 환경의 Redis를 사용해 락 없이 연산하여도 1초에 10만번씩 네트워크를 타고 메인 데이터베이스나 캐시 서버로 동기화 트래픽이 몰릴 것이다.
- CPU가 무한 재시도하거나 네트워크 대역폭이 폭발할 것이다.

해결책 - 주기적 배치와 비동기 처리

유튜브는 “조회수는 0.001초 단위로 정확할 필요가 없다”는 대전제를 활용해 시스템 설계

1. 로그 수집(write) : 사용자가 영상을 보면 DB를 직접 수정하는 게 아니라 “A가 B영상을 봤음”이라는 가벼운 로그만 카프카나 메시큐에 던지고 끝낸다.
2. 주기적 합산(Batch) : 수많은 로그를 모아두었다가 몇분 주기로 계산
3. 업데이트 : 계산된 숫자를 DB에 한 번의 쿼리로 업데이트
    
    → 조회수가 13,000 → 100,000으로 조회수가 점프하는 이유이다.
    

트레이드 오프

1. 실시간 조회가 사용자에게 필수적인가?
    1. 아니다. 수강신청 재고나 결제 잔액은 1원, 1개만 틀려도 대참사가 나서 ‘실시간성’이 필수지만 유튜브 조회수는 지금 당장 100개든 10,000개든 사용자의 동영상 시청 경험에 아무런 영향을 미치지 않는다.
2. 실시간성을 포기하고 얻는 이득이 무엇인가?
    1. 비용 절감과 시스템 안정성이다. DB인프라 비용을 아기면서도 전 세계 수억명의 동시 접속을 버텨내는 아키텍처를 얻을 수 있다.
