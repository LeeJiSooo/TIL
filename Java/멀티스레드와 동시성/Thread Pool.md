## Thread Pool

Thread는 프로그램 내에서 동시에 실행될 수 있는 독립적인 실행 흐름 단위이다.

<img width="1156" height="668" alt="image" src="https://github.com/user-attachments/assets/d1142eef-3bb0-4a3c-97f2-25f29524a44a" />


Thread Pool은 작업을 처리하기 위한 스레드를 매번 새로 생성하지 않고 미리 만들어둔 스레드 집합을 재사용하여 작업을 처리하는 구조이다.

스레드를 필요할 때마다 생성하고 제거하는 방식이 아니라 미리 지정된 수의 스레드를 풀 안에 유지하면서 요청된 작업을 스레드가 차례로 받아 실행하도록 한다.

작업이 들어오면 대기열에 저장되고 유휴 상태의 스레드가 해당 작업을 꺼내 실행한다.

작업이 끝난 스레드는 제거되지 않고 다시 풀로 돌아가 다음 작업을 기다린다.

## 사용 이유

스레드를 매번 생성하지 않고 재사용함으로써 시스템 자원을 효율적으로 관리하고 동시 처리량을 안정적으로 제어하기 위해서이다.

자바에서 멀티스레딩이 필요한 경우에는 nes Thread()를 사용해 작업을 실행할 수 있다.

하지만 이 방식은 작업마다 새로운 스레드를 생성하고 실행이 끝나면 제거하는 구조이기 때문에 작업 수가 많아지면 시스템 자원은 과도하게 소비하게 된다.

| **구분** | **스레드 풀 사용 시** | **Thread 직접 생성 시** |
| --- | --- | --- |
| 자원 관리 | 스레드 수를 제한하여 자원을 예측 가능하게 사용 | 스레드가 무제한 생성되며, 생성·제거 비용이 반복 발생 |
| 시스템 부하 | 병렬 처리량을 제어해 서버를 보호 | 과도한 문맥 전환으로 CPU 부하 증가, 성능 저하 |
| 작업 대기 처리 | BlockingQueue로 작업을 순차 대기 및 처리 | 작업마다 스레드 생성하기 때문에 스템 자원 고갈 위험 |
| 운영 추적 | 스레드 이름, 작업 흐름 등을 체계적으로 관리 | 흐름 추적, 로깅, 종료 처리가 어렵고 불안정 |
| 메모리 안정성 | 제한된 풀 내에서 스레드를 재사용 | OOM(Out Of Memory) 발생 가능성 증가 |

### 실무에서 Thread Pool이 사용되는 이유

- 스레드 수와 자원 사용량 제어 → 스레드 수를 제한하여 cpu, 메모리 등 시스템 자원을 예측 가능하게 관리할 수 있다.
- 요청 폭주 방어 → 풀 크기와 작업 큐를 통해 갑작스러운 요청 증가에도 처리량을 조절하고 시스템을 보호할 수 있다.
- 프레임워크와 통합 → ExecutorService, @Async, @Scheduled 등 대부분 비동기 기능은 스레드 풀 기반으로 동작한다.

## 동작 방식

<img width="1492" height="580" alt="image" src="https://github.com/user-attachments/assets/41debbba-fc20-43b7-af1c-d5732b93b651" />


1. 애플리케이션이 작업을 제출한다.
    
    Application 영역은 우리가 작성한 코드에서 ExecutorService.excute() 또는 submit()을 호출해 작업을 요청하는 주체를 의미한다. → 이때 작업은 Runnable 또는 Callable 객체의 형태로 전달된다.
    
2. 작업이 큐에 저장된다.

<img width="824" height="736" alt="image" src="https://github.com/user-attachments/assets/09199ce2-e946-47ba-82b5-72d05b4e9e58" />


작업이 제출되면 즉시 실행되는 것이 아니라 작업 대기열에 먼저 저장된다.

이 대기열은 자바에서 보통 BlockingQueue로 구현되며 작업은 도착한 순서대로 큐에 쌓인다.

이때 자바의 ThreadPoolExecutor는 내부적으로 다음과 같은 기준에 따라 작업을 처리한다.

<aside>
💡 ThreadPoolExecutor는 자바에서 스레드 풀을 직접 생성하고 제어할 수 있도록 설계된 클래스이다.

ExecutorService 인터페이스의 대표 구현체로 스레드 수, 작업 큐, 거부 정책 등 세부 설정을 직접 지정할 수 있다.

</aside>

현재 실행 중인 스레드 수가 corePoolSize보다 작다면, 새로운 스레드를 생성하여 해당 작업을 즉시 실행한다.

하지만 이미 corePoolSize만큼의 스레드가 바쁘게 동작중이라면 추가로 들어온 작업은 큐에 저장되어 대기 상태로 들어간다.

3. 유휴 스레드가 작업을 꺼내 실행한다.

풀에 있는 스레드는 하나라도 유휴 상태가 되면 큐에서 작업을 꺼내 실행한다.

스레드는 작업을 실행한 후 즉시 제거되지 않고 풀로 반환되어 다음 작업을 기다린다.

다만, maxinumPoolSize로 인해 추가 생성된 스레드는 일정 시간 동안 유휴 상태가 유지되면 자동으로 제거될 수 있다.(keepAliveTime)

<aside>
💡 keepAliveTime은 자바의 ThreadPoolExecutor에서 corePoolSize를 초과하면 생성된 스레드가 유휴 상태일 때, 얼마나 오래 유지할지를 설정하는 시간 값이다. 

이 시간이 지나도록 해당 스레드가 작업을 받지 않으면 자동으로 제거되어 자원을 반환한다.

</aside>

4. 작업이 완료되면 결과 또는 상태가 애플리케이션으로 반환된다.

execute()로 제출한 작업은 반환값이 없고 단순히 작업만 끝난다.

submit()으로 제출한 작업은 Future 객체를 통해 결과 조회나 예외 확인이 가능하다.

5. 스레드는 제거되지 않고 재사용된다.

스레드는 작업을 마친 후에도 제거되지 않고 Thread Pool에 유지되며 다음 작업을 기다린다.
