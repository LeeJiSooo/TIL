## Volatile정의

Volatile은 변수의 값을 모든 스레드가 항상 메인 메모리에서 직접 읽고 쓰도록 보장하는 키워드이다.

<img width="782" height="712" alt="image" src="https://github.com/user-attachments/assets/1111c9b9-3a06-4968-bb03-5be552e4e0db" />


일반적으로 자바 프로그램은 성능을 높이기 위해 변수 값을 CPU 내부 캐시에 복사해 두고 각 스레드가 해당 값을 빠르게 읽어온다.

→ 각 스레드가 서로 다른 복사본을 가지고 작업하는 문제를 유발할 수 있다.

어떤 스레드가 변수 값을 변경해도 다른 스레드는 여전히 이전 값을 보고 있을 수 있다. → 가시성 문제(visibility problem)

volatile가 붙은 변수는 항상 메인 메모리에서 직접 읽고 쓰도록 강제되며 어떤 스레드가 값을 변경하면 즉시 메인 메모리에 반영되고 다른 스레드들도 변경된 최신 값을 읽게된다.

## 사용 이유

`volatile` 키워드를 사용하면 변수의 값이 변경될 때, 그 변경 내용은 즉시 메인 메모리에 반영되며 다른 모든 스레드는 항상 메인 메모리로부터 최신 값을 읽게 된다.

→ 스레드 간의 상태 변경이 실시간으로 반영되고 값을 기준으로 조건을 분기하거나 종료 여부를 판단하는 실행 흐름도 예측 가능하게 유지할 수 있다.

→ 락을 사용하지 않고도 변수의 값 자체는 일관성을 보장받을 수 있으므로 코드가 더 간단해지고 성능에도 부담이 적은 구조를 만들 수 있다.

복잡한 동기화가 필요 없는 단순한 상태 전파나 제어 흐름 제어에 매우 효과적인 수단으로 사용된다.

멀티스레드 환경에서는 하나의 변수 값을 여러 스레드가 동시에 접근하는 상황이 매우 자주 발생한다.

자바는 기본적으로 성능을 높이기 위해, 공유 변수의 값을 각 스레드가 자신만의 CPU 캐시에 복사해 두고 사용한다. → 읽기 속도를 높이는 데에는 효과적이지만, 한 스레드에서 변수 값을 변경하더라도 다른 스레드는 여전히 이전 값을 사용하게 되는 문제가 발생할 수 있다.

그 결과, 스레드마다 서로 다른 값을 보고 판단하게 되면, 실행 흐름이 꼬이거나 루프가 종료되지 않는 등 의도치 않은 동작이 발생할 수 있다.

더 심각한 점은, 이 문제가 예외나 오류로 드러나지 않기 때문에 테스트 환경에서는 잘 동작하더라도 실제 운영 환경에서 무증상 장애로 이어질 수 있다.

—> 가시성 문제(visibility problem)로 분류된다.

## 사용 방법

### **기본 형식**

`volatile`은 변수 선언부에 키워드로 명시하여 사용한다.

변수 선언 시 `volatile` 키워드를 붙이면, 이 변수에 대한 모든 읽기와 쓰기는 CPU 캐시가 아닌 메인 메모리에서 이루어진다.

```jsx
private volatile 타입 변수명;
```

이렇게 선언된 변수는 모든 스레드가 항상 메인 메모리로부터 읽고, 메모리에 직접 쓰게 된다.

변수 값 변경이 즉시 모든 스레드에 반영되며, 스레드 간의 값 불일치 문제를 방지할 수 있다.

### **`volatile` 없이 사용할 경우 (문제 발생 가능)**

```jsx
private static boolean engineOn = true;

public static void main(String[] args) {
    new Thread(() -> {
        while (engineOn) {
            // 주행 중...
        }
        System.out.println("엔진 종료");
    }).start();

    Thread.sleep(1000);
    engineOn = false; // 시동 끔
}
```

이 코드는 겉보기에 문제가 없어 보이지만, `engineOn = false`로 변경한 값이 다른 스레드에 반영되지 않아, 스레드가 무한 루프에 빠질 수 있다.

자바가 변수 값을 각 스레드의 CPU 캐시에 저장해 사용하기 때문이며, 다른 스레드는 여전히 true라는 이전 값을 보고 있을 수 있다.

### **`volatile`을 붙인 경우 (문제 해결)**

```jsx
private static volatile boolean engineOn = true;
```

`volatile`을 붙이면 변수 값은 항상 메인 메모리에서 읽히기 때문에, 값 변경이 즉시 전파되어, 다른 스레드도 변경된 값을 정확히 감지한다.

위 예제에서 `engineOn = false`를 설정하면, 루프가 정상적으로 종료되고 "엔진 종료"가 출력된다.

| **비교 항목** | **volatile 없음** | **volatile 사용** |
| --- | --- | --- |
| 값 변경 전파 | CPU 캐시에 남아 최신 값 반영 안 됨 | 메인 메모리로 바로 반영됨 |
| 실행 결과 | 루프가 끝나지 않음 (무한 실행) | 루프 정상 종료 |
| 가시성 보장 | 불가 | 가능 |

## 사용하지 말아야 하는 경우

volatile은 가시성은 보장하지만 원자성은 보장하지 않는다.

count 변수 값이 있고 만약에 여러 스레드가 count+코드를 동시에 실행하면 count변수에 volatile을 붙이면 안전하지 않다.

count++은 실제로

1. 값을 읽고
2. 1을 더하고
3. 다시 저장한다.

**만약에 스레드 A와 스레드 B가 동시에 count++을 실행한다고 가정해보자**

1단계: Read (읽기) - volatile 덕분에 둘 다 최신값 10을 잘 읽음

- **스레드 A:** 메인 메모리에서 최신값 **10**을 읽는다. (volatile 덕분에 정확함)
- **스레드 B:** 메인 메모리에서 최신값 **10**을 읽는다. (volatile 덕분에 정확함)

2단계: Modify (수정) - 각자 연산 수행

- **스레드 A:** 10에다가 1을 더해서 **11**을 만든다.
- **스레드 B:** 10에다가 1을 더해서 **11**을 만든다.

3단계: Write (저장) - 메인 메모리에 덮어쓰기

- **스레드 A:** 계산 결과인 **11**을 메인 메모리의 `count`에 저장한다.
- **스레드 B:** 계산 결과인 **11**을 메인 메모리의 `count`에 저장한다.

두 번이나 더했으므로 데이터가 **12**가 되어야 정상인데, 최종 결과는 **11**이 된다.

스레드 A가 11을 저장하는 순간 메인 메모리는 즉시 11이 되었고, 스레드 B가 저장하는 순간에도 즉시 11이 되었다. 하지만 스레드 B는 **이미 1단계(읽기)에서 10을 리딩해 간 상태**였기 때문에, A가 업데이트한 최신값을 다시 읽을 기회가 없었다.

→ `volatile`은 값을 읽고 쓸 때 메인 메모리를 통하게만 해줄 뿐 줄을 세우는 **자물쇠(상호 배제)** 기능이 전혀 없다.

→ volatile은 읽을 때 최신 값을 읽어온다는 것만 보장한다.(가시성) 1을 더해서 저장하려는 찰나에 다른 스레드가 끼어드는 경쟁조건을 막아주지 않는다.(원자성)

→ volatile는 한 쓰레드는 쓰기만 하고 다른 쓰레드들은 읽기만 하는 플래그 변수 같은 상황에서 사용해야한다. → 여러 쓰레드가 동시에 값을 읽고 수정하는 복합 연산이 필요하면 volatile만으로는 의도한 동작을 보장할 수 없다.

## 정리

- 자바는 성능을 위해서 변수값을 cpu캐시에 두고 사용하는데 이로 인해 스레드간의 값이 다르게 보이는 가시성 문제가 발생할 수 있다.
- volatile을 변수에 붙이면 가시성을 보장하고 상태 변경이 즉각적으로 모든 스레드에 전파된다.
- volatile은 락 없이 상태를 공유할 수 있는 가벼운 동기화 도구이지만 원자성은 보장하지 않아서 제한적으로 사용해야한다.
